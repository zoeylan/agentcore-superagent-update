# 工业IoT与产线优化 标准操作规程（SOP）

## 文档信息
- **文档编号**: SOP-IOT-2024
- **版本**: V1.0
- **适用范围**: 工业IoT传感器数据监控、设备异常检测、预测性维护、产线效率优化
- **核心目标**: 通过IoT数据驱动实现设备健康度实时评估、故障提前7天预测、产线瓶颈自动识别与排产优化

## 二、SOP-IOT-001 设备异常告警处理流程

### 2.1 流程目标
确保设备异常在30秒内被检测并推送告警，紧急告警5分钟内得到响应确认，所有告警实现闭环管理。

### 2.2 触发条件
- 传感器数据异常检测算法输出异常结果
- 设备健康度评分跌破阈值（60分/40分）
- 多参数联合异常模式被识别

### 2.3 详细步骤

#### 步骤1：异常检测（IoT数据监控分析师）
- **动作**: 对实时传感器数据流应用异常检测算法（阈值检测+趋势检测+模式检测）
- **输出**: 异常检测结果（是否异常/异常类型/异常评分/数据证据）
- **时效**: 从数据到达到检测完成 <30秒
- **异常处理**: 如果检测算法执行超时（>30秒），记录超时事件并通知IoT运维排查系统性能

#### 步骤2：告警生成与分级（IoT数据监控分析师）
- **动作**: 基于异常检测结果和设备健康度评分，生成分级告警
- **分级标准**:
  - 观察级：异常特征存在但健康度>60分 → 增加监控频率，标记关注
  - 告警级：健康度降至40-60分 → 触发维护检查通知
  - 紧急级：健康度<40分 → 触发停机检修通知
- **输出**: 标准化告警记录（告警ID/设备/级别/时间/证据/建议动作）
- **异常处理**: 同一设备同类告警在未关闭状态下不重复告警，但累计次数+1

#### 步骤3：分级推送（系统自动）
- **动作**: 按告警级别路由到不同接收人
  - 观察级 → 设备责任工程师（系统消息）
  - 告警级 → 维护班组长 + 设备责任人（系统消息+即时通讯）
  - 紧急级 → 维护经理 + 班组长 + 设备责任人 + 生产主管（系统+即时通讯+电话）
- **时效**: 告警产生到推送完成 <30秒
- **异常处理**: 推送通道故障时启用备用通道（主通道企微→备用通道短信）

#### 步骤4：响应确认（接收人）
- **动作**: 告警接收人在规定时间内确认告警并标注初步判断
- **时效要求**:
  - 紧急级：5分钟内确认
  - 告警级：30分钟内确认
  - 观察级：当班次内确认
- **异常处理**: 超时未确认自动升级：
  - 紧急级5分钟未确认 → 升级到车间主任
  - 紧急级15分钟未确认 → 升级到工厂经理
  - 告警级30分钟未确认 → 升级到维护经理

#### 步骤5：现场处置（维护团队/操作人员）
- **动作**: 根据告警信息进行现场检查和处置
  - 观察级：增加巡检频率，持续监控趋势
  - 告警级：安排技术人员检查，评估是否需要停机维修
  - 紧急级：立即安排停机，联动故障诊断专家进行诊断
- **输出**: 处置记录（实际情况/处置措施/是否解决）
- **异常处理**: 现场无法处置时，升级到设备厂家技术支持

#### 步骤6：闭环记录（IoT数据监控分析师）
- **动作**: 确认告警处置完成，记录闭环信息
- **闭环分类**:
  - 真实异常-已修复
  - 真实异常-持续观察
  - 误报-模型需优化
  - 误报-传感器故障
- **输出**: 告警闭环记录，误报事件纳入模型优化反馈库
- **时效**: 告警从产生到闭环不超过7天

### 2.4 质量检查点
| 检查项 | 标准 | 检查频率 | 责任人 |
|--------|------|---------|--------|
| 告警产生到推送时间 | <30秒 | 实时监控 | IoT数据监控分析师 |
| 紧急告警响应确认时间 | <5分钟 | 每次告警 | 维护经理 |
| 告警误报率 | <15% | 月度统计 | IoT数据监控分析师 |
| 告警闭环率 | 100%（7天内） | 周度检查 | IoT数据监控分析师 |
| 告警升级规则执行率 | 100% | 每次超时 | 系统自动 |

## 三、SOP-IOT-002 预测性维护模型管理流程

### 3.1 流程目标
确保预测模型准确率>=80%，提前7天预测故障，模型性能持续监控和迭代优化。

### 3.2 触发条件
- 定期触发：每周运行RUL评估（所有A类设备）
- 事件触发：设备健康度快速下降（7天内降幅>15分）
- 反馈触发：预测准确率月度评估低于80%

### 3.3 详细步骤

#### 步骤1：数据准备（预测性维护分析师 + IoT数据监控分析师）
- **动作**: 从IoT数据平台提取目标设备的历史传感器特征数据
- **数据要求**:
  - 时间跨度：至少30天连续数据
  - 数据质量：完整率>95%，无大段缺失
  - 特征集：振动RMS/峰值/频谱特征、温度趋势、电流THD等
- **输出**: 结构化特征数据集
- **异常处理**: 数据质量不达标时通知IoT运维修复，暂缓预测

#### 步骤2：特征工程（预测性维护分析师）
- **动作**:
  - 选择退化相关特征（与历史故障强相关的指标）
  - 构建退化曲线（Health Indicator随时间变化）
  - 特征标准化和降维
- **输出**: 模型输入特征矩阵
- **质量要求**: 选取的退化指示器必须与故障模式有物理解释性

#### 步骤3：模型训练/更新（预测性维护分析师）
- **动作**:
  - 新设备：使用迁移学习从同类设备模型初始化
  - 已有模型：增量学习纳入新数据和反馈
  - 性能退化：完整重训练
- **模型类型**: 根据数据量选择：LSTM（数据量大）/ Weibull（数据量小）
- **输出**: 训练好的预测模型（含版本号）
- **异常处理**: 训练失败/不收敛时回退到上一版本模型

#### 步骤4：验证评估（预测性维护分析师）
- **动作**: 使用留出的验证集评估模型性能
- **评估指标**:
  - 准确率>=80%
  - 召回率>=85%
  - 平均预测提前期>=7天
  - 误报率<20%
- **通过标准**: 所有指标达标方可上线
- **异常处理**: 指标不达标→分析原因→调整特征/模型→重新训练（最多3轮）

#### 步骤5：上线部署（预测性维护分析师）
- **动作**: 将验证通过的模型部署到生产环境
- **部署策略**: 灰度发布，先在部分设备上运行观察1周
- **输出**: 上线记录（模型版本/上线时间/覆盖设备范围）
- **异常处理**: 灰度期间发现问题立即回滚

#### 步骤6：持续监控（预测性维护分析师）
- **动作**: 月度评估模型性能指标，对比预测结果与实际结果
- **监控指标**: 准确率趋势、误报率趋势、预测提前期趋势
- **触发迭代**:
  - 准确率连续2月<80% → 触发重训练
  - 新故障模式出现 → 30天内纳入训练集
  - 设备大修/改造后 → 重新建立基线
- **输出**: 月度模型性能报告

### 3.4 质量检查点
| 检查项 | 标准 | 检查频率 | 责任人 |
|--------|------|---------|--------|
| 模型预测准确率 | >=80% | 月度 | 预测性维护分析师 |
| 误报率 | <20% | 月度 | 预测性维护分析师 |
| 模型性能评估 | 按时完成 | 月度 | 预测性维护分析师 |
| 新故障模式纳入训练集 | 30天内 | 事件驱动 | 预测性维护分析师 |
| 模型版本变更记录 | 完整可追溯 | 每次变更 | 预测性维护分析师 |

## 四、SOP-IOT-003 产线瓶颈分析与排产优化流程

### 4.1 流程目标
每班次自动识别TOP3瓶颈工序并输出改善建议，排产方案综合考虑四要素（订单/产能/换线/物料），排产方案采纳率>=70%。

### 4.2 触发条件
- 定期触发：每班次结束后自动运行瓶颈分析
- 事件触发：新订单/插单需求到达
- 实时触发：OEE实时计算持续运行（看板延迟<5秒）

### 4.3 详细步骤

#### 步骤1：生产数据采集（产线效率优化Agent）
- **动作**: 从MES和IoT系统采集当班次生产数据
- **数据内容**:
  - 各工序实际节拍时间（CT）
  - 各工序前WIP堆积量
  - 各工序良率/直通率
  - 设备运行时间/停机时间/换型时间
  - 产出数量和计划完成率
- **输出**: 结构化生产数据集
- **异常处理**: MES数据缺失时从IoT数据补充，两者都缺失时标记数据不完整

#### 步骤2：瓶颈识别（产线效率优化Agent）
- **动作**: 应用多维度瓶颈识别方法
  - CT对比法：识别节拍时间最长的工序
  - WIP堆积法：识别WIP最多的工序前工序
  - 利用率法：识别设备利用率最高的工序
  - 综合评分排名输出TOP3
- **输出**: TOP3瓶颈工序报告（排名/工序/评分/数据依据）
- **时效**: 班次结束后30分钟内完成
- **异常处理**: 数据不完整时在报告中标注数据覆盖率，不影响可分析工序的输出

#### 步骤3：根因分类（产线效率优化Agent）
- **动作**: 对每个瓶颈工序分析根因类别
  - 设备因素：OEE低/故障频繁/速度未达标
  - 工艺因素：CT长/换型时间长/工艺复杂
  - 物料因素：缺料等待/来料质量问题
  - 人员因素：技能不足/人员不够
- **输出**: 根因分类结果和改善方向
- **质量要求**: 根因判定必须有数据支撑，不得仅凭假设

#### 步骤4：优化建议生成（产线效率优化Agent）
- **动作**: 针对每个瓶颈和根因生成改善建议
  - 短期建议（1周内）：排程调整/人员调配/参数微调
  - 中期建议（1-3月）：SMED换型改善/工艺优化/设备升级
  - 长期建议（>3月）：产线布局/设备扩能
- **必须量化预期效果**: "预计提升OEE X个百分点"或"减少换线时间X分钟"
- **输出**: 结构化优化建议报告
- **异常处理**: 对于无法量化的建议标注"预期效果待验证"

#### 步骤5：排产方案（新订单/插单场景）（产线效率优化Agent）
- **触发**: 收到新订单或插单需求
- **动作**:
  1. 评估当前产能可用情况
  2. 检查物料齐套状态
  3. 计算换线时间影响
  4. 综合约束生成2-3个排产方案
  5. 方案对比（交期满足率/利用率/换线次数）
- **输出**: 推荐排产方案+备选方案，标注各方案优劣
- **审批**: 提交生产计划员/生产经理人工确认
- **异常处理**: 产能完全不足时输出分析报告和应对建议（加班/委外/交期协商）

#### 步骤6：执行跟踪（产线效率优化Agent）
- **动作**: 排产方案确认执行后，跟踪实际执行情况与计划的偏差
- **监控项**: 实际CT vs计划CT、实际产出vs计划产出、实际换线时间vs计划
- **输出**: 执行偏差报告，为下次排产提供修正依据
- **异常处理**: 偏差>20%时触发重新排产评估

### 4.4 OEE实时计算（持续运行）

#### 计算逻辑
- **可用率** = (计划运行时间 - 停机时间) / 计划运行时间
- **性能率** = (实际产出 × 理论CT) / 实际运行时间
- **良品率** = 良品数 / 总产出数
- **OEE** = 可用率 × 性能率 × 良品率

#### 六大损失分解
每次OEE计算同时输出六大损失的时间分解和占比排名

#### 看板更新
- 数据刷新频率：<5秒
- 展示内容：实时OEE值、可用率/性能率/良品率、六大损失排名、与目标对比、趋势图

### 4.5 质量检查点
| 检查项 | 标准 | 检查频率 | 责任人 |
|--------|------|---------|--------|
| 瓶颈报告按时生成 | 班次结束后30min内 | 每班次 | 产线效率优化Agent |
| 排产方案四要素完整性 | 100%覆盖 | 每次排产 | 产线效率优化Agent |
| 优化建议量化预期效果 | 100%有量化指标 | 每次建议 | 产线效率优化Agent |
| OEE看板数据延迟 | <5秒 | 实时监控 | IoT运维团队 |
| 排产方案采纳率 | >=70% | 月度统计 | 产线效率优化Agent |

## 五、决策树

```
传感器数据到达
├── 数据质量检查
│   ├── 质量合格（完整率>95%）→ 进入正常处理流程
│   └── 质量不合格
│       ├── 传感器离线 → 数据质量告警 → 通知IoT运维
│       └── 数据漂移 → 标记异常数据 → 使用备用数据源
│
├── 异常检测
│   ├── 无异常 → 更新健康度评分 → 数据归档
│   └── 检测到异常 → 评估健康度分数
│       ├── 健康度 > 60（观察级）
│       │   → 增加监控频率 → 标记关注 → 纳入RUL评估队列
│       ├── 健康度 40-60（告警级）
│       │   → 生成告警 → 通知维护检查 → 继续监控
│       │   → 是否在7天内有RUL风险？
│       │       ├── 是 → 生成维护建议 → 推送维护计划调度员
│       │       └── 否 → 安排计划性检查
│       └── 健康度 < 40（紧急级）
│           → 生成紧急告警 → 建议停机 → 联动故障诊断专家
│           → 5分钟内有响应？
│               ├── 是 → 进入维修流程
│               └── 否 → 自动升级到车间主任/工厂经理
│
├── 预测性维护路径（定期/事件触发）
│   ├── RUL评估结果
│   │   ├── RUL > 30天 → 正常监控
│   │   ├── RUL 7-30天 → 生成维护建议（常规优先级）
│   │   └── RUL < 7天 → 生成紧急维护建议（高优先级）
│   │       → 备件是否在库？
│   │           ├── 是 → 安排维护时间窗口
│   │           └── 否 → 触发紧急采购 + 评估风险
│   └── 预测结果跟踪
│       ├── 预测准确 → 正反馈 → 强化模型
│       ├── 误报 → 记录反馈 → 纳入模型优化
│       └── 漏报 → 高优先级反馈 → 紧急模型复盘
│
└── 产线优化路径
    ├── 每班次瓶颈分析 → TOP3瓶颈
    │   ├── 设备因素 → 联动设备维护Scope
    │   ├── 工艺因素 → 输出工艺优化建议
    │   ├── 物料因素 → 联动供应链Scope
    │   └── 人员因素 → 输出人员配置建议
    └── 排产优化（新订单触发）
        ├── 产能充足 + 物料齐套 → 生成排产方案 → 人工确认 → 执行
        ├── 产能不足 → 评估应对（加班/委外/协商交期）
        └── 物料未齐套 → 联动供应链确认到料时间 → 延后排产
```

## 七、异常处理与升级机制

### 7.1 系统异常

| 异常情况 | 处理措施 | 升级条件 |
|---------|---------|---------|
| 传感器数据中断 | 标记数据质量告警→通知IoT运维→切换备用监控 | 中断>30分钟升级到IoT运维经理 |
| 检测算法超时 | 切换到简单阈值检测→记录性能事件→排查 | 频繁超时（>3次/天）升级到系统架构师 |
| 模型预测服务不可用 | 切换到规则引擎告警→记录服务中断→恢复 | 中断>1小时升级到IT运维 |
| OEE看板延迟>5秒 | 排查数据链路→临时降低刷新频率 | 持续>30分钟升级到IoT运维 |

### 7.2 业务异常

| 异常情况 | 处理措施 | 升级条件 |
|---------|---------|---------|
| 连续告警无人响应 | 自动升级推送→记录响应失效事件 | 按分级升级路径执行 |
| 模型误报率超标 | 分析误报原因→调整阈值/重训练模型 | 连续2月>20%升级到技术负责人 |
| 排产方案被频繁拒绝 | 分析拒绝原因→优化方案生成逻辑 | 采纳率<50%连续2月升级到工程经理 |
| 健康度评分与实际不符 | 检查传感器/评分模型→校准修正 | 严重偏差（>30分）升级到设备工程师 |

## 八、跨Scope协作接口

### 8.1 与设备维护管理Scope
- **输出**: 设备异常告警（含级别、数据证据、健康度）→ 故障诊断专家
- **输出**: 预测性维护建议（含RUL、备件需求、时间窗口）→ 维护计划调度员
- **输入**: 维护完成通知 → 重置设备基线、重新评估健康度

### 8.2 与供应链管理Scope
- **输出**: 物料消耗计划（排产结果驱动）→ 库存管理专员
- **输入**: 物料齐套状态 → 排产约束条件
- **输出**: 紧急备件需求 → 采购计划专员

### 8.3 与质量管理Scope
- **输出**: 工序节拍与质量关联分析 → 质量检验分析师
- **输入**: 工序良率数据 → OEE良品率计算
- **输出**: 异常工艺参数信息 → 质量异常处理专家

### 8.4 与客户反馈Scope
- **输出**: 缺陷批次的生产过程数据 → 缺陷模式分析Agent
- **输入**: 缺陷关联的工艺参数调整需求 → 工艺优化
