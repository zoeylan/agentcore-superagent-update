# 模型性能监控 (Model Performance Monitoring)

## 概述
持续监控预测性维护模型的运行表现，评估预测准确率、误报率、漏报率等关键指标，识别模型性能退化趋势并触发模型重训练流程。

## 适用场景
- 预测模型上线后的持续性能评估
- 模型预测结果与实际故障时间的对照验证
- 模型漂移（Concept Drift）检测
- 模型重训练时机判定
- 不同模型版本的A/B对比评估

## 执行步骤

### 步骤1：预测结果跟踪
- 记录每次预测输出（预测时间、目标设备、预测RUL、置信度）
- 跟踪实际结果：设备是否在预测时间窗口内故障、实际故障时间
- 分类预测结果：
  - 准确预测（TP）：预测有故障风险，实际发生了故障
  - 误报（FP）：预测有故障风险，实际未发生
  - 漏报（FN）：未预测到故障，但实际发生了
  - 正确否定（TN）：未预测故障，实际也未发生

### 步骤2：性能指标计算
- **准确率** = (TP + TN) / Total，目标>=80%
- **召回率** = TP / (TP + FN)，目标>=85%（漏报代价高）
- **精确率** = TP / (TP + FP)，目标>=70%
- **平均预测提前期**：预测到实际故障的时间差，目标>=7天
- **误报率** = FP / (FP + TN)，目标<20%
- 按月/季度滚动计算以上指标

### 步骤3：性能退化检测
- 对比当前周期与历史周期的指标变化趋势
- 检测模型漂移信号：
  - 准确率连续2个月下降>5%
  - 误报率连续上升
  - 预测提前期持续缩短（从7天→3天）
- 分析性能退化原因：数据分布变化、新设备/新工况引入、传感器更换

### 步骤4：触发与报告
- 性能达标：生成月度性能报告，无需干预
- 性能预警（某指标接近阈值）：增加监控频率，记录观察
- 性能不达标（准确率<80%或误报率>20%）：触发模型重训练流程
- 输出改善建议：增加训练数据、调整特征、更换模型架构等

## 输入规格
| 字段 | 类型 | 说明 |
|------|------|------|
| prediction_logs | array | 历史预测记录 |
| actual_outcomes | array | 实际故障/正常运行记录 |
| model_version | string | 当前模型版本号 |
| evaluation_period | object | 评估时间范围 |

## 输出规格
| 字段 | 类型 | 说明 |
|------|------|------|
| accuracy | number | 准确率 |
| recall | number | 召回率 |
| precision | number | 精确率 |
| false_alarm_rate | number | 误报率 |
| avg_lead_time_days | number | 平均预测提前期 |
| performance_trend | enum | 性能趋势（improving/stable/degrading） |
| action_required | enum | 需要的动作（none/monitor/retrain） |
| improvement_suggestions | array | 改善建议列表 |

## 最佳实践
- 模型每月重新评估一次性能（即使指标正常也需定期审视）
- 保留所有预测记录和反馈数据，用于模型迭代训练
- 误报和漏报对业务影响不同（漏报可能导致非计划停机，代价更高），评估时需加权考虑
- 新设备类型纳入预测范围前，需至少积累3个月运行数据

## 约束条件
- 模型性能评估至少需要30个预测样本才具有统计意义
- 性能不达标时的模型重训练需在30天内完成
- 新版本模型上线前必须通过离线回测验证（准确率>=80%）
- 模型版本切换需记录完整的变更日志和审批记录
