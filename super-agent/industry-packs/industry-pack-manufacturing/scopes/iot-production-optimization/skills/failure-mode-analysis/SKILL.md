# 故障模式分析 (Failure Mode Analysis)

## 概述
对设备历史故障数据进行系统性分析，识别高频故障模式、故障间关联性和故障演化规律，为预测性维护模型优化和维护策略制定提供数据支撑。

## 适用场景
- 设备MTBF（平均故障间隔）和MTTR（平均修复时间）趋势分析
- 故障模式Pareto分析（识别TOP故障类型）
- 故障关联分析（不同部件故障的前后关系）
- 新故障模式识别与特征签名提取
- 故障根因分类（机械磨损/电气老化/操作不当/环境因素）

## 执行步骤

### 步骤1：故障数据采集与整理
- 从CMMS（维护管理系统）提取设备故障工单历史数据
- 结构化处理：提取故障时间、设备编号、部件、故障现象描述、根因、修复措施
- 将非结构化故障描述进行标准化分类（建立故障模式编码体系）
- 关联故障前的传感器数据（故障发生前7天的特征趋势）

### 步骤2：统计分析
- **MTBF/MTTR计算**：按设备、按部件、按故障类型分别计算
- **Pareto分析**：识别占80%故障次数的TOP20%故障模式
- **趋势分析**：MTBF是否在改善（上升）或恶化（下降）
- **季节性分析**：某些故障是否与季节/温湿度相关（如夏季过热故障增多）

### 步骤3：模式关联挖掘
- 分析故障的时序关联：A部件故障是否常在B部件故障后出现
- 分析故障的条件关联：特定工况/负载下某类故障发生概率更高
- 构建故障传播图：主故障→次生故障的因果链路
- 提取故障前兆特征：故障发生前传感器数据的共性变化模式

### 步骤4：知识输出
- 生成故障模式知识库条目（故障模式编码、特征签名、根因分类、建议措施）
- 输出新故障模式识别报告（以前未出现的异常模式）
- 更新设备FMEA风险评估（频度/严重度/探测度评分）
- 为RUL预测模型提供标注数据和故障阈值参考

## 输入规格
| 字段 | 类型 | 说明 |
|------|------|------|
| device_id | string | 设备编号 |
| failure_history | array | 历史故障工单数据集 |
| sensor_data_before_failure | array | 故障前传感器数据快照 |
| maintenance_records | array | 维护保养记录 |

## 输出规格
| 字段 | 类型 | 说明 |
|------|------|------|
| mtbf | number | 平均故障间隔（小时） |
| mttr | number | 平均修复时间（小时） |
| top_failure_modes | array | TOP故障模式排名（含频次、占比） |
| failure_correlations | array | 故障关联关系 |
| precursor_signatures | array | 故障前兆特征签名 |
| new_patterns | array | 新发现的故障模式 |

## 最佳实践
- 故障数据至少需要6个月以上才能做有意义的统计分析
- FMEA风险优先级数（RPN）>200的故障模式需重点关注
- 新故障模式在30天内必须纳入预测模型训练集
- 故障分析结论应与维修技术人员沟通验证，避免纯数据导致的误判

## 约束条件
- 故障分类编码需遵循企业统一编码标准
- 分析结论必须有统计支撑（样本量>=5才能作为趋势性结论）
- 涉及安全类故障的分析结果需提交安全管理部门审核
- 故障知识库更新需经过技术评审后方可纳入正式库
