# OEE分析 (OEE Analysis)

## 概述
实时计算和深度分析设备综合效率（Overall Equipment Effectiveness），将OEE分解为可用率、性能率、良品率三维度，进一步拆解六大损失，精确定位效率提升机会。

## 适用场景
- 单台设备/产线OEE实时计算与看板展示
- OEE六大损失分解与根因定位
- OEE趋势分析和目标对比
- 设备群组OEE排名对比
- OEE改善措施效果验证

## 执行步骤

### 步骤1：数据采集与时间分类
按照SEMI E10标准对设备时间进行分类：
- **计划时间**：日历时间 - 计划停机（节假日/无订单/计划保养）
- **运行时间**：计划时间 - 非计划停机（故障停机/缺料等待）
- **净运行时间**：运行时间 - 速度损失（减速运行/空转暂停）
- **有价值运行时间**：净运行时间 - 质量损失（不良品/返工）

### 步骤2：三维度计算
- **可用率** = 运行时间 / 计划时间
  - 损失来源：故障停机、换型调整、物料等待、其他停机
- **性能率** = (实际产出 × 理论CT) / 运行时间
  - 损失来源：减速运行、短暂停机（<5分钟）、空转
- **良品率** = 良品数 / 总产出数
  - 损失来源：不良品、返工品、启动损失
- **OEE** = 可用率 × 性能率 × 良品率

### 步骤3：六大损失分解
将OEE gap（100% - 实际OEE）分解为六大损失时间：
1. **故障停机损失**：非计划停机时间（从IoT停机事件数据获取）
2. **换型调整损失**：产品切换和设备调整时间
3. **空转暂停损失**：短暂停机（<5分钟的微停）
4. **速度降低损失**：实际速度低于设计速度的部分
5. **不良品损失**：生产不良品消耗的时间
6. **启动损失**：开机暖机、首件调试消耗的时间

按各损失的时间占比排序，确定改善优先级

### 步骤4：分析输出与建议
- 输出OEE报告（实时/班次/日/周/月维度）
- 与目标值对比：A类设备目标>=85%，B类>=75%
- 标注最大损失项和改善方向
- 关联历史改善措施验证效果（改善前后OEE对比）
- OEE看板数据确保延迟<5秒

## 输入规格
| 字段 | 类型 | 说明 |
|------|------|------|
| device_id | string | 设备编号 |
| time_period | object | 分析时段 |
| planned_production_time | number | 计划生产时间（分钟） |
| downtime_events | array | 停机事件列表（含原因分类和时长） |
| actual_output | number | 实际产出数量 |
| theoretical_ct | number | 理论节拍时间（秒/件） |
| defect_count | number | 不良品数量 |
| changeover_events | array | 换型事件列表 |

## 输出规格
| 字段 | 类型 | 说明 |
|------|------|------|
| oee | number | OEE值（百分比） |
| availability | number | 可用率 |
| performance | number | 性能率 |
| quality | number | 良品率 |
| six_losses | object | 六大损失明细（各损失时间和占比） |
| vs_target | object | 与目标值对比 |
| top_loss | string | 最大损失项 |
| improvement_priority | array | 改善优先级排序 |
| trend | object | OEE趋势（对比上期） |

## 最佳实践
- OEE计算需要精确的停机时间记录，IoT自动采集比人工录入更准确
- 区分"计划停机"和"非计划停机"是OEE准确性的关键
- 世界级OEE水平为85%+，多数工厂实际在60-75%之间
- 改善聚焦最大损失项（通常是故障停机或换型），遵循帕累托法则

## 约束条件
- OEE看板数据延迟<5秒（实时刷新）
- 理论CT必须使用经过验证的标准工时，不得随意调整
- 停机原因分类必须使用统一编码，确保分析一致性
- OEE数据不得人工篡改，系统自动采集为准
