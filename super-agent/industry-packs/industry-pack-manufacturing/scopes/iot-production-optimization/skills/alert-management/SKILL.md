# 告警管理 (Alert Management)

## 概述
管理设备异常告警的完整生命周期——从告警生成、分级推送、响应确认到闭环记录，确保异常信息快速有效地触达正确的责任人并得到及时处置。

## 适用场景
- 异常检测触发后的告警生成与格式化
- 基于告警级别和设备归属的智能推送路由
- 告警响应状态跟踪与超时升级
- 告警误报统计与规则优化
- 连续告警无人响应时的自动升级

## 执行步骤

### 步骤1：告警生成
- 接收异常检测结果，生成标准化告警记录
- 告警内容包含：告警ID、设备信息、异常类型、严重级别、发生时间、数据证据、建议动作
- 判断是否为重复告警（同一设备同类告警在未关闭状态下不重复推送，但记录次数）
- 分配告警唯一编号，记录入告警数据库

### 步骤2：分级推送路由
- **观察级告警**：推送到设备责任人的消息队列，不强制响应
- **告警级（健康度40-60）**：推送到维护班组长+设备责任人，要求5分钟内确认
- **紧急级（健康度<40）**：推送到维护经理+班组长+设备责任人+生产主管，要求立即响应
- 推送渠道：系统消息+短信/企微通知（紧急级加电话呼叫）

### 步骤3：响应跟踪与升级
- 跟踪告警确认状态（未确认→已确认→处理中→已关闭）
- 超时升级机制：
  - 紧急级5分钟未确认 → 升级到车间主任
  - 紧急级15分钟未确认 → 升级到工厂经理
  - 告警级30分钟未确认 → 升级到维护经理
- 记录每个状态变化的时间戳和操作人

### 步骤4：闭环管理与分析
- 每条告警必须有闭环记录（处置结果：真实异常已修复/观察后恢复正常/误报）
- 统计告警处理时效（从产生到关闭的时间）
- 月度告警分析：误报率统计、高频告警设备排名、响应时效达标率
- 对误报率高的告警规则提出优化建议

## 输入规格
| 字段 | 类型 | 说明 |
|------|------|------|
| anomaly_result | object | 异常检测输出结果 |
| device_info | object | 设备信息（编号、名称、区域、责任人） |
| alert_rules | object | 告警规则配置（推送路由、升级策略） |

## 输出规格
| 字段 | 类型 | 说明 |
|------|------|------|
| alert_id | string | 告警唯一编号 |
| alert_level | enum | 告警级别 |
| recipients | array | 推送接收人列表 |
| push_channels | array | 推送渠道 |
| status | enum | 当前状态 |
| escalation_history | array | 升级记录 |
| closure_record | object | 闭环记录 |

## 最佳实践
- 告警从产生到推送的延迟必须<30秒
- 避免告警风暴：同一设备在短时间内的多条告警合并推送（聚合窗口60秒）
- 定期review告警规则，淘汰长期无触发或频繁误报的规则
- 建立告警分级的量化标准，避免人为主观判断级别

## 约束条件
- 紧急级告警5分钟内必须有人响应确认
- 告警误报率控制目标<15%
- 每条告警必须有闭环处理记录，不允许未关闭的告警积压超过7天
- 告警推送不得绕过已定义的路由规则（安全合规要求）
