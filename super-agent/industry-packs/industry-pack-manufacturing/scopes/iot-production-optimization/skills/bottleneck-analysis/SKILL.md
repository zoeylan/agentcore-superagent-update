# 产线瓶颈分析 (Bottleneck Analysis)

## 概述
基于TOC约束理论（Theory of Constraints），运用生产数据和设备实时运行数据，自动识别产线中制约整体产出的瓶颈工序，分析瓶颈根因并提出改善建议。

## 适用场景
- 每班次自动识别TOP3瓶颈工序
- 新产品导入后的产线平衡分析
- 瓶颈工序根因分类（设备/工艺/物料/人员）
- 瓶颈改善措施效果验证

## 执行步骤

### 步骤1：数据采集
- 从MES获取各工序实际节拍时间（Cycle Time）数据
- 从设备IoT数据获取工序等待时间（Queue Time/等待WIP数量）
- 从质量系统获取各工序直通率/良率数据
- 采集设备可用率数据（停机时间/计划运行时间）

### 步骤2：瓶颈识别
- **CT对比法**：将各工序实际CT与理论CT对比，差异最大的为潜在瓶颈
- **WIP堆积法**：工序前WIP堆积最多的为瓶颈下游工序（瓶颈=堆积最多处）
- **利用率法**：设备利用率最高（接近100%）的工序为瓶颈
- **综合评分**：结合以上三维度的综合评分排名，输出TOP3瓶颈工序
- 注意：瓶颈可能是动态的（不同时段/不同产品组合下瓶颈位置不同）

### 步骤3：瓶颈根因分类
对识别出的瓶颈工序进行根因分类：
- **设备因素**：设备故障停机多、设备速度未达设计值、设备OEE低
- **工艺因素**：工艺流程复杂/步骤多、工艺参数设定不优、换型时间过长
- **物料因素**：物料等待/缺料、来料质量问题导致返工
- **人员因素**：人员技能不足/效率低、人员配置不足
- 同时评估瓶颈对整体产出的影响程度（提升瓶颈X%可提升整体产出Y%）

### 步骤4：改善建议输出
- 针对每个瓶颈工序输出具体改善建议：
  - 短期（1周内可执行）：如调整排程优先级、增加人员配置、微调工艺参数
  - 中期（1-3月）：如SMED快速换型改善、工艺优化、设备升级
  - 长期（>3月）：如产线布局调整、设备扩能投资
- 量化预期效果（如'消除TOP1瓶颈可提升产出约X%'）

## 输入规格
| 字段 | 类型 | 说明 |
|------|------|------|
| production_line_id | string | 产线编号 |
| shift_info | object | 班次信息（日期、班别） |
| process_ct_data | array | 各工序实际节拍时间数据 |
| wip_data | array | 各工序前WIP堆积数据 |
| quality_data | array | 各工序良率数据 |
| equipment_availability | array | 设备可用率数据 |

## 输出规格
| 字段 | 类型 | 说明 |
|------|------|------|
| analysis_period | object | 分析时段（班次信息） |
| top_bottlenecks | array | TOP3瓶颈工序（排名、工序名、评分） |
| bottleneck_evidence | object | 瓶颈判定依据（CT/WIP/利用率数据） |
| root_cause_category | enum | 根因分类 |
| impact_assessment | object | 对整体产出的影响评估 |
| improvement_suggestions | array | 改善建议（短/中/长期） |
| expected_improvement | string | 预期改善效果量化 |

## 最佳实践
- 瓶颈分析应每班次运行一次，持续跟踪瓶颈是否转移
- 瓶颈消除后观察是否出现新瓶颈（TOC中的"约束转移"现象）
- 不要试图同时解决所有瓶颈，聚焦TOP1瓶颈效果最大
- 结合价值流图（VSM）做全流程视角的分析，避免局部优化

## 约束条件
- 瓶颈判定必须基于量化数据，不能仅凭主观感受
- 每班次瓶颈报告必须在班次结束后30分钟内生成
- 改善建议的预期效果必须有数据测算依据
- 涉及设备参数调整的改善建议需经过工程师确认后方可执行
