# 根因分析 (Root Cause Analysis)

## 技能概述
对设备故障执行系统化的根因分析（RCA），运用5Why分析法、鱼骨图、故障树分析（FTA）等方法论，找到故障的根本原因而非表面原因，制定针对性的纠正措施和预防措施，防止同类故障重复发生。

## 适用场景
- 每次非计划停机后的必做分析（5Why报告）
- A类设备故障后的深度根因分析
- 同一设备30天内故障>=3次的专项分析
- MTBF持续下降趋势的系统性原因调查

## 执行步骤

### 步骤1：问题定义
- 准确描述故障问题（What/When/Where/How much）
- 界定分析范围：单次故障分析 vs 反复故障模式分析
- 收集事实数据：故障时间线、维修记录、操作日志、传感器数据
- 确认影响程度：停机时长、产能损失、质量影响、安全事件

### 步骤2：原因分析（5Why展开）
- 第1个Why：故障的直接原因是什么？（物理层面）
- 第2个Why：为什么会出现这个直接原因？（条件/机制层面）
- 第3个Why：为什么条件会恶化到触发故障？（监控/维护层面）
- 第4个Why：为什么监控/维护没有提前发现？（系统/流程层面）
- 第5个Why：为什么系统/流程存在这个缺陷？（管理/设计层面）
- 每层Why必须有事实数据或证据支持，避免主观猜测

### 步骤3：原因验证
- 验证根因的逻辑性：从根因正推→能否必然导致故障？
- 检查是否为充分必要条件：消除此根因→故障是否不再发生？
- 排除混淆因素：是否存在其他同时变化的因素？
- 对比类似设备：相同条件下其他设备是否有同样问题？

### 步骤4：制定措施
- 纠正措施（Corrective Action）：消除已发生的问题后果
- 预防措施（Preventive Action）：消除根因，防止复发
- 措施分级：临时对策（24h内）→永久对策（30天内）
- 每项措施明确：责任人、完成日期、验证标准、验证方法
- 评估措施可行性和副作用

### 步骤5：效果验证标准
- 定义"成功"标准：X个月内同类故障为零 / MTBF恢复到目标值
- 设定验证周期：通常为措施实施后90天
- 确定监控指标和检查频率

## 输入规格
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| fault_record | object | 是 | 故障记录（时间/现象/处置/恢复）|
| equipment_info | object | 是 | 设备信息 |
| maintenance_history | array | 是 | 近期维护记录 |
| sensor_data_before_fault | object | 否 | 故障前传感器数据趋势 |
| similar_cases | array | 否 | 历史相似案例 |
| operation_log | array | 否 | 故障前操作日志 |

## 输出规格
| 字段 | 类型 | 说明 |
|------|------|------|
| problem_statement | string | 问题描述 |
| five_why_chain | array | 5Why逐层分析（含证据）|
| root_cause | string | 确定的根本原因 |
| contributing_factors | array | 促成因素 |
| corrective_actions | array | 纠正措施（含责任人/日期）|
| preventive_actions | array | 预防措施（含责任人/日期）|
| verification_criteria | object | 验证标准和周期 |
| lessons_learned | string | 经验教训总结 |

## 最佳实践
- 5Why分析应由跨职能小组共同完成（维修/操作/工艺/质量）
- 每层Why的答案必须有数据/事实支撑，禁止"因为人员不小心"等无法改善的结论
- 根因通常落在系统/流程/设计层面，而非个人操作失误层面
- 预防措施应优先采用防错（Poka-yoke）设计，其次是标准化流程
- 分析完成后必须更新故障知识库和PM标准（如调整PM频率/增加检查项）

## 约束条件
- 非计划停机的5Why报告必须在设备恢复后72小时内完成
- A类设备故障的深度RCA必须在7天内完成
- 措施验证周期不超过90天
- 分析报告需要维护主管审核签字确认
