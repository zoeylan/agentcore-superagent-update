# SCAR管理 (Supplier Corrective Action Request Management)

## 概述
本技能负责供应商纠正措施要求（SCAR）的全生命周期管理，包括SCAR的发起、跟踪、评审和关闭。SCAR是制造业中管控供应商质量的核心工具，当来料检验发现不合格或供应商相关质量问题发生时，通过正式的SCAR流程要求供应商进行根因分析并制定纠正/预防措施，确保质量问题不再复发。

## 适用场景
- 来料检验（IQC）判定不合格的批次
- 供应商来料导致的产线停线事件
- 客户投诉追溯到供应商来料原因
- 供应商交付的产品不符合PPAP批准的规格
- 供应商质量绩效评分连续下降触发的整改要求
- 来料中发现的安全或法规合规性问题

## 执行步骤

### 步骤1：SCAR发起条件评估
1. 确认不合格事实：
   - 来源：IQC检验报告/产线不良反馈/客户投诉
   - 不合格类型：尺寸超差/外观缺陷/功能失效/材质不符/标识错误
   - 严重程度评估：CRITICAL（安全/法规）/ MAJOR（功能影响）/ MINOR（外观/次要）
2. 判断是否需要发起SCAR：
   - 必须发起：CRITICAL缺陷、连续2批不合格、同一缺陷3个月内复发
   - 建议发起：MAJOR缺陷首次发生
   - 可选发起：MINOR缺陷但供应商无改善趋势
3. 收集支撑证据：不良品照片、检验数据、检验报告编号、相关标准/规格

### 步骤2：SCAR文档编制与发出
1. 填写SCAR表单核心内容：
   - SCAR编号（自动生成，格式：SCAR-YYYY-NNNN）
   - 供应商信息（名称/编号/联系人）
   - 不合格品信息（物料号/批次号/数量/不良率）
   - 问题描述（含5W1H：何物/何处/何时/何量/如何表现）
   - 不合格证据（照片/检测数据/对比标准值）
   - 要求回复内容：临时遏制措施、根因分析、纠正措施、预防措施、验证计划
   - 回复时限：7个工作日（CRITICAL级5个工作日）
2. 审核SCAR内容完整性（问题描述是否清晰、证据是否充分）
3. 正式发送供应商（邮件+系统通知），抄送采购人员
4. **时限要求：来料不合格确认后24小时内发出SCAR**

### 步骤3：供应商回复跟踪
1. 监控回复时效：
   - 7天到期提醒（到期前2天发送催促）
   - 超期未回复：标记为逾期，自动升级通知采购经理
   - 目标：95%以上供应商在7天内回复
2. 接收供应商回复后进行初步完整性检查：
   - 是否包含所有要求的内容（遏制/根因/纠正/预防）
   - 是否有具体的措施而非泛泛承诺
   - 是否有时间计划和责任人

### 步骤4：回复质量评审
1. 临时遏制措施评审：
   - 是否针对已生产/已发货的可疑品进行了排查
   - 遏制范围是否充分（不仅当前批次，还包括在途和在库品）
2. 根因分析评审（最关键环节）：
   - 5Why分析是否深入到第3层以上
   - 根因是否可解释问题现象（逻辑自洽性）
   - 拒绝表面原因（如"员工疏忽"、"偶发事件"）
   - 是否区分了直接原因和系统原因
3. 纠正措施评审：
   - 措施是否针对根本原因而非现象
   - 措施是否具体可执行（有责任人、时限、验证标准）
   - 措施是否有防错机制（Poka-Yoke）而非仅依赖人工注意力
4. 预防措施评审：
   - 是否有横展考虑（类似产品/工序是否检查）
   - 是否更新了相关控制文件（作业指导书/检验标准）
5. 评审结论：
   - 接受：分析充分、措施得当 → 进入验证阶段
   - 有条件接受：部分不足，限期补充 → 返回供应商修改
   - 拒绝：分析浮浅、措施无效 → 退回重做并记录不良记录

### 步骤5：措施有效性验证
1. 验证时间窗口：措施落实后连续3批来料检验（或2个月，取先到者）
2. 验证标准：
   - 同类缺陷不再出现（零复发）
   - 相关指标恢复或超过历史水平
3. 验证通过 → SCAR关闭
4. 验证未通过 → 重新开启SCAR或升级处理（约谈/减少份额/淘汰预警）

### 步骤6：SCAR关闭与数据归档
1. 填写关闭记录：验证结果、实际关闭日期、经验教训
2. 更新供应商质量档案：
   - SCAR历史记录（发起数/关闭率/平均关闭周期）
   - 对供应商季度评分的影响
3. 数据归档并纳入供应商质量绩效分析数据集

## 输入规格
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| supplier_code | string | 是 | 供应商编号 |
| supplier_name | string | 是 | 供应商名称 |
| material_code | string | 是 | 不合格物料编号 |
| batch_id | string | 是 | 不合格批次号 |
| defect_description | string | 是 | 缺陷描述 |
| defect_quantity | integer | 是 | 不合格品数量 |
| severity | enum | 是 | CRITICAL/MAJOR/MINOR |
| evidence | array | 是 | 证据（照片/检测数据） |
| iqc_report_id | string | 是 | 关联的IQC检验报告编号 |
| specification_ref | string | 是 | 对应的标准/规格文件 |

## 输出规格
| 字段 | 类型 | 说明 |
|------|------|------|
| scar_id | string | SCAR编号 |
| status | enum | OPEN/PENDING_REPLY/UNDER_REVIEW/VERIFICATION/CLOSED |
| issue_date | datetime | 发出日期 |
| due_date | datetime | 要求回复日期 |
| supplier_reply_date | datetime | 实际回复日期 |
| review_result | enum | ACCEPTED/CONDITIONAL/REJECTED |
| root_cause | string | 供应商分析的根本原因 |
| corrective_actions | array | 纠正措施清单 |
| verification_result | enum | PASS/FAIL |
| closure_date | datetime | 关闭日期 |
| days_to_close | integer | 总处理天数 |

## 最佳实践
1. **24小时黄金时间**：不合格确认后24小时内发出SCAR，确保供应商在问题新鲜时进行分析
2. **证据充分**：SCAR中的问题描述越详细、证据越充分，供应商越容易快速定位根因
3. **评审不妥协**：对供应商回复的评审是SCAR质量的核心，宁可退回重做也不接受浮浅分析
4. **趋势管理**：单个SCAR看个案，但要持续跟踪供应商的SCAR数量趋势和关闭质量趋势
5. **与8D联动**：供应商的SCAR回复本质是供应商侧的8D，评审标准应与内部8D一致

## 约束条件
- SCAR必须基于确认的不合格事实发起，不能基于怀疑或推测
- 不接受供应商"增加检验"作为唯一纠正措施（检验不增加价值，必须改善过程）
- 同一供应商同一问题6个月内复发，视为原SCAR措施无效，需升级处理
- SCAR关闭必须有验证数据支撑，不接受供应商自我声明"已改善"
- SCAR发起和关闭权限属于SQE，采购人员无权关闭质量类SCAR

## 示例

### 示例1：SCAR发起与评审
```
SCAR发起：
- SCAR编号: SCAR-2024-0156
- 供应商: 精密五金有限公司 (SUP-0023)
- 物料: 铝合金外壳 M-HSG-0045
- 批次: LOT-20240315-002
- 缺陷: 表面阳极氧化层厚度不足（规格要求≥15μm，实测8-12μm）
- 不合格率: 16%（抽检50件，8件不合格）
- 严重度: MAJOR（影响耐腐蚀性能）

供应商回复评审：
- 根因: "阳极氧化时间不足" → 评审结果：拒绝
  理由: 这是现象描述而非根因。需追问为什么时间不足？是设定参数错误？还是设备故障？还是员工未执行标准参数？

- 重新回复后根因: "氧化槽温度传感器偏移(+3°C)，导致实际槽温偏高，同等时间下膜层生长不足"
  → 评审结果：接受
  纠正措施: 更换温度传感器+校准验证+增加每日开班温度点检
  预防措施: 所有氧化槽传感器纳入月度校准计划
```
