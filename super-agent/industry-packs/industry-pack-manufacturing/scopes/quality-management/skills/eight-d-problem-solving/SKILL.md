# 8D问题解决 (Eight-D Problem Solving)

## 概述
本技能负责按照8D（Eight Disciplines）方法论系统性解决质量异常问题。8D是制造业中最广泛使用的结构化问题解决方法，适用于需要跨部门协作、需要深入根因分析的复杂质量问题。确保从问题识别到彻底预防的完整闭环。

## 适用场景
- 来料检验不合格且涉及批量性问题
- 过程检验发现系统性不良（非偶发性）
- 客户投诉/退货的正式质量问题处理
- 内部审核或第三方审核发现的不合格项
- 重复发生的慢性质量问题
- 产线停机相关的质量异常

## 执行步骤

### D0：问题感知与评估
1. 接收质量异常信号（检验不合格/产线停线/客诉/SPC失控）
2. 快速评估问题严重性：
   - 安全影响（是否涉及人身安全） → 最高优先级
   - 功能影响（产品是否可正常使用）→ 高优先级
   - 批量影响（影响范围多大）→ 评估追溯范围
   - 客户影响（是否已流出到客户）→ 决定响应速度
3. 确定是否需要启动8D（判断标准：非偶发性、原因未知、需跨部门协作）
4. **时限要求：问题确认后48小时内必须启动8D**

### D1：组建团队
1. 确定团队领导（通常为质量工程师）
2. 按问题类型选择团队成员：
   - 工艺问题：工艺工程师 + 操作技术骨干
   - 设备问题：设备工程师 + 维修技术员
   - 来料问题：SQE + 供应商代表
   - 设计问题：研发工程师 + 测试工程师
3. 确保团队具备问题解决所需的知识、权限和时间
4. 明确各成员职责和时间投入承诺

### D2：问题描述
1. 运用5W2H完整描述问题：
   - What：什么缺陷/什么规格偏离
   - Where：在哪个工序/产品/位置发现
   - When：什么时间首次/多次发生
   - Who：谁发现/影响谁
   - Why：为什么是问题（对照标准）
   - How：如何表现（缺陷形态）
   - How much：影响数量/比例/金额
2. 用Is/Is Not分析缩小问题边界
3. 附上不良品照片/数据/样品信息

### D3：临时遏制措施（ICA）
1. **时限要求：4小时内到位**
2. 典型遏制措施：
   - 可疑批次隔离标识
   - 加严检验/100%全检
   - 已发货产品追溯与通知
   - 切换合格物料/备用设备
3. 验证遏制措施有效性（实施后不良是否停止流出）
4. 评估遏制措施对产能/交期的影响

### D4：根因分析
1. 使用至少2种分析工具交叉验证：
   - **5Why分析**：逐层追问"为什么"至少到第3层
   - **鱼骨图**：从人/机/料/法/环/测六方面展开
   - **FTA故障树**：自顶向下逻辑分解
   - **DOE实验验证**：对可疑因素进行实验确认
2. 区分直接原因、根本原因和系统原因
3. 验证根因：能否解释问题的Is和Is Not？
4. **关联历史案例**：查询近1年同类问题的8D记录，加速定位

### D5：永久纠正措施（PCA）
1. 针对D4识别的根因制定纠正措施
2. 措施类型：
   - 工艺参数调整（附新参数范围和验证标准）
   - 设备改造/维修（附改造方案和验收标准）
   - 检验方案升级（附新检验标准）
   - 设计变更（附ECN编号）
   - 供应商整改（附SCAR编号）
3. 每项措施必须有：责任人、完成时限、量化验证标准
4. 评估措施的副作用和风险

### D6：实施与验证
1. 按计划实施永久纠正措施
2. 验证措施有效性：
   - 短期验证：实施后连续5批/7天无同类不良
   - 中期验证：1个月内不良率降至目标值
   - 长期验证：3个月内无同类问题复发
3. 确认遏制措施可以撤除
4. 更新相关文件（作业指导书/检验标准/FMEA等）

### D7：预防再发
1. 将改善措施横展到类似产品/工序/供应商
2. 更新FMEA风险评估（降低RPN值）
3. 更新控制计划和检验规范
4. 制定培训计划（如涉及人员操作问题）
5. 评估监控机制是否需要加强

### D8：团队表彰与关闭
1. 总结经验教训（Lessons Learned）
2. 将解决方案纳入知识库
3. 正式关闭8D报告
4. **时限要求：从启动到关闭不超过30天**

## 输入规格
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| trigger_source | enum | 是 | IQC/IPQC/OQC/CUSTOMER/LINE_STOP |
| problem_description | string | 是 | 问题初始描述 |
| affected_product | string | 是 | 受影响产品型号 |
| affected_batch | array | 是 | 受影响批次号列表 |
| defect_quantity | integer | 是 | 不良品数量 |
| severity | enum | 是 | CRITICAL/MAJOR/MINOR |
| customer_impact | boolean | 是 | 是否已影响客户 |
| discovery_date | datetime | 是 | 问题发现日期 |

## 输出规格
| 字段 | 类型 | 说明 |
|------|------|------|
| eight_d_id | string | 8D报告编号 |
| status | enum | D0~D8当前阶段 |
| team_members | array | 团队成员列表 |
| ica_measures | array | 临时遏制措施 |
| root_causes | array | 根本原因列表 |
| pca_measures | array | 永久纠正措施（含责任人/时限） |
| verification_results | object | 验证结果 |
| preventive_actions | array | 预防措施 |
| timeline | object | 各阶段完成时间 |
| closure_date | datetime | 关闭日期 |

## 最佳实践
1. **48-4-30规则**：48小时启动、4小时遏制到位、30天关闭
2. **客诉快速通道**：客户投诉类问题24小时内完成D1-D3，并向客户反馈初步分析
3. **避免伪根因**：如果根因可以用"人员疏忽/操作不当"概括，说明分析不够深入，应继续追问为什么会疏忽
4. **横展思维**：解决一个问题时思考类似问题在其他产品/工序/供应商是否存在
5. **知识积累**：每个关闭的8D都要形成标准化的经验教训并入库

## 约束条件
- 不简化跳过任何D步骤，每个步骤都必须有实质内容
- 根因分析不接受"操作员未遵守规程"作为最终根因，必须追问为什么未遵守
- 30天未关闭自动升级到质量经理审核
- D6验证必须基于数据，不接受主观判断"已改善"
- 安全相关问题的8D不设30天上限，但必须优先关闭
