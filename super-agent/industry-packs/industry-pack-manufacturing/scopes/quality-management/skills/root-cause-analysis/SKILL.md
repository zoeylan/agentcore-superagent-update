# 根因分析 (Root Cause Analysis)

## 概述
本技能专注于运用多种结构化分析工具深入挖掘质量问题的根本原因，区分表面现象、直接原因和系统性根因。核心目标是找到真正的根本原因，使纠正措施能够彻底消除问题而非仅仅抑制症状。

## 适用场景
- 8D流程中的D4阶段（根因分析）
- 产线异常原因调查
- 客户退货/投诉的原因分析
- SPC失控原因排查
- 设备故障后的5Why分析
- 反复发生问题的深度调查

## 执行步骤

### 步骤1：问题边界界定（Is/Is Not分析）
1. 明确"是什么"和"不是什么"：
   | 维度 | IS (是) | IS NOT (不是) | 差异 |
   |------|---------|---------------|------|
   | What | 什么产品有问题 | 什么产品没问题 | 两者差异 |
   | Where | 哪里发现 | 哪里没有 | 位置差异 |
   | When | 什么时候开始 | 之前是否有 | 时间差异 |
   | Extent | 影响多大范围 | 不受影响范围 | 程度差异 |
2. 从差异分析中寻找变化点（什么发生了变化？）
3. 变化点即为根因调查的重点方向

### 步骤2：5Why分析（纵向深挖）
1. 从问题现象开始，逐层追问"为什么"
2. 每一层"为什么"必须有事实或数据支撑
3. 至少追问到第3层，通常需要5层
4. 判断到达根因的标准：
   - 再问"为什么"只能得到不可控的答案（如物理定律）
   - 答案指向可以被消除或改变的系统/流程缺陷
5. 常见层次结构：
   - 第1层Why：直接原因（现象层面）
   - 第2-3层Why：过程原因（流程/方法层面）
   - 第4-5层Why：系统原因（管理/机制层面）

### 步骤3：鱼骨图分析（横向展开）
1. 围绕6M维度展开可能原因：
   - **Man（人）**：技能水平、培训状况、疲劳度、操作规范性
   - **Machine（机）**：设备状态、精度保持性、维护记录、工装夹具
   - **Material（料）**：来料质量、物料批次、存储条件、替代物料
   - **Method（法）**：工艺参数、作业标准、检验方法、程序文件
   - **Measurement（测）**：测量设备、校准状态、测量方法、标准件
   - **Milieu（环）**：温湿度、洁净度、照明、振动
2. 每个维度至少列出3个可能原因
3. 对每个可能原因进行验证排查（有数据支持/可排除）
4. 标记已验证的根因和已排除的非根因

### 步骤4：FTA故障树分析（逻辑分解）
1. 将顶事件（质量问题）进行逻辑分解
2. 使用AND门（所有条件同时满足才发生）和OR门（任一条件满足即发生）
3. 逐层分解直到基本事件（可直接验证的单一原因）
4. 对每个基本事件计算/估计发生概率
5. 通过最小割集找到关键路径

### 步骤5：根因验证
1. 确认根因能够解释：
   - 为什么问题在"IS"条件下发生
   - 为什么问题在"IS NOT"条件下不发生
   - 为什么问题在特定时间点开始发生
2. 如可能，通过实验或数据回归验证因果关系
3. 多个可能根因时，按贡献度排序

### 步骤6：关联历史案例
1. 在知识库中搜索类似问题的历史8D记录
2. 匹配条件：同一物料/同一工序/同一设备/同一供应商/同一缺陷类型
3. 对比历史根因是否适用于当前问题
4. 评估历史纠正措施是否已失效或未横展

## 输入规格
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| problem_statement | string | 是 | 问题描述（5W2H格式） |
| is_is_not | object | 否 | Is/Is Not分析数据 |
| defect_samples | array | 否 | 不良品信息/图片 |
| process_data | object | 否 | 相关工艺参数数据 |
| equipment_logs | array | 否 | 设备运行日志 |
| change_history | array | 否 | 近期变更记录（4M变更） |
| historical_cases | array | 否 | 系统检索的历史类似案例 |

## 输出规格
| 字段 | 类型 | 说明 |
|------|------|------|
| analysis_tools_used | array | 使用的分析工具列表 |
| is_is_not_result | object | Is/Is Not分析结果 |
| five_why_chain | array | 5Why分析链 |
| fishbone_diagram | object | 鱼骨图各分支分析 |
| verified_root_causes | array | 已验证的根本原因 |
| eliminated_causes | array | 已排除的可能原因（含排除依据） |
| confidence_level | enum | HIGH/MEDIUM/LOW |
| evidence_summary | string | 证据链总结 |
| related_cases | array | 相关历史案例及参考价值 |

## 最佳实践
1. **多工具交叉验证**：至少使用2种工具，一个根因如果只有一种工具指向它则置信度不足
2. **数据驱动**：避免依赖经验直觉，每个推论都应有数据佐证
3. **追问"为什么系统允许这种情况发生"**：超越个人行为层面，找到系统性漏洞
4. **警惕确认偏误**：不要先入为主地验证预设答案，要客观排查所有可能
5. **区分多因一果**：有时问题由多个原因共同导致，不要满足于找到一个就停止

## 约束条件
- 不接受"人为疏忽"、"操作失误"作为最终根因，必须继续追问为什么会疏忽/失误
- 不接受没有数据支撑的根因结论
- 分析过程必须记录所有被考虑和被排除的可能原因，形成完整的排查逻辑链
- 如果根因分析无法在5个工作日内完成，必须向8D团队领导报告并说明阻碍因素
