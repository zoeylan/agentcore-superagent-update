# 改进措施进度跟踪技能

## 概述
本技能负责对所有活跃的改进措施（主要来源于8D流程输出的纠正措施和预防措施）进行系统化的进度跟踪管理。确保每一项改善措施都有明确的责任人、计划时间和可衡量的完成标准，并通过定期进度检查和逾期升级机制，驱动改善措施按时落地。

## 适用场景
- 8D改进措施确定后的任务分配与跟踪
- 日常进度检查（每日自动执行）
- 逾期任务的提醒与升级处理
- 改进措施完成后的归档与移交（移交给效果验证环节）

## 执行步骤

### 步骤1：改进任务登记
接收新的改进措施信息并创建跟踪记录：
- 改进措施描述（具体、可执行、可验证）
- 来源8D工单编号
- 对应的缺陷模式/客户投诉
- 责任人（主责+协助）
- 计划开始日期
- 计划完成日期
- 关键里程碑（拆分为3-5个可检查节点）
- 完成标准（量化指标）
- 所需资源（设备/物料/预算/外部支持）

### 步骤2：日常进度监控
每日自动执行进度检查：
- 扫描所有"进行中"状态的改进任务
- 检查是否有新的进度更新记录
- 对比当前进度与计划进度：
  - 提前或按时 → 标记为"正常"
  - 延迟1-7天 → 标记为"轻度延迟"，发送提醒给责任人
  - 延迟7-14天 → 标记为"严重延迟"，发送预警给责任人和其上级
  - 延迟>14天 → 标记为"逾期升级"，正式升级到部门经理

### 步骤3：逾期升级处理
对逾期任务执行升级流程：
- **7天逾期提醒**：
  - 向责任人发送正式提醒（含任务详情/当前状态/原计划日期）
  - 要求24小时内更新进度或提出延期申请
- **14天逾期升级**：
  - 向部门经理发送升级通知
  - 要求48小时内召开专项沟通会
  - 在改进看板上标红警示
- **21天逾期再升级**：
  - 向质量总监/分管副总报告
  - 纳入月度管理评审重点关注事项

### 步骤4：完成确认与移交
当责任人提交完成报告时：
- 验证完成标准是否满足（基于登记时的量化指标）
- 确认证据材料完整（变更记录/验证数据/标准化文件）
- 标记任务为"待验证"状态
- 创建效果验证任务，设定3个月观察期
- 记录实际完成日期，计算改进周期

### 步骤5：看板数据更新
实时维护改进措施看板数据：
- 任务状态分布（未开始/进行中/待验证/已关闭/逾期）
- 本月新增/关闭数量
- 按时关闭率（目标>=85%）
- 平均改进周期（目标<=45天）
- 按责任部门分布的任务统计

## 输入规格
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| improvement_task | Object | 是（新建时）| 改进措施详细信息 |
| progress_update | Object | 是（更新时）| 进度更新内容 |
| completion_report | Object | 是（关闭时）| 完成报告和证据 |

## 输出规格
| 字段 | 类型 | 说明 |
|------|------|------|
| task_status | Object | 任务当前状态 |
| alerts | Array | 产生的提醒/预警/升级通知 |
| dashboard_data | Object | 看板统计数据 |
| handover_to_verification | Object | 移交效果验证的信息包 |

## 最佳实践
- 改进措施描述越具体，跟踪越容易——拒绝接受"加强管理"这类模糊描述
- 里程碑设置遵循SMART原则（Specific/Measurable/Achievable/Relevant/Time-bound）
- 对于跨部门协作的改善，明确接口人和交付物
- 逾期升级必须严格执行，不能因为"快完成了"而通融——纪律性是改善文化的基础
- 关闭任务时要求提供变更前后的对比数据，而非仅提交一句"已完成"

## 约束条件
- 改善措施按时关闭率目标>=85%
- 反馈到改进落地周期目标<=45天
- 逾期升级规则不可由跟踪Agent自行豁免，只能由管理层书面延期
- 任务一旦创建不可删除，只能关闭（关闭原因：完成/合并/取消-需审批）
- 进度更新频率要求：至少每周一次
