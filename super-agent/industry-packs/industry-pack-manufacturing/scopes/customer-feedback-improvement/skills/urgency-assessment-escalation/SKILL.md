# 紧急程度评估与升级技能

## 概述
本技能负责对客户反馈进行紧急程度评估，并按照预设规则触发相应的升级流程。在制造业场景中，紧急升级主要针对两类情况：安全类缺陷（涉及人身伤害风险，可能触发产品召回）和批量性问题（同批次多客户反馈，可能影响大量已交付产品）。准确及时的紧急评估直接关系到企业的产品责任风险和客户关系。

## 适用场景
- 新分类的产品缺陷类反馈需要紧急程度判定
- 多条反馈聚合后触发批量性问题告警
- 安全类缺陷的即时升级处理
- 升级后的跟踪确认（确保责任人已接收并响应）

## 执行步骤

### 步骤1：紧急程度初步评估
基于以下维度进行打分（各维度加权求和）：

**安全影响（权重40%）：**
- 0分：无安全影响
- 50分：潜在安全风险（需特定条件触发）
- 100分：直接安全威胁（已发生人身伤害或明确的伤害风险）

**影响范围（权重30%）：**
- 0分：个别产品/偶发
- 50分：多台产品/间歇发生
- 100分：批量性（同批次多客户报告/批量失效）

**业务影响（权重20%）：**
- 0分：轻微不便
- 50分：功能受限/客户产线受影响
- 100分：客户停产/质量索赔/合同违约风险

**时间紧迫性（权重10%）：**
- 0分：无时间压力
- 50分：客户要求一周内回复
- 100分：客户要求立即处理/已升级到高层

### 步骤2：紧急等级判定
基于评估得分划分紧急等级：
- **P1-紧急（>=80分）**：安全类缺陷或重大批量性问题 → 1小时内升级
- **P2-高（60-79分）**：批量性问题或重要客户重大投诉 → 4小时内升级
- **P3-中（40-59分）**：一般产品缺陷需关注 → 24小时内纳入分析
- **P4-低（<40分）**：轻微问题或使用建议 → 常规流程处理

### 步骤3：升级执行
按紧急等级执行对应升级路径：

**P1升级路径：**
1. 即时通知质量总监（电话+系统消息）
2. 同步通知质量管理Scope质量异常处理专家
3. 创建紧急事件工单（含所有已知信息）
4. 启动产品召回评估预流程
5. 冻结同批次在途/在库产品（如适用）

**P2升级路径：**
1. 通知质量经理（系统消息）
2. 推送给缺陷模式分析Agent优先处理
3. 24小时内启动调查
4. 评估是否需要批量追溯

**P3/P4路径：**
- 纳入常规缺陷分析流程

### 步骤4：升级确认跟踪
- P1：5分钟内必须有接收确认，否则自动升级到更高层级
- P2：30分钟内必须有接收确认
- 记录升级时间、接收确认时间、首次响应时间
- 生成升级处理时效报告

## 输入规格
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| feedback_record | Object | 是 | 已结构化的反馈记录 |
| classification | Object | 是 | 分类结果（必须为产品缺陷类）|
| historical_context | Object | 否 | 同型号/同批次历史反馈汇总 |
| customer_tier | String | 否 | 客户等级（战略客户加权处理）|

## 输出规格
| 字段 | 类型 | 说明 |
|------|------|------|
| urgency_score | Number | 综合紧急度评分（0-100）|
| urgency_level | String | P1/P2/P3/P4 |
| escalation_actions | Array | 已执行的升级动作列表 |
| escalation_status | Object | 升级状态（已通知/已确认/响应中）|
| rationale | String | 紧急度判定的理由说明 |

## 最佳实践
- 对于战略客户（大客户/OEM合作伙伴），紧急评分自动+10分
- 批量性问题的判定需要关联历史反馈，同一批次号出现3+条反馈即视为批量性
- 安全类缺陷宁可误报不可漏报——误报造成的成本远低于漏报的后果
- 建立升级通讯录的冗余机制（主要联系人+备份联系人）
- 所有P1事件必须有完整的事件时间线记录（for 后续复盘和法律合规）

## 约束条件
- 安全类缺陷（P1）升级时效必须<=1小时，0容忍延迟
- 升级判定必须有明确的数据依据，不能仅凭"感觉"
- 升级通知必须包含足够的上下文信息，让接收人能快速理解问题
- 不得自行做出"无需升级"的判定来覆盖安全类缺陷标记——这类判定只能由质量总监做出
