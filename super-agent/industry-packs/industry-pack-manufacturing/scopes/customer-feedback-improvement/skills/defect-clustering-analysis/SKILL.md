# 缺陷聚类分析技能

## 概述
本技能对累积的缺陷反馈数据执行系统性聚类分析，识别高频缺陷模式和新出现的缺陷趋势。通过文本语义聚类、时间序列分析和多维度交叉分析，将离散的客户投诉数据汇聚为有业务意义的缺陷模式，为后续的根因分析和改善决策提供方向指引。

## 适用场景
- 每周定期执行缺陷模式聚类分析
- 缺陷数据突然增加时的即时分析
- 新产品上市后的前3个月密集监控分析
- 改善措施落地后的效果验证（缺陷模式是否消失）

## 执行步骤

### 步骤1：数据准备
- 从反馈数据库中提取分析时间窗口内的产品缺陷类反馈
- 默认时间窗口：近4周（滚动）
- 数据清洗：排除重复记录、信息不完整记录（标记为待补全的除外）
- 统计本期数据量，与历史同期对比

### 步骤2：文本语义聚类
- 对缺陷描述文本进行向量化（TF-IDF/Embedding）
- 执行聚类算法（DBSCAN优先，可适应不规则形状的簇）
- 参数调优：基于silhouette score评估聚类质量
- 对每个聚类簇生成代表性标签（最频繁的关键词组合）
- 识别孤立点（不属于任何簇的异常反馈）→ 可能是新缺陷类型

### 步骤3：多维度交叉分析
对每个缺陷簇进行多维度特征分析：
- **产品维度**：哪些型号受影响？是通用问题还是特定型号？
- **时间维度**：趋势如何？（新增/上升/稳定/下降）
- **客户维度**：影响了多少客户？是否集中在特定地区/行业？
- **严重程度维度**：该缺陷模式的平均严重度？
- **频率维度**：月投诉次数是否达到高频阈值（>=5次/月）？

### 步骤4：TOP5排序与趋势判定
- 按影响度（=频率×严重度×影响客户数）综合排序
- 取TOP5高影响缺陷模式
- 对每个模式判定趋势状态：
  - **新增**：近2周首次出现且已有3+条
  - **恶化**：本周数量较上周增长>50%
  - **稳定**：变化在±20%以内
  - **改善**：本周数量较上周下降>30%

### 步骤5：报告生成
生成标准化的周度缺陷模式报告：
- 摘要：本周新增反馈总量、聚类模式数量、TOP5缺陷概览
- 详情：每个TOP5缺陷的完整分析卡片
- 趋势图：缺陷数量时间序列图
- 新发现：新出现的缺陷类型或异常信号
- 建议：需要重点关注或触发改进的缺陷

## 输入规格
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| feedback_data | Array | 是 | 分析时间窗口内的缺陷反馈记录 |
| analysis_window | Object | 否 | 分析时间窗口配置（默认4周滚动）|
| previous_clusters | Array | 否 | 上期聚类结果（用于趋势对比）|
| threshold_config | Object | 否 | 高频阈值配置（默认5次/月）|

## 输出规格
| 字段 | 类型 | 说明 |
|------|------|------|
| clusters | Array | 所有缺陷聚类簇（含标签/数量/特征）|
| top5_report | Object | TOP5缺陷模式详细报告 |
| trend_analysis | Object | 趋势分析结果 |
| new_patterns | Array | 新发现的缺陷模式 |
| trigger_alerts | Array | 达到阈值需触发8D的缺陷 |

## 最佳实践
- 聚类结果需要与上期做对比，观察缺陷模式的演变而非每次从零开始
- 对"新出现的缺陷类型"设置更低的关注阈值（3次即关注），因为可能是严重问题的早期信号
- 聚类标签应使用业务人员能理解的术语，而非技术性描述
- 保留聚类过程的中间数据，便于质疑时回溯验证
- 季节性因素需要考虑（如温度敏感产品在夏季可能缺陷率上升属正常）

## 约束条件
- 每周报告必须在周一上午9:00前生成完毕
- 聚类结果最小簇大小=3（少于3条不构成模式）
- 趋势判定需基于至少3个数据点（3周以上数据）
- 分析结论必须标注置信度和数据支撑量级
