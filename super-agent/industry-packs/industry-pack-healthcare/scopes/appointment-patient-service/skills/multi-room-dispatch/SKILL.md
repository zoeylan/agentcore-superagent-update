# 多诊室分流 (Multi-Room Dispatch)

## 概述
在同一科室开设多个诊室的场景下，智能调配患者分布，平衡各诊室的候诊负荷，减少整体等候时间，避免出现一个诊室排长队而另一个诊室空闲的情况。

## 适用场景
- 同科室多诊室的日常负载均衡
- 某诊室医生暂离时的患者分流
- 新增诊室开放时的存量患者重新分配
- 高峰时段的弹性诊室启用和分流

## 执行步骤

### 1. 诊室状态监控
- 实时获取同科室所有诊室的当前状态
- 监控指标：候诊人数、预计清空时间、医生当前进度
- 识别负载不均衡信号：某诊室候诊人数超过其他诊室50%以上

### 2. 分流策略制定
- 计算各诊室的标准化负载指数：候诊人数/医生接诊速度
- 确定分流方向：从高负载诊室向低负载诊室分流
- 评估分流收益：预计可缩短患者等候时间
- 确认分流可行性：目标诊室医生是否接受、患者是否同意

### 3. 分流对象选择
- 优先选择尚未到院或刚签到的患者（影响最小）
- 排除已等候超过30分钟的患者（转移增加不确定性）
- 排除指名预约特定医生的患者（除非患者同意）
- 按预约时间倒序选择（晚时段患者优先被分流到快队列）

### 4. 执行分流
- 向目标患者推送分流建议（含原因和预计等候时间对比）
- 患者确认后更新队列位置
- 不强制分流，尊重患者选择（提供"继续等候原诊室"选项）
- 更新两个诊室的排队序号和预计时间

### 5. 效果评估
- 记录分流前后各诊室的等候时间变化
- 统计患者接受分流建议的比例
- 评估分流是否达到预期效果（等候时间缩短≥20%才视为有效）
- 无效分流策略加入负面反馈用于策略优化

## 输入规范
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| departmentCode | string | 是 | 科室代码 |
| rooms | array | 是 | 该科室所有诊室状态列表 |
| threshold | number | 否 | 触发分流的不均衡阈值（默认50%） |
| dispatchMode | enum | 否 | auto/suggest/manual |

## 输出规范
| 字段 | 类型 | 说明 |
|------|------|------|
| dispatchPlan | array | 分流计划（患者ID、原诊室、目标诊室） |
| expectedBenefit | object | 预期收益（各诊室等候时间变化） |
| patientConsent | array | 患者确认状态 |
| actualResult | object | 实际效果（分流执行后统计） |

## 最佳实践
- 分流建议强调对患者的好处（"切换到X诊室可节省约15分钟"）
- 对不愿更换医生的患者不强制分流
- 高峰时段可考虑启动弹性诊室并分流部分患者
- 医生能力差异（主任vs主治）可能影响患者分流意愿，需考虑

## 约束条件
- 分流必须经患者同意，不得强制执行
- 已等候超过30分钟的患者原则上不再分流
- 指定医生预约的患者需特别确认后才可分流
- 分流后的队列位置必须公平（不得让分流患者插队）
- 分流操作记录完整日志
