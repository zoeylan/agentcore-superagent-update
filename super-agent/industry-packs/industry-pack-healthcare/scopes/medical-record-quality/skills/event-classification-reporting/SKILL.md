# 不良事件分级与上报

## 概述
本技能负责对识别到的不良事件进行标准化分级，并按照不同级别的时限要求完成规范化上报。遵循WHO患者安全分类框架和国家卫健委相关规定，确保事件信息完整、分级准确、上报及时。

## 适用场景
- 主动监测系统检测到的潜在不良事件分级
- 医护人员主动上报事件的标准化处理
- 警告事件的即时上报和应急响应启动
- 不良事件上报表的结构化填写辅助
- 上报时限监控和超时提醒

## 执行步骤

### 步骤1：事件分级判定
根据事件后果和性质进行三级分类：

**警告事件（Sentinel Event）- I级**
- 定义：导致患者死亡或永久性功能丧失的非预期事件
- 典型场景：手术部位/患者错误、院内非预期死亡、异物遗留体内、输错血型
- 上报时限：即时（发现后立即）
- 通知范围：医务部主任→分管院长→质控委员会

**不良事件（Adverse Event）- II级**
- 定义：造成患者临时性伤害，需要额外治疗或延长住院
- 典型场景：用药错误致不良反应、手术并发症（非预期）、跌倒致骨折、输血反应
- 上报时限：24小时内
- 通知范围：科室主任→医务部

**无伤害事件（Near Miss）- III级**
- 定义：事件已发生但未造成患者伤害，或在造成伤害前被及时发现和纠正
- 典型场景：处方差错被药师拦截、给药前发现患者识别错误、标本贴错标签被发现
- 上报时限：72小时内
- 通知范围：科室内部

### 步骤2：标准化上报表填写
上报信息包含：
- 基本信息：事件发生时间、地点、科室、涉及人员（匿名化）
- 患者信息：年龄、性别、诊断（脱敏处理）
- 事件描述：发生经过的客观描述（5W1H）
- 事件分类：按WHO分类（药物事件/手术事件/跌倒/感染/设备相关等）
- 后果评估：对患者的实际影响程度
- 即时处理：已采取的补救措施
- 初步原因：事件发生的直接原因

### 步骤3：上报流程执行
- I级（警告事件）：系统自动触发即时通知链（电话+短信+系统消息），同时通知医务部主任和值班院领导
- II级（不良事件）：生成标准上报表推送至责任科室，科室在24小时内完成填写提交
- III级（无伤害事件）：简化上报流程，提供快速上报入口，72小时内完成

### 步骤4：上报时限监控
- 对已识别但未完成上报的事件进行倒计时跟踪
- 距时限前4小时发送提醒
- 超时未上报自动升级通知科室主任
- 记录上报及时率统计数据

## 输入规格
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| event_source | enum | 是 | 事件来源（主动监测/被动上报） |
| event_description | string | 是 | 事件描述 |
| patient_outcome | enum | 是 | 患者结局（死亡/永久伤害/临时伤害/无伤害） |
| event_category | enum | 是 | 事件类别 |
| department_id | string | 是 | 涉及科室 |
| reporter_id | string | 否 | 上报人（被动上报时提供） |

## 输出规格
| 字段 | 类型 | 说明 |
|------|------|------|
| event_level | enum | I级/II级/III级 |
| report_deadline | datetime | 上报截止时间 |
| notification_targets | array | 通知对象列表 |
| report_form | object | 标准化上报表（含已自动填充字段） |
| reporting_status | enum | 已上报/待上报/超时 |
| follow_up_required | boolean | 是否需要后续跟踪 |

## 最佳实践
- 上报流程设计应尽量简化，降低上报门槛（目标：3分钟内完成基本上报）
- 强调非惩罚性文化，上报界面明确提示"上报不追责"
- 匿名上报通道应独立于实名通道，保护上报人
- 上报率季度环比增长视为积极信号（说明上报文化在建设中）
- 对高频上报的科室给予正面反馈而非质疑

## 约束条件
- 上报人身份信息严格保密，仅质控管理员可查看
- 事件信息不得用于对当事人的行政处罚
- 涉及患者死亡的事件必须同步进行死亡讨论
- 上报数据的存储和传输应符合医疗信息安全规范
