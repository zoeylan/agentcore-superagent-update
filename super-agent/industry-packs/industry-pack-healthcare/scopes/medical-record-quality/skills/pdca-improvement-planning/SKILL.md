# PDCA持续改进规划

## 概述
本技能基于PDCA（Plan-Do-Check-Act）方法论，将质量监测中发现的趋势性问题转化为结构化的改进方案。从问题识别到方案设计、执行跟踪、效果验证形成完整闭环，是医疗质量持续改进的核心方法论工具。

## 适用场景
- 质量指标连续偏离阈值，被判定为趋势性问题
- 月度质量分析会议确定的重点改进项目
- 等级评审中发现的系统性缺陷改进
- 不良事件RCA分析后的系统性整改
- 新制度/流程推行后的效果优化

## 执行步骤

### 步骤1：Plan（计划）- 问题定义与方案设计
- **问题描述**：用数据明确问题的范围、程度和影响（如"外科病历时限达标率从95%下降至87%，持续4周"）
- **目标设定**：遵循SMART原则（具体目标值、达成时限、可衡量标准）
- **根因分析**：引用趋势分析的根因假设，确定关键因素
- **对策制定**：针对每个根因制定具体措施
  - 短期措施（1-2周内可落地）
  - 中期措施（1-3月内完成）
  - 长期措施（3-6月，涉及制度/系统变更）
- **责任分配**：每项措施明确负责人、协作人、完成时限
- **资源需求**：人力、系统支持、培训等需求清单

### 步骤2：Do（执行）- 方案实施跟踪
- 创建改进项目任务看板
- 跟踪各措施的执行进度（未开始/进行中/已完成/延期）
- 记录执行过程中的困难和调整
- 阶段性检查点（如两周一次进展汇报）

### 步骤3：Check（检查）- 效果验证
- 设定观察期（通常4-8周）
- 对比改进前后的指标变化
- 评估是否达到预设目标
- 识别残留问题和新发现问题
- 统计分析确认改善的显著性

### 步骤4：Act（处理）- 标准化或调整
- **达标**：将有效措施纳入标准流程/制度
- **部分达标**：分析未达标部分原因，进入下一个PDCA循环
- **未达标**：重新审视根因分析，调整改进策略
- 总结经验教训，形成知识库沉淀

## 输入规格
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| problem_description | string | 是 | 问题描述（含数据） |
| root_causes | array | 是 | 根因分析结果 |
| target_indicator | string | 是 | 目标指标 |
| target_value | float | 是 | 目标值 |
| current_value | float | 是 | 当前值 |
| deadline | date | 是 | 预期达成日期 |

## 输出规格
| 字段 | 类型 | 说明 |
|------|------|------|
| pdca_plan | object | 完整PDCA方案 |
| pdca_plan.measures | array | 改进措施列表 |
| pdca_plan.measures[].action | string | 具体行动 |
| pdca_plan.measures[].owner | string | 责任人/部门 |
| pdca_plan.measures[].deadline | date | 完成时限 |
| pdca_plan.measures[].priority | enum | 高/中/低 |
| pdca_plan.checkpoints | array | 检查点日程 |
| pdca_plan.success_criteria | object | 成功标准 |
| tracking_status | object | 当前执行状态 |

## 最佳实践
- 每个PDCA周期聚焦1-3个核心问题，避免目标过多导致资源分散
- 改进措施应从"系统层面"思考，避免仅归因于"个人不努力"
- 设定合理的改善幅度预期（通常首轮PDCA改善30-50%即为成功）
- PDCA方案应获得科室/医务部的正式审批后再执行
- 建立改进案例库，将成功经验在全院推广

## 约束条件
- PDCA方案的最终审批权在医务部/质控委员会，系统仅提供方案建议
- 涉及制度变更的措施需走正式审批流程
- 改进措施不得增加一线医护人员不合理的工作负担
- 效果验证需要足够的观察期，不得过早宣布成功
