# 根因分析（RCA）

## 概述
本技能运用系统化的根因分析方法（Root Cause Analysis），对已发生的不良事件进行深度分析，识别事件背后的系统性根本原因而非表面直接原因。RCA是患者安全改进的核心方法论，通过对系统缺陷的识别推动从根本上预防同类事件再次发生。

## 适用场景
- 所有警告事件（I级）的强制性RCA分析
- 重复发生的同类不良事件（II级）
- 涉及多环节的复杂不良事件
- 医务部指定的专项分析
- 科室层面的事件复盘学习

## 执行步骤

### 步骤1：组建RCA团队与信息收集
- 确定RCA负责人（通常为医务部指定的质控专家）
- 组建多学科团队（临床医生、护士、药师、管理人员等相关方）
- 收集事件相关信息：
  - 当事人访谈记录（非评判性访谈）
  - 相关病历和医嘱记录
  - 护理记录和交班记录
  - 相关制度和流程文件
  - 设备/环境信息
  - 类似既往事件资料

### 步骤2：事件时间线重建
- 按时间顺序还原事件全过程
- 标注关键节点（决策点、失误点、救济点）
- 识别事件链中的每个环节
- 区分事实与推测，标注信息来源

### 步骤3：根因分析（鱼骨图+5-Why）

**鱼骨图分析维度：**
- **人员因素**：知识/技能不足、疲劳、沟通失误、工作负荷
- **流程因素**：制度缺失/不明确、流程设计缺陷、执行偏差
- **设备因素**：设备故障、设计缺陷、维护不当
- **环境因素**：空间布局不合理、噪音/干扰、人员配比不足
- **制度因素**：培训不足、监督机制缺失、考核激励偏差

**5-Why深挖示例：**
- Why1: 为什么给错药？→ 护士拿错了药物
- Why2: 为什么拿错？→ 两种药物外观包装相似且相邻存放
- Why3: 为什么相邻存放？→ 药房按字母排序而非风险等级
- Why4: 为什么按字母排序？→ 没有高警示药物分开管理的制度
- Why5: 为什么没有制度？→ 管理层未意识到LASA药物混淆风险
→ **根因：高警示药物管理制度缺失**

### 步骤4：生成RCA报告
- 事件概述和时间线
- 根因分析过程和结论
- 根因分类（系统因素 vs 个人因素，强调系统改进）
- 整改建议（针对每个根因）
- 预防措施建议
- 类似风险排查建议

## 输入规格
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| event_id | string | 是 | 不良事件ID |
| event_description | string | 是 | 事件详细描述 |
| event_timeline | array | 是 | 事件时间线 |
| involved_parties | array | 否 | 涉及人员（匿名） |
| relevant_records | array | 否 | 相关记录引用 |
| similar_events | array | 否 | 历史类似事件 |

## 输出规格
| 字段 | 类型 | 说明 |
|------|------|------|
| rca_report | object | 完整RCA报告 |
| root_causes | array | 根因列表 |
| root_causes[].category | enum | 人员/流程/设备/环境/制度 |
| root_causes[].description | string | 根因描述 |
| root_causes[].evidence | string | 支持证据 |
| contributing_factors | array | 促成因素 |
| corrective_actions | array | 整改建议 |
| preventive_measures | array | 预防措施 |
| lessons_learned | string | 经验教训总结 |

## 最佳实践
- RCA应聚焦系统缺陷而非个人责任，避免"怪罪文化"
- 5-Why分析至少追问到第3层才可能触及根因
- 每个根因应有对应的整改措施，整改应针对系统层面
- 警告事件的RCA应在72小时内完成初步分析
- RCA结论应在科室范围内分享学习（脱敏后），促进全员安全意识

## 约束条件
- RCA过程和结论的保密性必须保证，不得外泄
- 当事人访谈应在非评判性氛围下进行，保护心理安全
- RCA结论不得作为法律诉讼中的证据使用（遵循相关法规）
- 整改措施应可行性评估后再确认，避免不切实际的要求
- AI辅助的RCA分析仅作为参考，最终结论需团队讨论确认
