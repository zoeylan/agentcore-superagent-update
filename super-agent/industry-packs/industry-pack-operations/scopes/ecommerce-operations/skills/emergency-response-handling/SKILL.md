# 应急响应处理 (Emergency Response Handling)

## 概述
本技能用于处理直播间突发紧急情况，包括技术故障、违规预警、平台处罚风险和异常流量事件。目标是在最短时间内控制损失、恢复直播间正常运营状态，并通知相关人员介入处理。

## 使用场景
- 数字主播技术故障（卡顿/断流/画面异常）
- 平台违规关键词触发预警
- 直播间被限流或面临处罚风险
- 恶意刷屏/攻击导致直播间秩序混乱
- 商品链接异常（库存/价格错误）
- 投放异常（消耗暴增/效果骤降）

## 执行步骤

### Step 1: 事件识别与分级
- **P0级（立即处理，<10秒响应）**：
  - 违规内容播出（可能导致封号）
  - 主播断流/黑屏
  - 商品价格错误（亏本售卖）
- **P1级（5分钟内处理）**：
  - 平台限流预警
  - 恶意刷屏/弹幕攻击
  - 投放消耗异常暴增
- **P2级（15分钟内处理）**：
  - 商品库存不足预警
  - 画面轻微卡顿
  - 非核心功能异常

### Step 2: 即时应急动作

**违规预警应急：**
1. 立即向数字主播发送静音指令
2. 切换画面至安全场景（品牌logo/商品静态展示）
3. 通知人工运营团队介入复核
4. 准备合规话术，等待确认后恢复

**技术故障应急：**
1. 切换至备用主播实例（如有）
2. 如无备用实例，启动备用话术循环（预录内容）
3. 通知技术团队紧急修复
4. 向观众播放"稍等片刻"安抚话术
5. 修复后执行平滑恢复流程

**商品异常应急：**
1. 立即下架问题商品链接
2. 通知数字主播跳过该品
3. 通知店铺运营管家修复商品信息
4. 如已产生错误订单，记录并通知客服处理

**投放异常应急：**
1. ROI暴跌/消耗暴增→立即暂停相关计划
2. 通知流量投放操盘手排查原因
3. 评估是否影响整体预算执行
4. 确认恢复条件后逐步重启

### Step 3: 通知与升级
- P0事件：即时通知运营主管+技术负责人（电话/IM）
- P1事件：5分钟内通知相关责任人（IM消息）
- P2事件：记录事件日志，在下一个数据快照中汇报
- 所有事件记录完整时间线（发现时间→处理动作→恢复时间）

### Step 4: 恢复与复盘
- 确认问题完全解决后才恢复正常流程
- 恢复时采用渐进策略（不要一步到位，先低强度验证）
- 事后输出事件复盘报告（根因/影响/处理时效/改进措施）
- 更新应急预案库

## 输入规格
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| event_type | enum | 是 | 事件类型 |
| severity | enum | 是 | 严重等级（P0/P1/P2） |
| event_detail | object | 是 | 事件详细信息 |
| current_livestream_state | object | 是 | 当前直播状态 |
| available_backup | object | 否 | 可用备用资源 |

## 输出规格
| 字段 | 类型 | 说明 |
|------|------|------|
| emergency_actions | array | 应急动作序列 |
| notifications | array | 通知列表（对象+方式+内容） |
| recovery_plan | object | 恢复计划 |
| estimated_impact | object | 预估影响（时长/GMV损失） |
| incident_log | object | 事件日志记录 |

## 最佳实践
1. **速度第一**：P0事件处理速度>处理完美度，先止损再优化
2. **预案驱动**：所有常见故障应有预设处理流程，减少决策时间
3. **最小影响**：应急处理以最小范围影响直播间体验为原则
4. **通知到位**：确保相关人员第一时间知晓，不做信息孤岛
5. **事后复盘**：每次事件都是优化预案的机会，不放过任何教训

## 约束条件
- P0事件响应时间≤10秒
- 违规内容必须在5秒内静音/切画面
- 不得在未确认安全的情况下自行恢复
- 所有应急操作必须记录完整日志
- 涉及资金损失的事件必须通知管理层
