# 会员分层管理 (Membership Tier Management)

## 技能概述

会员分层管理技能负责维护和运行5级会员分层模型（普通/银卡/金卡/铂金/钻石），实现会员等级的自动计算、变动处理和权益匹配。该技能是会员运营体系的基础能力，所有后续的精细化运营策略（挽留、权益优化、定向营销）都依赖于准确的分层结果。

## 适用场景

- 每周一凌晨例行运行分层模型，批量更新全量会员等级
- 会员行为数据实时变化触发的即时等级重新评估
- 新会员注册后的初始分层定级
- 分层规则调整后的全量重新计算
- 会员等级争议的数据复核

## 执行步骤

### 步骤1：数据采集与预处理
1. 从客户运营域获取最新客户画像数据（基础属性、消费偏好）
2. 采集会员行为数据：近90天登录频率、消费金额、消费频次、权益使用率
3. 采集会员交互数据：客服互动记录、活动参与次数、社交分享行为
4. 数据清洗：处理缺失值（默认填充为该等级均值的50%）、去重、异常值检测
5. 数据标准化：将各维度数据归一化到0-100分范围

### 步骤2：分层模型运算
1. 运行RFM模型：计算最近消费时间(R)、消费频率(F)、消费金额(M)得分
2. 叠加行为特征：权益使用率权重20%、活动参与度权重15%、社交传播力权重10%
3. 计算综合会员价值分(MVP Score)：RFM占55% + 行为特征占45%
4. 按照分层阈值进行等级划分：
   - 钻石：MVP ≥ 90（Top 5%）
   - 铂金：MVP 75-89（Top 15%）
   - 金卡：MVP 55-74（Top 35%）
   - 银卡：MVP 35-54（Top 60%）
   - 普通：MVP < 35（其余）
5. 校验分层覆盖率 = 100%（不允许出现未归类会员）

### 步骤3：等级变动处理
1. 对比上周等级，生成升级/降级/维持清单
2. 升级处理：标记祝贺通知任务、配置新等级权益、记录升级原因
3. 降级处理：检查是否有冻结标记（投诉中的会员不降级）、推送保级任务、设置限时挽留方案
4. 权益变更系统配置需在分层完成后2小时内完成

### 步骤4：质量校验与输出
1. 校验分层结果：等级分布是否符合预设比例（允许±5%浮动）
2. 校验权益发放准确率 = 100%
3. 输出分层报告：等级分布统计、变动明细、趋势分析（至少4周历史对比）
4. 异常标记：等级跳跃2级以上的会员需人工复核

## 输入规格

| 输入项 | 类型 | 说明 |
|--------|------|------|
| 会员基础数据 | 结构化数据 | 会员ID、注册时间、基础属性 |
| 消费行为数据 | 时序数据 | 近90天消费记录（金额、频次、品类） |
| 平台行为数据 | 事件数据 | 登录、浏览、收藏、分享等行为日志 |
| 权益使用记录 | 结构化数据 | 各权益项的领取和使用状态 |
| 客户画像数据 | 外部输入 | 来自客户运营域的画像标签 |

## 输出规格

| 输出项 | 类型 | 说明 |
|--------|------|------|
| 分层结果清单 | 结构化数据 | 全量会员的当前等级和MVP分数 |
| 等级变动明细 | 列表 | 升级/降级会员清单及变动原因 |
| 分层分布报告 | 报告文档 | 各等级人数/占比/趋势 |
| 权益配置指令 | 系统指令 | 需变更权益的会员列表和目标权益包 |
| 异常标记清单 | 列表 | 需人工复核的异常变动记录 |

## 最佳实践

1. **平滑降级机制**：不要一次性降多级，每周最多降1级，给会员保级缓冲期
2. **升级即时性**：升级应实时触发（不等到周批次），提升用户体验
3. **分层阈值动态调整**：每季度根据会员基数变化微调分层阈值，避免某一层级人数过多或过少
4. **新会员保护期**：注册30天内的新会员不参与降级评估
5. **数据回溯窗口**：使用90天滑动窗口避免短期波动导致频繁变动

## 约束条件

- 分层覆盖率必须为100%，不允许任何会员处于"未分类"状态
- 投诉处理中的会员冻结降级操作
- 权益变更必须在分层完成后2小时内配置完毕
- 等级变动需保留完整审计日志
- 分层模型参数调整需经过至少2周A/B测试验证
