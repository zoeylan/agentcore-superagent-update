# 并发执行编排 (Concurrent Execution Orchestration)

## 概述

本技能负责管理多设备并发控制时的执行编排，包括依赖关系分析、执行顺序规划、冲突检测与解决、部分失败时的处理策略（回滚/继续/降级）。在场景控制和批量操作中，一次用户意图可能涉及5-10个设备的协调控制，需要确保执行的正确性和一致性。

### 适用场景
- 场景模式触发（如"回家模式"同时控制灯光/空调/窗帘/音箱）
- 用户发出多设备批量指令
- 自动化规则触发的多设备联动
- 有执行依赖关系的设备操作序列

## 执行步骤

### Step 1: 依赖关系分析
1. 解析意图列表中标注的依赖关系（parallel/sequential/conditional）
2. 补充隐含依赖：
   - 物理约束依赖：开空调前应关窗（防止冷气流失）
   - 时序逻辑依赖：开投影前应先关灯
   - 资源互斥依赖：同一功率回路不能同时开多台大功率设备
3. 构建执行依赖图（DAG）
4. 执行拓扑排序确定执行层级

### Step 2: 冲突检测
1. 扫描意图列表检测以下冲突类型：
   - **直接冲突**：同一设备的矛盾操作（开+关）
   - **语义冲突**：逻辑矛盾的操作组合（开暖气+开窗）
   - **资源冲突**：共享资源的竞争（电力负载超限）
   - **时间冲突**：定时规则与实时指令重叠
2. 冲突解决策略（按优先级）：
   - 安全优先：涉及安全的操作无条件优先
   - 用户实时指令优先于自动化规则
   - 后发指令覆盖先发指令（同一设备）
   - 无法自动解决时请求用户确认

### Step 3: 执行计划生成
1. 将DAG按层级分组：
   - Layer 0：无依赖的操作（可并行执行）
   - Layer 1：依赖Layer 0完成的操作
   - Layer N：依赖Layer N-1的操作
2. 为每层生成并行执行集合
3. 设定全局执行超时（所有层执行完成的最大允许时间）
4. 为每个操作设定独立超时和重试策略

### Step 4: 执行与监控
1. 按层级顺序执行，层内并行下发
2. 实时收集每个操作的执行状态
3. 层内全部完成后检查结果：
   - 全部成功：进入下一层
   - 部分失败：评估是否影响下一层依赖
   - 全部失败：中止执行，进入异常处理
4. 部分失败时的策略选择：
   - **继续模式**（默认）：跳过失败设备，继续执行不依赖它的操作
   - **回滚模式**：撤销已成功的操作，恢复初始状态
   - **降级模式**：选择替代设备或参数完成核心目标

### Step 5: 执行结果汇总
1. 汇总所有操作的执行结果
2. 生成人类可读的执行摘要（成功N个/失败M个/跳过K个）
3. 失败操作标注失败原因和建议措施
4. 整体执行耗时统计

## 输入规范

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| intent_list | array | 是 | 待执行的意图列表 |
| dependency_graph | object | 否 | 预定义的依赖关系 |
| failure_strategy | enum | 否 | continue/rollback/degrade，默认continue |
| global_timeout_ms | integer | 否 | 全局执行超时，默认5000ms |

## 输出规范

| 字段 | 类型 | 说明 |
|------|------|------|
| execution_summary | object | 执行结果汇总 |
| success_count | integer | 成功操作数 |
| failed_count | integer | 失败操作数 |
| total_latency_ms | integer | 总执行耗时 |
| per_device_results | array | 每设备的详细执行结果 |
| rollback_actions | array | 已执行的回滚操作（如有） |

## 最佳实践

1. **快速失败**：单设备超时不应阻塞其他设备的并行执行
2. **幂等设计**：重试和回滚操作必须幂等，避免重复执行副作用
3. **渐进式执行**：大批量操作分批执行（每批≤10设备），避免网关拥塞
4. **状态快照**：执行前对涉及设备做状态快照，用于回滚恢复
5. **用户感知优先**：优先执行用户能直接感知的操作（如灯光），后台操作延后

## 约束条件

- 单批次最大并发设备数：10
- 全局执行超时：5000ms（可配置）
- 单设备指令超时：500ms（含1次重试共1000ms）
- 回滚操作必须在3秒内完成
- 执行过程中如收到用户新指令，优先处理新指令
