# 设备指令路由 (Device Command Routing)

## 概述

本技能负责将标准化的控制意图精确路由到目标设备，通过查询设备注册表确定设备的通信协议和控制接口，并将通用意图参数转换为设备特定的指令格式。这是连接"理解层"和"执行层"的关键桥梁，确保正确的指令以正确的格式到达正确的设备。

### 适用场景
- 将用户意图转换为具体设备的控制指令
- 处理同一品类不同品牌设备的参数映射差异
- 设备不可达时的备选路由选择
- 批量控制指令的分发路由

## 执行步骤

### Step 1: 设备寻址与可达性确认
1. 根据意图中的 `target_device` ID 查询设备注册表
2. 获取设备元信息：品牌/型号/通信协议/网关地址/在线状态
3. 确认设备当前在线状态（调用状态同步监控器实时查询）
4. 设备离线时的处理：
   - 检查是否有备选设备可达（如同房间另一台空调）
   - 无备选则返回设备不可达错误码及最后在线时间
   - 建议用户检查设备电源或网络连接

### Step 2: 能力校验
1. 查询目标设备的能力模型（支持的操作和参数范围）
2. 校验请求的操作是否在设备能力范围内：
   - 操作类型校验：该设备是否支持此操作（如风扇不支持制冷模式）
   - 参数范围校验：参数值是否在允许范围内（如温度16-30°C）
   - 模式兼容校验：当前模式下是否允许此参数调整
3. 参数越界时的处理：自动clip到最近合法值并通知用户

### Step 3: 指令格式转换
1. 根据设备品牌/型号选择对应的指令编码器（Codec）
2. 执行参数映射：
   - 温度：通用26°C → 美的0x1A / 格力编码 / Homekit targetTemperature
   - 模式：通用"cooling" → 品牌特定模式编码
   - 亮度：通用50% → 设备特定的0-255范围值
3. 组装完整的设备指令报文（含校验和/序列号/时间戳）
4. 选择通信通道：WiFi直连 / Zigbee网关转发 / BLE代理 / Matter协议

### Step 4: 指令下发与确认
1. 通过选定的通信通道发送指令报文
2. 启动ACK超时计时器（500ms）
3. 收到ACK：标记指令送达成功，记录往返时延
4. 超时未收到ACK：执行一次重试（使用相同序列号防重复执行）
5. 重试仍失败：标记设备指令执行失败，返回错误详情

## 输入规范

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| intent | object | 是 | 标准化意图结构体 |
| device_registry | object | 是 | 设备注册表引用 |
| priority | enum | 是 | 执行优先级(user/safety/scene/automation/remote) |
| request_source | string | 是 | 请求来源标识 |

## 输出规范

| 字段 | 类型 | 说明 |
|------|------|------|
| execution_status | enum | success/failed/partial/device_offline |
| device_ack | boolean | 设备是否ACK确认 |
| actual_command | object | 实际下发的指令报文（脱敏） |
| latency_ms | integer | 指令往返时延（毫秒） |
| error_detail | object | 失败时的错误详情 |

## 最佳实践

1. **指令幂等性**：使用唯一序列号确保重试不会导致重复执行（如灯闪烁）
2. **协议降级**：首选协议不可达时尝试备选协议（如WiFi断线尝试BLE直连）
3. **批量优化**：同一网关下的多设备指令打包发送减少通信开销
4. **缓存热点**：频繁操作设备的指令编码结果缓存，降低编码延迟
5. **灰度验证**：新增设备型号的指令编码器上线后先在单设备验证再全量启用

## 约束条件

- 指令路由决策时间 < 50ms
- 指令格式转换时间 < 30ms
- ACK等待超时 500ms，重试1次
- 设备注册表查询缓存TTL 60秒
- 不得绕过能力模型发送超范围指令
