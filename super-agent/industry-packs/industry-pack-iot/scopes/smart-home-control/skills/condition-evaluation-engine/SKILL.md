# 条件评估引擎 (Condition Evaluation Engine)

## 概述

本技能负责实时评估自动化规则的触发条件，通过订阅设备状态变更事件和传感器数据流，持续计算各条件表达式的布尔结果。核心挑战是在保证评估实时性的同时控制计算开销，并通过防抖机制避免数据波动导致的规则误触发。

### 适用场景
- 传感器阈值条件的实时判断（温度/湿度/光照超限）
- 设备状态变更条件的事件驱动评估
- 复合条件的逻辑组合评估（AND/OR/NOT）
- 时间窗口条件的持续性验证
- 条件表达式的语法解析和预编译

## 执行步骤

### Step 1: 条件表达式解析
1. 接收注册的条件规则表达式并解析为语法树：
   - 简单条件：`sensor.temperature > 28`
   - 比较操作：`>`, `<`, `>=`, `<=`, `==`, `!=`
   - 逻辑组合：`condition_A AND condition_B`, `condition_A OR condition_B`
   - 时间约束：`condition持续>=60秒`
   - 状态变更：`device.status CHANGED_TO offline`
2. 预编译条件表达式为可高效评估的内部表示
3. 提取条件依赖的数据源列表（哪些设备/传感器的数据变更会触发重新评估）
4. 建立数据源→规则的倒排索引（数据源变更时快速定位需要重评估的规则）

### Step 2: 事件驱动评估
1. 订阅状态同步监控器的事件流
2. 收到事件后通过倒排索引定位关联规则
3. 对关联规则执行条件表达式评估：
   - 读取条件引用的所有数据源当前值
   - 按语法树递归计算布尔结果
   - 复合条件采用短路求值优化（AND遇false提前返回）
4. 评估结果与上次状态对比：
   - false→true：条件满足，进入防抖验证
   - true→false：条件不再满足，重置防抖计时器
   - 无变化：跳过，不做任何操作

### Step 3: 防抖验证
1. 条件首次满足后启动防抖计时器（默认60秒，可配置）
2. 防抖期间持续验证条件是否持续满足：
   - 每次相关数据源更新时重新评估
   - 条件中途变为false：重置计时器，本次不触发
   - 条件持续为true直到计时器到期：确认触发
3. 防抖目的：
   - 传感器数据波动（如温度在阈值附近震荡）不导致频繁触发
   - 瞬时状态变更（如门短暂开关）不误触发
4. 特殊条件类型的防抖策略：
   - 安全类条件：防抖时间缩短为5秒（快速响应）
   - 环境类条件：标准60秒防抖
   - 统计类条件：防抖时间延长为5分钟

### Step 4: 触发与冷却
1. 防抖验证通过后正式触发规则
2. 生成触发事件（含触发时的数据快照作为上下文）
3. 启动冷却计时器（默认30分钟）：
   - 冷却期内同一规则即使条件再次满足也不重复触发
   - 冷却时间可由用户自定义
   - 安全类规则冷却时间较短（5分钟）
4. 触发事件发送给规则执行调度器

## 输入规范

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| rule_id | string | 是 | 规则标识 |
| condition_expression | string | 是 | 条件表达式文本 |
| debounce_seconds | integer | 否 | 防抖时间（默认60） |
| cooldown_minutes | integer | 否 | 冷却时间（默认30） |
| data_event | object | 是 | 触发评估的数据事件 |

## 输出规范

| 字段 | 类型 | 说明 |
|------|------|------|
| evaluation_result | boolean | 条件表达式当前布尔值 |
| state_changed | boolean | 评估结果是否发生变化 |
| trigger_confirmed | boolean | 是否通过防抖确认为正式触发 |
| trigger_context | object | 触发时的数据快照 |
| next_evaluation_eta | long | 下次可触发时间（冷却中时） |

## 最佳实践

1. **懒评估**：只在相关数据源变更时重新评估，不做定时轮询
2. **短路优化**：复合条件中概率最低的子条件放在最前面（AND场景）减少计算量
3. **缓存友好**：对频繁评估的条件缓存中间结果，只重算发生变化的部分
4. **溢出保护**：限制单次评估的数据源查询数（最多20个），超过时拒绝规则注册
5. **可观测性**：每次评估记录耗时和结果，用于规则性能分析和优化建议

## 约束条件

- 单条规则评估耗时 < 50ms
- 100条规则全量评估 < 5秒
- 防抖期间评估频率 ≤ 1次/秒（避免CPU过载）
- 冷却期最短5分钟最长24小时
- 条件表达式最大嵌套深度 5层
- 单条规则最大引用数据源 20个
