# 设备生命周期管理 (Device Lifecycle Management)

## 概述

本技能负责管理智能家居设备从发现、注册、运行到退役的完整生命周期。核心价值是让新设备"即插即用"——标准协议设备配置接入时间控制在1小时以内，同时确保设备能力模型的准确性和完整性，为上层控制逻辑提供可靠的设备元数据支撑。

### 适用场景
- 新设备首次接入网络时的自动发现和注册
- 设备能力模型的创建和更新
- 设备固件升级管理
- 设备迁移（更换房间/更换网关）
- 设备退役和数据清理

## 执行步骤

### Step 1: 设备发现
1. 持续监听网络中的设备广播/配对请求：
   - WiFi设备：mDNS/SSDP发现
   - Zigbee设备：网关报告新设备入网
   - BLE设备：广播包扫描
   - Matter设备：Commissioner发现
2. 提取设备基本信息：品牌/型号/MAC地址/协议版本
3. 与已注册设备列表比对，确认是否为新设备
4. 新设备标记为"待注册"状态，通知用户确认接入

### Step 2: 设备注册
1. 用户确认接入后，执行注册流程：
   - 分配唯一设备ID
   - 建立安全配对（密钥交换/证书颁发）
   - 配置通信参数（上报频率/重连策略）
2. 读取设备能力描述符（Device Descriptor）：
   - 支持的操作类型列表
   - 每个操作的参数类型和范围
   - 设备额定参数（功率/电压/频率）
   - 支持的通信模式
3. 用户补充配置：
   - 设备名称/别名设置
   - 所在房间/区域分配
   - 设备分组归属

### Step 3: 能力模型构建
1. 基于设备描述符生成标准化能力模型：
```json
{
  "device_id": "uuid",
  "brand": "midea",
  "model": "KFR-35GW",
  "category": "air_conditioner",
  "capabilities": [
    {
      "action": "set_temperature",
      "param_type": "integer",
      "range": [16, 30],
      "unit": "celsius",
      "step": 1
    },
    {
      "action": "set_mode",
      "param_type": "enum",
      "options": ["cooling", "heating", "dehumidify", "fan", "auto"]
    }
  ],
  "constraints": ["需要关窗才能高效制冷"],
  "rated_power_w": 1200
}
```
2. 验证能力模型完整性（所有必填字段有值/参数范围合理）
3. 注册完成后通知相关系统：
   - 意图解析引擎：更新可控设备列表
   - 设备调度控制器：更新路由表
   - 自动化引擎：更新可用设备条件选项

### Step 4: 运行维护
1. 固件升级管理：
   - 检测可用固件更新
   - 评估升级风险（是否有已知问题）
   - 用户确认后执行OTA升级
   - 升级后验证设备功能正常
2. 设备健康度持续评估：
   - 通信质量趋势（丢包率/延迟）
   - 性能劣化监测（响应变慢/功耗异常上升）
   - 使用寿命估算（基于运行时长/开关次数）
3. 设备退役处理：
   - 解除所有关联规则和场景
   - 安全注销（撤销密钥/证书）
   - 清理历史数据（按用户选择保留或删除）
   - 从各系统注册表中移除

## 输入规范

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| action | enum | 是 | discover/register/update/decommission |
| device_info | object | 是 | 设备基本信息 |
| user_config | object | 否 | 用户配置（名称/房间/分组） |
| capability_descriptor | object | 否 | 设备原始能力描述符 |

## 输出规范

| 字段 | 类型 | 说明 |
|------|------|------|
| device_id | string | 分配的设备ID |
| registration_status | enum | pending/active/decommissioned |
| capability_model | object | 生成的标准化能力模型 |
| validation_result | object | 能力模型验证结果 |
| notifications_sent | array | 已通知的下游系统列表 |

## 最佳实践

1. **安全第一**：设备注册必须经用户确认，防止恶意设备自动接入
2. **能力验证**：注册后执行一次能力验证测试（对每个声明的能力做一次调用确认）
3. **兼容性列表**：维护已验证设备型号列表，未验证型号标记为"实验性支持"
4. **渐进接入**：新设备先进入受限模式（仅基本操作），运行稳定48小时后开放全部能力
5. **退役缓冲**：设备退役后保留7天恢复窗口，防止误操作

## 约束条件

- 标准协议设备接入配置时间 < 1小时
- 设备发现到通知用户延迟 < 30秒
- 能力模型完整度校验必须100%通过才能标记为active
- 同时支持在线设备数 ≥ 200台
- 设备注册表查询响应 < 50ms
