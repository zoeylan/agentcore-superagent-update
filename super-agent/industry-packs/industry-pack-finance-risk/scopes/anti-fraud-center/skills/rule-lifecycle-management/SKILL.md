# 规则全生命周期管理 (Rule Lifecycle Management)

## 技能概述

规则全生命周期管理是规则策略Agent的核心能力，覆盖反欺诈规则从创建、测试、上线、监控到淘汰的完整生命周期。规则库是反欺诈系统的"确定性防线"，规则的质量和时效性直接影响欺诈检出率和误报率。

### 适用场景
- 基于新发现的欺诈模式创建新检测规则
- 现有规则的阈值调优和逻辑修改
- 规则效果评估和失效规则淘汰
- 规则库季度全面审查（Rule Book Review）
- 紧急规则发布（应对突发欺诈事件）

## 执行步骤

### 第一步：规则需求分析
1. 接收规则需求来源：
   - 案件调查Agent提交的新欺诈模式发现
   - 误报案件反馈的规则优化需求
   - 欺诈情报分析Agent推送的前瞻性防御需求
   - 规则效果监控发现的性能退化告警
2. 分析欺诈模式特征，提炼可量化的检测条件
3. 评估现有规则库是否已有覆盖，避免规则冗余

### 第二步：规则设计
1. 定义规则基本属性：
   - 规则ID（唯一标识，格式：RULE-{类型}-{序号}）
   - 规则名称（简明描述检测目标）
   - 规则类型：速度规则/金额规则/黑名单规则/地理规则/设备规则/组合规则
   - 适用渠道：ALL / MOBILE / WEB / POS / ATM / API
   - 优先级权重：1-100（影响综合评分的权重）
2. 编写规则逻辑表达式（伪代码或DSL格式），例如：
   ```
   IF account.txn_count_1h > 10
   AND account.txn_total_1h > account.avg_daily_amount * 5
   AND device.is_new = true
   THEN risk_score += 35
   ```
3. 设定触发阈值参数，基于历史数据的统计分析确定（如用P99分位值作为异常阈值）

### 第三步：规则回测验证
1. 在历史交易数据上执行规则回测（至少回测最近3个月数据）
2. 统计回测指标：
   - 命中率（Hit Rate）：命中交易占总交易的比例
   - 精确率（Precision）：命中交易中确实是欺诈的比例
   - 召回率增量（Recall Lift）：新规则相比现有规则库的额外检出
   - 误报率（FPR）：命中但非欺诈的比例，必须≤5%
3. 若回测指标不达标，调整阈值参数后重新回测

### 第四步：灰度发布
1. 编制规则更新提案，提交审批：
   - 规则详情、回测结果、预期影响、灰度方案、回滚方案
2. 审批通过后，以5%流量灰度上线
3. 灰度期至少24小时，监控指标：
   - 灰度组vs对照组的欺诈检出率差异
   - 灰度组的误报率是否在可接受范围
   - 规则的实际命中率是否与回测预期一致
4. 灰度效果达标 → 全量发布；不达标 → 回滚并分析原因

### 第五步：在线监控与优化
1. 规则上线后持续监控：
   - 日/周命中率趋势（命中率骤降可能意味着黑产已绕过该规则）
   - 误报率趋势（误报率上升可能意味着正常交易模式发生了变化）
   - 规则贡献度排名（识别高价值规则和低效规则）
2. 触发规则淘汰条件：
   - 连续30天零命中 → 标记为僵尸规则，评估淘汰
   - 误报率持续>10% → 标记为高误报规则，要求调优或淘汰
   - 被新规则完全覆盖 → 标记为冗余规则，评估淘汰

### 第六步：季度审查
1. 每季度执行规则库全面审查（Rule Book Review）
2. 审查内容：所有规则的活跃度、有效性、冗余度
3. 输出审查报告：活跃规则数、新增/淘汰规则数、Top10高效规则、Bottom10低效规则
4. 更新规则库文档和版本号

## 输入/输出规范

### 输入
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| requestType | enum | 是 | CREATE/MODIFY/DEPRECATE/REVIEW |
| fraudPattern | object | 条件必填 | 欺诈模式描述（创建时必填） |
| ruleId | string | 条件必填 | 目标规则ID（修改/淘汰时必填） |
| backtestPeriod | integer | 否 | 回测月数（默认3） |

### 输出
| 字段 | 类型 | 说明 |
|------|------|------|
| ruleId | string | 规则唯一标识 |
| ruleDefinition | object | 规则完整定义 |
| backtestResults | object | 回测指标结果 |
| deploymentStatus | enum | DRAFT/TESTING/GRAYSCALE/ACTIVE/DEPRECATED |
| changeProposal | object | 变更提案文档 |

## 最佳实践
- 规则命名规范统一，便于检索和管理（如"VEL-001: 同账户1小时内交易次数超限"）
- 复杂组合规则拆分为多个简单子规则的组合，便于独立评估和维护
- 保留完整的规则变更历史（Git版本管理），支持快速回滚
- 建立"规则效果看板"，实时展示各规则的贡献度排名

## 约束条件
- 任何规则变更必须经过审批流程，禁止直接修改生产环境
- 新规则上线前误报率评估必须≤5%
- 灰度测试期间不少于24小时
- 规则库总规模控制在合理范围（建议不超过500条活跃规则），避免规则膨胀影响性能
