# 实时交易风险评分 (Realtime Transaction Scoring)

## 技能概述

实时交易风险评分是反欺诈中心的核心检测能力，通过规则引擎和机器学习模型的并行评估，在200ms内对每笔资金交易完成风险量化评分。该技能是交易反欺诈的"第一道防线"，直接决定了交易的放行、复核或拦截。

### 适用场景
- 网银/手机银行转账交易的实时风险评估
- POS/扫码支付交易的欺诈检测
- ATM取现/跨行转账的异常识别
- API开放平台接入的第三方交易风控
- 跨境汇款的可疑交易筛查

## 执行步骤

### 第一步：交易数据接收与预处理
1. 接收交易报文，解析标准字段：交易ID、交易类型、金额、币种、发起方账户、接收方账户、交易渠道、设备信息、IP地址、地理位置、时间戳
2. 数据完整性校验——关键字段缺失时标记为异常并走降级评估通道
3. 数据标准化处理——金额统一为人民币计价，时间统一为UTC+8

### 第二步：实时特征计算
1. 查询预计算的时序特征缓存（Redis/Flink State）：
   - 账户维度：近5分钟/1小时/24小时交易次数、累计金额、不同对手方数量
   - 设备维度：近1小时同设备关联账户数、设备首次出现时间
   - IP维度：近1小时同IP交易次数、IP地理位置与账户常驻地距离
2. 计算即时特征：交易金额与账户历史均值的偏离度（Z-Score）、交易时间异常度（凌晨交易权重提升）
3. 特征向量组装，确保特征维度与模型训练时一致

### 第三步：双层检测并行执行
1. **规则引擎检测**（目标延迟 <50ms）：
   - 按优先级顺序执行规则组：黑名单规则 → 速度规则 → 金额规则 → 地理规则 → 设备规则 → 组合规则
   - 记录每条命中规则的ID、名称、风险权重
   - 规则评分 = Σ(命中规则权重)，归一化至0-100
2. **ML模型预测**（目标延迟 <100ms）：
   - 将特征向量送入在线推理服务（Ensemble模型）
   - 获取欺诈概率预测值（0-1），转换为0-100评分
   - 记录Top5特征贡献度（SHAP值）

### 第四步：综合评分合成
1. 综合评分 = α × 规则评分 + β × 模型评分（默认 α=0.4, β=0.6，可按渠道调整）
2. 当规则引擎命中"一票否决"类规则（如确认黑名单）时，综合评分直接设为100
3. 当模型服务不可用时，综合评分 = 规则评分 × 1.2（适度放大，补偿模型缺失）

### 第五步：评分结果输出
1. 输出结构化评分结果：综合评分、风险等级、命中规则详情、模型评分及特征贡献
2. 将评分结果写入评分日志（Kafka Topic），供下游消费
3. 更新实时特征缓存（本次交易计入后续交易的时序特征）

## 输入/输出规范

### 输入
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| transactionId | string | 是 | 交易唯一标识 |
| transactionType | enum | 是 | 交易类型：TRANSFER/PAYMENT/WITHDRAWAL/REMITTANCE |
| amount | decimal | 是 | 交易金额 |
| currency | string | 是 | 币种代码 |
| senderAccount | string | 是 | 发起方账户号 |
| receiverAccount | string | 是 | 接收方账户号 |
| channel | enum | 是 | 交易渠道：MOBILE/WEB/POS/ATM/API |
| deviceFingerprint | string | 否 | 设备指纹 |
| ipAddress | string | 否 | 客户端IP |
| geoLocation | object | 否 | 经纬度 |
| timestamp | datetime | 是 | 交易发起时间 |

### 输出
| 字段 | 类型 | 说明 |
|------|------|------|
| transactionId | string | 交易唯一标识 |
| riskScore | integer (0-100) | 综合风险评分 |
| riskLevel | enum | LOW(<50) / MEDIUM(50-80) / HIGH(>80) |
| ruleEngineScore | integer | 规则引擎评分 |
| mlModelScore | integer | ML模型评分 |
| hitRules | array | 命中规则列表（规则ID/名称/权重/命中详情） |
| topFeatures | array | 模型Top5特征贡献度 |
| processingTimeMs | integer | 处理耗时（毫秒） |
| degraded | boolean | 是否走降级通道 |

## 最佳实践
- 特征缓存使用Redis Cluster，设置合理的TTL（时序特征按时间窗口自动过期）
- 模型推理使用TensorFlow Serving或Triton，支持GPU加速和动态批处理
- 规则引擎使用Drools或自研高性能引擎，规则编译为字节码提升执行速度
- 建立完善的降级链路：模型超时→纯规则模式→核心规则模式→全部放行（最后一级需人工授权）

## 约束条件
- 端到端处理延迟必须 ≤200ms（P99），其中特征计算≤50ms、规则引擎≤50ms、模型推理≤100ms
- 评分日志100%落盘，保留期≥5年
- 不得在评分过程中修改规则或模型配置
- 降级模式切换必须同步触发告警通知
