# 系统健康监控 (System Health Monitoring)

## 技能概述

系统健康监控是实时监控Agent维护自身检测系统可用性和性能达标的核心能力。通过持续监控交易检测系统的各项健康指标，确保系统在99.99%可用性SLA下稳定运行，并在异常发生时自动触发降级和告警。

### 适用场景
- 实时交易检测系统的延迟、吞吐量、错误率监控
- ML模型推理服务的响应时间和可用性监控
- 规则引擎的执行性能和命中率异常检测
- 系统降级策略的自动触发与恢复
- 容量预警与扩容建议

## 执行步骤

### 第一步：指标采集
1. 每10秒采集一次核心指标：
   - **延迟指标**：交易评分P50/P90/P99/P999延迟，模型推理延迟，规则引擎执行延迟
   - **吞吐量指标**：当前TPS（每秒交易数）、峰值TPS、队列积压深度
   - **错误率指标**：评分失败率、模型超时率、规则引擎异常率
   - **资源指标**：CPU使用率、内存使用率、Redis缓存命中率、网络带宽利用率
2. 每分钟汇总一次统计指标写入时序数据库（如InfluxDB/Prometheus）

### 第二步：阈值检测与告警
1. 将实时指标与预设阈值对比：
   - P99延迟 >180ms → 黄色告警（接近SLA上限）
   - P99延迟 >200ms → 红色告警（SLA突破）
   - 模型超时率 >5% → 模型服务告警
   - TPS超过容量80% → 容量预警
   - 评分失败率 >1% → 系统异常告警
2. 告警分级推送：黄色告警→运维团队，红色告警→运维+反欺诈负责人+CTO

### 第三步：自动降级管理
1. **一级降级**（模型超时率>10%或模型服务不可用）：
   - 自动切换为纯规则引擎模式
   - 规则评分权重调整为1.2倍补偿
   - 触发告警并记录降级开始时间
2. **二级降级**（规则引擎也出现性能问题）：
   - 切换为核心规则子集模式（仅执行黑名单+高权重速度规则）
   - 触发紧急告警
3. **三级降级**（系统全面不可用）：
   - 启动熔断保护，需人工授权决定全放行或全拦截
   - 触发最高级别告警通知至管理层

### 第四步：恢复与报告
1. 降级触发后，每30秒探测故障服务的恢复状态
2. 确认恢复后自动切回正常模式，但保持10分钟观察期
3. 生成降级事件报告：降级原因、持续时长、影响交易笔数、降级期间的检测效果评估

## 输入/输出规范

### 输入
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| metricsSource | enum | 是 | 指标来源：PROMETHEUS/INFLUXDB/CUSTOM |
| monitoringInterval | integer | 否 | 采集间隔秒数（默认10） |
| alertThresholds | object | 否 | 自定义告警阈值覆盖 |

### 输出
| 字段 | 类型 | 说明 |
|------|------|------|
| systemStatus | enum | HEALTHY / DEGRADED_L1 / DEGRADED_L2 / CRITICAL |
| currentMetrics | object | 当前各项指标快照 |
| activeAlerts | array | 当前活跃告警列表 |
| degradationInfo | object | 降级状态详情（如有） |
| healthReport | object | 分钟级/小时级健康报告 |

## 最佳实践
- 监控系统本身必须与被监控系统解耦部署，避免同时故障
- 建立"金丝雀交易"机制——定期发送测试交易验证全链路健康
- 降级策略预先演练，每季度至少进行一次降级演练
- 容量规划基于峰值TPS的1.5倍冗余

## 约束条件
- 监控不得对交易处理性能产生可观测影响（额外延迟 <1ms）
- 降级决策日志必须完整记录，供事后审计
- 三级降级（全放行）必须有管理层实时授权
- 降级恢复后72小时内完成事件复盘报告
