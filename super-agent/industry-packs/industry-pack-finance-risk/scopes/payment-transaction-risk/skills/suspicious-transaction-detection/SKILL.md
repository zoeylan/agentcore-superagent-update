# 可疑交易检测 (Suspicious Transaction Detection)

## 概述
可疑交易检测是AML监测Agent的核心技能，通过规则引擎与机器学习模型协同工作，识别具有洗钱特征的异常交易模式。检测到的可疑交易将生成标准化分析报告，为合规人员的STR（可疑交易报告）提交决策提供依据。

## 适用场景
- 日常交易流的批量扫描（T+1模式）
- 高风险客户交易的准实时监控
- 支付授权Agent推送的高风险拦截交易深度分析
- AML规则命中后的深度分析确认
- 定期客户行为模式回溯分析

## 执行步骤

### 步骤1：交易数据聚合与特征工程
- 汇集目标客户/交易的多维度数据：
  - 交易流水（金额、频次、时间分布、交易对手）
  - 客户KYC信息（职业、收入、业务性质）
  - 账户行为历史（开户时间、日均余额、交易模式变迁）
- 计算洗钱相关特征：
  - 资金归集-分散比率（入金集中度 vs 出金分散度）
  - 阈值规避指标（金额分布在报告阈值附近的交易占比）
  - 循环交易检测（资金闭环路径分析）
  - 交易对手风险加权值

### 步骤2：规则引擎筛查
- 执行预设的可疑交易规则集（100+条规则），包括：
  - 结构化交易（Structuring）：多笔金额略低于5万/50万阈值的交易
  - 快进快出：大额资金入账后短时间内（24小时内）分散转出
  - 与高风险实体交易：对手方属于高风险国家/行业/人员
  - 现金异常：与申报业务性质不符的大额现金交易
  - 循环转账：资金在关联账户间循环流转无明确商业目的
- 记录命中的规则编号和命中详情

### 步骤3：ML模型评估
- 对规则命中或高风险客户的交易调用AML检测模型
- 模型输出可疑概率（0-1）及主要贡献特征
- 综合规则命中和模型评分生成最终可疑度评级：
  - 高可疑（规则命中+模型>0.8）：优先处理
  - 中可疑（规则命中 或 模型0.5-0.8）：常规复核
  - 低可疑（仅单条边缘规则命中，模型<0.5）：归档观察

### 步骤4：可疑交易分析报告生成
- 按标准模板生成分析报告：
  - 客户识别信息和风险评级
  - 可疑交易明细（时间线、金额、对手方）
  - 命中规则和模型评分详情
  - 资金流向图谱（关键节点和路径）
  - 可疑原因分析和定性建议
  - 建议处置措施（提交STR/增强监控/关闭告警）

## 输入规格
```json
{
  "analysis_type": "string - BATCH_SCAN/REALTIME_ALERT/DEEP_ANALYSIS",
  "customer_id": "string - 目标客户",
  "transaction_ids": "array - 相关交易列表",
  "time_window": "object - 分析时间窗口",
  "trigger_source": "string - 触发来源(RULE/MODEL/MANUAL/PAYMENT_PUSH)"
}
```

## 输出规格
```json
{
  "suspicion_level": "string - HIGH/MEDIUM/LOW",
  "suspicion_score": "number - 0-1可疑度",
  "rules_hit": [{"rule_id": "string", "description": "string", "severity": "string"}],
  "model_score": "number",
  "recommended_action": "string - SUBMIT_STR/ENHANCED_MONITORING/CLOSE_ALERT",
  "analysis_report": "object - 完整分析报告",
  "related_accounts": "array - 关联可疑账户"
}
```

## 最佳实践
- 规则和模型应并行执行以提升效率，不应串行依赖
- 高可疑案件应优先排序处理，避免24小时STR提交时限超期
- 分析报告应包含足够上下文信息，减少合规人员二次查询
- 定期（月度）统计规则命中率和最终确认率，淘汰低效规则

## 约束条件
- 批量扫描在T+1凌晨完成，不影响日间交易处理性能
- 高风险告警准实时处理延迟≤30分钟
- 分析报告必须可追溯、不可篡改
- 不得自行决定是否提交STR，仅提供分析和建议
