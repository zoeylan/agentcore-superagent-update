# 风控指标监控 (Risk Metrics Monitoring)

## 概述
风控指标监控是策略调优Agent的核心观测技能，通过构建完整的风控KPI体系，每日汇总交易风控核心指标、建立基线和异常检测机制，第一时间发现指标异常波动并启动根因分析，确保风控体系始终运行在健康状态。

## 适用场景
- 日度风控指标汇总与日报生成（T+1）
- 指标异常的实时检测与告警
- 指标趋势分析与预测
- 策略变更前后的效果对比
- 月度/季度风控运营报告的数据支撑

## 执行步骤

### 步骤1：指标采集与计算
- 从各Agent和系统采集原始数据并计算核心KPI：
  - **授权通过率**（Authorization Rate）= 授权通过笔数 / 总授权请求笔数
  - **拦截率**（Decline Rate）= 拦截笔数 / 总授权请求笔数
  - **欺诈漏放率**（False Negative Rate）= 漏放欺诈笔数 / 总欺诈笔数
  - **客户验证放弃率**（Drop-off Rate）= 放弃验证笔数 / 触发验证笔数
  - **误拦截投诉率** = 误拦截投诉笔数 / 总拦截笔数
  - **P99授权延迟** = 第99百分位的授权响应时间
  - **系统可用性** = 正常服务时间 / 总时间
  - **STR及时提交率** = 24小时内提交STR笔数 / 总需提交STR笔数
  - **大额报告合规率** = 5工作日内报送笔数 / 总需报送笔数

### 步骤2：基线建立与异常检测
- 为每个指标建立动态基线：
  - 计算过去30天的均值和标准差
  - 区分工作日/节假日/大促等不同时段的基线
  - 考虑季节性因素和长期趋势
- 异常检测逻辑：
  - 偏离基线超过2个标准差 → 黄色告警
  - 偏离基线超过3个标准差 → 红色告警
  - 连续3天同方向偏离超过1个标准差 → 趋势告警
  - 特定指标突破硬限（如验证放弃率>15%、P99>100ms）→ 立即告警

### 步骤3：异常根因分析
- 发现异常后启动多维度下钻分析：
  - **时间维度**：精确定位异常开始时间点
  - **客群维度**：哪类客户（新/老、高净值/普通）受影响
  - **渠道维度**：哪个支付渠道（APP/网银/POS/QR）异常
  - **地区维度**：是否为区域性异常
  - **规则维度**：是否某条新规则导致大量命中
  - **系统维度**：是否基础设施变更引起
- 输出根因分析结论和建议措施

### 步骤4：报告生成与分发
- **日报**（T+1产出）：各指标当日值、环比、同比、基线偏离度
- **周报**：趋势分析、异常事件汇总、策略变更效果跟踪
- **月度报告**（T+5工作日内）：全面效能评估、合规数据汇总、优化建议
- 按权限分发至管理层、风控团队、合规部门

## 输入规格
```json
{
  "report_type": "string - DAILY/WEEKLY/MONTHLY/ALERT",
  "date_range": {"start": "date", "end": "date"},
  "metrics_scope": "array - 指定监控的指标列表（空为全量）",
  "comparison_period": "string - 对比周期(DOD/WOW/MOM/YOY)"
}
```

## 输出规格
```json
{
  "report_date": "date",
  "metrics": [
    {
      "metric_name": "string",
      "current_value": "number",
      "baseline_value": "number",
      "deviation": "number - 偏离百分比",
      "trend": "string - UP/DOWN/STABLE",
      "alert_level": "string - NORMAL/YELLOW/RED",
      "dod_change": "number - 日环比",
      "wow_change": "number - 周环比"
    }
  ],
  "alerts": [{"metric": "string", "level": "string", "description": "string", "timestamp": "ISO8601"}],
  "root_cause_analysis": "object - 异常根因分析（如有告警时）",
  "recommendations": "array - 建议措施"
}
```

## 最佳实践
- 指标采集应覆盖从交易发起到最终决策的全链路
- 基线计算时排除已知异常日（如系统故障日）的数据
- 建立指标间的关联分析（如通过率下降时检查是否拦截率对应上升）
- 告警应分级分人推送，避免告警疲劳

## 约束条件
- 日报T+1上午10:00前产出
- 异常指标2小时内启动分析
- 红色告警5分钟内推送至风控负责人
- 指标数据保留≥3年用于趋势分析
