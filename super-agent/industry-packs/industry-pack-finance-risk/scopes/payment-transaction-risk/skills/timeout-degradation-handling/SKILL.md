# 授权超时降级处理 (Timeout Degradation Handling)

## 概述
授权超时降级处理是支付授权系统的容灾保障技能，在主评分模型超时、依赖服务不可用等异常场景下，通过预设的降级策略链确保系统仍能在SLA内返回授权决策，遵循"宁可误拦不可漏放"的安全底线原则。

## 适用场景
- 主评分模型响应超时（>80ms）
- 外部数据源（征信、设备指纹服务等）调用失败
- 系统负载过高触发限流保护
- 部分服务节点宕机（非全局故障）
- 网络抖动导致上游数据获取异常

## 执行步骤

### 步骤1：超时检测与降级触发
- 设定各环节超时阈值：
  - 主模型推理：80ms
  - 特征查询：30ms
  - 外部服务调用：50ms
- 超时触发时记录降级事件（级别、触发原因、时间戳）
- 启动降级策略链的下一级

### 步骤2：多级降级策略执行
- **L1降级（主模型超时）**：切换至备用规则引擎
  - 使用预加载的规则集进行评估
  - 规则覆盖核心风险维度：金额阈值、黑名单、频次限制
  - 预期评估耗时≤20ms
- **L2降级（备用规则引擎也不可用）**：最保守策略
  - 小额交易（<1000元）且非黑名单：放行
  - 其他交易一律要求短信验证
  - 大额交易一律拦截
- **L3降级（全系统故障）**：紧急熔断
  - 全部交易进入排队模式
  - 启动灾备通知流程
  - 触发运维紧急响应

### 步骤3：降级恢复与回归
- 持续探测主服务健康状态（每5秒探活）
- 主服务恢复后不立即全量切回，采用渐进恢复（10%→30%→50%→100%）
- 恢复期间对比主服务和降级策略的决策差异，验证一致性

### 步骤4：降级事件报告
- 实时告警通知运维团队和风控管理人员
- 记录降级期间的交易处理统计（降级处理量、拦截率变化）
- 降级结束后生成事后分析报告

## 输入规格
```json
{
  "trigger_reason": "string - MODEL_TIMEOUT/SERVICE_UNAVAILABLE/OVERLOAD/NETWORK_ERROR",
  "degradation_level": "string - L1/L2/L3",
  "transaction_data": "object - 原始交易数据",
  "available_services": "array - 当前可用的服务列表"
}
```

## 输出规格
```json
{
  "decision": "string - APPROVE/CHALLENGE/DENY/BLOCK/QUEUE",
  "degradation_applied": "boolean - 是否启用了降级",
  "degradation_level": "string - 实际使用的降级级别",
  "original_service_status": "string - 主服务当前状态",
  "recovery_eta": "string - 预计恢复时间",
  "alert_sent": "boolean - 是否已发送告警"
}
```

## 最佳实践
- 降级规则集应定期与主模型对齐（每月审查），避免降级期间策略过度保守或宽松
- 定期进行降级演练（每季度），验证降级链路的完整性
- 降级期间的决策数据应单独标记，便于事后评估降级策略的合理性
- 设置降级持续时间阈值（如超过5分钟），自动升级告警级别

## 约束条件
- 降级决策必须在100ms总SLA内完成
- 降级期间不得放行任何命中黑名单的交易
- 全部降级事件必须100%记录日志和告警
- 降级恢复后必须验证主服务决策质量再全量切回
