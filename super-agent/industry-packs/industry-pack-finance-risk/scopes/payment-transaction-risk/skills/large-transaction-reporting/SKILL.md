# 大额交易报告生成 (Large Transaction Reporting)

## 概述
大额交易报告生成技能负责实时识别达到法定报告标准的交易，自动生成符合人民银行规定格式的大额交易报告，并确保在交易发生后5个工作日内完成报送。这是金融机构的法定合规义务，报送不及时将面临严重监管处罚。

## 适用场景
- 个人现金交易金额 ≥ 5万元人民币
- 企业现金交易金额 ≥ 20万元人民币
- 个人转账交易金额 ≥ 50万元人民币
- 企业转账交易金额 ≥ 200万元人民币
- 外币交易折合人民币达到上述标准的等值交易

## 执行步骤

### 步骤1：大额交易实时识别
- 对每笔完成授权的交易进行阈值判定：
  - 确定客户类型（个人/企业）
  - 确定交易类型（现金/转账）
  - 与对应阈值比较
  - 外币交易按当日汇率中间价折算人民币
- 标记达标交易进入报告生成队列

### 步骤2：交易信息补全与校验
- 补全报告所需字段：
  - 付款人：姓名/名称、证件类型及号码、账号、开户行
  - 收款人：姓名/名称、证件类型及号码、账号、开户行
  - 交易信息：金额、币种、时间、用途/备注
- 交叉校验数据一致性（客户信息与KYC档案匹配）
- 标记信息缺失项，触发补全流程

### 步骤3：报告生成
- 按照人民银行规定的数据格式和字段规范生成报告
- 自动填充机构信息（报送机构代码、报送人员等）
- 生成报告编号（唯一、连续、可追溯）
- 对同一客户当日多笔大额交易进行合并报告处理

### 步骤4：报送管理
- 将生成的报告加入报送队列
- 监控5个工作日报送时限（T+5个工作日）
  - T+3：正常状态
  - T+4：预警提醒
  - T+5之前：紧急催办
- 记录报送状态（待报送/已报送/报送失败/已确认）
- 报送失败时自动重试并告警

## 输入规格
```json
{
  "transaction_id": "string",
  "amount": "number - 交易金额",
  "currency": "string - 币种",
  "transaction_type": "string - CASH/TRANSFER",
  "customer_type": "string - PERSONAL/ENTERPRISE",
  "payer": {
    "name": "string",
    "id_type": "string",
    "id_number": "string",
    "account": "string",
    "bank": "string"
  },
  "payee": {
    "name": "string",
    "id_type": "string",
    "id_number": "string",
    "account": "string",
    "bank": "string"
  },
  "purpose": "string - 交易用途",
  "transaction_time": "ISO8601"
}
```

## 输出规格
```json
{
  "report_id": "string - 报告唯一编号",
  "report_status": "string - GENERATED/QUEUED/SUBMITTED/CONFIRMED/FAILED",
  "submission_deadline": "date - 最晚报送日期",
  "days_remaining": "integer - 剩余报送天数",
  "data_completeness": "number - 数据完整度百分比",
  "missing_fields": "array - 缺失字段列表（如有）"
}
```

## 最佳实践
- 大额交易识别应与授权流程同步进行，避免遗漏
- 建立报送日历管理（排除节假日），精确计算工作日期限
- 信息不完整时立即触发补全流程，不应等到临近截止日
- 定期与人民银行反馈核对，确认报送格式合规性

## 约束条件
- 大额交易识别覆盖率必须100%，零遗漏
- 5个工作日内报送率达100%
- 报告数据与原始交易流水必须可一一对应
- 报告生成和报送全程留痕，满足审计要求
