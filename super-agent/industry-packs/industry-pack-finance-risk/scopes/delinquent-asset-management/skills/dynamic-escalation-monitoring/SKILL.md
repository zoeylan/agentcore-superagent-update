# 动态升级监控技能 (Dynamic Escalation Monitoring)

## 技能概述

动态升级监控是智能分案Agent（case-allocator）的闭环管控技能，负责对全量在催案件进行持续的进度监控，当案件在当前催收策略下未能在规定时间窗口内取得预期进展时，自动触发策略升级（Strategy Escalation）。该技能实现了逾期案件在"M1自动化催收 → M2人工催收 → M3+法务处置"完整升级链路上的动态流转，确保没有任何案件因策略失效而被遗漏或延误。

本技能的核心职责包括：
- **进度监控**：实时追踪每个在催案件的催收响应状态、还款进展和关键时间节点
- **升级触发**：当案件达到预设的升级触发条件（时间阈值、响应失败次数、风险信号变化）时，自动发起策略升级
- **冷却期管理**：在策略升级前后设置合理的冷却期（Cooldown Period），避免策略频繁切换导致客户体验恶化
- **异常处理**：对投诉、争议、特殊情况等异常事件执行暂停/降级/特殊处理逻辑
- **升级效果追踪**：监控每次策略升级后的实际效果，为升级规则的优化提供数据支撑

动态升级监控贯穿逾期案件的全生命周期，是保障催收效率和回收率的关键控制机制。

## 适用场景

| 场景编号 | 场景名称 | 触发条件 | 说明 |
|---------|---------|---------|------|
| UC-01 | M1阶段内策略升级 | M1案件催收无响应超过时间阈值 | M1-Low→M1-Mid→M1-High的阶段内升级 |
| UC-02 | M1→M2跨阶段升级 | M1案件DPD达到30天且催收效果不佳 | 从自动化催收通道升级至人工催收通道 |
| UC-03 | M2阶段内策略升级 | M2案件协商未果或承诺不履约 | M2-Low→M2-Mid→M2-High的阶段内升级 |
| UC-04 | M2→M3+跨阶段升级 | M2案件DPD达到60天且无实质回款 | 从人工催收通道升级至法务处置通道 |
| UC-05 | 紧急升级（风险信号触发） | 客户外部风险信号突变（如资产转移、失联） | 不等待时间阈值，直接触发升级 |
| UC-06 | 策略降级（正向信号） | 客户恢复还款或主动沟通 | 将已升级的策略适度降回 |
| UC-07 | 升级暂停（异常事件） | 客户投诉/争议/不可抗力 | 暂停升级流程，进入特殊处理 |
| UC-08 | 升级超时兜底 | 案件在某阶段滞留超过最大时限 | 强制升级，防止案件"僵死" |

## 详细执行步骤

### 步骤一：监控对象注册与状态初始化

**目标**：将完成策略匹配和资源分配的案件纳入监控池，初始化监控参数。

**监控注册流程**：

1. **接收新分配案件通知**
   - 从策略匹配技能接收案件的策略分配结果
   - 从资源配置技能接收案件的资源分配结果

2. **初始化监控参数**

```
为每个案件创建监控记录：

{
  case_id: "CASE-2024-001234",
  monitor_id: "MON-2024-0115-001234",
  registered_time: "2024-01-15T01:10:00+08:00",

  // 当前策略状态
  current_stage: "M1",
  current_risk_level: "M1-Mid",
  current_strategy: "M1_STANDARD_COLLECTION",
  strategy_start_time: "2024-01-15T01:10:00+08:00",

  // 升级触发配置
  escalation_config: {
    intra_stage_escalation: {       // 阶段内升级
      trigger_days: 10,             // 策略执行10天无响应触发阶段内升级
      trigger_no_response_count: 5, // 连续5次触达无响应触发
      target_level: "M1-High"
    },
    cross_stage_escalation: {       // 跨阶段升级
      trigger_dpd: 30,             // DPD达到30天触发M1→M2
      trigger_no_payment_days: 25,  // 25天无任何还款触发提前升级评估
      target_stage: "M2"
    },
    emergency_escalation: {         // 紧急升级
      triggers: ["ASSET_TRANSFER", "CONTACT_LOST_CONFIRMED", "JUDICIAL_RISK_NEW"]
    }
  },

  // 冷却期配置
  cooldown_config: {
    post_escalation_cooldown_hours: 48,  // 升级后48小时冷却期
    post_contact_cooldown_hours: 4,      // 成功联系后4小时不重复触达
    max_escalation_count_per_month: 3    // 每月最多升级3次
  },

  // 监控状态
  monitor_status: "ACTIVE",
  last_check_time: null,
  escalation_history: [],
  exception_flags: []
}
```

3. **设置监控时钟**
   - 为每个案件注册定时检查任务
   - 检查频率根据案件优先级和阶段确定：
     - M1-Low: 每24小时检查一次
     - M1-Mid/High: 每12小时检查一次
     - M2: 每8小时检查一次
     - M3+: 每24小时检查一次（法务流程周期较长）

### 步骤二：催收进度实时追踪

**目标**：持续采集案件的催收执行数据，更新案件状态。

**追踪数据源**：

| 数据类型 | 数据源 | 采集频率 | 关键字段 |
|---------|--------|---------|---------|
| 触达记录 | 催收系统 | 实时 | 触达渠道、触达结果（接通/未接/拒接）、通话时长 |
| 客户响应 | 催收员录入 | 实时 | 响应态度（配合/消极/拒绝/失联）、沟通要点 |
| 承诺还款 | 催收系统 | 实时 | 承诺金额、承诺还款日期、承诺类型（口头/书面） |
| 还款入账 | 核心系统 | T+0 | 还款金额、还款时间、还款渠道 |
| 减免/方案 | 审批系统 | 实时 | 方案类型、审批状态、方案参数 |
| 外部信号 | 风控系统 | T+1 | 征信变化、司法信息、多头借贷变化 |
| 投诉/争议 | 客诉系统 | 实时 | 投诉渠道、投诉内容、处理状态 |

**案件状态机**：

```
案件催收状态流转：

INITIATED (已分配)
  │
  ├──触达成功──→ CONTACTED (已联系)
  │                │
  │                ├──客户承诺还款──→ PROMISE_MADE (已承诺)
  │                │                    │
  │                │                    ├──按期还款──→ PROMISE_FULFILLED (承诺履约)
  │                │                    │               └──全额还清──→ RESOLVED (结案)
  │                │                    │
  │                │                    └──逾期未还──→ PROMISE_BREACHED (承诺违约)
  │                │                                    └──重新进入催收循环
  │                │
  │                ├──客户拒绝还款──→ REFUSED (拒绝还款)
  │                │                    └──触发升级评估
  │                │
  │                ├──客户协商方案──→ NEGOTIATING (协商中)
  │                │                    ├──方案达成──→ PLAN_AGREED (方案达成)
  │                │                    └──协商破裂──→ 触发升级评估
  │                │
  │                └──客户投诉/争议──→ DISPUTED (争议中)
  │                                      └──暂停催收，进入异常处理
  │
  ├──触达失败──→ UNREACHABLE (未触达)
  │               └──累计N次──→ 触发升级评估
  │
  └──持续无响应──→ NO_RESPONSE (无响应)
                    └──超过时间阈值──→ 触发升级评估
```

### 步骤三：升级触发条件评估

**目标**：按照预设规则，对每个案件评估是否满足策略升级条件。

#### 3.1 M1阶段内升级触发规则

**M1-Low → M1-Mid 升级**：

| 触发条件编号 | 触发条件 | 触发阈值 | 优先级 |
|------------|---------|---------|--------|
| M1-L2M-01 | 策略执行天数无响应 | 策略执行≥7天，客户零响应 | P2 |
| M1-L2M-02 | 触达失败次数 | 连续3次触达失败（含未接、拒接） | P2 |
| M1-L2M-03 | 部分还款后停止 | 客户部分还款后≥5天无后续还款 | P3 |

**M1-Mid → M1-High 升级**：

| 触发条件编号 | 触发条件 | 触发阈值 | 优先级 |
|------------|---------|---------|--------|
| M1-M2H-01 | AI外呼无效 | AI外呼≥5次均未达成承诺 | P2 |
| M1-M2H-02 | 承诺违约 | 客户承诺还款后≥2次未履约 | P1 |
| M1-M2H-03 | DPD接近阈值 | DPD≥20天且无任何还款行为 | P1 |
| M1-M2H-04 | 外部风险信号恶化 | 新增多头借贷/征信异常查询 | P2 |

#### 3.2 M1→M2 跨阶段升级触发规则

| 触发条件编号 | 触发条件 | 触发阈值 | 说明 |
|------------|---------|---------|------|
| M1-TO-M2-01 | 自然迁徙 | DPD > 30天 | 基础触发条件，DPD超过M1阶段上限 |
| M1-TO-M2-02 | 提前升级（无响应） | DPD ≥ 25天 且 T+15仍无任何客户响应 | 在M1末期提前启动M2策略预热 |
| M1-TO-M2-03 | 提前升级（高风险） | M1-High案件，DPD ≥ 20天 且 催收效果评分 < 30 | 高风险案件提前升级，争取时间 |
| M1-TO-M2-04 | 紧急升级（失联） | 确认客户失联（所有联系方式失效） | 立即升级至M2人工催收+信息修复 |
| M1-TO-M2-05 | 紧急升级（资产风险） | 发现客户有资产转移迹象 | 立即升级，启动资产保全预评估 |

**T+15无响应升级详细逻辑**：

```
M1案件 T+15 无响应升级评估：

判定条件（须同时满足）：
  1. 案件策略执行已满15个自然日
  2. 期间所有触达尝试（短信+电话+APP）客户均无任何响应
     定义"无响应"：
     ├── 短信：未点击还款链接
     ├── 电话：未接听 或 接听后未做出任何承诺
     ├── APP推送：未点击/未阅读
     └── 无任何主动还款行为（含部分还款）
  3. 案件不处于冷却期内
  4. 案件不存在异常标记（COMPLAINT_ACTIVE等）

升级动作：
  1. 将案件风险等级提升至M1-High（如已是M1-High则标记为M2预备）
  2. 停止当前自动化催收策略
  3. 分配人工催收员，48小时内完成首次人工跟进
  4. 同时保留自动化渠道作为辅助（降低频率为每3天1次）
  5. 设置M2升级倒计时（DPD 30天时正式升级）
```

#### 3.3 M2阶段内升级触发规则

**M2-Low → M2-Mid 升级**：

| 触发条件编号 | 触发条件 | 触发阈值 | 优先级 |
|------------|---------|---------|--------|
| M2-L2M-01 | 协商方案未达成 | 催收员跟进≥10天，未能达成任何还款方案 | P2 |
| M2-L2M-02 | 方案达成但未执行 | 达成还款方案后7天内未按约执行首期 | P1 |
| M2-L2M-03 | 客户配合度下降 | 从"配合"转为"消极"态度，连续3次不接电话 | P2 |

**M2-Mid → M2-High 升级**：

| 触发条件编号 | 触发条件 | 触发阈值 | 优先级 |
|------------|---------|---------|--------|
| M2-M2H-01 | 催收员施压无效 | 资深催收员跟进≥15天，客户明确拒绝还款 | P1 |
| M2-M2H-02 | 多次承诺违约 | 累计承诺还款≥3次均未履约 | P1 |
| M2-M2H-03 | DPD接近M3阈值 | DPD ≥ 55天且无实质回款 | P0 |
| M2-M2H-04 | 失联预警 | 连续7天所有渠道无法联系客户 | P0 |

#### 3.4 M2→M3+ 跨阶段升级触发规则

| 触发条件编号 | 触发条件 | 触发阈值 | 说明 |
|------------|---------|---------|------|
| M2-TO-M3-01 | 自然迁徙 | DPD > 60天 | 基础触发条件 |
| M2-TO-M3-02 | 提前升级（施压无效） | DPD ≥ 50天 且 M2-High策略执行无效 | 高风险案件提前启动法务评估 |
| M2-TO-M3-03 | 提前升级（金额触发） | 逾期金额 > 100,000元 且 DPD ≥ 45天 且 无任何回款 | 大额案件加速法务流程 |
| M2-TO-M3-04 | 紧急升级（资产转移） | 确认客户正在转移资产 | 立即启动诉前保全 |
| M2-TO-M3-05 | 紧急升级（跑路信号） | 客户确认失联 + 工作单位离职 + 住址变更 | 立即启动法务追索 |

**Day 60无响应强制升级逻辑**：

```
M2案件 Day 60 强制升级至M3+：

前置评估（升级前必须完成）：
  1. 确认M2阶段催收记录完整性（至少尝试过10次人工联系）
  2. 核实客户还款能力评估（最新收入/资产信息）
  3. 检查是否有进行中的还款方案（如有，评估方案执行情况）
  4. 确认无投诉/争议等异常标记

升级决策矩阵：
  ├── 有还款方案执行中 且 执行良好 → 不升级，继续M2跟进
  ├── 有还款方案但执行不力 → 升级至M3+，但保留方案协商窗口
  ├── 无任何还款方案 且 客户配合 → 最后协商机会（7天），失败则升级
  ├── 无任何还款方案 且 客户不配合/失联 → 立即升级至M3+
  └── 特殊情况（客户出具困难证明）→ 暂缓升级，转客户沟通Agent评估

升级执行动作：
  1. 生成M2阶段催收总结报告（含全部沟通记录、客户态度分析、失败原因分析）
  2. 触发案件分级评估技能重新评估（按M3+标准）
  3. 触发策略匹配技能匹配M3+策略
  4. 触发资源配置技能分配法务资源
  5. 通知原M2催收员案件已升级（释放其产能）
  6. 记录升级事件至升级日志
```

#### 3.5 M3+阶段内监控规则

M3+阶段升级逻辑与M1/M2不同，主要监控法务处置进度：

| 监控节点 | 时间阈值 | 触发动作 |
|---------|---------|---------|
| 法务评估完成 | 案件进入M3+后≤5个工作日 | 超时告警，催促法务团队 |
| 起诉材料准备 | 决定起诉后≤10个工作日 | 超时告警，评估是否更换律所 |
| 法院立案 | 材料提交后≤15个工作日 | 超时跟进法院进度 |
| 开庭审理 | 立案后≤90天 | 按法院排期，到期前提醒 |
| 判决执行申请 | 判决书下达后≤15天 | 超时告警，督促及时申请执行 |
| 委外催收进度 | 每30天 | 检查委外机构回款进展 |
| 资产包转让 | 挂牌后≤60天 | 检查是否有摘牌方 |

### 步骤四：升级执行与协调

**目标**：当升级条件触发后，执行完整的策略升级流程，协调各关联系统和Agent。

#### 升级执行标准流程

```
升级执行流程（6步标准流程）：

Step 1: 升级预检（Pre-Escalation Check）
  ├── 确认案件未处于冷却期
  ├── 确认案件无异常标记阻断升级
  ├── 确认本月升级次数未超限（≤3次/月）
  └── 确认触发条件仍然成立（避免延迟触发导致误升级）

Step 2: 当前策略终止（Current Strategy Termination）
  ├── 通知当前催收资源停止催收动作
  ├── AI系统：从任务队列中移除该案件
  ├── 人工催收员：标记案件为"已升级-待交接"
  └── 记录当前策略的执行摘要（执行时长、触达次数、客户响应等）

Step 3: 重新评估（Re-Assessment）
  ├── 触发案件分级评估技能，按新阶段标准重新评分
  ├── 更新案件的M-stage和Risk Level
  └── 检查是否有新的特殊标记需要设置

Step 4: 新策略匹配（New Strategy Matching）
  ├── 触发催收策略匹配技能，匹配新阶段的策略方案
  ├── 生成新的渠道编排和执行参数
  └── 新策略应考虑旧策略的失败原因（避免重复无效策略）

Step 5: 新资源分配（New Resource Allocation）
  ├── 触发资源优化配置技能，分配新阶段的催收资源
  ├── M1→M2：分配人工催收员
  ├── M2→M3+：分配法务团队/委外机构
  └── 释放原资源的产能

Step 6: 升级后监控初始化（Post-Escalation Monitoring）
  ├── 重新注册监控参数（新阶段的升级触发条件）
  ├── 启动升级后冷却期（48小时内不再触发新的升级）
  ├── 设置升级效果观察期（7天内重点关注新策略效果）
  └── 记录完整的升级事件日志
```

#### 升级协调通知

| 通知对象 | 通知内容 | 通知方式 | 时效要求 |
|---------|---------|---------|---------|
| 催收执行Agent | 案件策略变更，停止当前催收动作 | 系统消息 | 实时 |
| 新分配催收员 | 案件信息包 + 前序催收记录 + 升级原因 | 工作台推送 | 升级后30分钟内 |
| 法务处置Agent | M3+升级案件信息 + 法务评估请求 | 工单推送 | 升级后1小时内 |
| 客户沟通Agent | 升级涉及的客户沟通策略变更 | 系统消息 | 升级后1小时内 |
| 催收数据分析Agent | 升级事件数据（用于迁徙率统计） | 数据流 | T+0 |
| 催收主管 | P0/P1升级事件汇总 | 邮件/IM | 每日汇总 |

### 步骤五：冷却期与频控管理

**目标**：合理控制策略升级频率和催收接触频率，避免过度催收和策略频繁切换。

#### 冷却期规则体系

```
冷却期类型与时长：

1. 升级后冷却期（Post-Escalation Cooldown）
   ├── 阶段内升级（如M1-Mid→M1-High）：24小时
   ├── 跨阶段升级（如M1→M2）：48小时
   └── 紧急升级：无冷却期（但需在升级后24小时内完成首次新策略触达）

   冷却期内行为：
   ├── 不触发新的升级评估
   ├── 不执行旧策略的催收动作
   ├── 允许新策略的首次触达（建立沟通）
   └── 允许客户主动联系时的响应

2. 触达后冷却期（Post-Contact Cooldown）
   ├── 成功通话后：4小时不重复致电（同一渠道）
   ├── 短信发送后：24小时不重复发送同类短信
   ├── 客户明确表示"稍后联系"：按约定时间回拨
   └── 客户情绪激动挂断：48小时后再联系

3. 承诺还款等待期（Promise Waiting Period）
   ├── 客户承诺还款日期前：不催收（仅到期前1天发送提醒）
   ├── 承诺日当天：发送提醒，不催收
   └── 承诺日后次日：如未还款，恢复催收并触发升级评估

4. 投诉处理冷却期（Complaint Cooldown）
   ├── 投诉受理期间：全面暂停催收
   ├── 投诉结案后：48小时冷却期
   └── 投诉成立后：降低催收强度，7天观察期
```

#### 频控（Frequency Control）规则

```
全局频控规则：

日频控：
├── 总触达次数（所有渠道总和）≤ 3次/天
├── 电话催收（AI+人工）≤ 2次/天
├── 短信 ≤ 2条/天
└── APP推送 ≤ 1次/天

周频控：
├── 总触达次数 ≤ 7次/周
├── 电话催收 ≤ 5次/周
└── 连续无响应超过5次 → 降低频率至2次/周

月频控：
├── 策略升级次数 ≤ 3次/月
├── 催收员更换次数 ≤ 2次/月
└── 第三方联系人通知 ≤ 1次/月

特殊频控：
├── 节假日：降低至工作日频率的50%
├── 春节期间（除夕-初七）：仅允许短信提醒，不进行电话催收
└── 客户明确要求的特定时段：标记免打扰
```

### 步骤六：异常事件处理

**目标**：对催收过程中的异常事件进行识别、暂停和特殊处理。

#### 异常事件分类与处理矩阵

| 异常类型 | 识别方式 | 紧急程度 | 处理流程 | 升级影响 |
|---------|---------|---------|---------|---------|
| 客户投诉 | 客诉系统推送 | P0 | 立即暂停催收→转客户沟通Agent→投诉结案后恢复 | 暂停升级流程 |
| 账务争议 | 客户提出/催收员发现 | P1 | 暂停催收→核实账务→争议解决后恢复 | 暂停升级，争议期不计入DPD |
| 客户重大变故 | 催收员上报/系统识别 | P1 | 暂停常规催收→启动困难客户援助评估 | 转入特殊流程 |
| 客户身故 | 催收员核实/系统推送 | P0 | 立即停止催收→转入遗产继承流程 | 终止常规升级 |
| 客户被刑事拘留 | 催收员核实 | P1 | 暂停催收→等待司法程序结论 | 暂停升级 |
| 催收合规违规 | 质检系统/客户投诉 | P0 | 立即停止催收→更换催收员→内部审查 | 暂停升级48小时 |
| 系统故障 | 监控系统告警 | P1 | 切换至备用渠道/人工催收兜底 | 故障期间升级评估暂停 |
| 客户提起诉讼 | 法务系统推送 | P0 | 暂停催收→转法务处置Agent应诉 | 转入法务流程 |

#### 投诉处理详细流程

```
投诉触发的升级暂停流程：

1. 投诉接收（T+0小时）
   ├── 系统自动识别案件关联的投诉
   ├── 立即设置 COMPLAINT_ACTIVE 标记
   ├── 暂停该案件的所有催收动作（含自动化+人工）
   └── 暂停该案件的升级评估时钟

2. 投诉处理（T+0 ~ T+48小时）
   ├── 客户沟通Agent接管投诉处理
   ├── 调取催收录音/记录进行合规审查
   ├── 形成投诉调查报告
   └── 在48小时内出具正式回复

3. 投诉结案后恢复（T+48 ~ T+96小时）
   ├── 投诉不成立：
   │   ├── 48小时冷却期后恢复催收
   │   ├── 更换催收员（避免冲突升级）
   │   ├── 恢复升级评估时钟（投诉处理期间不计入无响应天数）
   │   └── 降低催收强度1个等级，观察7天
   │
   ├── 投诉成立（轻微违规）：
   │   ├── 72小时冷却期
   │   ├── 更换催收员 + 降低催收策略1个等级
   │   ├── 升级评估时钟重置（从零开始计算）
   │   └── 设置14天观察期，期间重点质检
   │
   └── 投诉成立（严重违规）：
       ├── 7天全面暂停催收
       ├── 内部合规审查 + 催收员处分
       ├── 由主管指定恢复催收方案
       └── 升级评估需人工审批（不再自动触发）
```

### 步骤七：升级效果追踪与规则优化

**目标**：追踪每次策略升级的实际效果，为升级规则的持续优化提供数据支持。

**效果追踪指标**：

| 指标名称 | 计算方式 | 目标值 | 用途 |
|---------|---------|--------|------|
| 升级后回款率 | 升级后30天内回款金额/升级时逾期金额 | M1→M2: ≥40%; M2→M3+: ≥20% | 评估升级时机是否恰当 |
| 升级后首次响应时间 | 升级后客户首次响应的天数 | ≤7天 | 评估新策略的触达效率 |
| 升级后客户态度变化 | 升级前后客户配合度评分变化 | 正向变化 | 评估策略升级的施压效果 |
| 无效升级率 | 升级后30天内仍无任何进展的案件占比 | ≤20% | 识别升级规则是否过于宽松/严格 |
| 升级导致投诉率 | 升级后7天内产生投诉的案件占比 | ≤1% | 评估升级是否过于激进 |

**规则优化机制**：

```
月度升级规则回检流程：

1. 数据汇总
   ├── 统计本月各类升级的触发次数、触发条件分布
   ├── 统计各类升级的效果指标（回款率、响应时间、投诉率）
   └── 对比不同触发条件的升级效果差异

2. 规则评估
   ├── 触发过早的升级（升级后客户在短期内即还款）→ 考虑延长触发阈值
   ├── 触发过晚的升级（升级前已有明确恶化信号但未及时升级）→ 考虑新增触发条件
   ├── 无效升级过多 → 审查触发条件是否过于灵敏
   └── 升级导致投诉率偏高 → 审查升级后的策略强度是否过激

3. 参数调优
   ├── 调整时间阈值（如T+15→T+12或T+18）
   ├── 调整触发组合条件（增加或减少必要条件）
   ├── 调整冷却期时长
   └── 调整频控参数

4. 审批上线
   ├── 参数调整方案须经催收业务负责人审批
   ├── 重大规则变更须经风控委员会审批
   ├── 新规则先在10%案件上灰度验证，确认效果后全量推广
   └── 保留上一版本规则，支持快速回滚
```

## 最佳实践与约束条件

### 最佳实践

1. **渐进式升级优于跳跃式升级**
   - 除紧急情况外，升级应遵循渐进路径：M1-Low → M1-Mid → M1-High → M2-Low → ...
   - 避免从M1-Low直接跳至M2-High，策略断崖式变化会恶化客户体验和合规风险
   - 每次升级一个级别，给客户和催收团队适应新策略的缓冲时间

2. **时间触发与行为触发相结合**
   - 不应仅依赖DPD时间触发升级（可能遗漏高风险行为信号）
   - 不应仅依赖行为触发（可能导致低活跃案件长期滞留在低效策略中）
   - 最佳实践：时间阈值作为"底线"，行为信号作为"加速器"

3. **升级前的"最后一次机会"机制**
   - 在跨阶段升级（M1→M2、M2→M3+）前，给客户一次"最后通牒"式的沟通机会
   - 例如：M2→M3+升级前，发送法务催收预告函，给予7天宽限期
   - 这既是合规需要（给予客户合理的通知和机会），也有助于提高回收率

4. **升级数据的闭环利用**
   - 每次升级都是模型优化的宝贵数据样本
   - 追踪"升级后回款"vs"未升级但自然回款"的对比，评估升级的边际价值
   - 利用A/B测试验证不同升级触发时机的效果差异

5. **异常处理的合规优先原则**
   - 当升级流程与合规要求冲突时（如投诉中案件到了升级时间点），合规要求绝对优先
   - 异常处理期间的DPD天数照常累计，但升级评估时钟暂停
   - 异常解除后的恢复策略应考虑异常期间累计的DPD变化

### 约束条件

1. **升级频率限制**
   - 单个案件每月升级次数 ≤ 3次（阶段内升级 + 跨阶段升级合计）
   - 跨阶段升级在案件生命周期内通常只发生1-2次（M1→M2、M2→M3+）
   - 短期内连续升级（7天内升级2次以上）需人工审批

2. **冷却期不可跳过**
   - 升级后冷却期为硬性约束，即使案件在冷却期内满足了新的升级条件，也必须等待冷却期结束
   - 唯一例外：紧急升级（资产转移、跑路信号等P0事件）可跳过冷却期
   - 冷却期内的时间不计入下一次升级的触发时间窗口

3. **降级限制**
   - 跨阶段降级（如M2降回M1）原则上不允许，只有在特殊审批后才可执行
   - 阶段内降级（如M1-High降为M1-Mid）允许但须有明确的正向信号支持
   - 降级后至少观察7天，期间不再触发升级评估

4. **合规红线**
   - 投诉中案件的升级评估必须完全冻结，不得绕过
   - 升级不得作为对客户投诉的"报复"手段（审计重点关注投诉后立即升级的案件）
   - 法务升级（M3+）前必须完成法务可行性评估，不得盲目起诉
   - 所有升级决策须留痕可追溯，保留至少5年

5. **系统稳定性约束**
   - 批量升级检查不得在业务高峰期（09:00-11:00）执行大量案件迁移
   - 单次批量升级处理案件数不超过500件，超过时分批执行
   - 升级执行失败（系统异常）时，案件保持原状态，触发告警等待人工处理

## 示例

### 示例一：M1 T+15无响应升级至M1-High

**场景**：一笔M1-Mid案件执行标准催收策略15天，客户完全无响应。

**案件信息**：
- 案件：M1-Mid，DPD=20天，逾期金额5,200元，消费分期
- 策略执行记录（15天内）：
  - APP推送：5次，均已送达但未点击
  - 短信：4条，均已送达但未点击还款链接
  - AI外呼：5次，其中3次未接听、2次接通后客户说"知道了"但无后续行动
  - 还款行为：无任何还款

**升级评估过程**：

```

  [MATCH] M1-M2H-03: DPD≥20天且无任何还款 ✓（DPD=20，零还款）
  [NOT MATCH] M1-M2H-02: 承诺违约 ✗（客户未做出过承诺）
  [NOT MATCH] M1-M2H-04: 外部风险信号恶化 ✗（无新增风险信号）

冷却期检查：
  上次升级时间：无（首次升级）
  当前是否处于冷却期：否
  本月升级次数：0次

升级决策：触发M1-Mid → M1-High升级 ✓
```

**升级执行结果**：

```

新策略要点：
  - Day 1（明天）：人工催收员首次致电，了解客户不还款原因
  - Day 1-3：SMS + APP推送持续提醒
  - Day 3-7：人工催收每2天跟进1次，施压+提供还款方案
  - Day 7+：如仍无进展，启动M2升级预热

下次升级评估：
  - 触发条件：DPD≥25天且人工催收3次无效 → 升级至M2预备状态
```

### 示例二：M2→M3+ Day 60强制升级

**场景**：一笔M2-High案件DPD达到60天，触发M2→M3+强制升级评估。

**案件信息**：

### 示例三：投诉导致升级暂停与恢复
