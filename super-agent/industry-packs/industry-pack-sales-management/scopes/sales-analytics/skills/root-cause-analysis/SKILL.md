# 根因分析

## 技能概述

本技能用于对销售数据中发现的异常现象进行系统性的根因追溯和原因分析，通过结构化的分析框架（5-Why、鱼骨图、帕累托分析）将表面现象层层深入到根本原因，并输出具有针对性的改善建议。该技能是将"发现问题"升级为"解决问题"的关键分析能力。

## 适用场景

- 预测偏差超过±10%后的偏差原因分析
- 大单丢失后的复盘分析
- 赢率异常下降的原因诊断
- 系统性批量丢单的竞品/市场分析
- Pipeline Coverage不足的供给侧分析
- 团队效能指标持续下滑的根因定位

## 执行步骤

### 步骤1：问题定义与量化
1. 明确分析对象：具体是什么异常/偏差
2. 量化问题程度：偏差了多少、影响范围多大、持续多久
3. 确定问题边界：是全局性还是局部性、是一次性还是持续性
4. 设定分析目标：需要回答什么问题、支撑什么决策

### 步骤2：数据收集与事实梳理
1. 收集与问题相关的多维度数据：
   - 时间维度：问题什么时候开始的？之前是什么状态？
   - 主体维度：涉及哪些人/团队/客户/区域？
   - 变化维度：同期有什么其他变化发生？（人员变动/产品变化/竞品动作/市场环境）
2. 建立时间线：将关键事件按时间顺序排列
3. 区分事实和推测：标记哪些信息是确认的数据，哪些是推断

### 步骤3：假设生成
1. 基于数据和经验生成3-5个可能的根因假设
2. 常见假设维度：
   - 人员因素（能力/态度/资源不足）
   - 流程因素（流程缺失/流程不合理/执行不到位）
   - 产品因素（功能缺失/质量问题/定价问题）
   - 市场因素（竞品动作/客户需求变化/经济环境）
   - 数据因素（数据质量问题/统计方法问题）
3. 为每个假设标注初始概率估计

### 步骤4：假设验证
1. 对每个假设设计验证方法：
   - 数据验证：通过数据分析确认或排除
   - 对比验证：控制变量后看差异是否消失
   - 时间验证：假设原因出现的时点是否与问题时点吻合
   - 相关验证：是否能解释观察到的所有异常
2. 逐一执行验证，更新各假设的概率
3. 运用5-Why方法对最可能的原因继续深挖

### 步骤5：结论与建议输出
1. 确定最终根因（可能是多因素组合）
2. 评估各因素的贡献度（帕累托原则：聚焦关键少数）
3. 针对根因给出改善建议：
   - 短期行动（立即可执行的止血措施）
   - 中期优化（1-3个月内的流程/能力提升）
   - 长期改进（需要组织/产品/策略层面的系统性变革）
4. 为每条建议标注：责任人、时间节点、预期效果

## 输入规格

| 输入项 | 类型 | 必需 | 说明 |
|--------|------|------|------|
| 异常/问题描述 | Text | 是 | 需要分析的问题定义 |
| 相关数据集 | Dataset | 是 | 与问题相关的多维度数据 |
| 时间范围 | DateRange | 是 | 分析的时间窗口 |
| 历史基线 | Dataset | 是 | 正常状态下的参考数据 |
| 业务上下文 | Text | 否 | 已知的相关背景信息 |

## 输出规格

| 输出项 | 类型 | 说明 |
|--------|------|------|
| 问题定义 | Text | 清晰的问题陈述和量化 |
| 根因分析树 | Structure | 5-Why或鱼骨图形式的分析链 |
| 原因排序 | List | 按贡献度排序的原因列表 |
| 证据链 | List | 支撑结论的数据证据 |
| 改善建议 | List | 分短/中/长期的行动建议 |
| 预期效果 | Estimate | 改善后的预期指标提升 |

## 最佳实践

- **数据先行**：先看数据再推原因，避免"先有结论再找证据"的确认偏差
- **多因素思维**：销售问题通常是多因素叠加的结果，避免过度简化为单一原因
- **区分相关和因果**：两个指标同时变化不代表有因果关系，需要进一步验证
- **可证伪原则**：好的假设必须是可以被数据证伪的，不可证伪的假设没有分析价值
- **行动导向**：根因分析的价值在于驱动改善，如果找到的"原因"无法行动，需要继续深挖到可行动的层面

## 约束条件

- 分析结论必须有数据证据支撑，不接受纯主观推断
- 涉及个人绩效问题时，分析聚焦于行为模式而非个人品质判断
- 涉及竞品分析时，基于可获取的公开信息和客户反馈，不涉及非法情报获取
- 分析时间要求：P1类问题24小时内出初步分析，完整报告3个工作日内
- 不替代管理层做最终决策——提供分析支撑而非代行管理职能
