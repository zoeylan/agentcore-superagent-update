# 异常检测与预警

## 技能概述

本技能用于实时监控销售数据中的异常模式，通过统计方法和业务规则的组合检测机制，在数据异常发生的第一时间生成分级预警，帮助管理层和相关团队快速识别风险并采取行动。该技能是销售管理"早期预警系统"的核心能力，其价值在于将被动发现问题转变为主动防控风险。

## 适用场景

- 7×24小时实时数据异常监控
- 大单丢失的即时预警
- Pipeline突变的检测和影响评估
- 销售行为异常的识别（如数据造假信号）
- 系统性市场变化的早期发现（如竞品动作导致批量丢单）
- 数据录入错误的快速识别

## 执行步骤

### 步骤1：建立检测规则库
1. 金额类异常规则：
   - 商机金额单次变更>50%（可能是录入错误或需求重大变化）
   - 商机金额变更为0或极小值（可能是放弃前兆）
   - 新建商机金额远超同类型平均值（P99以上，可能虚报）
2. 阶段类异常规则：
   - 商机阶段跳跃（跳过中间阶段直接推进）
   - 商机阶段回退（从后期阶段退回前期）
   - 同一阶段停留超过平均值2倍标准差
3. Pipeline类异常规则：
   - 周环比Pipeline总量变化>20%
   - 某个阶段商机数量突然大幅增减
   - Pipeline Coverage跌破3倍阈值
4. 个人行为类异常规则：
   - 个人赢率异常偏离团队平均>2个标准差
   - 短时间内批量变更商机状态
   - 连续多周无新建商机

### 步骤2：实时数据监听
1. 监听CRM系统的数据变更事件流
2. 监听各Scope同步的数据更新
3. 对每条数据变更事件进行规则匹配
4. 计算滑动窗口统计指标（7天/30天移动平均）

### 步骤3：异常确认与分级
1. 规则触发后进行二次确认（排除已知的正常操作，如批量导入）
2. 严重性评级：
   - P1（紧急）：大单丢失（>季度Quota的10%）、Pipeline断崖式下跌（>30%）、系统性批量异常
   - P2（重要）：单指标偏离>2σ、阶段跳跃、金额异常变更
   - P3（关注）：轻微偏离、数据质量问题
3. 评估影响范围：影响到个人/团队/全公司

### 步骤4：预警报告生成
1. 按标准模板生成预警通知：
   - 【严重性】P1/P2/P3
   - 【异常类型】简明分类标签
   - 【异常描述】What happened（具体数据变化）
   - 【影响评估】Impact（对预测/目标的影响）
   - 【可能原因】Why（2-3种最可能的原因）
   - 【建议行动】Action（具体的响应建议）
2. P1预警15分钟内发送，P2预警1小时内发送

### 步骤5：追踪与闭环
1. 记录预警发送时间和接收方
2. 追踪预警响应状态（已确认/处理中/已解决/误报）
3. 对确认为误报的预警分析原因，优化规则
4. 统计预警准确率（真阳性率目标>80%）
5. 月度输出预警效果报告

## 输入规格

| 输入项 | 类型 | 必需 | 说明 |
|--------|------|------|------|
| 实时数据变更事件 | EventStream | 是 | CRM和各系统的数据变更流 |
| 检测规则配置 | Rules | 是 | 异常检测阈值和逻辑 |
| 历史统计基线 | Dataset | 是 | 用于计算标准差和移动平均 |
| 预警接收人配置 | Config | 是 | 各类预警的通知对象 |
| 白名单/排除规则 | Config | 否 | 已知正常操作的排除条件 |

## 输出规格

| 输出项 | 类型 | 说明 |
|--------|------|------|
| 预警通知 | Alert | 按标准模板的预警消息 |
| 影响评估 | Report | 异常对业务指标的影响量化 |
| 原因分析 | List | 可能原因的概率排序 |
| 建议行动 | List | 具体响应建议 |
| 月度预警统计 | Report | 预警数量、准确率、响应率 |

## 最佳实践

- **宁可多报不漏报**（初期）：新规则上线初期适当放宽阈值，通过实际运行逐步收紧以减少误报
- **分级响应**：不同级别预警有不同的响应SLA和升级路径，避免"狼来了"效应
- **上下文丰富**：预警不是简单的触发通知，必须附带足够的上下文信息让接收者能快速判断
- **规则迭代**：每月review一次规则有效性，根据误报/漏报情况调整阈值
- **白名单管理**：对于已知的周期性波动（如季度末冲刺、年初pipeline重建）设置排除规则

## 约束条件

- 预警误报率需控制在20%以下，超过时需立即优化规则
- P1预警必须在15分钟内发送，延迟发送需记录原因
- 不擅自修改检测规则阈值——规则变更需经过数据验证和管理层确认
- 预警只提供分析和建议，不执行任何修正操作
- 单日P1预警超过3条时需人工确认是否存在系统性问题
