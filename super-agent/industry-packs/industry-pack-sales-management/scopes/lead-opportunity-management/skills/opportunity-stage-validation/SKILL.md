# 商机阶段校验技能 (Opportunity Stage Validation)

## 概述

对商机在销售漏斗中的阶段推进进行合规性校验，确保每个阶段的进入/退出条件得到满足，防止阶段跳跃或在信息不完整时强行推进。这是保证Pipeline数据准确性和预测可靠性的关键控制点。

## 适用场景

- 商机从当前阶段推进到下一阶段时的准入校验
- 每周Pipeline Review前的批量阶段合理性检查
- 商机阶段回退时的原因记录和审批
- 新商机创建时的初始阶段校验
- 异常阶段跳跃的自动检测和告警

## 执行步骤

### 步骤1：定义阶段推进矩阵
标准B2B商机阶段及其进入条件：

| 阶段 | 进入必要条件 | 平均停留时长 |
|------|-------------|-------------|
| 初步接触 | SQL判定合格、基础信息完整 | 7天 |
| 需求确认 | 明确客户痛点记录、关键联系人建立 | 14天 |
| 方案演示 | Decision Criteria确认、Champion识别、痛点文档 | 21天 |
| 报价谈判 | 方案确认、Economic Buyer接触、预算范围确认 | 14天 |
| 合同签署 | 报价确认、商务条款达成一致 | 7天 |
| 赢单 | 合同签署完成、首款到账确认 | - |

### 步骤2：推进请求校验
当销售请求推进商机阶段时：
1. 识别目标阶段的进入条件清单
2. 逐项检查当前商机信息是否满足条件
3. 对于每项条件标注：已满足/未满足/部分满足
4. 综合判定：全部满足→允许推进/关键项未满足→阻止/非关键项未满足→允许但标注风险

### 步骤3：阶段跳跃检测
1. 识别非相邻阶段的推进请求（如从"初步接触"直接跳到"报价谈判"）
2. 阶段跳跃一律标记为异常，需销售经理审批
3. 审批通过后需补充中间阶段的关键信息
4. 记录跳跃原因（客户紧急/高层关系/特批等）

### 步骤4：停滞检测
1. 计算商机在当前阶段的停留天数
2. 对比该阶段的平均停留时长
3. 超过1.5倍平均时长 → 黄色预警
4. 超过2倍平均时长 → 红色预警
5. 预警附带可能原因分析和建议行动

### 步骤5：校验结果输出
1. 生成校验报告（通过/阻塞/警告）
2. 阻塞时明确列出未满足的条件和补充建议
3. 预警时提供停滞原因假设和破局建议
4. 记录校验日志用于流程合规审计

## 输入规格

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| opportunity_id | String | 是 | 商机唯一标识 |
| current_stage | String | 是 | 当前阶段 |
| target_stage | String | 是 | 目标阶段 |
| opportunity_data | Object | 是 | 商机完整数据（含MEDDIC字段） |
| validation_rules | Object | 否 | 自定义校验规则（覆盖默认规则） |

## 输出规格

| 字段 | 类型 | 说明 |
|------|------|------|
| validation_result | String | 校验结果（pass/blocked/warning） |
| conditions_met | Array | 已满足的条件列表 |
| conditions_unmet | Array | 未满足的条件及补充建议 |
| stagnation_alert | Object | 停滞预警信息（如适用） |
| recommendations | Array | 改善建议列表 |

## 最佳实践

1. **刚性与弹性结合**：核心条件（如需求确认→方案演示必须有痛点文档）为刚性要求，辅助条件（如Champion明确程度）可弹性处理
2. **预防性校验**：不仅在推进时校验，每周主动扫描全部商机识别潜在问题
3. **数据驱动阈值**：停滞预警的时间阈值应基于本公司历史数据动态调整，而非行业通用值
4. **教育性反馈**：阻塞时不仅说"不可以"，更要说"需要做什么才可以"——赋能销售而非阻碍
5. **例外记录**：对于审批通过的例外推进，必须记录审批人和原因，用于后续流程优化

## 约束条件

- 校验规则变更需经销售VP审批
- 阶段跳跃审批权限不可下放到一线销售经理以下
- 校验日志保留至少2年，用于合规审计
- 系统故障时不可绕过校验直接修改阶段（需人工审批流程兜底）
