# SLA监控技能 (SLA Monitoring)

## 概述

实时监控线索跟进的SLA（服务等级协议）执行情况，核心监控指标为4小时首次触达时效。通过预警、催办、升级和自动回收机制确保线索得到及时响应，防止线索因跟进延迟而冷却流失。

## 适用场景

- 新分配MQL线索的4小时首次触达SLA监控
- SDR跟进间隔的24-48小时节奏监控
- 48小时未跟进线索的自动回收触发
- SLA达标率的统计报告生成
- 超时事件的升级通知和闭环追踪

## 执行步骤

### 步骤1：SLA计时器启动
1. 线索分配完成后自动启动4小时倒计时
2. 记录分配时间、SLA截止时间、负责SDR
3. 将线索加入监控队列，按截止时间排序

### 步骤2：实时状态检查
每15分钟扫描一次监控队列：
1. 检查线索是否已记录首次触达（电话/邮件/微信）
2. 已触达 → 标记SLA达标，从监控队列移除
3. 未触达 → 计算剩余时间，进入预警判定

### 步骤3：分级预警机制
- **黄色预警**（剩余1小时）：发送提醒通知给SDR
- **橙色预警**（剩余30分钟）：发送紧急提醒给SDR + 抄送其直属主管
- **红色预警**（已超时）：发送超时通知给销售经理 + 标记SLA违规
- **自动升级**（超时2小时无响应）：触发线索重新分配流程

### 步骤4：48小时回收判定
1. 首次触达后，监控后续跟进节奏
2. 如48小时内无任何跟进记录（通话/邮件/微信/笔记）
3. 发送48小时回收预警给SDR（给予4小时缓冲）
4. 缓冲期后仍无跟进 → 自动回收线索至重新分配队列

### 步骤5：SLA报告生成
1. 每日汇总：当日SLA达标率、超时线索列表、平均响应时间
2. 每周汇总：周度趋势、各SDR达标率排名、超时原因分析
3. 异常模式识别：某SDR连续超时、某时段集中超时等

## 输入规格

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| lead_id | String | 是 | 线索唯一标识 |
| assigned_sdr | String | 是 | 被分配SDR标识 |
| assignment_time | DateTime | 是 | 分配时间 |
| sla_config | Object | 否 | SLA配置（默认4小时首触/48小时回收） |

## 输出规格

| 字段 | 类型 | 说明 |
|------|------|------|
| sla_status | String | SLA状态（on_track/warning/breached） |
| time_remaining | Number | 剩余时间（分钟） |
| alerts_sent | Array | 已发送的预警记录 |
| daily_report | Object | 每日SLA达标统计 |
| recycle_candidates | Array | 待回收线索列表 |

## 最佳实践

1. **时区感知**：SLA计时需考虑工作时间（9:00-18:00），非工作时间不计入SLA时长
2. **假期处理**：法定假日和SDR请假期间，自动将线索转移给backup SDR
3. **分级SLA**：VIP线索SLA可缩短为2小时，低优先级线索可放宽至8小时
4. **根因分析**：SLA违规不仅要处理当次事件，更要分析系统性原因（如分配不均、某时段线索暴增）
5. **正向激励**：SLA达标率纳入SDR绩效考核，连续达标给予正向反馈

## 约束条件

- SLA监控必须7×24运行，不可因系统维护中断
- 预警通知必须通过多渠道（系统消息+企业微信+邮件），确保送达
- 自动回收操作必须留有缓冲期，不可无预警直接回收
- SLA配置变更需销售运营负责人审批
