# 智能线索分配技能 (Intelligent Lead Routing)

## 概述

基于线索属性与SDR能力模型的多维匹配算法，实现线索到最优SDR的智能分配。核心目标是在保证分配速度（<30分钟）的同时最大化匹配质量，提升线索转化率。

## 适用场景

- MQL线索的首次自动分配
- 超时或回收线索的重新分配
- 新SDR入职后的线索池初始分配
- SDR离职/调岗时的线索批量转移
- 战略大客户线索的VIP分配

## 执行步骤

### 步骤1：线索特征提取
从线索数据中提取分配维度：
- 行业属性：所属行业大类和细分领域
- 地理区域：公司所在城市/省份
- 企业规模：员工数/营收规模/融资阶段
- 产品意向：感兴趣的产品线或解决方案
- 语言偏好：沟通语言（中/英/粤语等）
- 线索优先级：BANT评分高低/VIP标记

### 步骤2：SDR能力模型匹配
加载SDR能力画像进行匹配计算：
- 行业专长匹配度（历史该行业赢率加权）
- 区域覆盖匹配度（是否为该SDR覆盖区域）
- 产品认证匹配度（是否具备对应产品Demo能力）
- 当前负载评估（在手MQL数量/本周外呼完成率）
- 历史转化率（该SDR过去90天MQL→SQL转化率）

### 步骤3：匹配评分计算
```
匹配得分 = 行业匹配×30% + 区域匹配×25% + 产品匹配×20% + 负载系数×15% + 历史赢率×10%
```
- 当最高匹配得分>80%，直接分配
- 当最高匹配得分60-80%，分配但标注"次优匹配"
- 当最高匹配得分<60%，升级到销售经理手动分配

### 步骤4：负载均衡校验
1. 检查候选SDR当前在手MQL数量（上限20条）
2. 检查今日已分配数量（单日分配上限8条新MQL）
3. 如最优SDR已满载，选择次优SDR并记录原因
4. 如所有匹配SDR均满载，触发容量预警并升级处理

### 步骤5：分配执行与确认
1. 执行分配操作，更新CRM记录
2. 发送分配通知给SDR（含线索概要和建议首次触达策略）
3. 启动4小时首次触达SLA计时
4. 记录分配决策日志（线索ID/SDR/匹配得分/分配理由）

## 输入规格

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| lead | Object | 是 | 已评分的MQL线索完整数据 |
| available_sdrs | Array | 是 | 可用SDR列表及其能力模型 |
| routing_rules | Object | 否 | 自定义分配规则（如大客户指定SDR） |
| priority_override | String | 否 | 优先级覆盖（VIP/Normal） |

## 输出规格

| 字段 | 类型 | 说明 |
|------|------|------|
| assigned_sdr | Object | 被分配SDR信息 |
| match_score | Number | 匹配得分 |
| match_details | Object | 各维度匹配明细 |
| routing_reason | String | 分配理由说明 |
| sla_deadline | DateTime | SLA截止时间 |
| alternative_sdrs | Array | 备选SDR列表（如需重分配） |

## 最佳实践

1. **速度优先原则**：在匹配度差距<10%时，优先选择当前负载更低（能更快响应）的SDR
2. **连续性原则**：对于老客户新线索，优先分配给之前服务过该客户的SDR
3. **公平性兜底**：长时间未获得新线索的SDR应适当提高分配权重，避免"马太效应"
4. **AB测试**：定期对分配算法进行AB测试，对比不同权重组合对最终转化率的影响
5. **特殊规则优先**：VIP客户/战略客户的指定分配规则优先于算法匹配结果

## 约束条件

- 分配决策必须在30分钟内完成，不可阻塞线索流转
- 同一线索不可分配给同一SDR两次（如已回收则不可回分配给原SDR）
- 跨区域分配需销售经理确认（涉及业绩归属问题）
- VIP线索分配失败必须立即升级，不可进入普通排队
