# SQL资质评估技能 (SQL Qualification Assessment)

## 概述

在SDR完成3次有效沟通后，基于BANT标准综合评估线索是否达到SQL（Sales Qualified Lead）资质，决定是否创建正式商机。SQL评估是线索从Marketing ownership转移到Sales ownership的关键判定节点，直接影响销售团队的时间投入质量。

## 适用场景

- 3次有效沟通后的标准SQL评估
- 客户主动表达强购买意向时的快速评估（可提前于3次）
- 定期Nurture线索的重新评估（每月对nurture pool扫描）
- SQL标准的月度校准（对比实际商机转化数据）

## 执行步骤

### 步骤1：沟通记录汇总
1. 收集该线索的全部沟通记录（通话录音摘要/邮件往来/微信对话记录）
2. 提取关键信息：客户明确表达的内容（需求/预算/时间/人员）
3. 区分"客户明确表达"vs"SDR推断"的信息（前者权重更高）
4. 验证有效沟通次数是否≥3次（有效=双向交流超过2分钟）

### 步骤2：BANT逐项深度评估

**Budget（预算）评估**：
- 是否有明确的预算范围或采购计划？
- 预算与我方产品定价是否匹配（±30%范围内）？
- 预算来源是否明确（已批复/待申请/无预算概念）？
- 评估结论：明确/模糊/无预算

**Authority（决策权）评估**：
- 是否已识别最终决策人？
- 当前联系人在决策链中的角色？
- 是否了解完整决策流程（几轮评估/谁参与/审批层级）？
- 评估结论：决策人已接触/决策人已识别未接触/决策链不明

**Need（需求）评估**：
- 是否有明确的业务痛点或需求？
- 痛点与我方产品能力是否匹配？
- 需求的紧迫性如何（必须解决/最好有/nice-to-have）？
- 评估结论：强需求/一般需求/弱需求

**Timeline（时间线）评估**：
- 是否有明确的采购/上线时间计划？
- 是否有驱动时间线的外部因素（合规要求/合同到期/业务目标）？
- 当前处于评估流程的哪个阶段？
- 评估结论：3个月内/6个月内/无明确时间线

### 步骤3：SQL判定决策
综合四维评估结果做出判定：
- **合格SQL**：至少3个维度评估为正面（明确需求+预算范围+决策时间线+关键决策人识别），创建商机
- **条件SQL**：2个维度正面，其余维度有提升空间，可创建商机但标记"早期阶段"
- **暂不合格**：正面维度<2，但有长期价值，转入nurture计划
- **无效**：明确无需求或无购买可能，标记关闭

### 步骤4：商机创建（如合格）
如判定为SQL，生成商机创建信息：
1. 必填字段：客户名称、预计金额、预计日期、商机阶段（初步接触）、关键联系人
2. MEDDIC要素填写（至少3项）
3. 商机摘要和赢单策略初步建议
4. 后续行动计划（下一步做什么/谁来做/什么时候）

### 步骤5：评估报告输出
1. 生成SQL评估报告卡
2. 记录评估依据和判定理由
3. 对于暂不合格线索，输出nurture策略建议
4. 将评估结果同步给相关stakeholders

## 输入规格

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| lead_id | String | 是 | 线索唯一标识 |
| communication_records | Array | 是 | 全部沟通记录（≥3条有效记录） |
| lead_profile | Object | 是 | 线索画像和最新评分 |
| qualification_criteria | Object | 否 | SQL判定标准配置（可自定义阈值） |

## 输出规格

| 字段 | 类型 | 说明 |
|------|------|------|
| qualification_result | String | 判定结果（qualified/conditional/nurture/disqualified） |
| bant_assessment | Object | BANT四维评估详情 |
| opportunity_data | Object | 商机创建数据（如合格） |
| nurture_plan | Object | nurture计划（如暂不合格） |
| assessment_rationale | String | 判定综合理由 |
| confidence_level | String | 判定置信度 |

## 最佳实践

1. **证据驱动**：每个维度的评估必须基于客户明确表达的信息，而非SDR的主观乐观判断
2. **宁缺毋滥**：SQL标准宁可严格不可宽松——低质量SQL进入销售pipeline会浪费Account Executive的宝贵时间
3. **持续跟踪**：暂不合格不等于永久不合格，nurture pool的线索应每月重新评估
4. **校准闭环**：每月对比SQL判定与最终成单结果，识别过严/过松的维度并调整标准
5. **特殊情况处理**：对于高层直接推荐或战略客户，即使BANT不完全满足也可创建商机（标记特殊来源）

## 约束条件

- 有效沟通不足3次时不可做出最终SQL判定（紧急情况需主管审批）
- SQL评估结果必须经SDR主管review确认后方可创建商机
- 被判定为无效的线索进入6个月冷却期，期间不可重新激活
- 评估报告必须可溯源，记录每项判断的信息来源
