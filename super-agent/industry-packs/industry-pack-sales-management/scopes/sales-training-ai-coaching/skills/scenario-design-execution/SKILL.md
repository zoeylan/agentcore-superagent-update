# 场景设计与执行 (Scenario Design & Execution)

## 技能概述

本技能负责基于向量知识库中的真实客户画像、赢单/丢单案例和行业数据，动态生成高仿真的销售陪练场景，并管理场景执行的完整生命周期。核心价值在于确保陪练场景"像真的一样"——还原真实客户的行业背景、决策压力、沟通风格和关注重点。

## 适用场景

- 新人入职首周5大核心场景的初始化配置
- 基于丢单案例构建专项训练场景
- 根据学员能力画像动态推荐训练场景
- 竞品变化导致的场景紧急更新
- 进阶场景设计（多角色联合决策等复杂场景）

## 执行步骤

### Step 1: 场景需求分析
- 获取学员当前能力画像（从能力评估教练）
- 确定训练目标场景类型（开场白/需求挖掘/产品演示/异议处理/促单）
- 确定难度级别（Level 1-5）
- 检查是否有特定训练需求（如针对某行业客户的专项训练）

### Step 2: 场景要素配置
- 从知识库检索匹配的客户画像模板
- 设定客户角色参数：行业、职位、决策权限、性格类型、痛点假设、预算范围
- 设定场景背景参数：会议目的、已有接触历史、竞品现状、时间紧迫度
- 设定对话约束：时长限制（15-30分钟）、最大轮次、结束条件

### Step 3: 对话策略设计
- 定义客户的核心关注点和隐藏需求（需要学员通过提问才能发现）
- 预设2-3个典型异议的触发条件和表达方式
- 设定客户的防御级别和信任建立阈值
- 设计关键转折点（如客户突然提到竞品、预算被砍等）

### Step 4: 场景启动与执行
- 向学员展示场景briefing（客户基本信息、会议背景）
- 启动计时器和轮次计数器
- 进入角色扮演对话模式
- 实时记录对话元数据（时间戳、轮次、说听比）

### Step 5: 场景结束与数据输出
- 触发结束条件：时间到达/对话自然结束/学员请求结束/偏离主题超限
- 生成结构化对话记录包
- 标记关键转折点和评分参考点
- 将数据包传递给能力评估教练

## 输入规格

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| trainee_id | string | 是 | 学员唯一标识 |
| scenario_type | enum | 是 | opening/discovery/demo/objection/closing |
| difficulty_level | int(1-5) | 是 | 难度级别 |
| customer_persona | object | 否 | 自定义客户画像（为空则自动匹配） |
| time_limit_minutes | int | 否 | 时长限制，默认20分钟 |
| focus_skills | array | 否 | 重点考察的子技能列表 |

## 输出规格

| 字段 | 类型 | 说明 |
|------|------|------|
| session_id | string | 陪练会话唯一标识 |
| dialogue_log | array | 完整对话记录（含时间戳和角色标识） |
| scenario_config | object | 场景配置参数快照 |
| key_moments | array | 关键转折点标记（含位置和类型） |
| metrics_raw | object | 原始指标数据（说听比、轮次数、时长等） |
| termination_reason | string | 结束原因 |

## 最佳实践

1. **场景真实度优于难度**：宁可降低难度也不要让场景显得虚假——真实感是训练有效性的前提
2. **渐进式信息披露**：客户不会一开始就告诉你全部信息，需要学员通过提问逐步获取
3. **合理的不确定性**：真实客户不会按照剧本走，场景设计需要留有一定的随机性和意外因素
4. **避免过度设计**：每个场景聚焦1-2个核心训练目标，不要试图在一个场景中覆盖所有技能
5. **基于数据迭代**：通过分析学员表现数据不断优化场景设计——过简单（通过率>90%）的场景需要升级，过难（通过率<20%）的场景需要拆解

## 约束条件

- 每次场景执行必须在30分钟内完成
- 场景配置中不得包含真实客户的隐私信息（姓名、电话等需脱敏）
- 竞品相关信息必须基于公开资料，不得虚构竞品功能
- 场景剧本中不得引导学员做出违反商业道德的行为（如贬低竞品人格、利用客户隐私等）
