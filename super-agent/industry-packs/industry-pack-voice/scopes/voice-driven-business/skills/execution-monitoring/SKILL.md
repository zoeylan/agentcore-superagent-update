# 执行监控与告警 (Execution Monitoring & Alerting)

## 概述

本技能负责对所有业务事务的执行过程进行实时监控，跟踪关键指标（成功率、延迟、错误率），在异常发生时及时触发告警并启动应急预案。同时生成执行统计报告供运营决策使用。

## 适用场景

- 实时监控API调用成功率，发现下游服务异常
- 跟踪事务执行延迟，识别性能瓶颈
- 检测异常操作模式（如短时间大量退款请求）
- 生成业务操作的日报/周报统计

## 执行步骤

### Step 1: 指标采集
- 从每个事务执行记录中提取关键指标：
  - 执行时长（端到端、各阶段分别）
  - 执行结果（成功/失败/超时）
  - 错误类型和错误码分布
  - 业务类型分布
- 实时计算滑动窗口统计值（1分钟/5分钟/1小时）

### Step 2: 异常检测
- **成功率异常**：5分钟窗口内成功率低于95%触发黄色告警，低于90%触发红色告警
- **延迟异常**：P95延迟超过正常值2倍触发告警
- **错误聚集**：同一错误码短时间内出现5次以上触发告警
- **流量异常**：请求量突增/突降超过正常值3倍标准差触发告警
- **业务异常**：单用户短时间频繁操作（可能是攻击或系统Bug导致的重复请求）

### Step 3: 告警处理
- 根据告警级别选择通知渠道：
  - 黄色：记录日志 + 运维群消息
  - 红色：记录日志 + 运维群消息 + 短信/电话通知值班人员
  - 紧急：自动触发熔断 + 所有渠道通知
- 告警内容包含：时间、指标值、影响范围、建议操作
- 告警去重：同一问题5分钟内只发送一次通知

### Step 4: 应急预案触发
- 下游服务不可用：自动切换到降级模式（排队等待/拒绝新请求并告知用户）
- 资金类API异常：立即暂停所有资金操作，等待人工确认恢复
- 性能下降：自动扩容请求/限流保护

### Step 5: 报告生成
- 日报：当日事务总量、成功率、平均延迟、Top5错误原因、Top5操作类型
- 周报：趋势分析、同比环比、SLA达成情况、需优化事项
- 实时大盘：核心指标实时展示（供运营人员监控）

## 输入规格

```json
{
  "monitoring_scope": "all | transaction_type | specific_api",
  "time_window": "1m | 5m | 1h | 1d",
  "alert_rules": [
    {
      "metric": "success_rate",
      "threshold": 0.95,
      "operator": "less_than",
      "severity": "warning"
    }
  ]
}
```

## 输出规格

```json
{
  "monitoring_report": {
    "period": "2024-03-15T10:00:00Z/2024-03-15T11:00:00Z",
    "total_transactions": 1250,
    "success_rate": 0.986,
    "avg_latency_ms": 850,
    "p95_latency_ms": 2100,
    "error_distribution": {"TIMEOUT": 5, "API_ERROR": 8, "BUSINESS_RULE": 2},
    "alerts_triggered": 0
  },
  "active_alerts": [],
  "recommendations": ["建议关注支付网关P95延迟上升趋势"]
}
```

## 最佳实践

1. **分级告警**：避免告警疲劳，只有真正需要关注的问题才通知人工
2. **根因关联**：多个指标同时异常时，尝试关联分析找出根因而非逐个告警
3. **自愈优先**：能自动恢复的问题（如短暂网络抖动）自动处理，不必通知人工
4. **历史基线**：告警阈值基于历史数据动态调整，而非静态固定值
5. **事后复盘**：每次重大告警后生成事件报告，分析根因并更新防御策略

## 约束条件

- 监控系统本身的可用性必须>=99.99%（比业务系统更高）
- 告警延迟不超过30秒（从异常发生到告警发出）
- 监控数据存储保留期：明细数据7天，聚合数据90天
- 不得在监控过程中修改或干预事务执行流程
- 敏感业务数据（金额、用户信息）在监控日志中需脱敏处理
