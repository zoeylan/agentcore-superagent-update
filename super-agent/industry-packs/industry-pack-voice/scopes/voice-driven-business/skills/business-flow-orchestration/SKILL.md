# 业务流程编排 (Business Flow Orchestration)

## 概述

本技能负责管理语音驱动业务场景下的多步骤流程编排，将复杂的业务操作拆解为有序的对话步骤，动态管理流程状态、处理分支跳转和中断恢复。这是流程引导Agent的核心能力，确保即使是最复杂的业务操作也能通过自然语音对话顺利完成。

## 适用场景

- 航班改签：查询原订单→检查规则→搜索航班→计算差价→确认→执行
- 电商退货：确认商品→选择原因→确认退款方式→提交→物流安排
- 医疗预约：选择科室→选择医生→确认时间→身份验证→创建预约
- 银行转账：确认收款方→确认金额→身份验证→二次确认→执行转账

## 执行步骤

### Step 1: 流程模板匹配
- 根据业务意图匹配预定义的流程模板
- 加载模板定义：步骤序列、必填参数、确认检查点、异常分支
- 初始化流程实例（flow_instance_id）
- 设置流程超时计时器（默认5分钟）

### Step 2: 流程状态管理
- 维护流程状态机：
  - INITIATED → COLLECTING_PARAMS → VERIFYING → CONFIRMING → EXECUTING → COMPLETED/FAILED
- 每个步骤完成后持久化当前状态（支持中断恢复）
- 记录已收集的参数快照（支持用户回溯修改）

### Step 3: 步骤执行与引导
- 按模板顺序执行每个步骤
- 每步生成对应的对话引导话术
- 处理步骤内的分支逻辑（如根据改签规则决定是否允许免费改签）
- 步骤完成条件满足后自动推进到下一步

### Step 4: 流程变更处理
- 用户中途变更意图时：
  - 评估变更范围（同流程内参数修改 vs 切换到新流程）
  - 同流程内修改：回退到对应步骤，保留不受影响的参数
  - 切换新流程：保存当前流程状态（可恢复），启动新流程
- 超时处理：
  - 3分钟无交互：主动提醒用户
  - 5分钟无交互：暂停流程并通知用户可稍后恢复

### Step 5: 流程完成与清理
- 流程正常完成：生成操作摘要，清理临时状态
- 流程异常终止：记录失败原因，保留状态快照供分析
- 流程放弃：用户主动取消时，执行必要的资源释放

## 输入规格

```json
{
  "business_intent": "flight_change | order_return | medical_booking | bank_transfer",
  "initial_params": {
    "从用户首句话中提取的初始参数"
  },
  "user_context": {
    "user_id": "用户ID",
    "history_orders": ["相关历史订单"],
    "preferences": "用户偏好数据"
  },
  "resume_from": "flow_instance_id（中断恢复时提供）"
}
```

## 输出规格

```json
{
  "flow_instance_id": "FLOW-20240315-uuid",
  "current_step": "confirming",
  "next_action": {
    "type": "ask_user | call_agent | execute",
    "target": "身份验证Agent | 事务执行Agent",
    "dialogue_prompt": "需要播报给用户的话术",
    "required_input": "期望的用户回应类型"
  },
  "collected_params": {
    "所有已收集并确认的参数"
  },
  "flow_progress": "4/6 steps completed"
}
```

## 最佳实践

1. **渐进式信息收集**：不要一次性要求用户提供所有参数，按优先级逐步收集
2. **智能默认值**：基于用户历史行为和常见选择提供默认值，减少交互轮次
3. **快速路径**：为回头客提供"和上次一样"的快捷确认路径
4. **优雅降级**：当某个步骤的依赖服务不可用时，跳过或提供替代路径
5. **进度透明**：复杂流程中适时告知用户当前进度（"好的，基本信息已确认，接下来需要验证一下您的身份"）

## 约束条件

- 单个流程实例的最大生命周期为30分钟（含暂停时间）
- 同一用户同时只能有1个活跃流程（避免操作冲突）
- 流程状态持久化延迟不超过1秒（确保中断恢复的数据完整性）
- 流程模板变更需经过版本管理，旧版本流程实例不受新版本影响
- 所有流程变更（跳转/回退/取消）必须记录原因供事后分析
