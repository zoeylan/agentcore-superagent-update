# 确认协议管理 (Confirmation Protocol Management)

## 概述

本技能负责在语音驱动业务操作中执行严格的多级确认机制，确保每一个关键业务操作都经过用户明确授权。该技能是防止语音交互中因误识别、误理解导致错误操作的核心安全保障。关键设计原则是：确认必须严格但对话必须自然。

## 适用场景

- 订单类操作的关键信息复述确认（商品、数量、金额、时间）
- 资金类操作的二次确认（支付/转账/退款金额和对象）
- 不可逆操作的最终确认（订单取消、账户注销）
- 批量操作的逐项或汇总确认

## 执行步骤

### Step 1: 确认级别判定
- 根据操作类型和风险等级确定确认级别：
  - **L1-轻量确认**：查询类操作确认（简短复述+"对吗？"）
  - **L2-标准确认**：修改类操作（完整复述关键信息+等待"确认/对"）
  - **L3-严格确认**：资金类操作（完整复述+金额对象单独强调+等待"确认支付/确认转账"）
  - **L4-双重确认**：高风险操作（L3确认后，间隔2秒再次确认"涉及XX元操作，请再次说确认"）

### Step 2: 确认话术生成
- 组装确认内容（必须包含的三要素：金额/数量、时间/日期、对象/目标）
- 生成自然语言确认话术：
  - 避免机械式："请确认以下信息..." ❌
  - 使用对话式："好的，我帮您确认一下：..." ✓
  - 使用复述式："那就是说，您要把...改到...，需要补...元，对吗？" ✓
- 关键数字必须用清晰表述避免歧义（"一百二十元"而非"一百二"）

### Step 3: 用户响应解析
- 识别确认响应：
  - 肯定：对/是/确认/没问题/嗯/好的/可以 → CONFIRMED
  - 否定：不对/错了/不是/等等/慢着 → REJECTED
  - 修正：对但XX改成YY/不是XX是YY → PARTIALLY_CONFIRMED
  - 模糊：嗯...（犹豫语气）/算了吧 → UNCERTAIN
- 对UNCERTAIN响应进行二次确认（"您是要继续还是取消操作呢？"）

### Step 4: 确认结果处理
- CONFIRMED：生成确认凭证，准许进入下一步骤
- REJECTED：返回参数修改步骤，询问需要修改哪个部分
- PARTIALLY_CONFIRMED：更新对应参数后重新进入确认流程
- UNCERTAIN：最多追问2次，仍不明确则建议"要不我们先暂停，您考虑好了再操作？"

### Step 5: 确认记录归档
- 记录完整的确认过程（时间、确认内容、用户响应、结果）
- 确认音频片段保存（供争议时回溯）
- 生成确认凭证（confirmation_token）供事务执行阶段校验

## 输入规格

```json
{
  "operation_type": "payment | transfer | order_change | cancellation",
  "risk_level": "low | medium | high | critical",
  "confirmation_content": {
    "amount": 120.00,
    "currency": "CNY",
    "target": "3月17日CA1234航班",
    "action": "改签",
    "additional_info": "需补差价"
  },
  "user_response_audio": "用户响应的音频数据",
  "user_response_text": "ASR转写的用户响应文本"
}
```

## 输出规格

```json
{
  "confirmation_result": "CONFIRMED | REJECTED | PARTIALLY_CONFIRMED | UNCERTAIN",
  "confirmation_token": "CONF-20240315-uuid（仅CONFIRMED时生成）",
  "token_expiry": "2024-03-15T10:05:00Z（5分钟有效期）",
  "dialogue_response": "下一步应播报给用户的话术",
  "modification_request": {
    "field": "需修改的参数名",
    "original_value": "原值",
    "new_value": "用户要求的新值（如适用）"
  },
  "audit_record": {
    "timestamp": "2024-03-15T10:00:15Z",
    "content_confirmed": "完整确认内容",
    "user_response": "用户原始响应",
    "result": "CONFIRMED"
  }
}
```

## 最佳实践

1. **三要素原则**：关键操作确认必须包含金额+时间+对象三要素，缺一不可
2. **清晰数字表述**：金额必须用完整数字表述（"一百二十元"），避免口语简称
3. **间隔确认**：L4级确认中，两次确认之间至少间隔2秒，防止用户惯性确认
4. **语境适配**：确认话术的语气和详细程度根据用户特征调整（老年用户更详细更慢）
5. **不可跳过**：任何情况下不允许跳过确认步骤，即使系统负载高或对话时间已长

## 约束条件

- 确认token有效期5分钟，过期需重新确认
- L3及以上级别确认中，用户必须说出特定确认词（"确认/确认支付"），"嗯/对"不够格
- 确认过程中系统不得插入广告、推荐等非确认相关内容
- 确认音频必须保存至少90天
- 同一操作的确认不超过3次（3次REJECTED后建议转人工或终止）
- 二次确认跳过率必须为0%（系统层面强制，不依赖Agent自觉）
