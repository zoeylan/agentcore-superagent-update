# 安全审计日志 (Security Audit Logging)

## 概述

本技能负责记录语音驱动业务操作中所有安全相关事件的完整审计日志，支持异常模式检测、合规审计、争议回溯和风控分析。审计日志是安全体系的"黑匣子"，确保所有敏感操作都有完整的可追溯记录。

## 适用场景

- 身份验证过程的完整记录（每次尝试的详情）
- 业务操作执行的审计追踪（谁在何时做了什么）
- 异常安全事件的检测和告警（频繁失败、可疑行为模式）
- 合规审计报告的自动生成
- 用户争议处理时的操作回溯

## 执行步骤

### Step 1: 事件采集
- 接收来自各Agent的安全事件数据：
  - 身份验证事件：验证尝试、结果、置信度、方式
  - 操作执行事件：事务发起、确认、执行、结果
  - 异常事件：验证失败、操作回滚、系统错误
- 为每个事件添加标准化元数据：
  - 时间戳（精确到毫秒，UTC时区）
  - 事件类型（AUTH/OPERATION/ANOMALY/SYSTEM）
  - 严重级别（INFO/WARNING/ERROR/CRITICAL）
  - 关联ID（session_id, transaction_id, user_id）

### Step 2: 日志结构化与存储
- 将事件转化为标准审计日志格式
- 敏感信息脱敏处理：
  - 手机号：138****1234
  - 金额：保留（审计需要）
  - 声纹特征：不记录原始数据，仅记录比对结果
  - 口头密码：不记录任何内容，仅记录"验证通过/失败"
- 日志分级存储：
  - 热存储（7天）：全量详细日志，支持实时查询
  - 温存储（90天）：聚合摘要+关键事件详情
  - 冷存储（3年）：合规要求的最小保留集

### Step 3: 异常模式检测
- 实时分析日志流，检测以下异常模式：
  - **暴力攻击**：同一用户24h内验证失败>=3次
  - **账户扫描**：同一来源尝试验证多个不同用户
  - **声纹异常**：用户声纹突然大幅变化（可能被冒用）
  - **操作异常**：非常规时段的大额操作
  - **频率异常**：短时间内异常高频操作
- 检测到异常立即触发告警（30秒内）

### Step 4: 风控联动
- 异常模式确认后的自动响应：
  - 黄色告警：记录+通知安全运营团队
  - 红色告警：临时冻结可疑操作+通知安全运营+推送用户确认
  - 紧急告警：全面冻结账户+即时通知多方
- 向风控系统推送实时风险评分更新
- 维护用户安全信誉分（基于历史行为）

### Step 5: 审计报告生成
- 定期生成合规审计报告（日报/月报）
- 报告内容：
  - 验证统计：各方式的使用量、通过率、失败原因分布
  - 操作统计：操作类型分布、成功率、异常操作比例
  - 安全事件：告警数量、处理结果、未关闭事件
  - 趋势分析：与前期对比的变化趋势
- 支持按需生成特定用户/时段的操作轨迹报告（用于争议处理）

## 输入规格

```json
{
  "event_type": "AUTH_ATTEMPT | AUTH_RESULT | OPERATION_EXECUTE | OPERATION_RESULT | ANOMALY_DETECTED | SYSTEM_EVENT",
  "severity": "INFO | WARNING | ERROR | CRITICAL",
  "event_data": {
    "user_id": "user-123",
    "session_id": "sess-456",
    "transaction_id": "TXN-789",
    "action": "声纹验证",
    "result": "PASS",
    "details": {
      "confidence": 96.5,
      "verification_mode": "silent",
      "risk_level": "medium"
    }
  },
  "context": {
    "device_info": "设备信息",
    "network_info": "网络环境",
    "timestamp": "2024-03-15T10:00:00.123Z"
  }
}
```

## 输出规格

```json
{
  "log_id": "AUDIT-20240315-100000-uuid",
  "stored": true,
  "anomaly_detected": false,
  "alert_triggered": null,
  "risk_score_update": {
    "user_id": "user-123",
    "new_score": 85,
    "change": 0,
    "reason": "正常操作，信誉分不变"
  }
}
```

## 最佳实践

1. **不可篡改**：审计日志一旦写入不允许修改或删除（append-only）
2. **时钟同步**：所有参与系统必须使用NTP同步时钟，确保日志时序准确
3. **关联查询**：通过session_id/transaction_id可以串联一次完整操作的所有日志
4. **最小敏感**：日志中记录的信息应足够审计但不包含不必要的敏感数据
5. **独立存储**：审计日志应存储在独立于业务系统的安全存储中，防止业务侧篡改

## 约束条件

- 日志写入延迟不超过1秒（事件发生到日志可查询）
- 日志存储的可用性>=99.99%
- 审计日志保留期限符合行业监管要求（金融：至少5年；通用：至少3年）
- 日志访问需要独立的审计员权限，运维人员不可直接访问
- 异常检测的误报率应<5%（避免告警疲劳）
- 任何Agent不得关闭或绕过审计日志记录
