# 智能参数收集 (Intelligent Parameter Collection)

## 概述

本技能负责在语音对话中智能收集业务操作所需的参数，处理语音交互特有的挑战：同音词歧义、口误修正、模糊表达、省略表达等。核心目标是在最少的交互轮次内收集完整且准确的业务参数，同时确保用户体验自然流畅。

## 适用场景

- 从用户语音中提取日期、时间、金额、数量等结构化参数
- 处理"后天下午三点"等相对时间表达的绝对化转换
- 解决"一百二"(120) vs "一百零二"(102)等数字歧义
- 批量参数提取（用户一句话包含多个参数时的完整提取）
- 参数验证失败时的自然语言追问

## 执行步骤

### Step 1: 参数需求分析
- 从流程模板中获取当前步骤需要的参数列表
- 区分必填参数和可选参数
- 确定每个参数的数据类型和验证规则
- 检查是否有参数可从上下文或用户历史中自动填充

### Step 2: 语义提取
- 对用户语音的ASR转写文本进行NLU解析
- 提取实体（日期、时间、地点、人名、金额、商品名等）
- 处理特殊表达：
  - 相对时间："后天" → 转换为绝对日期
  - 模糊数量："几个" → 标记为待确认
  - 省略表达："还是那个" → 从上下文中解析指代
  - 口语化数字："一百二" → 120

### Step 3: 参数验证
- 格式校验：日期合法性、手机号位数、金额范围
- 业务规则校验：航班日期不能是过去、转账金额不超过日限额
- 逻辑一致性：出发日期不能晚于返回日期
- 验证失败时生成具体的纠正引导话术

### Step 4: 歧义消解与确认
- 检测潜在歧义（同音词：'式'vs'十'、'一'vs'意'）
- 对关键参数（金额、日期、数量）主动确认
- 确认话术设计：
  - 单参数："您是说3月15号对吗？"
  - 多参数批量："好的，3月17号下午3点，北京到上海，对吗？"
  - 金额类（必须复述）："确认金额是一百二十元，对吗？"

### Step 5: 缺失参数追问
- 识别流程继续所需但尚未获取的必填参数
- 生成自然的追问话术（非机械式"请提供XXX"）：
  - "请问您想改到哪一天呢？"
  - "好的，那您希望几点的航班？"
- 提供选项引导（当选项有限时）：
  - "下午有3点和5点两个航班，您选哪个？"
- 超过3轮仍无法获取某参数时，建议转人工

## 输入规格

```json
{
  "user_utterance": "帮我把明天下午三点北京到上海的机票改到后天",
  "required_params": [
    {"name": "original_flight_date", "type": "date", "required": true},
    {"name": "new_flight_date", "type": "date", "required": true},
    {"name": "departure", "type": "city", "required": true},
    {"name": "destination", "type": "city", "required": true},
    {"name": "preferred_time", "type": "time", "required": false}
  ],
  "context": {
    "current_date": "2024-03-15",
    "previously_collected": {},
    "user_history": "最近订单信息"
  }
}
```

## 输出规格

```json
{
  "extracted_params": {
    "original_flight_date": {"value": "2024-03-16", "confidence": 0.95},
    "new_flight_date": {"value": "2024-03-17", "confidence": 0.92},
    "departure": {"value": "北京", "confidence": 0.99},
    "destination": {"value": "上海", "confidence": 0.99},
    "preferred_time": {"value": "15:00", "confidence": 0.90}
  },
  "missing_params": [],
  "needs_confirmation": ["new_flight_date"],
  "confirmation_prompt": "您说的后天是3月17号周日对吗？",
  "ambiguities": []
}
```

## 最佳实践

1. **一次问一个**：避免一次追问多个参数导致用户记忆负担
2. **利用上下文**：能从历史或上文推断的参数不要重复询问
3. **确认而非追问**：当有高置信度猜测时，用确认式而非开放式提问
4. **数字必确认**：所有涉及金额、数量的数字参数必须明确确认
5. **容错设计**：用户说错时提供"没关系，那应该是…？"的友好纠正引导

## 约束条件

- 参数提取的整体置信度低于80%时，必须进行确认
- 金额类参数无论置信度多高都必须确认
- 单个参数的追问不超过3轮，超过建议转人工
- 参数收集全过程不超过2分钟（超时提醒用户）
- ASR转写结果的同音词歧义必须通过上下文消歧或主动确认解决
