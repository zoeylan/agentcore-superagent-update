# KPI指标计算 (KPI Calculation)

## 概述
统计和计算通话级别和系统级别的关键绩效指标（KPI），包括平均处理时长(AHT)、首次解决率(FCR)、客户满意度(CSAT)预测等，为运营决策提供数据支撑。

## 适用场景
- 每通通话结束后的KPI数据更新
- 实时运营大盘的指标计算
- 坐席/系统绩效评估
- 服务质量趋势分析
- 异常指标告警

## 执行步骤

### 1. 通话级指标计算

**平均处理时长(AHT)**：
- AHT = 通话时长 + 通话后处理时长
- 按业务类型基线对比：查询类基线90秒、操作类基线180秒、投诉类基线300秒
- 超过基线150%标记为"超时通话"

**首次解决率(FCR)预测**：
基于以下信号综合判断：
- 用户在通话结束时明确表示问题已解决 → FCR=Yes，置信度高
- 用户说"好的/谢谢/那没事了" → FCR=Likely Yes，置信度中
- 通话被转人工 → FCR=No
- 24小时内同一问题再次来电 → FCR=No（回溯修正）
- 通话中途挂断 → FCR=Likely No

**客户满意度(CSAT)预测**：
基于多信号预测1-5分：
- 情绪轨迹结束时的情绪值（权重0.3）
- 问题是否解决（权重0.4）
- 通话时长合理性（权重0.1）
- 服务态度评分（权重0.2）

### 2. 系统级指标聚合

**端到端响应延迟(P95)**：
- 从用户说完话到系统开始回复的时间
- 目标：P95<500ms
- 按链路分段统计：ASR延迟+意图识别延迟+知识检索延迟+TTS首包延迟

**意图识别准确率**：
- 定义：用户首次表达后系统正确理解意图的比例
- 计算：正确识别数/总识别尝试数
- 正确性标准：后续对话未出现"不是"、"我说的是"等否定纠正

**转人工率**：
- 转人工通话数/总通话数
- 目标：<15%
- 按转人工原因分类：情绪升级/意图不明/系统故障/用户主动要求

**自然打断成功率**：
- 成功响应打断数/总打断次数
- 成功标准：100ms内停止播报+正确理解打断内容

### 3. 指标异常检测
对各项KPI设置动态阈值：
- 基于最近7天均值±2倍标准差
- 指标突破阈值立即告警
- 持续偏移（趋势性恶化）也需要告警

### 4. 报表输出
按不同粒度输出KPI报表：
- 实时：每5分钟刷新的运营大盘
- 日报：每日汇总+环比变化
- 周报：趋势分析+改进建议
- 月报：月度KPI达标情况+下月目标

## 输入规格
| 字段 | 类型 | 说明 |
|------|------|------|
| call_record | object | 通话完整记录 |
| quality_score | object | 质检评分结果 |
| emotion_summary | object | 情绪摘要 |
| system_metrics | object | 系统性能指标（延迟等） |
| historical_baseline | object | 历史基线数据 |

## 输出规格
| 字段 | 类型 | 说明 |
|------|------|------|
| call_kpis | object | 本通通话KPI指标 |
| fcr_prediction | object | FCR预测（值+置信度） |
| csat_prediction | object | CSAT预测（值+置信度） |
| anomaly_alerts | array | 异常指标告警 |
| dashboard_update | object | 大盘更新数据 |

## 最佳实践
- FCR需要24小时后回溯修正（用户可能次日再来电），初始值为预测值
- CSAT预测模型定期用真实满意度调查数据校准
- 指标异常时自动关联可能原因（如ASR准确率下降导致意图识别率下降）
- 避免单一指标优化导致其他指标恶化（如缩短AHT可能降低FCR）

## 约束
- KPI计算延迟<5秒（通话结束到指标更新）
- CSAT预测与实际评分相关系数>=0.75
- FCR 24小时回溯修正率（预测vs实际差异）<10%
