# 知识版本管理 (Knowledge Versioning)

## 概述
管理业务知识库的多版本生命周期，包括知识条目的创建、更新、过期和废弃。确保在任何时间点检索到的知识都是当前有效的最新版本，同时支持政策变更过渡期的新旧版本并存。

## 适用场景
- 促销活动FAQ的定时生效和过期
- 政策变更时新旧规则的过渡期管理
- 产品更新后文档的版本切换
- 季节性/周期性知识的激活与归档

## 执行步骤

### 1. 知识条目生命周期管理
每条知识条目包含版本元数据：
```json
{
  "knowledge_id": "KB-001",
  "version": "2.1",
  "status": "active",  // draft/active/deprecated/archived
  "effective_from": "2024-01-01T00:00:00Z",
  "effective_until": "2024-06-30T23:59:59Z",
  "superseded_by": "KB-001-v3",
  "change_reason": "2024年新政策生效",
  "business_line": "insurance"
}
```

### 2. 时效性检查
每次检索时自动校验：
- 当前时间是否在有效期内
- 条目状态是否为active
- 是否存在更新版本（superseded_by字段）
- 标记即将过期的条目（7天内过期）给出预警

### 3. 过渡期管理
政策变更场景下的双版本并存策略：
- 新政策生效日前：返回旧版本
- 生效日起过渡期内（通常30天）：同时返回新旧版本，标明适用条件
  - "如果您的业务办理在X日前，适用旧规则..."
  - "X日后办理的，适用新规则..."
- 过渡期结束：旧版本自动归档，仅返回新版本

### 4. 临时知识管理
促销活动、限时政策等临时知识：
- 设置精确的生效和失效时间
- 失效后自动从检索索引中移除
- 保留在归档库中用于事后审计
- 活动结束后如用户仍问相关问题，返回"该活动已结束"提示

### 5. 版本冲突解决
当同一问题存在多个版本答案时：
- 优先返回最新active版本
- 过渡期内标注版本差异
- 多个并行active版本（如不同地区政策）通过业务线/地域标签区分

## 输入规格
| 字段 | 类型 | 说明 |
|------|------|------|
| knowledge_id | string | 知识条目ID |
| operation | enum | CHECK_VALIDITY/GET_LATEST/LIST_VERSIONS/TRANSITION_STATUS |
| current_time | datetime | 当前时间 |
| business_context | object | 业务上下文（地区、用户类型等） |

## 输出规格
| 字段 | 类型 | 说明 |
|------|------|------|
| valid_version | object | 当前有效版本 |
| version_status | enum | CURRENT/TRANSITION/EXPIRED/UPCOMING |
| transition_info | object | 过渡期信息（如适用） |
| expiry_warning | bool | 是否即将过期 |

## 最佳实践
- 版本变更时同步更新向量索引，避免旧版本向量继续参与检索
- 临时知识设置自动过期提醒，防止"僵尸知识"
- 高频检索的知识条目变更需要人工复核确认后才切换
- 版本变更记录完整保留审计轨迹

## 约束
- 版本检查增加的延迟<10ms
- 支持同一知识条目最多保留10个历史版本
- 过渡期管理支持最长90天
