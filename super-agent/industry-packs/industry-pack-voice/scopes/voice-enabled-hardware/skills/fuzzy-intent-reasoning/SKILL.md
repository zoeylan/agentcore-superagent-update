# 模糊意图推理 (Fuzzy Intent Reasoning)

## 概述

模糊意图推理技能处理用户非明确、主观性的语音表达，将其转化为具体的设备操作。当用户说"太暗了""有点冷""不舒服"这类描述感受而非指定操作的指令时，系统需要结合环境传感器数据、用户历史偏好、当前时间和场景等多维信息，推理出用户最可能期望的操作。

## 适用场景

- 感受类表达："太暗了""好冷""太吵了""闷得慌"
- 状态类表达："想睡觉了""准备出门""在看电影"
- 请求类模糊表达："搞舒服一点""调一下""弄好一点"
- 否定类模糊表达："不太对""不是这样""不舒服"
- 多义表达消歧："冷了"→关窗?开空调?加热?

## 执行步骤

### Step 1: 模糊意图分类
1. 接收解析置信度<0.7的意图
2. 分类模糊意图类型：
   - 感知类（冷/热/暗/亮/吵/闷）→环境调节
   - 活动类（睡觉/工作/运动/看电影）→场景匹配
   - 否定类（不对/不好/不舒服）→纠正上一操作
   - 抽象类（舒服一点/好一点）→综合环境优化

### Step 2: 环境数据采集
1. 获取当前环境传感器数据：
   - 温度/湿度/光照度/噪声级别/空气质量/CO2浓度
2. 获取当前时间和日期（工作日/周末、早/午/晚）
3. 获取用户当前位置（哪个房间）
4. 获取相关设备当前状态

### Step 3: 偏好数据加载
1. 向个性化学习Agent请求当前用户+当前时段+当前场景下的偏好参数
2. 加载用户对类似模糊表达的历史操作记录（最近30天）
3. 加载环境舒适度阈值（温度22-26°、湿度40-60%、光照200-500lux）

### Step 4: 推理决策
1. 将感知类表达映射为环境参数差异：
   - "太暗了"→当前光照 < 用户偏好阈值→需要增加光照
   - "好冷"→当前温度 < 用户舒适阈值→需要升温
2. 确定具体操作：
   - 优先选择最直接的设备（"暗了"→先尝试调高当前房间灯光亮度，而非开启其他灯）
   - 参数值参考用户偏好（有历史记录用历史值，无则用环境舒适度标准值）
3. 活动类映射到预设/学习到的场景模板
4. 否定类根据last_action执行逆向操作或小幅调整

### Step 5: 置信度评估与输出
1. 评估推理结果的合理性：
   - 与环境数据一致→高置信度(0.8+)
   - 有用户历史支撑→中高置信度(0.7-0.8)
   - 纯推理无依据→低置信度(0.5-0.7)
2. 置信度>=0.75→直接执行
3. 置信度0.6-0.75→执行但语音确认（"为您调亮客厅灯到80%"）
4. 置信度<0.6→追问（"您是想调亮灯光还是打开窗帘？"）

## 输入规格

```json
{
  "fuzzy_expression": "太暗了",
  "fuzzy_type": "perception|activity|negation|abstract",
  "environment": {
    "temperature": 24.5,
    "humidity": 55,
    "illuminance_lux": 120,
    "noise_db": 35,
    "time": "2024-01-15T21:30:00",
    "room": "living_room"
  },
  "user_preferences": {
    "illuminance_comfort": {"min": 200, "max": 500, "preferred": 350},
    "last_similar_action": {"action": "set_brightness", "value": 80}
  },
  "available_devices": []
}
```

## 输出规格

```json
{
  "reasoning_result": {
    "inferred_intent": "increase_illuminance",
    "reasoning_chain": "当前光照120lux < 用户偏好最低阈值200lux，用户表达'太暗了'确认光照不足",
    "recommended_action": {
      "device_id": "light_01",
      "action": "set_brightness",
      "parameters": {"brightness": 80},
      "alternative_actions": [
        {"device_id": "curtain_01", "action": "open", "reason": "白天可拉开窗帘"}
      ]
    },
    "confidence": 0.88,
    "confidence_factors": {
      "environment_match": 0.95,
      "user_history_match": 0.85,
      "expression_clarity": 0.8
    },
    "action_mode": "execute|confirm|ask"
  }
}
```

## 最佳实践

- 推理链（reasoning_chain）必须记录，用于事后分析推理质量和模型优化
- 提供备选操作（alternative_actions），当主推荐被用户否决时可快速提供替代方案
- 时间因素很重要："太暗了"在白天可能意味着开窗帘，在晚上意味着开灯
- 对于多义表达，优先选择"最小干预"的操作（能调亮就不开新灯，能开窗帘就不开灯）
- 累积用户对推理结果的反馈（采纳/否决/修正），持续优化推理模型

## 约束条件

- 推理总耗时<200ms（含环境数据获取和偏好查询）
- 追问最多1次，不能连续追问导致体验割裂
- 推理结果不能涉及安全设备的主动操作（不能因为"不安全"就自动锁门）
- 推理操作的参数值必须在设备允许范围内
- 对于同一模糊表达，在环境不变的情况下应产生一致的推理结果（可重复性）
