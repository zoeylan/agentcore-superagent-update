# 数据洞察与决策智能 — 标准操作规程 (SOP)

## 1. 目的与范围

本SOP定义了数据洞察与决策智能Scope的标准化工作流程，覆盖数据源健康管理、竞品监测、归因分析、爆款预测、报告交付和自然语言查询服务六大核心业务环节。目标是确保营销数据分析的及时性、准确性和可执行性，为营销决策提供可靠的数据智能支撑。

---

## 2. RACI矩阵

|----------|:---:|:---:|:---:|:---:|:---:|
| SOP-1 数据源健康检查 | I | I | R | I | A |

## 3. SOP-1：数据源健康检查

### 触发条件
- 每日定时触发：08:00 自动执行全量数据源健康检查
- 事件触发：任何数据源连续2小时无数据到达

### 执行步骤

1. **API连接状态检测**
   - 逐一验证各广告平台API连接：Google Ads、Meta Ads、TikTok/巨量引擎、快手磁力引擎
   - 验证网站分析数据源：GA4、自建埋点系统
   - 验证CRM系统数据接口
   - 验证社交媒体数据源连接

2. **数据完整性校验**
   - 对比当日数据量与近7天均值，偏差>30%标记异常
   - 检查关键字段空值率（user_id、timestamp、event_type），空值率>5%标记异常
   - 验证数据时间连续性，检测是否存在时间段缺失

3. **数据时效性检查**
   - 各数据源最新数据的时间戳 vs 当前时间
   - 延迟>2小时 → 黄色告警
   - 延迟>4小时 → 红色告警 + 自动切换备用数据路径（如有）

4. **健康报告输出**
   - 生成数据健康度评分（0-100分）
   - 异常项清单及影响评估
   - 推送至相关负责人

### 输出物
- 每日数据健康报告（含评分和异常清单）
- 异常告警通知（如触发）

### 异常处理
- 单一数据源中断：标记受影响时段，切换备用数据路径，告警数据工程团队
- 多数据源同时中断：触发紧急响应流程，暂停依赖该数据的分析任务，升级处理
- 数据质量严重退化（健康度<60分）：暂停基于该数据源的报告输出，标记为"数据不可靠"

### KPI指标
- 数据源可用性：>99.5%
- 异常发现时效：数据中断后30分钟内告警
- 健康检查覆盖率：100%数据源每日检查

## 4. SOP-2：竞品监测

### 触发条件
- 每日定时触发：09:00 执行Top5竞品日常监测
- 每周定时触发：周一09:00 执行Top10竞品全面监测
- 事件触发：行业重大事件（如双11、618大促前一周加密监测频率）

### 执行步骤

1. **日常监测（Top5竞品）**
   - 扫描竞品广告透明度库（Meta Ad Library、TikTok Creative Center等）
   - 记录新增/下架的广告素材（数量、风格、渠道）
   - 监测竞品社交媒体官方账号动态（发布内容、互动数据）
   - 扫描竞品电商旗舰店（价格、促销标签、新品上架）

2. **周度监测（Top10竞品）**
   - 执行日常监测所有项目
   - 补充分析：竞品投放渠道组合变化
   - 补充分析：竞品受众定向策略推断（基于广告展示场景）
   - 补充分析：竞品内容策略方向变化（话题、风格、合作KOL）

3. **变化评级与处理**
   - L1（常规）：记录归档，纳入数据库
   - L2（值得关注）：纳入周报分析模块
   - L3（重大事件）：24小时内生成即时告警 + 影响评估 + 建议应对

4. **情报归档**
   - 所有监测结果结构化入库
   - 更新竞品"当前状态快照"
   - 维护竞品策略时间线

### 输出物
- 每日竞品动态简报（Top5）
- 每周竞品深度分析（Top10，纳入周报）
- L3级即时告警及应对建议

### 异常处理
- 监测源不可用（如平台关闭透明度接口）：标记数据缺失，启用备选数据源
- 竞品数量级策略变化（如突然大幅度缩减投放）：立即启动专题分析，24小时内输出分析报告

### KPI指标
- 竞品重大动态发现及时性：24小时内识别
- 监测覆盖完整度：Top5每日100%覆盖，Top10每周100%覆盖
- L3级告警准确率：>80%（避免过多误报）

## 5. SOP-3：归因模型校准

### 触发条件
- 每月定时触发：每月1日启动归因模型校准分析
- 事件触发：营销渠道组合发生重大变化（新增/撤出主要渠道）
- 事件触发：归因结果与实际业务感知严重偏离

### 执行步骤

1. **多模型并行计算**
   - 使用上月全量转化路径数据
   - 并行运行6种归因模型（首次/末次/线性/时间衰减/位置/数据驱动）
   - 输出各模型下各渠道的贡献度分配

2. **模型对比分析**
   - 生成模型对比矩阵：各渠道在不同模型下的贡献差异
   - 标记"归因敏感渠道"（模型间贡献差异>30%）
   - 分析差异原因（该渠道在路径中的位置分布特征）

3. **模型选择与校准**
   - 评估当前主模型的适用性：
     - 对比模型建议的预算分配 vs 上月实际执行后的效果变化
     - 如果模型建议方向与实际效果一致 → 维持当前主模型
     - 如果出现系统性偏差 → 评估切换为更适配的模型
   - 校准数据驱动模型参数（如Markov链转移概率更新）

4. **结果确认与文档化**
   - 输出月度归因校准报告
   - 记录模型变更决策和原因
   - 更新归因结果并通知下游消费者（投放策略Agent）

### 输出物
- 月度归因模型校准报告
- 模型选择决策记录
- 更新后的归因结果分发

### 异常处理
- 数据驱动模型训练样本不足（<1000条路径）：降级使用规则类模型（位置归因），标记"模型精度受限"
- 跨设备匹配率骤降：排查原因（Cookie政策变化？），评估对归因准确性的影响

### KPI指标
- 归因模型覆盖率：≥95%的转化路径可被归因
- 模型校准按时执行率：100%（每月1日）
- 归因建议有效性：执行归因建议后ROAS改善>5%

## 6. SOP-4：爆款预测验证

### 触发条件
- 每月定时触发：每月15日执行上月预测结果回溯验证
- 事件触发：连续两月准确率<60%触发模型重训练

### 执行步骤

1. **预测结果回溯**
   - 提取上月所有爆款预测记录（含预测对象、置信度评分、预测时间）
   - 收集实际市场表现数据（搜索量变化、销量数据、社交热度）
   - 判定预测结果：预测为爆款(>60分)且实际爆发 → 命中；预测为爆款但未爆发 → 假阳性；未预测但实际爆发 → 假阴性

2. **准确率计算**
   - 整体准确率 = 命中数 / 总预测数
   - 分品类准确率（识别薄弱品类）
   - 分置信度区间准确率（验证评分校准度）

3. **误判案例分析**
   - 假阳性分析：过度依赖了哪些信号？是否存在信号噪音源？
   - 假阴性分析：遗漏了哪些早期信号？信号采集覆盖面是否足够？
   - 提取改进方向和模型调优建议

4. **决策与行动**
   - 准确率≥70%：模型健康，记录验证结果，微调信号权重
   - 准确率60-70%：标记为"需关注"，分析退化原因，小幅调参
   - 准确率<60%（连续两月）：触发模型重训练流程
     - 更新训练数据集（纳入最近3月数据）
     - 重新评估特征工程（是否需要新增信号源）
     - 重训练完成后进行A/B验证

### 输出物
- 月度预测验证报告（含准确率、误判分析、改进建议）
- 模型重训练决策（如触发）

### 异常处理
- 验证数据不完整（某些预测对象缺乏实际表现数据）：基于可获得数据计算，标注覆盖率
- 外部黑天鹅事件导致预测全面失准（如突发政策变化）：标记为"非模型问题"，不计入准确率

### KPI指标
- 爆款预测准确率：≥70%
- 验证报告按时交付率：100%（每月15日）
- 模型迭代响应时间：准确率<60%后2周内完成重训练

## 7. SOP-5：报告交付

### 触发条件
- 周报：每周日22:00开始自动生成，周一10:00前交付
- 月报：每月2日开始自动生成，每月5日前交付
- 专题报告：按需求触发，确认需求后3个工作日内交付

### 执行步骤

1. **数据准备（T-1日/T-3日）**
   - 确认报告期数据完整性（周报：上周全7天；月报：上月全部天数）
   - 标注数据质量问题和缺失时段
   - 从各Agent获取分析结果：竞品动态、归因分析、趋势洞察

2. **报告草稿生成**
   - 按标准三段式结构（数据摘要→关键洞察→建议行动项）生成初稿
   - 洞察数量控制：周报3-5条，月报5-8条
   - 每条建议必须包含：具体操作、执行主体、预期收益、优先级

3. **质量审核**
   - 数据准确性校验：抽查核心数字与原始数据一致性
   - 逻辑一致性校验：洞察结论与数据支撑是否逻辑自洽
   - 可执行性校验：每条建议是否明确到可直接执行的粒度
   - 格式规范性校验：标题、图表标注、数据来源脚注完整

4. **交付与确认**
   - 按预定渠道推送报告
   - 等待接收确认
   - 记录反馈意见，纳入下期改进

### 输出物
- 周度市场趋势报告（每周一10:00前）
- 月度ROI分析报告（每月5日前）
- 专题分析报告（按需）

### 异常处理
- 数据不完整导致报告无法按时生成：在截止时间前2小时通知相关方，说明延迟原因和预计交付时间
- 报告结论与上期建议矛盾：在报告中主动说明原因（市场环境变化/数据修正/模型更新）

### KPI指标
- 周报/月报按时交付率：100%
- 报告质量满意度（接收方评分）：≥8/10
- 建议采纳率：≥60%

## 8. SOP-6：自然语言查询服务

### 触发条件
- 实时触发：接收到用户的自然语言数据查询请求
- 持续运行：7×24小时在线

### 执行步骤

1. **查询接收与解析**
   - 接收自然语言查询
   - 意图识别：确定查询类型（单值/对比/排名/趋势/原因）
   - 实体提取：时间范围、指标、维度、筛选条件
   - 歧义检测：如存在歧义，在3秒内反馈确认问题

2. **查询执行**
   - 确定数据源和查询路径
   - 执行结构化查询
   - 计时监控：
     - <10秒：正常返回完整结果
     - 10-15秒：返回"正在分析中"中间状态
     - >30秒：返回部分可用结果+完整结果预计时间

3. **结果返回**
   - 按查询类型适配返回格式
   - 自动补充上下文信息（环比/同比、行业基准、历史位置）
   - 附带2-3个追问建议
   - 标注数据来源和时间范围

4. **质量追踪**
   - 记录每次查询的响应时间
   - 记录查询是否成功解析和回答
   - 用户反馈采集（是否有用）

### 输出物
- 结构化数据回答
- 上下文补充信息
- 追问引导建议

### 异常处理
- 查询无法解析：返回"我不确定您的问题，您是想问…？"并给出2-3个最接近的猜测
- 数据源不可用：告知用户"该数据暂时不可用，预计恢复时间为…"，并提供替代数据
- 超时未返回：30秒后返回"查询超时"，建议简化查询条件或拆分查询

### KPI指标
- 响应时间：<10秒（简单查询），<30秒（复杂查询）
- 查询失败率：<5%
- 查询成功解析率：>95%

## 9. 决策树

```
[数据采集与清洗]
    │
    ├── 数据源健康？
    │   ├── YES → 继续正常流程
    │   └── NO → 触发SOP-1异常处理
    │           ├── 单一数据源中断 → 切换备用+告警
    │           └── 多源中断 → 紧急响应+暂停分析
    │
    ├── [分支A：竞品监测 - SOP-2]
    │   ├── 发现竞品变化？
    │   │   ├── L1常规 → 归档
    │   │   ├── L2关注 → 纳入周报
    │   │   └── L3重大 → 24h即时告警
    │   └── 竞品变化影响自身策略？
    │       ├── YES → 推送策略调整建议至投放Scope
    │       └── NO → 记录观察
    │
    ├── [分支B：归因分析 - SOP-3]
    │   ├── 归因结果与当前预算匹配？
    │   │   ├── 匹配（偏差<15%）→ 维持当前分配
    │   │   └── 不匹配（偏差>15%）→ 生成预算调整建议
    │   │       └── 推送投放Scope执行
    │   └── 发现数据异常？
    │       ├── YES → 触发异常检测流程
    │       └── NO → 正常输出归因报告
    │
    ├── [分支C：爆款预测 - SOP-4]
    │   ├── 预测模型准确率≥70%？
    │   │   ├── YES → 正常运行，输出预测结果
    │   │   └── NO → 连续两月<60%？
    │   │       ├── YES → 触发模型重训练
    │   │       └── NO → 微调参数，持续观察
    │   └── 识别到高潜力趋势(>80分)？
    │       ├── YES → 生成创意方向建议 → 推送创意Scope
    │       └── NO → 加入观察清单
    │
    ├── [实时响应：查询服务 - SOP-6]
    │   ├── 查询可解析？
    │   │   ├── YES → 执行查询 → 返回结果
    │   │   └── NO → 返回猜测选项让用户确认
    │   └── 10秒内返回？
    │       ├── YES → 正常交付
    │       └── NO → 返回中间状态 → 后台继续计算
    │
    └── [定时报告：SOP-5]
        ├── 数据完整性OK？
        │   ├── YES → 生成报告 → 质量审核 → 交付
        │   └── NO → 标注数据局限 → 延迟通知 → 补全后交付
        └── 按时交付？
            ├── YES → 完成
            └── NO → 通知相关方 + 记录延迟原因
```
