# 数据异常检测与根因分析 (Anomaly Detection)

## 概述

数据异常检测技能用于自动识别营销数据中的异常模式，包括数据缺失、跟踪代码失效、作弊流量、指标突变等问题。快速定位异常根因并评估其对分析结论的影响范围，确保数据驱动决策建立在可靠的数据基础上。

## 适用场景

- 每日数据质量巡检：识别数据缺失、延迟、格式错误
- 实时异常告警：关键指标突变（CTR/CVR/消耗速度异常波动）
- 作弊流量识别：点击劫持、归因欺诈、机器人流量
- 跟踪代码健康检查：UTM参数丢失、Pixel触发异常、SDK版本过期
- 异常影响评估：量化数据异常对归因和ROI分析的偏差影响

## 执行步骤

### 步骤1：数据完整性检查
- **数据到达检查**：各数据源是否按预期频率到达（延迟>2小时触发告警）
- **数据量级校验**：对比历史同期数据量，偏离>30%标记异常（如某渠道数据量突降50%）
- **字段完整性**：关键字段（user_id, timestamp, event_type, source, medium）空值率监控
- **格式一致性**：检测时间戳格式混乱、数值类型字段出现非数字值等

### 步骤2：统计异常检测
- **单指标异常**：基于历史均值±3σ（或IQR方法）识别指标异常值
- **比率异常**：CTR/CVR/跳出率等比率指标偏离历史基线>30%
- **趋势异常**：时间序列中的突变点检测（如ROAS从3.0骤降到1.5）
- **关联异常**：本应相关的指标出现背离（如点击大涨但转化不变 → 可能作弊）

### 步骤3：根因分析
- **跟踪代码问题**：检查对应时段的代码部署变更记录、SDK版本更新
- **平台侧变化**：查询广告平台是否有算法更新、政策调整、系统故障公告
- **竞争环境变化**：核查是否有竞品集中投放导致CPC上升
- **作弊行为特征**：IP集中度、设备重复率、点击时间分布（深夜异常高频）、转化后行为缺失
- **内部操作失误**：投放设置变更（出价误调、受众误操作）、落地页故障

### 步骤4：影响评估与处理建议
- 量化异常数据的影响范围：受影响的时间段、渠道、数据条数
- 评估对当期归因分析和ROI计算的偏差：如"该异常导致某渠道ROAS被高估约15%"
- 生成处理建议：
  - 数据缺失 → 标记时段、使用历史均值补全或排除该时段
  - 跟踪失效 → 紧急修复通知+受影响数据隔离
  - 作弊流量 → 标记并排除、向平台申请退款、更新反作弊规则
  - 指标突变 → 确认是真实变化还是数据问题，排除数据问题后升级为业务告警

## 输入规格

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| data_streams | Array | 是 | 待检测的数据流（含数据源、时间范围） |
| baseline_period | String | 否 | 基线对比周期（默认最近30天） |
| sensitivity | Enum | 否 | 检测灵敏度（high/medium/low，默认medium） |
| metrics_to_monitor | Array | 否 | 指定监控的指标（默认全部核心指标） |
| known_changes | Array | 否 | 已知变更事件（用于排除已知原因的"异常"） |

## 输出规格

| 字段 | 类型 | 说明 |
|------|------|------|
| anomalies | Array | 检测到的异常列表 |
| anomalies[].type | Enum | 异常类型（data_missing/tracking_failure/fraud/metric_spike/metric_drop） |
| anomalies[].severity | Enum | 严重程度（critical/warning/info） |
| anomalies[].affected_scope | Object | 影响范围（时间段、渠道、数据量） |
| anomalies[].root_cause | String | 根因判断 |
| anomalies[].root_cause_confidence | Number | 根因判断置信度（0-1） |
| anomalies[].impact_on_analysis | String | 对分析结论的影响评估 |
| anomalies[].recommended_action | String | 建议处理措施 |
| data_health_score | Number | 整体数据健康度评分（0-100） |
| integrity_report | Object | 数据完整性报告 |

## 最佳实践

1. **分层告警**：Critical级（数据源中断、大规模作弊）立即通知，Warning级进入日报，Info级归档记录
2. **已知变更排除**：维护"变更日历"（投放调整、网站改版、平台更新），避免已知变更被误报为异常
3. **趋势结合点值**：单日异常可能是波动，连续3天趋势性偏离才确认为真实异常
4. **影响量化优先**：先告知"这个异常会导致什么后果"，再分析"为什么发生"
5. **自动修复与人工确认**：低风险异常自动处理（如补全缺失值），高风险异常必须人工确认后执行

## 约束条件

- 数据延迟>2小时即触发告警，不等待自动恢复
- 作弊判定需要多个特征同时满足（≥3个特征），避免误判正常流量
- 根因分析结果必须标注置信度，低置信度（<0.6）的判断标记为"待确认"
- 不得自动删除或修改原始数据，只能标记和隔离
- Critical级异常30分钟内必须完成初步根因分析并推送告警
