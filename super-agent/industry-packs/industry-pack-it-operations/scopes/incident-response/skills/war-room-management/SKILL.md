# War Room管理 (War Room Management)

## 技能概述

War Room管理技能负责在P0/P1故障确认后快速创建应急作战室（War Room），拉通所有必要的参与方，建立高效的信息共享和协作机制。War Room是故障应急响应期间的核心协作枢纽，所有关键信息、决策和操作都在此集中进行。

## 使用场景

- P0/P1故障确认后立即创建War Room
- 故障升级时扩展War Room参与人员
- 故障恢复后归档War Room记录
- 大促保障期间预创建备用War Room

## 执行步骤

### 步骤1：创建War Room
- 在即时通讯平台（如飞书/钉钉/企业微信）创建临时群组
- 命名规范：`[P0/P1] 故障简述 - YYYYMMDD-HHmm`
- 设置群公告：故障摘要、当前状态、指挥官、各角色分工
- 关联故障工单号和监控大盘链接

### 步骤2：拉通参与人员
- 必须参与方（根据故障类型确定）：
  - 故障指挥官（Incident Commander）
  - 受影响服务的OnCall工程师
  - 相关领域专家（DBA/网络/安全，根据故障类型）
  - 对应业务线的技术负责人
- 信息知悉方：
  - 相关VP/总监（仅接收通报，不参与排查）
  - 客服团队负责人（用于同步用户影响）
  - PR/公关（如有舆情风险）

### 步骤3：建立协作规范
- 发布War Room协作规则：
  - 所有排查进展在群内同步（避免私聊导致信息不对称）
  - 重要发现和决策使用`@所有人`确保全员知悉
  - 使用约定格式：`[进展]`、`[发现]`、`[决策]`、`[求助]`前缀
  - 非相关讨论禁止在War Room中进行

### 步骤4：信息看板维护
- 实时维护故障状态看板（置顶消息）：
  - 当前状态：排查中/恢复中/已恢复
  - 故障影响：实时影响面数据
  - 当前排查方向和负责人
  - 待确认事项清单
  - 下次通报时间

### 步骤5：归档与关闭
- 故障恢复确认后，发布最终通报
- 导出War Room全部聊天记录（用于复盘）
- 标记War Room为"已归档"
- 生成War Room活动摘要（参与人数、持续时间、关键决策点）

## 输入规格

```json
{
  "incident": {
    "id": "故障ID",
    "severity": "P0|P1",
    "summary": "故障摘要",
    "affected_services": ["服务列表"],
    "initial_impact": "初始影响描述"
  },
  "participants": {
    "commander": "指挥官",
    "oncall_engineers": ["值班工程师"],
    "domain_experts": ["领域专家"],
    "stakeholders": ["利益相关方"]
  },
  "platform": "feishu|dingtalk|wecom"
}
```

```json

    "name": "群组名称",

```

## 最佳实践

1. **5分钟内创建**：P0故障确认后5分钟内War Room必须上线，预模板化可加速
2. **精准拉人**：避免拉入过多无关人员导致信息噪声，但宁多勿少
3. **信息结构化**：使用统一格式前缀，便于后续信息检索和时间线梳理
4. **定期刷新看板**：每15-30分钟更新置顶状态看板，新加入者可快速了解全貌

## 约束条件

- War Room中的所有信息视为内部机密，不得外传
- War Room创建后不可删除，只能归档（保留完整记录）
- 每个故障事件只有一个活跃War Room，避免信息分散
- 归档的War Room记录至少保留90天
