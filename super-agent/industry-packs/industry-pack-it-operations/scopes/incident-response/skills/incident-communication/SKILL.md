# 故障通报 (Incident Communication)

## 技能概述

故障通报技能负责在故障响应全生命周期中，按照规定频率和格式向管理层、利益相关方和相关团队同步故障进展。高质量的通报确保信息透明、减少焦虑、避免重复询问，让所有人对故障状态有一致的认知。

## 使用场景

- P0/P1故障启动后的首次通报
- 每30分钟（P0）/每1小时（P1）的定时进展通报
- 关键状态变更时的即时通报（如发现根因、开始恢复、恢复成功）
- 故障恢复后的最终通报
- 面向不同受众的差异化通报（技术团队 vs 管理层 vs 客服）

## 执行步骤

### 步骤1：确定通报类型和受众
- 首次通报：全员（技术+管理+业务）
- 进展通报：技术团队（详细）+ 管理层（摘要）
- 恢复通报：全员
- 确定通报渠道：War Room（技术详情）、管理群（摘要）、邮件（正式记录）

### 步骤2：收集通报素材
- 从故障时间线获取最新节点
- 从排查团队获取当前进展和下一步计划
- 获取实时业务影响数据（用户影响面、核心指标状态）
- 确认预计恢复时间（如可评估）

### 步骤3：生成通报内容
- 管理层通报格式（简洁版）：
  ```
  【P0故障进展通报 #序号】
  时间：YYYY-MM-DD HH:mm
  状态：排查中/恢复中/已恢复
  影响：影响XX万用户，核心交易成功率下降至XX%
  进展：已定位根因为XXX/正在执行XXX恢复方案
  预计：预计X分钟内恢复/暂无法评估
  下次通报：HH:mm
  ```
- 技术团队通报格式（详细版）：额外包含排查细节、已排除方向、技术指标数据

### 步骤4：发送通报
- 按照确定的渠道和受众发送
- 管理层通报需抄送指定的邮件列表
- 在War Room中同步通报已发送的确认
- 标记时间线：通报发送节点

### 步骤5：通报质量管控
- 确保通报间隔不超过约定频率（P0每30分钟）
- 通报内容需要有实质性进展更新（避免"仍在排查中"的空洞通报）
- 如无实质进展，需明确说明"已尝试XX方向无果，正在转向YY方向"
- 预计恢复时间的承诺要保守（宁晚勿早）

## 输入规格

```json
{
  "incident": {
    "id": "故障ID",
    "severity": "P0|P1",
    "summary": "故障摘要",
    "started_at": "故障开始时间",
    "duration_minutes": 45
  },
  "communication_type": "initial|progress|status_change|resolution",
  "audience": "management|technical|all",
  "current_status": {
    "phase": "investigating|identified|recovering|resolved",
    "impact_metrics": {"affected_users": 100000, "success_rate": "85%"},

    "next_steps": "下一步计划",
    "eta_minutes": 30
  },
  "timeline_updates": ["最新时间线条目"]
}
```

```json
{

    "type": "通报类型",

    "content_technical": "技术团队版本内容",

}
```

## 最佳实践

1. **首报速度**：首次通报在故障确认后10分钟内发出，即使信息不完整
2. **坏消息不隐藏**：如实通报影响面，不缩小不夸大
3. **避免技术黑话**：管理层通报使用业务语言，避免过多技术术语
4. **主动预期管理**：明确告知下次通报时间，减少焦虑和追问

## 约束条件

- P0故障通报间隔不超过30分钟
- P1故障通报间隔不超过1小时
- 通报中不得包含未经确认的猜测（可包含"疑似"但需标注）
- 对外通报（如客户公告）需要PR/法务审核后发出
- 所有通报必须存档，用于复盘评估通报质量
