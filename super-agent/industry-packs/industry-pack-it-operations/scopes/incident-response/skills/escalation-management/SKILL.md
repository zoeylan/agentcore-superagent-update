# 升级管理 (Escalation Management)

## 技能概述

升级管理技能负责在故障响应超时或排查受阻时，按照预定义的升级链路将故障升级至更高级别的决策者或专家。升级管理是确保故障不会因为"在错误的层级停留过久"而延误恢复的关键机制。

## 使用场景

- 故障响应超时未恢复时的自动升级
- OnCall人员未确认响应时的通知升级
- 技术排查受阻需要更高级别专家介入
- 恢复决策需要更高权限审批（如全站切流、数据回滚）
- 业务影响超预期需要管理层介入

## 执行步骤

### 步骤1：升级触发条件检测
- 时间触发：
  - P0：响应超5分钟 → 升级OnCall；排查超30分钟 → 升级至L2；恢复超2小时 → 升级至VP/CTO
  - P1：响应超15分钟 → 升级OnCall；恢复超2小时 → 升级至L2/L3
- 状态触发：
  - 排查无方向（>20分钟无新线索）→ 引入更多专家
  - 恢复方案需要高权限操作 → 升级审批
  - 影响面扩大（定级升高）→ 同步升级响应级别

### 步骤2：确定升级目标
- 技术升级链路：L1值班工程师 → L2领域专家/TL → L3架构师/总监
- 管理升级链路：TL → 部门总监 → VP → CTO
- 选择依据：技术受阻走技术链路，决策需要走管理链路，通常并行

### 步骤3：准备升级材料
- 升级通报必须包含：
  - 故障时间线摘要（关键节点）
  - 当前状态和已尝试的方案
  - 升级原因（超时/受阻/需要决策/影响扩大）
  - 需要被升级者做什么（具体的求助内容）
  - 当前的恢复选项和各选项风险

### 步骤4：执行升级通知
- 通知被升级者（电话+IM双通道）
- 邀请加入War Room
- 更新War Room状态看板中的升级记录
- 在时间线中标注升级节点

### 步骤5：升级后跟进
- 确认被升级者已响应并介入
- 更新角色分工（如需要）
- 持续监控是否需要进一步升级
- 故障恢复后评估升级是否及时、合理

## 输入规格

```json
{
  "incident": {
    "id": "故障ID",
    "severity": "P0|P1|P2",
    "started_at": "故障开始时间",
    "current_duration_minutes": 45,
    "current_status": "investigating|recovering|blocked"
  },
  "escalation_trigger": "timeout|blocked|impact_expansion|approval_needed",
  "current_responders": ["当前响应人员"],
  "escalation_chain": {
    "L1": "当前值班",
    "L2": "领域专家",
    "L3": "架构师/总监",
    "management": ["VP", "CTO"]
  },
  "context_summary": "当前排查进展摘要"
}
```

```json
  "escalated_to": ["被升级通知的人员"],
  "escalation_reason": "升级原因",

  "acknowledged_at": "确认时间",
  "escalation_brief": "升级通报内容摘要"
}
```

## 最佳实践

1. **自动+人工结合**：超时自动触发升级，但排查受阻的判断需要人工触发
2. **升级≠问责**：升级是正常机制，建立"升级是负责任的行为"的文化
3. **升级通报要精练**：被升级者通常刚介入，需要最精简的上下文快速进入状态
4. **并行升级**：技术升级和管理升级可以同时进行，不需要串行等待

## 约束条件

- 升级通知必须在触发后1分钟内发出
- 不可跳级升级（除非下一级明确不可达）
- 每次升级必须记录在故障时间线中
- 夜间P0升级不受免打扰设置限制（直接电话）
- 升级决策一旦执行不可撤回（但可以在介入后判断降级处理）
