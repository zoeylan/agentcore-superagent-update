# 变更关联分析 (Change Correlation Analysis)

## 技能概述

变更关联分析技能用于在故障发生后快速检索近期变更记录，分析故障与变更之间的因果关系。业界统计60-80%的生产故障由变更引起，因此变更回溯是根因分析的第一优先级排查方向，能够在最短时间内锁定（或排除）最可能的根因。

## 使用场景

- 故障发生后第一时间进行变更回溯
- 判断故障是否由近期变更引起
- 为回滚决策提供依据
- 故障恢复后确认根因变更用于复盘

## 执行步骤

### 步骤1：变更记录检索
- 检索故障前24小时内的所有变更记录：
  - 代码发布（应用部署、版本升级、热修复）
  - 配置变更（参数调整、开关切换、路由规则变更）
  - 数据库变更（DDL执行、索引调整、数据迁移）
  - 基础设施变更（网络调整、扩缩容、硬件更换）
  - 安全变更（ACL调整、证书更新）
- 按时间倒序排列，重点关注故障前2小时内的变更

### 步骤2：时间相关性分析
- 计算各变更执行时间与故障首次告警时间的时间差
- 标注时间窗口内的高可疑变更（变更完成后5-30分钟内出现故障）
- 考虑延迟效应：某些变更（如内存泄漏）可能数小时后才表现
- 排除故障后的变更（时序不符的排除）

### 步骤3：影响面匹配分析
- 对比变更影响的服务/组件与故障涉及的服务/组件
- 若变更影响的服务与故障服务存在上下游依赖关系，标记为"间接相关"
- 若变更直接修改了故障服务的代码/配置，标记为"高度可疑"
- 检查变更内容与故障现象的逻辑相关性

### 步骤4：变更内容分析
- 对高可疑变更进行内容详查：
  - 代码变更：分析diff内容，识别高风险修改（并发逻辑、资源管理、超时配置）
  - 配置变更：对比变更前后值，评估变更幅度是否异常
  - 数据库变更：分析SQL内容，评估对性能的潜在影响
- 检查变更是否有灰度验证、变更成功率如何

### 步骤5：输出关联分析结论
- 明确关联结论：确认关联/高度可疑/可能相关/排除关联
- 若确认变更引起故障，给出回滚建议（含回滚方案和预计时间）
- 若排除变更因素，说明排除理由，转向其他排查方向

## 输入规格

```json
{
  "incident": {
    "first_alert_time": "首次告警时间",
    "affected_services": ["故障涉及服务"],
    "symptoms": ["故障现象描述"],
    "affected_metrics": ["异常指标"]
  },
  "change_records": [
    {
      "change_id": "变更ID",
      "type": "code_deploy|config_change|db_change|infra_change",
      "executed_at": "执行时间",
      "completed_at": "完成时间",
      "target_services": ["变更目标服务"],
      "executor": "执行人",
      "description": "变更描述",
      "rollback_plan": "回滚方案",
      "change_diff": "变更详情/diff"
    }
  ],
  "lookback_hours": 24
}
```

```json
{
  "correlation_result": "confirmed|highly_suspected|possibly_related|excluded",
  "confidence": "high|medium|low",
  "suspected_changes": [
    {
      "change_id": "变更ID",

      "time_gap_minutes": 15,

```

## 最佳实践

1. **时间窗口适配**：大部分变更引发的故障在30分钟内暴露，但内存泄漏类可延迟数小时
2. **不要忽视间接影响**：变更A的服务与故障服务无直接关系，但可能通过共享资源（如数据库连接池）间接影响
3. **灰度发布的特殊性**：灰度中的变更可能只影响部分用户，需对照灰度比例和影响面
4. **多变更并存**：若时间窗口内存在多个变更，需分析哪个更可能是根因（而非简单选最近的）

## 约束条件

- 变更回溯分析必须在5分钟内完成初步结论
- 分析必须基于事实和数据，不能仅凭直觉
- 若变更管理系统不可用，需要通知并转向其他排查手段
- 回滚建议需要评估回滚本身的风险（如回滚数据库DDL可能导致不兼容）
