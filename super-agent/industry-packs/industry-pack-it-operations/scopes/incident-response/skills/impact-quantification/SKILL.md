# 影响面量化 (Impact Quantification)

## 技能概述

影响面量化技能负责将故障的业务影响从定性描述转化为可度量的数字，包括影响用户数、订单损失、GMV损失、SLA违反情况等维度。精确的影响面量化是复盘报告的核心数据支撑，也是故障定级合理性验证和改进措施优先级排序的关键依据。

## 使用场景

- 故障复盘报告中的影响面章节
- 故障进行中的实时影响评估（辅助定级决策）
- 年度/季度故障影响汇总统计
- 改进措施的ROI评估（投入产出对比）

## 执行步骤

### 步骤1：确定影响时间窗口
- 故障开始时间：首次出现业务影响的时刻（可能早于首条告警）
- 故障结束时间：业务指标完全恢复正常的时刻（可能晚于"恢复操作"完成时间）
- 考虑长尾影响：如缓存重建期间的性能劣化、用户重试导致的体验下降
- 总影响时长 = 结束时间 - 开始时间

### 步骤2：用户影响量化
- 直接受影响用户数：在影响时段内实际遇到问题的独立用户数
- 潜在受影响用户数：在影响时段内访问了受影响功能的用户数
- 影响比例：受影响用户数 / 同时段总活跃用户数 × 100%
- 分地域/分客户端统计（如仅Android受影响则需分端计算）

### 步骤3：业务影响量化
- 交易影响：
  - 失败订单数 = 影响时段失败数 - 同期基线失败数
  - 交易成功率下降幅度：从基线XX% 下降至 YY%
  - GMV损失估算：失败订单数 × 平均客单价
- 功能影响：
  - 不可用的功能列表及其业务价值
  - 降级运行期间的体验损失度
- 间接影响：
  - 用户流失风险（如用户转向竞品）
  - 品牌信誉影响（如有舆情）
  - 赔偿/违约金（如有SLA合同）

### 步骤4：SLA违反评估
- 对照服务等级协议检查：
  - 可用性SLA：本月可用时间 / 总时间，是否跌破承诺值
  - 响应时间SLA：故障期间响应时间是否超出承诺值
  - 内部SLA：故障响应时效是否满足P0/P1的内部标准
- 计算SLA信用额度消耗（如99.9% SLA年度仅允许8.76小时不可用）

### 步骤5：输出量化报告
- 以数据表格形式呈现各维度影响
- 提供同比/环比参照（与历史同类故障对比）
- 标注数据来源和计算方法
- 给出影响等级评定（与定级是否吻合）

## 输入规格

```json
{
  "incident": {
    "start_time": "故障开始时间",
    "end_time": "故障恢复时间",
    "affected_services": ["受影响服务"],
    "severity": "P0|P1|P2|P3"
  },
  "metrics_data": {
    "affected_users": "受影响用户数",
    "total_active_users": "同时段总活跃用户数",
    "failed_transactions": "失败交易数",
    "baseline_failure_rate": "基线失败率",
    "average_order_value": "平均客单价"
  },
  "sla_config": {
    "availability_target": 99.9,
    "response_time_target_ms": 500,
    "monthly_budget_minutes": 43.2
  }
}
```

## 输出规格

```json
{
  "impact_summary": {
    "duration_minutes": 37,
    "affected_users": 150000,
    "user_impact_ratio": "12.5%",
    "failed_transactions": 8500,
    "estimated_gmv_loss": 4250000,
    "sla_violated": true,
    "sla_budget_consumed_minutes": 37
  },
  "detailed_breakdown": {
    "by_time": [{"period": "14:30-14:45", "error_rate": "45%", "affected_users": 80000}],
    "by_region": [{"region": "华东", "impact_ratio": "20%"}],
    "by_function": [{"function": "下单", "status": "完全不可用"}]
  },
  "comparison": {
    "vs_similar_incidents": "与历史同类故障对比",
    "severity_validation": "实际影响与定级是否匹配"
  },
  "data_sources": ["数据来源说明"],
  "confidence": "high|medium|low"
}
```

## 最佳实践

1. **多维度量化**：不只看用户数，还要看业务价值（GMV/转化率）
2. **考虑长尾效应**：故障恢复后仍可能有残留影响（如缓存重建/用户重试/SEO权重）
3. **保守估算**：当精确数据不可得时，取影响范围的上限估算
4. **数据可追溯**：每个数字都要能追溯到具体的数据源和计算公式

## 约束条件

- 影响面数据必须来自可信数据源（监控系统/业务数据库），不接受纯估算
- 涉及金额的损失计算需要财务确认（复盘报告中使用估算值需标注"待财务确认"）
- 用户数等敏感数据在对外通报中需脱敏处理
- 量化结果用于改进决策，不得用于个人绩效惩罚
