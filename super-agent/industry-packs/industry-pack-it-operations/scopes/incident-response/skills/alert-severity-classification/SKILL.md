# 告警定级 (Alert Severity Classification)

## 技能概述

告警定级技能负责根据告警的影响面、业务关键性和紧迫度，将聚合后的故障事件准确分类为P0-P3四个等级。准确的定级直接决定了后续响应流程的启动级别、参与人员范围和时间约束，是故障响应链路中最关键的决策节点之一。

## 使用场景

- 聚合告警事件的自动定级
- 故障升级/降级时的重新评估
- 定级争议时的辅助判断
- 定级标准的持续校准

## 执行步骤

### 步骤1：影响面评估
- 确定受影响的用户规模：全量用户 / >10%用户 / <10%用户 / 个别用户
- 确定受影响的功能类型：核心交易链路 / 核心功能 / 非核心功能 / 辅助功能
- 确定数据安全影响：数据丢失 / 数据不一致 / 无数据影响
- 确定是否有舆情风险：已出现社交媒体传播 / 可能引发传播 / 无风险

### 步骤2：定级矩阵匹配

| 等级 | 用户影响 | 功能影响 | 数据影响 | 典型场景 |
|------|----------|----------|----------|----------|
| P0 | 全量或>50% | 核心交易中断 | 数据丢失/泄露 | 全站不可用、支付链路中断、数据库主库宕机 |
| P1 | >10%用户 | 核心功能不可用 | 数据不一致 | 登录失败、搜索不可用、单机房故障 |
| P2 | <10%用户 | 非核心功能异常 | 无 | 个别页面报错、非核心接口超时、边缘节点异常 |
| P3 | 个别用户 | 辅助功能 | 无 | 单机故障已自动容灾、间歇性异常、性能轻微劣化 |

### 步骤3：加权因子调整
- 时间因子：业务高峰期（如电商大促、早晚高峰）自动升一级
- 趋势因子：问题在快速恶化（影响面持续扩大）自动升一级
- 累积因子：同一服务7天内第3次告警，考虑升级
- 业务日历：重大营销活动期间，核心链路告警默认按P1处理

### 步骤4：置信度评估
- 高置信度：多个独立信号源确认，影响面明确可量化
- 中置信度：部分信号确认，影响面通过推断评估
- 低置信度：信息不完整，需要进一步确认

### 步骤5：输出定级结果
- 明确最终定级（P0/P1/P2/P3）
- 列出定级依据（每个维度的评估结果）
- 标注置信度和可能需要升降级的条件
- 触发对应级别的响应流程

  "aggregated_event": {

    "first_trigger_time": "首次触发时间"

## 最佳实践

1. **宁高勿低**：在信息不完整时优先按更高级别响应，后续可以降级，但P0漏报不可接受
2. **多信号交叉验证**：不仅依赖告警，还要综合用户反馈、客服工单、业务指标等多源信号
3. **定级≠不变**：随着排查深入，影响面可能扩大或缩小，定级应动态调整
4. **标准统一**：全公司必须使用统一的定级标准，避免各团队标准不一导致的响应混乱

## 约束条件

- 定级决策必须在接收到聚合告警后2分钟内完成
- P0定级决策自动触发应急响应流程，不可拦截
- 降级操作需要明确说明原因并记录在时间线中
- 定级依据必须可追溯，供复盘评估响应合理性
