# 告警路由 (Alert Routing)

## 技能概述

告警路由技能负责将定级后的故障事件准确分发给对应的OnCall人员或团队，确保正确的人在最短时间内获知故障信息并开始响应。路由策略需要考虑OnCall排班、技能匹配、人员可用性和升级链路。

## 使用场景

- 告警定级后的自动路由分发
- OnCall人员未响应时的自动升级
- 跨团队故障的多路通知
- 紧急情况下的全员广播

## 执行步骤

### 步骤1：确定路由目标
- 根据告警涉及的服务，查询服务归属团队（Service Owner）
- 查询当前OnCall排班表，确定值班人员（L1）
- 若涉及多个团队，确定主责团队和协助团队

### 步骤2：确定通知方式
- P0：电话+短信+即时通讯（IM）+邮件，同时创建War Room群组
- P1：电话+即时通讯+邮件
- P2：即时通讯+邮件
- P3：即时通讯或工单系统

### 步骤3：构建通知内容
- 包含：故障等级、故障摘要、影响面、当前状态、期望动作、响应确认链接
- 内容简洁明确，一屏可读完
- 附带快捷操作入口（确认响应、查看详情、加入War Room）

### 步骤4：执行通知并跟踪确认
- 发送通知并启动确认计时器
- P0：5分钟内未确认→自动升级至L2（领域专家）
- P1：15分钟内未确认→自动升级
- 记录通知发送时间和确认时间

### 步骤5：升级链路处理
- L1（值班工程师）→ L2（领域专家/Team Lead）→ L3（架构师/总监）
- 每级未响应按超时时间自动升级
- P0超过30分钟未恢复，自动通知至VP级别
- P0超过2小时未恢复，自动通知至CTO

## 输入规格

```json
{
  "incident": {
    "severity": "P0|P1|P2|P3",
    "summary": "故障摘要",
    "affected_services": ["服务列表"],
    "impact": "影响描述"
  },
  "oncall_schedule": {
    "primary": "主OnCall人员",
    "secondary": "备份OnCall",
    "escalation_chain": ["L1", "L2", "L3"]
  },
  "notification_preferences": {
    "channels": ["phone", "sms", "im", "email"],
    "language": "zh-CN"
  }
}
```

## 输出规格

```json
{
  "notifications_sent": [
    {
      "recipient": "接收人",
      "channel": "通知渠道",
      "sent_at": "发送时间",
      "confirmed_at": "确认时间（null表示未确认）",
      "content_summary": "通知内容摘要"
    }
  ],
  "escalation_triggered": false,
  "war_room_created": false,
  "war_room_link": "War Room链接（如有）"
}
```

## 最佳实践

1. **多通道冗余**：P0/P1告警使用多通道通知（电话+IM），避免单一通道漏读
2. **避免通知疲劳**：对同一故障的更新通知合并发送，避免刷屏
3. **上下文充分**：通知中包含足够信息让接收者不需要额外查询就能开始响应
4. **确认机制**：必须有明确的"已收到"确认机制，未确认自动升级

## 约束条件

- 通知发送延迟不超过30秒
- 不得向非当班人员发送P3级别告警（减少打扰）
- 夜间（23:00-07:00）P2及以下告警仅发送IM，不拨打电话
- 保留所有通知记录用于复盘评估响应时效
