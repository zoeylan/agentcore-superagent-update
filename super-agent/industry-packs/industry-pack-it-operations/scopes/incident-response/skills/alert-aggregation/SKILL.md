# 告警聚合 (Alert Aggregation)

## 技能概述

告警聚合技能用于将来自多源监控系统（Prometheus、Zabbix、云监控、APM等）的海量原始告警进行智能聚合，减少告警数量级的同时保留有效信息。一次级联故障可能触发数百甚至上千条告警，聚合后通常可归纳为1-5个核心故障事件。

## 使用场景

- 多源告警实时流入时的自动聚合处理
- 告警风暴识别与合并（5分钟内同类告警>50条）
- 级联故障中的根源告警识别
- 告警去重与抑制（同一故障点的重复告警合并）

## 执行步骤

### 步骤1：告警接收与标准化
- 从多源告警通道接收原始告警事件
- 将不同格式的告警统一为标准化结构：`{source, timestamp, severity, target_host, target_service, metric, current_value, threshold, description}`
- 对缺失字段进行补全（如通过CMDB补全服务归属信息）

### 步骤2：时间窗口聚合
- 使用5分钟滑动窗口对告警进行时间聚合
- 相同目标（host+service+metric）在窗口内的多次告警合并为一条，保留首次触发时间和最新状态
- 计算窗口内的告警频率，识别突增模式

### 步骤3：拓扑关联聚合
- 基于服务拓扑图（Service Dependency Map）分析告警之间的上下游关系
- 识别级联告警：如果上游服务A告警，下游服务B/C/D的超时告警大概率是A的级联影响
- 将级联告警归并至根源节点，标注"疑似级联"标签

### 步骤4：历史模式匹配
- 将当前告警组合与历史故障模式库比对
- 若匹配到已知模式（如"数据库连接池耗尽→应用超时→前端502"），标注模式名称和历史处理方案
- 计算模式匹配置信度

### 步骤5：告警风暴检测
- 当5分钟窗口内告警数量>50条时标记为告警风暴
- 分析风暴中告警的共性特征（同一机房？同一服务？同一时间点？）
- 输出风暴摘要：触发时间、涉及范围、疑似根源

## 输入规格

```json
{
  "alerts": [
    {
      "source": "prometheus|zabbix|cloudwatch|custom",
      "timestamp": "ISO8601",
      "severity": "critical|warning|info",
      "host": "目标主机",
      "service": "目标服务",
      "metric": "告警指标名",
      "current_value": "当前值",
      "threshold": "阈值",
      "description": "告警描述"
    }
  ],
  "time_window_minutes": 5,
  "topology_data": "服务拓扑关系数据（可选）"
}
```

## 输出规格

```json
{
  "aggregated_events": [
    {
      "event_id": "聚合事件ID",
      "summary": "事件摘要",
      "root_alert": "根源告警",
      "related_alerts_count": "关联告警数量",
      "affected_services": ["受影响服务列表"],
      "first_trigger_time": "首次触发时间",
      "pattern_match": "匹配的历史模式（如有）",
      "is_storm": false
    }
  ],
  "storm_detected": false,
  "storm_summary": "风暴摘要（如检测到）",
  "suppressed_count": "被抑制的重复告警数量"
}
```

## 最佳实践

1. **时间窗口调优**：默认5分钟窗口适用于大部分场景，但对于网络类告警可缩短至2分钟（网络故障传播更快）
2. **拓扑数据新鲜度**：服务拓扑图需要与实际调用关系保持同步，建议每日从APM系统更新
3. **避免过度聚合**：不同根因导致的告警不应合并，聚合时关注时间相关性和因果关系
4. **告警风暴阈值**：50条/5分钟是推荐起始值，应根据企业监控规模调整

## 约束条件

- 聚合延迟不超过30秒（从告警接收到聚合结果输出）
- 不得丢弃任何原始告警数据，聚合仅是逻辑分组
- 当拓扑数据不可用时，降级为纯时间窗口+主机维度聚合
- P0级别的告警即使在聚合过程中也必须立即转发，不等待窗口关闭
