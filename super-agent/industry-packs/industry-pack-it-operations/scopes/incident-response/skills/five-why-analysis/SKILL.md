# 5-Why根因分析 (Five-Why Analysis)

## 技能概述

5-Why分析技能运用丰田生产方式中的"五问法"（Five Whys），从故障的直接技术原因逐层追溯至系统性根因，揭示流程、机制和文化层面的深层问题。目标不是停留在"代码有BUG"这个表面答案，而是追问"为什么BUG能上线？为什么上线后没有被及时发现？为什么恢复用了这么久？"

## 使用场景

- 故障复盘报告中的根因分析章节
- P0/P1故障的深度原因追溯
- 识别系统性改进机会
- 建立故障预防机制

## 执行步骤

### 步骤1：确定分析起点
- 明确故障的直接表现（What happened）
- 确定直接技术原因（由根因分析Agent提供）
- 作为5-Why的第一个"Why"的起点

### 步骤2：逐层追问（至少3层，推荐5层）

**第1层Why - 直接原因（技术层）**
- 为什么发生了这个技术故障？
- 例：为什么数据库连接池耗尽？→ 因为慢查询占用连接时间过长

**第2层Why - 触发原因（变更/外部层）**
- 为什么会出现第1层的问题？
- 例：为什么出现慢查询？→ 因为新上线的功能缺少索引

**第3层Why - 防御失效（检测/预防层）**
- 为什么第2层的问题没有被预防或提前发现？
- 例：为什么缺失索引没有在上线前发现？→ 因为SQL Review流程没有覆盖该类变更

**第4层Why - 流程/机制层**
- 为什么防御机制会有缺口？
- 例：为什么SQL Review没有覆盖？→ 因为该功能走了紧急发布通道跳过了Review

**第5层Why - 系统/文化层**
- 为什么流程存在这样的缺口？
- 例：为什么紧急发布可以跳过Review？→ 因为紧急发布流程没有定义最低质量门槛

### 步骤3：识别多条因果链
- 一次故障通常有多条并行的因果链：
  - 为什么发生？（预防失效链）
  - 为什么没有提前发现？（检测失效链）
  - 为什么恢复这么慢？（恢复效率链）
- 每条链路独立追问5-Why

### 步骤4：标注根因类型
对最终追溯到的根因进行分类：
- 流程缺陷：规范/流程有缺口或未被遵循
- 技术债务：已知风险但未及时治理
- 能力缺失：缺乏必要的工具/自动化/知识
- 资源不足：人员/时间/预算投入不够
- 文化/意识：安全/质量意识不足

### 步骤5：输出分析结论
- 以树状结构展示5-Why分析过程
- 每一层附带证据支撑
- 最终根因需要具备"可行动性"——能够对应到具体的改进措施
- 区分"直接原因"和"根本原因"

## 输入规格

```json
{
  "incident_summary": "故障概述",
  "direct_cause": {
    "technical_cause": "技术直接原因",
    "evidence": "支撑证据"
  },
  "context": {
    "recent_changes": ["近期变更"],
    "existing_safeguards": ["现有防护机制"],
    "response_timeline": "响应时间线摘要"
  }
}
```

## 最佳实践

1. **不止步于技术原因**：如果5-Why分析的结论是"代码BUG"，说明分析不够深入
2. **Blameless原则**：分析对象是系统和流程，不是人；"某人操作失误"不是有效根因
3. **证据驱动**：每一层的回答必须有数据或事实支撑，不能是臆测
4. **多链路分析**：一次故障通常需要分析2-3条不同维度的因果链
5. **可行动性检验**：如果根因无法对应到具体改进措施，需要继续追问

## 约束条件

- 最少追溯3层，推荐5层
- 每一层必须有事实依据，不接受主观猜测
- 分析结论不得指向具体个人的"态度"或"能力"问题
- 5-Why分析应在故障恢复后48小时内完成（记忆衰减前）
