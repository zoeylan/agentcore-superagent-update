## 3. SOP-1: 每日自动巡检

### 3.1 触发条件
- **定时触发**：每日06:00自动启动
- **手动触发**：运维主管根据需要手动触发额外巡检

### 3.2 执行流程

```
06:00 数据采集启动
  │
  ├─→ 基础设施层采集 (Prometheus/Zabbix)
  ├─→ 数据库层采集 (DB监控/慢查询日志)
  ├─→ 网络层采集 (SNMP/NetFlow)
  ├─→ 应用层采集 (APM/日志平台)
  └─→ 业务层采集 (业务监控系统)
  │
06:25 数据采集完成（≤25分钟）
  │
06:25-06:30 异常检测与关联分析
  │
  ├─→ 静态阈值比对
  ├─→ 动态基线比对（7天/30天）
  ├─→ 趋势恶化识别
  └─→ 跨层关联分析
  │
  ├── [严重异常] → 立即推送故障响应Scope
  ├── [安全异常] → 同步安全运维Scope
  └── [一般异常/趋势恶化] → 纳入报告
  │
07:00 报告初稿生成
  │
08:00 运维工程师审核确认
  │
  ├── [确认无误] → 发送最终报告
  └── [发现遗漏/误判] → 修正后重新生成
  │
09:00 最终报告发送（邮件+IM）
```

### 3.3 详细操作步骤

#### 步骤1：数据采集（06:00-06:25）
- **执行者**：自动巡检Agent
- **操作**：
  1. 加载当日巡检模板（标准模板含约200个检查项）
  2. 并行连接各数据源执行数据采集
  3. 记录采集成功/失败状态
- **输出**：结构化原始数据JSON
- **异常处理**：
  - 数据源连接超时（30s）→ 重试3次，间隔10s
  - 3次重试失败 → 标记该项为"未完成"，继续其他项
  - 总体完成率<95% → 通知运维工程师排查数据源故障

#### 步骤2：异常检测（06:25-06:30）
- **执行者**：自动巡检Agent
- **操作**：
  1. 与预设阈值比对（CPU>80%持续5min、磁盘>85%、慢查询>100/h、错误率>1%、P99>3s）
  2. 与7天/30天动态基线比对（偏离>2σ标记异常）
  3. 执行趋势分析（线性回归识别恶化趋势）
  4. 执行跨层关联（识别因果链）
- **输出**：异常分析结果（分级分类）
- **异常处理**：
  - 发现Critical级异常 → 立即（≤1分钟）推送告警至故障响应Scope
  - 发现安全类异常 → 同步通知安全运维Scope

#### 步骤3：报告生成（07:00）
- **执行者**：自动巡检Agent
- **操作**：
  1. 按优先级排列异常项
  2. 生成结构化报告（执行摘要/告警项/风险项/建议项/正常项）
  3. 生成面向运维团队的完整版和面向业务负责人的摘要版
- **输出**：巡检报告初稿

#### 步骤4：人工审核（08:00）
- **执行者**：运维工程师
- **操作**：
  1. 审核异常项的准确性（是否有误判）
  2. 补充业务上下文信息
  3. 确认建议操作的合理性
  4. 标注需要额外跟进的事项
- **输出**：审核通过的最终报告
- **异常处理**：
  - 发现严重遗漏 → 标记需修正项，通知Agent重新分析
  - 审核未在08:30前完成 → 升级至运维主管

#### 步骤5：报告发送（09:00）
- **执行者**：自动巡检Agent
- **操作**：
  1. 通过邮件发送完整版给运维团队
  2. 通过IM（企业微信/钉钉/飞书）发送摘要版给业务负责人
  3. 记录发送状态和时间
- **输出**：发送确认记录

### 3.4 质量检查点

| KPI指标 | 目标值 | 计算方式 | 检查频率 |
|---------|--------|----------|----------|
| 巡检完成率 | 100% | 成功采集项/总检查项 | 每日 |
| 报告准时发送率 | ≥99% | 09:00前发送的天数/总工作日 | 每月 |
| 异常标注准确率 | ≥95% | 人工确认有效异常/Agent标注异常数 | 每月 |
| 严重异常推送及时率 | 100% | 1分钟内推送数/总严重异常数 | 每日 |
| 趋势预警有效率 | ≥80% | 7天内确认为真实恶化的预警/总趋势预警 | 每月 |

## 4. SOP-2: 月度容量评估

### 4.1 触发条件
- **定时触发**：每月第一个工作日启动
- **事件触发**：大促前30天启动专项评估
- **异常触发**：巡检发现容量趋势告警时提前触发

### 4.2 执行流程

```
Day 1: 数据采集与预处理
  │
  ├─→ 采集90天历史资源使用数据
  ├─→ 数据清洗与异常点剔除
  └─→ 标记已知事件（故障/大促/扩容操作）
  │
Day 2-3: 趋势建模与预测
  │
  ├─→ STL时序分解（趋势+季节+残差）
  ├─→ 线性回归拟合增长趋势
  ├─→ 业务增长因子叠加
  └─→ 容量触达时间预测
  │
  ├── [30天内触达水位线] → 生成紧急扩容申请
  ├── [60天内触达水位线] → 标记重点关注
  └── [资源浪费识别] → 生成缩容建议
  │
Day 3-4: 资源优化分析
  │
  ├─→ 低利用率实例识别
  ├─→ 合理性排查（灾备/批处理/预留）
  ├─→ 缩容方案制定
  └─→ 成本影响计算
  │
Day 4-5: 报告生成与评审
  │
  ├─→ 整合趋势预测和优化建议
  ├─→ 生成容量评估报告
  ├─→ 运维主管评审
  └─→ 发送给相关干系人
```

### 4.3 详细操作步骤

#### 步骤1：数据采集与预处理
- **执行者**：容量规划Agent
- **操作**：
  1. 从监控系统采集90天历史数据（1小时粒度）
  2. 数据清洗：缺失值插值、异常点剔除（3σ原则）
  3. 标记历史事件点（避免大促/故障数据干扰趋势分析）
- **输出**：清洁的历史时序数据集
- **异常处理**：
  - 数据缺失>10% → 标注数据质量风险，降低预测置信度
  - 数据源不可用 → 使用备用数据源或降级为近30天数据分析

#### 步骤2：趋势建模
- **执行者**：容量规划Agent
- **操作**：
  1. STL分解提取趋势项
  2. 对趋势项进行线性回归，计算日增长率
  3. 叠加业务增长因子和大促峰值预估
  4. 模型验证：用后30天数据回测（R²≥0.7）
- **输出**：容量预测模型及预测结果
- **异常处理**：
  - R²<0.7 → 尝试分段拟合或非线性模型，标注预测不确定性高

#### 步骤3：资源优化分析
- **执行者**：容量规划Agent
- **操作**：
  1. 识别CPU<10%且内存<20%的实例（极低利用率）
  2. 排查合理性（灾备/批处理/预留buffer）
  3. 制定缩容/释放/降配方案
  4. 计算预计节省成本
- **输出**：优化建议列表（含成本影响）
- **异常处理**：
  - 无法确认实例用途 → 标注"需业务方确认"，不直接建议释放

#### 步骤4：报告生成与评审
- **执行者**：容量规划Agent + 运维主管
- **操作**：
  1. 整合预测结果和优化建议为月度报告
  2. 标注需要管理层决策的事项（扩容预算/缩容确认）
  3. 运维主管审核报告合理性
  4. 紧急扩容需求生成变更申请提交变更管理Scope
- **输出**：月度容量评估报告

### 4.4 质量检查点

| KPI指标 | 目标值 | 计算方式 | 检查频率 |
|---------|--------|----------|----------|
| 预测准确率 | ±20%以内 | \|实际值-预测值\|/实际值 | 每月回溯 |
| 提前30天预警达成率 | ≥90% | 提前30天预警的容量事件/总容量事件 | 每季度 |
| 资源浪费识别覆盖率 | ≥80% | 识别到的浪费实例/实际浪费实例 | 每季度 |
| 扩容建议采纳率 | ≥85% | 被采纳的扩容建议/总扩容建议 | 每月 |
| 报告按时产出率 | 100% | 5个工作日内完成/应产出报告数 | 每月 |

## 5. SOP-3: 监控治理（季度）

### 5.1 触发条件
- **定时触发**：每季度第一个月启动
- **事件触发**：故障复盘中产生监控改进Action时
- **指标触发**：告警噪声比超过30%时提前触发优化

### 5.2 执行流程

```
Week 1: 覆盖率评估
  │
  ├─→ CMDB服务清单获取
  ├─→ 监控系统接入清单获取
  ├─→ 交叉比对计算覆盖率
  └─→ 识别监控盲区
  │
  ├── [核心系统未覆盖] → 立即启动补全
  └── [非核心系统未达标] → 纳入补全计划
  │
Week 2: 告警质量分析
  │
  ├─→ 采集30天告警数据
  ├─→ 有效性分析（准确率/噪声比）
  ├─→ 问题规则识别
  ├─→ 漏报回溯分析
  └─→ 输出优化方案
  │
Week 3: 大盘规范化审查
  │
  ├─→ Golden Signals覆盖检查
  ├─→ 大盘模板合规性验证
  ├─→ 数据保留策略检查
  └─→ 生成不达标项清单
  │
Week 4: 治理报告与改进跟踪
  │
  ├─→ 整合评估结果
  ├─→ 跟踪历史改进Action状态
  ├─→ 制定新季度改进路线图
  └─→ 输出季度治理报告
```

### 5.3 详细操作步骤

#### 步骤1：覆盖率评估
- **执行者**：监控治理Agent
- **操作**：
  1. 从CMDB导出当前在线资产清单
  2. 从各监控系统获取已接入对象列表
  3. 交叉比对，计算分层/分级覆盖率
  4. 识别完全未覆盖、部分覆盖、Golden Signals缺失的服务
  5. 对每个gap生成补全方案
- **输出**：覆盖率评估报告 + 补全计划
- **异常处理**：
  - CMDB数据不准确 → 标注并反馈给CMDB管理员更新
  - 核心系统覆盖率<100% → 升级为P0优先级，运维主管立即审批补全

#### 步骤2：告警质量分析
- **执行者**：监控治理Agent
- **操作**：
  1. 采集30天全量告警记录及处理结果
  2. 计算各规则有效率和整体质量指标
  3. 识别高噪声规则（有效率<50%）
  4. 回溯故障记录发现漏报场景
  5. 对问题规则输出具体优化建议（含调整后的阈值/条件）
- **输出**：告警质量分析报告 + 优化方案
- **异常处理**：
  - 告警噪声比>50% → 升级为紧急优化项，2周内必须完成首批规则优化

#### 步骤3：监控大盘规范化
- **执行者**：监控治理Agent + 运维工程师
- **操作**：
  1. 检查核心服务的Dashboard是否符合标准模板
  2. 验证Golden Signals四大类指标完整性
  3. 检查时间粒度支持和同比/环比功能
  4. 对不合规大盘生成改进方案
  5. 运维工程师执行大盘优化
- **输出**：大盘合规性报告 + 改进方案

#### 步骤4：治理报告与路线图
- **执行者**：监控治理Agent
- **操作**：
  1. 整合覆盖率、告警质量、大盘合规性评估结果
  2. 跟踪历史改进Action状态（从故障复盘中提取的Action）
  3. 对逾期Action发出提醒
  4. 制定下季度改进路线图
  5. 运维主管审批路线图
- **输出**：季度监控治理报告 + 改进路线图

### 5.4 质量检查点

| KPI指标 | 目标值 | 计算方式 | 检查频率 |
|---------|--------|----------|----------|
| 核心系统监控覆盖率 | 100% | 已覆盖核心服务/核心服务总数 | 每季度 |
| 非核心系统覆盖率 | ≥95% | 已覆盖非核心服务/非核心服务总数 | 每季度 |
| 告警准确率 | ≥90% | 有效告警/总告警 | 每月 |
| 告警噪声比 | ≤30% | 噪声告警/总告警 | 每月 |
| 季度改进项完成率 | ≥80% | 已完成改进项/总改进项 | 每季度 |
| 大盘合规率 | ≥90% | 合规大盘数/核心服务数 | 每季度 |

## 6. SOP-4: 专项巡检

### 6.1 触发条件
- **大促前触发**：大促前7天启动专项评估
- **变更后触发**：重大架构变更完成后24小时内启动稳定性验证
- **管理层指令**：运维主管/CTO要求的临时专项巡检

### 6.2 执行流程

```
触发事件
  │
  ├── [大促场景] → 压力评估模板
  ├── [变更后] → 稳定性验证模板
  └── [其他] → 自定义检查模板
  │
Step 1: 巡检范围确定
  │
  ├─→ 确定目标系统/服务范围
  ├─→ 选择/定制巡检模板
  └─→ 确认关注的核心指标
  │
Step 2: 深度指标采集
  │
  ├─→ 基础资源深度采集（1分钟粒度）
  ├─→ 应用性能深度分析
  ├─→ 依赖服务可用性检查
  └─→ 历史峰值数据获取
  │
Step 3: 压力/稳定性评估
  │
  ├─→ 对比历史峰值（大促场景）
  ├─→ 对比变更前基线（变更场景）
  ├─→ 容量余量计算
  └─→ 单点故障风险识别
  │
Step 4: 风险清单输出
  │
  ├─→ 风险分级（高/中/低）
  ├─→ 每项风险的影响评估
  ├─→ 整改建议及优先级
  └─→ 整改责任人指定
  │
Step 5: 整改跟踪
  │
  ├─→ 每日跟踪整改进度
  ├─→ 整改完成后复检
  └─→ 大促前/变更后确认所有风险已消除
```

### 6.3 详细操作步骤

#### 步骤1：巡检范围确定
- **执行者**：运维主管 + 自动巡检Agent
- **操作**：
  1. 根据触发场景确定巡检目标（涉及的系统/服务/链路）
  2. 选择或定制巡检模板
  3. 明确关注的核心指标和验收标准
  4. 确定巡检完成截止时间
- **输出**：专项巡检计划

#### 步骤2：深度指标采集
- **执行者**：自动巡检Agent
- **操作**：
  1. 按更高频率（1分钟级）采集目标系统指标
  2. 采集应用链路调用关系和性能数据
  3. 检查依赖服务的可用性和性能
  4. 获取历史峰值数据作为对比基线
- **输出**：深度巡检数据

#### 步骤3：评估分析
- **执行者**：自动巡检Agent + 容量规划Agent
- **操作**：
  1. 大促场景：对比预估流量与当前容量余量，计算缺口
  2. 变更场景：对比变更前后核心指标是否有显著偏移
  3. 识别单点故障风险（无冗余的关键组件）
  4. 评估极端场景下的系统韧性
- **输出**：评估分析报告

#### 步骤4：风险清单输出
- **执行者**：自动巡检Agent
- **操作**：
  1. 按高/中/低分级所有识别到的风险
  2. 每项风险标注：影响范围、发生概率、建议整改方案
  3. 指定整改责任人和截止日期
  4. 高风险项需立即通报运维主管
- **输出**：风险清单 + 整改计划

#### 步骤5：整改跟踪
- **执行者**：自动巡检Agent + 运维主管
- **操作**：
  1. 每日跟踪各整改项进度
  2. 整改完成后执行复检验证
  3. 大促前确认所有高风险项已消除
  4. 生成专项巡检闭环报告
- **输出**：整改完成确认书
- **异常处理**：
  - 整改逾期 → 升级至运维主管介入
  - 高风险项无法按时整改 → 制定临时缓解方案 + 报告CTO

### 6.4 质量检查点

| KPI指标 | 目标值 | 计算方式 | 检查频率 |
|---------|--------|----------|----------|
| 大促前专项巡检提前完成 | ≥7天 | 巡检完成日 - 大促开始日 | 每次大促 |
| 风险项识别覆盖率 | 100%（零遗漏） | 巡检发现风险/实际风险 | 每次 |
| 高风险整改率 | 100% | 已整改高风险项/总高风险项 | 每次 |
| 整改复检通过率 | ≥95% | 一次复检通过项/总整改项 | 每次 |
| 专项巡检按时交付率 | 100% | 按时完成/应完成 | 每次 |

## 8. 跨Scope协作接口

### 8.1 向故障响应Scope输出
- **触发条件**：巡检发现Critical级异常
- **输出内容**：异常指标详情、影响面评估、已知上下文
- **时效要求**：发现后≤1分钟推送
- **接收方**：故障响应Scope → 告警分诊Agent

### 8.2 向变更管理Scope输出
- **触发条件**：容量规划生成扩容/缩容建议
- **输出内容**：变更申请（含规格/数量/优先级/回滚方案）
- **时效要求**：月度报告发布后3个工作日内提交
- **接收方**：变更管理Scope → 变更评估Agent

### 8.3 向安全运维Scope输出
- **触发条件**：巡检发现安全类异常（异常登录/权限变更/可疑进程）
- **输出内容**：安全异常详情、涉及资产、时间线
- **时效要求**：发现后≤5分钟通知
- **接收方**：安全运维Scope → 安全事件响应Agent

### 8.4 从故障响应Scope输入
- **触发条件**：故障复盘完成后产生监控改进Action
- **输入内容**：改进Action描述、优先级、截止日期
- **处理方**：监控治理Agent跟踪落地
