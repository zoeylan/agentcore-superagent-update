# 告警质量优化

## 技能概述

本技能负责系统性分析告警规则的有效性，识别噪声告警、误报规则和缺失告警，输出告警规则优化方案。目标是将告警准确率提升至90%以上、噪声比控制在30%以下，确保每一条到达OnCall人员的告警都值得关注和处理。

## 适用场景

- 季度监控治理中的告警质量分析
- 告警风暴事后的规则优化
- 故障复盘中发现的告警缺失补充
- 新服务上线后的告警规则审查
- OnCall团队反馈的噪声告警处理

## 执行步骤

### 步骤1：告警数据采集
1. 采集近30天所有告警记录（触发时间/恢复时间/规则名/目标/处理结果）
2. 采集告警处理反馈数据（确认有效/标记误报/忽略/自动恢复）
3. 获取当前所有告警规则配置（阈值/持续时间/通知方式/抑制规则）
4. 获取同期故障记录（用于发现漏报情况）

### 步骤2：告警有效性分析
1. 计算每条告警规则的有效率：`有效告警数 / 总触发次数 × 100%`
2. 分类统计：
   - **有效告警**：触发后需要人工介入处理的（有效率目标≥90%）
   - **噪声告警**：触发后无需操作、自动恢复的（目标≤30%）
   - **重复告警**：同一问题的重复通知
   - **误报告警**：系统正常但触发的误判
3. 计算整体告警质量指标：准确率、噪声比、人均日告警量

### 步骤3：问题规则识别
1. **高噪声规则**：有效率<50%的规则 → 需要调整阈值或增加持续时间窗口
2. **频繁触发规则**：日触发>10次的规则 → 可能阈值设置过低或存在系统性问题
3. **从不触发规则**：90天内未触发的规则 → 可能阈值设置过高或已失效
4. **告警风暴源**：单次事件触发>50条告警的规则组 → 需要配置聚合和抑制

### 步骤4：漏报分析
1. 回溯近30天故障记录，检查故障发生前是否有对应告警
2. 对未产生告警的故障进行根因分析：
   - 缺少对应监控指标 → 需要补充指标和规则
   - 有指标但阈值过高导致未触发 → 需要调低阈值
   - 有指标但缺少告警规则 → 需要补充规则
3. 从故障复盘报告中提取"本应提前发现"的案例作为补充依据

### 步骤5：优化方案输出
1. 对每个问题规则输出具体优化建议：
   - **调整阈值**：给出建议值（基于历史数据P95/P99分位数）
   - **增加持续窗口**：建议持续时间（避免瞬时波动误触发）
   - **配置抑制规则**：定义抑制条件和时间窗口
   - **规则重构**：如改用基于动态基线的检测替代静态阈值
   - **新增规则**：针对漏报场景补充的新告警规则
2. 按优化收益排序（预计减少告警数×单条告警的人力成本）
3. 标注每个优化的风险（调高阈值可能增加漏报风险）

### 步骤6：质量报告生成
1. 输出告警质量仪表盘指标
2. 与上期对比展示改进趋势
3. 对OnCall团队的告警负荷进行评估
4. 预估优化后的目标改善值

## 输入规格

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| alert_records | array | 是 | 近30天告警记录 |
| alert_rules | array | 是 | 当前所有告警规则配置 |
| incident_records | array | 是 | 近30天故障记录 |
| oncall_feedback | array | 否 | OnCall人员的告警反馈 |
| postmortem_actions | array | 否 | 故障复盘中的监控改进Action |

## 最佳实践

1. **渐进调整**：阈值调整建议先在非核心系统验证，再推广到核心系统
2. **双向优化**：既要降低噪声（减少误报），也要减少漏报（补充缺失告警）
3. **OnCall反馈闭环**：建立OnCall人员对告警的反馈机制，持续积累有效性数据
4. **告警分级**：不是所有告警都需要即时通知，低优先级告警可聚合为日报
5. **季节性考量**：大促期间的告警规则应有专门的调整策略

## 约束条件

- 分析周期：至少30天数据才具有统计意义
- 阈值调整需经过评审，不可自动生效
- 漏报分析仅基于已知故障，未发现的静默故障不在分析范围
- 核心系统的告警规则修改需额外的影响评估
