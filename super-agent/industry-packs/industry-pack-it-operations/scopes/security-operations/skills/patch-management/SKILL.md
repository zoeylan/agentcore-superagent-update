# 补丁管理 (Patch Management)

## 概述
补丁管理技能覆盖安全补丁的评估、测试、部署规划和验证全流程，确保在满足修复时效要求的同时最小化补丁部署对业务连续性的影响。核心挑战在于平衡修复速度与稳定性风险。

## 使用场景
- 厂商发布安全补丁需要评估和部署
- 高危漏洞需要在24小时内完成修复或缓解
- 需要制定批量补丁的滚动升级计划
- 补丁部署后出现兼容性问题需要回退

## 执行步骤

### 第一步：补丁评估
1. 获取补丁详情（修复的CVE列表、版本要求、依赖条件）
2. 确认补丁适用范围（受影响的系统/组件清单对照资产库）
3. 评估补丁兼容性风险（版本兼容/依赖冲突/配置变更）
4. 查询补丁社区反馈（是否有已知的部署问题）
5. 确定补丁是否需要重启服务/重启系统

### 第二步：测试验证
1. 在测试环境部署补丁
2. 执行功能回归测试（核心业务流程验证）
3. 执行性能基准测试（对比补丁前后的性能指标）
4. 验证与其他组件的兼容性（特别是自研组件和第三方集成）
5. 记录测试结果和发现的问题

### 第三步：部署规划
1. 确定部署策略（滚动升级/蓝绿部署/金丝雀发布）
2. 规划部署顺序（非核心系统先行→核心系统跟进）
3. 确定部署窗口（高危补丁可能需要紧急窗口）
4. 制定回滚方案（含回滚触发条件和执行步骤）
5. 准备部署前检查清单和部署后验证清单

### 第四步：部署执行与验证
1. 执行Pre-check（系统状态/备份/监控就绪）
2. 按计划分批执行补丁部署
3. 每批次完成后执行功能验证和指标监控
4. 确认所有目标系统补丁部署成功
5. 执行漏洞复扫验证修复有效性

## 输入规格
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| patch_id | string | 是 | 补丁标识 |
| cve_list | array | 是 | 修复的CVE列表 |
| affected_systems | array | 是 | 受影响系统清单 |
| priority | enum | 是 | 修复优先级（critical/high/medium/low） |
| patch_details | object | 是 | 补丁技术细节 |
| rollback_requirements | object | 否 | 回滚要求 |

## 输出规格
| 字段 | 类型 | 说明 |
|------|------|------|
| compatibility_assessment | object | 兼容性评估结果 |
| test_results | object | 测试验证结果 |
| deployment_plan | object | 部署计划（含时间表/策略/回滚方案） |
| deployment_status | object | 部署执行状态 |
| verification_results | object | 部署后验证结果 |

## 最佳实践
- 高危补丁的缓解措施（虚拟补丁/ACL限制）应在正式修复前24小时内到位
- 永远不要跳过测试环境验证，即使时间紧迫
- 补丁部署顺序：先测试环境→预发布环境→非核心生产→核心生产
- 保持补丁回退能力至少72小时（观察期）
- 记录补丁部署影响的配置变更，便于问题排查

## 约束条件
- 补丁部署必须通过变更管理流程（紧急修复走紧急变更通道）
- 核心系统补丁必须在维护窗口执行（周二/周四 22:00-06:00）
- 回滚时间不得超过部署时间的50%
- 批量补丁部署每批次不超过总量的25%
