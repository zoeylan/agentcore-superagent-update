# 取证保全 (Forensics Preservation)

## 概述
取证保全技能确保在安全事件处置过程中正确保留数字证据，维护证据链完整性（Chain of Custody），为后续深度调查、攻击溯源和法律诉讼提供可靠的证据基础。

## 使用场景
- S1/S2安全事件确认后需要保留取证现场
- 需要进行攻击溯源和攻击链还原
- 可能涉及法律诉讼需要保留法律效力的数字证据
- 监管机构要求提供安全事件相关证据材料

## 执行步骤

### 第一步：取证范围确定
1. 识别所有受影响系统和关联系统
2. 确定需要保留的证据类型（日志/内存/磁盘/网络流量）
3. 评估证据时效性（易失性证据优先：内存>运行状态>日志>磁盘）
4. 制定取证计划和优先级顺序

### 第二步：易失性证据采集
1. 内存转储（Memory Dump）——使用专业工具（如LiME/WinPmem）获取完整物理内存镜像
2. 运行进程列表、网络连接状态、路由表、ARP缓存
3. 当前登录会话和用户活动
4. 计划任务/定时任务列表
5. 系统时间同步验证（确保时间戳准确性）

### 第三步：非易失性证据采集
1. 磁盘镜像——对受影响系统磁盘进行位级别（bit-by-bit）镜像
2. 关键日志文件保全（系统日志/应用日志/安全日志/Web访问日志）
3. 安全设备日志（WAF/IDS/防火墙/VPN日志）导出
4. 网络流量包（PCAP）保存
5. 相关数据库审计日志

### 第四步：证据链管理
1. 对所有采集的证据计算哈希值（SHA-256）
2. 记录证据清单（含采集时间/操作人/存储位置/哈希值）
3. 证据存储到独立的只读存储介质
4. 建立证据访问控制（仅授权人员可访问）
5. 填写证据移交记录（每次转移都需要双方签字确认）

## 输入规格
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| incident_id | string | 是 | 关联事件编号 |
| affected_systems | array | 是 | 需要取证的系统列表 |
| evidence_scope | enum | 是 | 取证范围：full/targeted |
| legal_hold | boolean | 否 | 是否需要法律级别保全 |
| priority_evidence | array | 否 | 优先采集的证据类型 |

## 输出规格
| 字段 | 类型 | 说明 |
|------|------|------|
| evidence_inventory | array | 证据清单（含哈希值） |
| collection_timeline | array | 采集时间线 |
| chain_of_custody | object | 证据链记录 |
| storage_location | string | 证据存储位置 |
| integrity_verification | object | 完整性验证结果 |

## 最佳实践
- 按照易失性从高到低的顺序采集（RFC 3227）
- 绝对不要在原始证据上进行分析操作
- 取证操作本身也会产生痕迹，需要最小化对证据的影响
- 安全设备日志需确保保留≥6个月（等保2.0要求）
- 数据泄露事件的取证需要特别注意个人信息保护合规

## 约束条件
- 取证操作不得影响证据完整性
- 采集前必须验证系统时间准确性
- 证据存储必须使用加密存储且定期验证哈希一致性
- 涉及个人信息的证据需遵守《个人信息保护法》相关规定
- 未经授权不得自行销毁任何证据
