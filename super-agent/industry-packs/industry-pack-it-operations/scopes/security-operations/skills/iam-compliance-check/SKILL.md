# IAM合规检查 (IAM Compliance Check)

## 概述
IAM合规检查技能专注于验证身份与访问管理体系是否满足等保2.0和相关法规的合规要求，是连接访问控制审计与整体合规评估的桥梁，为合规审计Agent提供IAM领域的专项合规数据。

## 使用场景
- 等保2.0中身份鉴别/访问控制/安全审计相关控制项的验证
- 年度等保复评前的IAM专项自查
- 新系统上线前的IAM合规验收
- 法规变更后的合规差距评估

## 执行步骤

### 第一步：控制项映射
1. 梳理等保2.0三级中IAM相关控制项：
   - 身份鉴别（8.1.4.1）：唯一身份标识/双因子认证/登录失败处理/远程访问加密
   - 访问控制（8.1.4.2）：最小权限/默认拒绝/安全标记/授权管理
   - 安全审计（8.1.4.3）：审计日志/操作记录/日志保护
2. 将每个控制项分解为可检查的验证条目
3. 区分技术验证项（可自动化）和管理验证项（需人工确认）

### 第二步：自动化检查执行
1. 检查身份鉴别机制：密码复杂度策略/登录失败锁定/会话超时设置
2. 检查双因子认证启用状态（核心系统MFA覆盖率）
3. 检查权限配置：默认权限是否最小化/是否存在默认账号未修改密码
4. 检查审计日志：是否完整记录特权操作/日志保留期是否≥6个月
5. 检查远程访问：VPN/堡垒机配置是否符合加密传输要求

### 第三步：管理措施验证
1. 验证权限管理制度文档是否完善且现行有效
2. 验证账号申请/变更/注销流程是否规范执行
3. 验证定期审计制度是否按要求执行（频率/覆盖范围）
4. 验证安全培训记录（用户安全意识教育）
5. 验证第三方人员账号管理规范是否到位

### 第四步：合规报告输出
1. 生成各控制项达标状态（达标/部分达标/不达标）
2. 计算IAM领域整体合规率
3. 列出不合规项及整改建议
4. 提供证据材料清单（截图/配置导出/日志样本）
5. 输出数据供合规审计Agent用于整体合规评估

## 输入规格
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| systems_in_scope | array | 是 | 需要检查的系统列表 |
| control_baseline | object | 是 | 等保2.0控制项基线 |
| system_configs | array | 是 | 系统配置数据 |
| policy_documents | array | 否 | 管理制度文档 |
| previous_results | object | 否 | 上次检查结果 |

## 输出规格
| 字段 | 类型 | 说明 |
|------|------|------|
| compliance_status | object | 各控制项达标状态 |
| overall_rate | number | IAM整体合规率 |
| non_compliant_items | array | 不合规项清单 |
| evidence_list | array | 合规证据材料 |
| remediation_suggestions | array | 整改建议 |

## 最佳实践
- 自动化检查覆盖率目标≥80%（IAM领域技术控制项占比较高）
- 合规检查不是pass/fail二元结果，需要评估控制措施的有效性程度
- 证据收集要有时效性标注——过期的证据不能用于当年复评
- 新上线系统应在上线前完成IAM合规验收
- 合规状态是动态的——配置漂移可能导致原来达标的项变为不达标

## 约束条件
- 检查过程不得修改系统配置
- 证据材料需脱敏处理（隐藏真实密码/密钥信息）
- 检查结果的访问权限限制在安全团队和合规团队
- 等保复评前90天必须完成全量自查
- 发现的高危不合规项需24小时内启动整改
