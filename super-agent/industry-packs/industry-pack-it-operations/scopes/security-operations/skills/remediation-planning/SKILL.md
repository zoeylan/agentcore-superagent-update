# 修复方案规划 (Remediation Planning)

## 概述
修复方案规划技能用于为已确认的漏洞制定最优修复策略，综合考虑修复效果、业务影响、实施复杂度和时间约束，为每个漏洞提供多种修复选项及其风险评估。

## 使用场景
- 高危漏洞确认后需要在24小时内制定修复或缓解方案
- 需要在正式修复和临时缓解之间做出选择
- 修复方案可能影响业务连续性需要评估风险
- 批量漏洞需要统一规划修复窗口和顺序

## 执行步骤

### 第一步：修复方案制定
1. 确定可用的修复手段：
   - **正式补丁**：厂商发布的安全更新
   - **配置加固**：修改系统配置消除漏洞利用条件
   - **虚拟补丁**：WAF/IPS规则层面阻断利用尝试
   - **网络隔离**：限制暴露面降低被攻击概率
   - **组件升级/替换**：升级到不受影响的版本或更换组件
2. 评估每种方案的适用性和有效性
3. 为每种方案标注预估修复时间

### 第二步：修复风险评估
1. 兼容性风险——补丁/升级是否可能破坏现有功能
2. 性能影响——新规则/配置是否影响系统性能
3. 重启需求——是否需要服务重启/系统重启
4. 回退可能性——方案是否可以快速回退
5. 依赖影响——修复操作是否影响上下游系统

### 第三步：修复计划编排
1. 确定修复Owner（负责实施的团队/个人）
2. 确定修复窗口（常规窗口/紧急窗口）
3. 制定实施步骤清单（Pre-check/执行/Post-check）
4. 制定回滚方案和触发条件
5. 确定验证方式（复扫/功能测试/监控观察）

### 第四步：修复跟踪与升级
1. 设定修复截止时间（高危24h/中危7天/低危30天）
2. 配置超时提醒和自动升级机制
3. 跟踪修复执行进度
4. 超时未完成时评估临时缓解措施并强制排期

## 输入规格
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| vulnerability_id | string | 是 | 漏洞编号 |
| cve_id | string | 否 | CVE编号 |
| affected_systems | array | 是 | 受影响系统 |
| priority | enum | 是 | 修复优先级 |
| system_dependencies | object | 否 | 系统依赖关系 |
| maintenance_windows | array | 否 | 可用维护窗口 |

## 输出规格
| 字段 | 类型 | 说明 |
|------|------|------|
| remediation_options | array | 可选修复方案列表（含优缺点） |
| recommended_option | object | 推荐方案及理由 |
| implementation_plan | object | 实施计划（步骤/Owner/时间表） |
| rollback_plan | object | 回滚方案 |
| risk_assessment | object | 修复风险评估 |
| tracking_config | object | 跟踪和升级配置 |

## 最佳实践
- 高危漏洞应同时准备正式修复方案和临时缓解方案（后者24小时内生效）
- 修复方案不是越激进越好——稳定的缓解措施比仓促的补丁更安全
- 批量修复应考虑分组策略，避免同时修复关联系统导致全面不可用
- 修复计划提交变更管理前需要漏洞管理Agent确认方案的有效性
- 每次修复后必须执行复扫验证，不能仅凭操作完成就关闭漏洞

## 约束条件
- 修复方案必须通过变更管理Scope审批后方可执行
- 紧急修复（24h SLA）可走紧急变更通道但需事后补审
- 修复操作的回滚时间不得超过修复时间的50%
- 涉及核心交易系统的修复必须在维护窗口执行
- 修复方案不得引入新的安全风险
