# 访问控制审计 (Access Audit)

## 概述
访问控制审计技能用于对组织的身份认证和访问授权体系进行系统性审计，验证权限分配是否符合最小权限原则和职责分离要求，发现权限配置中的安全隐患。

## 使用场景
- 月度定期访问控制审计
- 组织架构调整后的权限审查
- 等保合规中访问控制类控制项的验证
- 安全事件后的权限回溯和紧急审计

## 执行步骤

### 第一步：审计范围确定
1. 确定本次审计涵盖的系统范围（全量/特定系统/特定业务线）
2. 获取最新的组织架构和人员信息
3. 获取各系统的权限配置数据（AD/LDAP/IAM平台导出）
4. 确定审计基准（上次审计结果、权限策略文档）

### 第二步：权限清单盘点
1. 列出所有用户及其权限映射
2. 标注特权账号和高权限用户
3. 识别共享账号和服务账号
4. 统计各权限级别的用户分布
5. 核对权限申请记录与实际权限的一致性

### 第三步：合规性检查
1. **最小权限原则验证**：用户权限是否超出工作需要
2. **职责分离验证**：是否存在权限冲突（如同一人拥有开发和生产部署权限）
3. **离职/调岗处理**：已离职人员是否仍有活跃账号；已调岗人员是否仍保留旧部门权限
4. **密码策略合规**：特权账号90天轮换是否执行；密码复杂度是否达标
5. **MFA启用状态**：核心系统是否全部启用多因子认证

### 第四步：审计报告与整改
1. 生成审计发现清单（含风险等级/影响描述/整改建议）
2. 计算权限合规率和各维度达标情况
3. 为每个发现项指定整改Owner和Deadline
4. 与上期审计结果对比，验证历史整改项是否闭环

## 输入规格
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| audit_scope | object | 是 | 审计范围定义 |
| org_structure | object | 是 | 组织架构数据 |
| permission_data | array | 是 | 各系统权限配置数据 |
| previous_audit | object | 否 | 上期审计结果 |
| policy_baseline | object | 否 | 权限策略基线 |

## 输出规格
| 字段 | 类型 | 说明 |
|------|------|------|
| audit_findings | array | 审计发现清单 |
| compliance_rate | object | 各维度合规率 |
| privilege_inventory | array | 特权账号清单 |
| remediation_items | array | 整改项及Owner/Deadline |
| trend_comparison | object | 与上期对比结果 |

## 最佳实践
- 审计频率至少每月一次，核心系统可每周
- 不仅审计谁有什么权限，还要审计权限是否被使用（长期未使用的高权限是风险）
- 服务账号和共享账号是常见的审计盲区，需要特别关注
- 审计发现的高风险项（如离职人员仍有活跃账号）应24小时内处理
- 自动化能力覆盖80%的审计工作，但人工Review仍不可或缺

## 约束条件
- 审计过程中不得修改任何权限配置
- 审计数据需脱敏后才能生成报告（不暴露具体密码信息）
- 特权账号清单为高敏感数据，仅授权人员可查看
- 审计结果保留≥2年用于合规追溯
