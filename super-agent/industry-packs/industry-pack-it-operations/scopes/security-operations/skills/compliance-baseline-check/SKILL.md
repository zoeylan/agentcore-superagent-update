# 合规基线检查 (Compliance Baseline Check)

## 概述
合规基线检查技能用于对照等保2.0三级系统要求的240+控制项，执行持续性的合规状态验证。区别于一次性的等保评测，本技能强调合规的持续性和动态性——确保控制措施在任何时间点都处于有效状态。

## 使用场景
- 季度合规评估的技术控制项自动化检查
- 年度等保复评前的全量基线验证
- 新控制措施上线后的有效性验证
- 配置变更后的合规状态确认
- 法规更新后新增控制项的基线补充

## 执行步骤

### 第一步：控制项分类与自动化映射
1. 将240+等保控制项分为技术控制项和管理控制项
2. 技术控制项进一步分为：可完全自动化检查 / 半自动化（需人工辅助验证） / 需人工检查
3. 为每个自动化检查项定义检查逻辑（命令/脚本/API调用）
4. 确定检查频率：高风险项每日、中风险项每周、低风险项每月

### 第二步：自动化检查执行
1. **安全通信网络**：TLS配置/加密算法强度/证书有效期
2. **安全区域边界**：防火墙规则/入侵检测状态/恶意代码防护更新
3. **安全计算环境**：
   - 身份鉴别：密码策略/登录失败处理/远程加密
   - 访问控制：权限配置/默认账号/安全标记
   - 安全审计：日志开启/保留期限/日志完整性
   - 入侵防范：漏洞修复情况/不必要服务关闭
   - 数据保密性：敏感数据加密存储/传输加密
   - 数据备份恢复：备份策略执行/恢复测试
4. **安全管理中心**：集中管控策略/审计日志集中管理

### 第三步：结果评估与状态标注
1. 为每个检查项标注状态：达标(G)/部分达标(Y)/不达标(R)/不适用(N/A)
2. 部分达标需注明差距描述
3. 计算各控制域的达标率
4. 计算整体合规分数（达标=100%/部分达标=50%/不达标=0%）
5. 与上次检查结果对比，标注状态变化（恶化/改善/不变）

### 第四步：基线维护
1. 法规更新时及时补充新控制项
2. 系统架构变更时调整检查项的适用性
3. 定期优化自动化检查的准确性（减少误报）
4. 记录例外项及其补偿控制措施
5. 维护控制项与检查工具/脚本的映射关系

## 输入规格
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| check_scope | enum | 是 | 检查范围：full/domain/system |
| target_systems | array | 是 | 目标系统列表 |
| baseline_version | string | 是 | 基线版本号 |
| previous_results | object | 否 | 上次检查结果 |
| exceptions | array | 否 | 已知例外项 |

## 输出规格
| 字段 | 类型 | 说明 |
|------|------|------|
| check_results | array | 各控制项检查结果 |
| domain_scores | object | 各控制域达标率 |
| overall_score | number | 整体合规分数 |
| status_changes | array | 状态变化项 |
| automation_coverage | number | 自动化检查覆盖率 |

## 最佳实践
- 自动化检查覆盖率目标≥70%，持续提升
- 检查结果异常时先确认是真正不合规还是检查逻辑有问题
- 基线不是一成不变的——法规更新/架构变更都需要调整
- 配置漂移是合规退化的主要原因，需要持续监测
- 等保复评前90天启动全量检查，预留整改时间

## 约束条件
- 检查过程不得影响生产系统性能
- 自动化脚本的变更需要经过安全评审
- 检查结果数据保留≥3年
- 不适用项(N/A)需要有明确的理由说明和审批
- 技术控制项自动化覆盖率不得低于70%
