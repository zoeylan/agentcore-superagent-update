# 安全应急响应协调 (Incident Response Coordination)

## 概述
安全应急响应协调技能用于在确认的安全事件（特别是S1/S2级别）中驱动完整的应急响应流程，包括团队拉通、处置协调、进展同步和事件闭环。确保在SLA时间内完成事件处置并最小化业务影响。

## 使用场景
- S1级安全事件确认后需要立即启动应急响应
- S2级安全事件需要在4小时内完成协调处置
- 安全事件持续升级需要扩大响应范围
- 安全事件处置完成后需要进行闭环管理

## 执行步骤

### 第一步：应急响应启动（S1事件≤5分钟内完成）
1. 创建安全事件工单（编号规则：SEC-YYYYMMDD-序号）
2. 确认事件等级和影响范围
3. 创建应急响应通信群（拉入安全团队、受影响系统Owner、必要时包含CISO）
4. 发送事件通告（含事件概述、当前影响、初步处置动作）

### 第二步：即时处置动作协调
1. **系统隔离**：协调网络团队对受影响系统实施ACL隔离（保持系统运行以保留内存状态）
2. **攻击阻断**：协调在WAF/防火墙层面封禁攻击源IP/域名
3. **取证保留**：确认关键日志已保存、磁盘快照已创建、必要时安排内存dump
4. **影响遏制**：评估攻击是否已横向移动，必要时扩大隔离范围

### 第三步：进展同步与升级管理
1. 每30分钟向管理层同步处置进展（S1事件）
2. 关键决策节点实时通报（如：需要断网、需要关闭业务系统）
3. 超时未解决自动触发升级（S1事件2小时未解决→升级至CTO/CEO）
4. 记录完整的事件时间线（Timeline精确到分钟）

### 第四步：事件闭环
1. 确认威胁已完全清除（所有后门/恶意文件/持久化机制已移除）
2. 受影响系统安全加固后恢复服务
3. 通知所有相关方事件已关闭
4. 整理事件材料移交给事件报告编写
5. S1事件需评估是否需要向监管机构报告

## 输入规格
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| incident_id | string | 是 | 事件编号 |
| severity | enum | 是 | 事件等级S1-S4 |
| attack_type | string | 是 | 攻击类型 |
| affected_systems | array | 是 | 受影响系统列表 |
| threat_analysis | object | 是 | 威胁分析结果 |
| initial_indicators | array | 是 | 初始攻击指标（IoC） |

## 输出规格
| 字段 | 类型 | 说明 |
|------|------|------|
| response_actions | array | 已执行的响应动作清单 |
| timeline | array | 事件时间线 |
| containment_status | enum | 遏制状态：contained/partially/uncontained |
| communication_log | array | 通信记录 |
| escalation_history | array | 升级记录 |
| closure_report | object | 事件关闭报告 |

## 最佳实践
- S1事件1小时内必须上报CISO，这是硬性合规要求
- 隔离操作优先使用网络ACL而非关机，保留取证现场
- 应急响应期间所有操作必须记录，作为后续取证和合规的证据
- 先遏制再调查——确保攻击不再继续扩大是第一优先级
- 涉及数据泄露的事件需评估是否触发《个人信息保护法》的72小时通知义务

## 约束条件
- 不得在未经CISO授权的情况下执行断网操作
- 取证现场保留优先于系统恢复
- 所有隔离操作必须通过变更管理的紧急变更通道记录
- 事件通报中不得包含具体攻击细节（防止信息泄露扩大影响）
