# 变更影响面分析 (Change Impact Analysis)

## 概述

变更影响面分析是变更评估的核心技能，通过分析服务拓扑、调用链和依赖关系，精确识别一个变更可能影响的所有上下游服务、用户群体和业务指标。在微服务架构下，一个看似简单的配置变更可能通过级联效应影响数十个下游服务，因此影响面分析的准确性直接决定了变更风险评估的可靠性。

## 适用场景

- 新变更申请提交后的影响评估
- 变更冲突检测（多个变更是否影响相同服务/资源）
- 紧急变更的快速影响判定
- 变更回滚影响范围评估

## 执行步骤

### 步骤1：变更内容解析
- 提取变更申请中的核心信息：变更对象（服务名/配置项/数据库表）、变更类型（代码/配置/数据/基础设施）、变更内容摘要
- 识别变更的直接影响对象（第一层影响）

### 步骤2：服务依赖关系查询
- 从CMDB/服务目录中查询变更对象的服务拓扑信息
- 获取该服务的上游调用方列表（谁在调用它）
- 获取该服务的下游依赖列表（它在调用谁）
- 识别共享资源依赖（数据库、缓存、消息队列、配置中心）

### 步骤3：级联影响评估
- 基于调用链数据，向上追溯影响传播路径（最多追溯3层）
- 评估每层影响的严重程度衰减系数
- 识别关键路径上的服务（单点依赖、无降级方案的强依赖）
- 标记影响到核心交易链路的路径（支付/下单/登录等）

### 步骤4：影响面量化
- 估算影响的用户量（基于服务的DAU/MAU数据）
- 估算影响的交易量（基于服务的QPS/TPS数据）
- 评估对SLA的潜在影响（如该服务中断将导致整体可用性下降多少）
- 计算影响面综合评分（用户量权重40% + 交易量权重30% + SLA影响权重30%）

### 步骤5：冲突与窗口检测
- 检查是否存在同时段影响相同服务的其他变更
- 检查是否在变更冻结期内
- 评估最优执行窗口（低流量时段、非关键业务时间）

## 输入规范

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| change_id | string | 是 | 变更工单唯一标识 |
| change_target | string | 是 | 变更对象（服务名/资源名） |
| change_type | enum | 是 | 代码/配置/数据库/基础设施/安全 |
| change_description | string | 是 | 变更内容描述 |
| planned_time | datetime | 否 | 计划执行时间 |

## 输出规范

```json
{
  "impact_summary": {
    "direct_services": ["直接影响的服务列表"],
    "indirect_services": ["间接影响的服务列表（2-3层）"],
    "affected_users_estimate": "预估影响用户数",
    "affected_transactions_estimate": "预估影响交易量/小时",
    "sla_impact": "对整体SLA的影响评估",
    "critical_path_hit": true/false,
    "impact_score": 0-100
  },
  "dependency_topology": {
    "upstream": ["上游服务列表及调用量"],
    "downstream": ["下游服务列表及依赖强度"],
    "shared_resources": ["共享资源列表"]
  },
  "conflicts": ["冲突变更列表"],
  "recommended_window": "建议执行窗口"
}
```

## 最佳实践

1. **宁可高估不可低估**：影响面评估宁可偏保守，因为低估影响面导致的生产故障代价远大于高估带来的审批延迟
2. **关注间接依赖**：特别是配置中心变更，可能通过配置热更新同时影响所有订阅该配置的服务
3. **时间维度考量**：数据库DDL变更在执行期间可能锁表，影响面不仅是逻辑依赖还有时间窗口内的性能影响
4. **历史数据参考**：参考同类型变更的历史影响数据，特别是曾导致故障的变更模式
5. **人工确认环节**：对于影响面评分超过70分的变更，建议人工确认服务依赖关系的准确性

## 约束与限制

- 服务拓扑数据依赖CMDB的准确性和时效性，需注意数据可能滞后
- 间接影响评估最多追溯3层，超过3层的级联效应通常可忽略
- 影响用户量为估算值，基于历史流量数据，实际值可能因业务波动而偏差
- 对于全新上线的服务（无历史数据），影响评估需依赖架构文档和人工判断
