# 变更日历管理 (Change Calendar Management)

## 概述

变更日历管理是对生产环境所有计划变更进行时间维度统一调度和冲突管控的技能。通过维护全局变更日历视图，确保变更在合适的时间窗口执行、避免高风险变更并行、遵守变更冻结策略。变更日历是变更管理的"交通调度中心"——没有有效的日历管控，多团队并行变更极易引发级联故障或互相干扰。

## 适用场景

- 变更时间窗口的合规性检查
- 多变更并行冲突检测
- 维护窗口容量管理
- 变更冻结期（封版期）管控
- 最优执行时间推荐

## 执行步骤

### 步骤1：时间合规性检查
检查变更计划时间是否满足以下硬性规则：
- **周五禁令**：周五全天禁止非紧急变更执行
- **节假日禁令**：法定节假日前一天及节假日期间禁止非紧急变更
- **维护窗口限制**：高风险变更仅允许在维护窗口内执行（周二/周四 22:00-06:00）
- **大促冻结期**：大促前7天至大促结束后3天为变更冻结期
- **其他冻结期**：重要会议/审计/等保评测期间的临时冻结

### 步骤2：并行冲突检测
- 查询同一时间窗口内的所有已排期变更
- 检测资源冲突：同一服务/数据库/网络设备的并行变更
- 检测依赖冲突：变更A的对象是变更B的依赖
- 检测窗口容量：同一维护窗口内高风险变更≤2个
- 检测人员冲突：同一执行人/复核人被分配多个并行变更

### 步骤3：冲突解决建议
- 对检测到的冲突，生成解决方案建议：
  - 时间错开：建议变更A先执行，变更B在A完成验证后再执行
  - 窗口调整：建议移至下一个可用维护窗口
  - 合并执行：同一服务的多个变更考虑合并为一次变更

### 步骤4：日历视图更新
- 变更审批通过后，将变更加入全局日历
- 更新维护窗口的剩余容量
- 标记变更间的依赖关系（B须在A成功后执行）
- 变更完成/取消后释放日历占位

### 步骤5：最优时间推荐
基于以下因素推荐最优执行时间：
- 业务流量低谷期（通常凌晨2:00-5:00）
- 运维团队人员充足的时间段
- 无其他高风险变更并行的窗口
- 距离最近一次同类变更的间隔（避免变更过密）

## 输入规范

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| change_id | string | 是 | 变更工单标识 |
| planned_start | datetime | 是 | 计划开始时间 |
| planned_end | datetime | 是 | 计划结束时间 |
| risk_level | enum | 是 | 风险等级 |
| target_resources | array | 是 | 变更涉及的资源列表 |
| executor | string | 是 | 执行人 |
| is_emergency | boolean | 是 | 是否紧急变更 |

## 最佳实践

1. **提前锁定窗口**：高风险变更应提前3个工作日预约维护窗口，先到先得
2. **缓冲时间**：相邻变更之间至少留30分钟缓冲（用于前一个变更的Post-check）
3. **冻结期通知**：变更冻结期应提前7天通知所有变更申请人
4. **日历可视化**：变更日历应对所有团队公开可见，提升协调效率
5. **紧急变更不受限**：紧急变更可突破日历限制，但须记录原因并事后复盘

## 约束与限制

- 变更日历准确性依赖所有团队按规范提交和更新变更计划
- 时间合规性检查基于公开的节假日安排和维护窗口定义
- 对于跨时区团队的变更协调，需特别注意时区转换
- 变更日历容量限制是软限制，特殊情况可由运维总监审批突破
