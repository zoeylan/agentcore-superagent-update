# 回滚管理 (Rollback Management)

## 概述

回滚管理是在变更执行过程中检测到异常后，快速决策并执行回滚操作以恢复系统到变更前状态的技能。回滚是变更管理的"最后一道保险"——当变更导致的问题超出预期时，快速回滚是最有效的止损手段。回滚决策的时效性（≤5分钟）和回滚执行的成功率（≥99%）直接决定了故障影响时间。

## 适用场景

- 灰度发布过程中检测到核心指标异常
- 变更完成Post-check验证失败
- 变更执行过程中出现未预期的错误
- 上下游服务报告受到变更影响的异常

## 执行步骤

### 步骤1：回滚决策（≤5分钟完成）
触发回滚评估的条件：
- 灰度监控触发回滚告警（指标偏离基线>30%）
- 用户/客服反馈异常
- 上下游团队报告受影响
- 执行人/复核人主观判断异常

决策框架：
- **立即回滚**：核心交易受影响（错误率>5%）、数据异常、安全问题
- **暂停+评估**：指标轻微异常但趋势恶化、非核心功能受影响
- **继续观察**：指标波动在可接受范围、已确认非变更引起的抖动

### 步骤2：回滚准备
- 确认回滚方案可用（脚本/工具/目标版本就绪）
- 通知相关方即将回滚（执行团队/业务方/值班运维）
- 暂停所有正在进行的放量操作
- 确认回滚执行人和复核人就位

### 步骤3：回滚执行
- 按回滚方案的步骤逐步执行
- 实时监控回滚过程中的系统状态
- 记录每个回滚步骤的执行时间和结果
- 回滚总时间控制在变更执行时间的50%以内

### 步骤4：回滚验证
- 确认系统恢复到变更前的版本/配置状态
- 验证核心指标恢复到基线水平（±10%以内）
- 检查无残留数据不一致或脏数据
- 验证上下游服务恢复正常通信
- 确认用户功能恢复正常可用

### 步骤5：回滚后处理
- 标记变更工单为"已回滚/失败"
- 记录回滚原因和过程时间线
- 评估回滚影响面（是否有用户在变更期间受到影响）
- 如影响面大（影响用户>1000或持续时间>15分钟），触发故障响应流程
- 通知相关方回滚完成及后续处理计划
- 发起变更失败分析（为什么变更失败、如何改进）

## 输入规范

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| change_id | string | 是 | 变更工单标识 |
| rollback_trigger | string | 是 | 回滚触发原因 |
| current_state | object | 是 | 当前系统状态（已放量比例、执行阶段） |
| rollback_plan | object | 是 | 回滚方案详情 |
| metrics_snapshot | object | 是 | 触发回滚时的指标快照 |

## 最佳实践

1. **决策果断**：宁可过早回滚也不要犹豫不决——回滚的成本远小于故障持续的成本
2. **自动化回滚**：对标准变更类型（如容器化服务的版本回退），应实现一键自动回滚
3. **回滚演练**：定期在测试环境演练回滚操作，确保回滚方案的有效性
4. **先止血后分析**：回滚的第一优先级是恢复服务，原因分析是回滚完成后的事
5. **回滚也要灰度**：对于大规模服务，回滚也建议分批进行，避免回滚本身引发问题

## 约束与限制

- 回滚决策5分钟时效是硬性要求，超时需升级决策
- 某些变更本质上不可完全回滚（如已发送的通知、已产生的交易数据）
- 回滚到旧版本可能引入已修复的旧bug，需权衡
- 数据库Schema变更的回滚需要特别谨慎（可能有新数据写入）
- 回滚执行本身也是一次变更，回滚失败需要有plan-B
