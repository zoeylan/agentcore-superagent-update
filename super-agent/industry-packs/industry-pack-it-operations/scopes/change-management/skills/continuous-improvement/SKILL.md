# 持续改进 (Continuous Improvement)

## 概述

持续改进是基于ITIL CSI（持续服务改进）方法论，通过系统化的改进识别、评估、实施和验证循环，不断提升变更管理体系的效率和质量。核心理念是：没有终极完美的流程，只有持续趋近最优的过程。本技能将数据分析洞察转化为具体的改进计划，并跟踪改进效果的闭环验证。

## 适用场景

- 月度改进项识别和优先级排序
- 标准变更预授权目录的定期评审和更新
- 变更管理成熟度评估和提升路线图
- 故障复盘中变更管理相关Action的落地推进
- 流程优化实验的设计和效果评估

## 执行步骤

### 步骤1：改进机会识别
来源渠道：
- 指标分析报告中的未达标项
- 高风险模式识别的改进建议
- 故障复盘中的变更相关Action
- 团队反馈（审批效率投诉、流程繁琐反馈）
- 行业对标差距（vs ITIL最佳实践、vs 行业头部企业）

### 步骤2：改进项评估与优先级
评估维度：
- **影响度**：改进后对核心指标的预期提升幅度
- **紧迫度**：当前问题是否在恶化、是否有合规风险
- **可行性**：技术实现难度、资源需求、组织变革阻力
- **成本收益比**：投入成本 vs 预期收益（量化或定性）

优先级矩阵：
- P0（紧急）：影响高+紧迫度高（如合规违规、成功率持续下降）
- P1（重要）：影响高+可行性好（如自动化提升、工具优化）
- P2（计划）：影响中+需要长期投入（如架构优化、文化建设）
- P3（观察）：影响低或可行性差（记录但不立即行动）

### 步骤3：改进计划制定
每个改进项必须包含：
- 目标定义：SMART原则（具体、可衡量、可达成、相关性、时限性）
- 实施方案：具体步骤、责任人、里程碑
- 验证标准：如何衡量改进是否达到预期效果
- 风险评估：改进本身可能带来的负面影响
- 回退方案：如改进效果不佳如何回退

### 步骤4：标准变更目录管理
预授权目录纳入评估标准：
- 历史执行成功率≥99.9%（近6个月）
- 连续成功次数≥20次
- 操作步骤已标准化且自动化程度高
- 影响面可控且一致

预授权目录移出条件：
- 连续出现失败（2次/季度）
- 环境变化导致操作性质改变
- 长期（6个月）未使用

### 步骤5：效果验证与闭环
- 改进实施后观察期：至少1个完整月度周期
- 对比改进前后的相关指标变化
- 确认改进效果稳定后标记"已验证"
- 未达预期的改进：分析原因、调整方案或关闭
- 季度汇总改进成效，向管理层汇报

## 输入规范

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| improvement_source | enum | 是 | 来源（指标分析/故障复盘/团队反馈/对标/审计） |
| current_state | string | 是 | 当前状态描述 |
| target_state | string | 是 | 期望目标状态 |
| metrics_impact | array | 否 | 影响的指标列表 |

## 最佳实践

1. **小步快跑**：优先实施投入小、见效快的改进，积累势能后再推大改进
2. **数据说话**：改进前后必须有可量化的对比数据，避免"感觉变好了"
3. **闭环管理**：每个改进项必须有明确的结论（成功/失败/暂停），不允许无限期"进行中"
4. **避免过度优化**：当指标已达到优秀水平（如99.8%成功率），进一步提升的边际成本极高
5. **组织学习**：将改进经验形成文档，避免重复踩坑

## 约束与限制

- 改进效果的验证需要足够的观察周期和样本量
- 流程变更可能遭遇组织阻力，需要管理层支持
- 并行推进的改进不宜超过5个，避免变量过多无法归因
- 某些改进（如文化建设）难以量化，需使用定性评估方法
