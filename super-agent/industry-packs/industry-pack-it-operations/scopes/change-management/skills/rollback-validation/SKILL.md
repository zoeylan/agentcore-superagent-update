# 回滚方案验证 (Rollback Validation)

## 概述

回滚方案验证是确保每个变更都有可靠的"安全网"的关键技能。在生产环境中，回滚方案的有效性直接决定了故障恢复时间（MTTR）。一个不完整或不可行的回滚方案比没有方案更危险——它会给执行团队带来虚假的安全感。本技能通过多维度审核回滚方案的完整性、可行性和时效性，确保在需要回滚时能够快速、安全地将系统恢复到变更前状态。

## 适用场景

- 变更申请提交后的回滚方案审核
- 变更执行前的回滚就绪确认（Pre-check环节）
- 回滚方案的定期有效性复验
- 变更失败后的回滚可行性快速评估

## 执行步骤

### 步骤1：回滚方案完整性检查
必须包含的内容项：
- [ ] 回滚触发条件（什么情况下执行回滚）
- [ ] 回滚决策人（谁有权决定回滚）
- [ ] 回滚步骤（具体操作步骤，精确到命令级别）
- [ ] 回滚顺序（多步骤变更的逆序操作）
- [ ] 回滚验证方法（如何确认回滚成功）
- [ ] 回滚预估时间（必须≤变更执行时间的50%）
- [ ] 数据兼容性说明（新老版本数据格式是否兼容）
- [ ] 不可回滚场景的应急方案（如有不可逆步骤）

### 步骤2：可行性验证
- 验证回滚脚本/工具是否已准备就绪（不接受"到时候再写"）
- 验证回滚目标版本/配置是否可获取（制品库/配置中心有留存）
- 验证回滚操作的权限是否已准备（是否需要额外授权）
- 验证回滚是否有环境依赖（如依赖某个中间状态的数据）
- 对数据库变更：验证是否有反向DDL/DML脚本并已通过DBA审核

### 步骤3：时效性验证
- 回滚预估时间是否满足≤变更时间50%的硬性要求
- 对灰度发布类变更：各批次的回滚时间是否独立评估
- 回滚操作是否可在维护窗口剩余时间内完成
- 是否考虑了回滚后的验证时间

### 步骤4：风险评估
- 回滚操作本身是否有副作用（如数据库回滚可能丢失增量数据）
- 是否存在"回滚陷阱"（回滚后系统进入更糟糕的状态）
- 多服务并行变更时，部分回滚是否导致不一致
- 回滚失败的应急plan-B

### 步骤5：审核结论输出
- 通过：回滚方案完整且可行
- 待完善：列出具体缺失项，退回补充
- 不通过：存在关键风险，需重新设计回滚方案

## 输入规范

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| change_id | string | 是 | 变更工单标识 |
| change_type | enum | 是 | 变更类型 |
| change_duration_min | number | 是 | 预计变更执行时长（分钟） |
| rollback_plan | object | 是 | 回滚方案详情 |
| deployment_strategy | enum | 否 | 发布策略（灰度/蓝绿/滚动） |

## 最佳实践

1. **回滚方案不接受"人工判断"**：回滚步骤必须明确到可执行的命令或操作，不接受"根据情况处理"
2. **预演验证**：高风险变更的回滚方案应在测试环境预演验证过
3. **自动化优先**：鼓励使用自动化回滚脚本，减少人工操作环节的出错概率
4. **时间要留余量**：回滚预估时间建议在实际测试基础上增加50%的buffer
5. **数据变更特殊关注**：涉及数据变更的回滚方案是审核重点，需确认数据备份和恢复可行性

## 约束与限制

- 回滚方案验证不替代实际的回滚测试（特别是数据库变更）
- 对于全新技术栈的变更，回滚方案的可行性判断可能需要技术专家协助
- 不可逆变更（如数据删除、Schema破坏性变更）必须有替代的应急方案
- 回滚方案的有效性有时效性（如依赖的旧版本制品可能被清理），需关注保留期限
