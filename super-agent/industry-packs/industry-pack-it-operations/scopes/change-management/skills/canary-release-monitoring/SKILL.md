# 灰度发布监控 (Canary Release Monitoring)

## 概述

灰度发布监控是在变更执行过程中，通过分批放量和实时指标监控相结合，确保变更的影响在可控范围内逐步验证的技能。核心理念是"小批量验证，快速反馈"——通过1%→10%→50%→100%的分批放量策略，在每个批次验证核心指标稳定后再扩大影响范围。一旦发现异常，立即暂停放量并评估回滚，将"爆炸半径"控制在最小范围。

## 适用场景

- 代码发布的分批灰度验证
- 配置变更的逐步生效验证
- 流量切换的渐进式推进
- 任何需要分阶段验证的生产变更

## 执行步骤

### 步骤1：灰度策略配置
- 确定灰度批次和比例：最少3批（如1%→10%→50%→100%）
- 确定每批次的观察时间：至少覆盖一个完整业务周期（最短5分钟）
- 确定灰度维度：按机器/按用户/按流量比例/按地域
- 确定放量判定标准：各批次通过的具体指标条件

### 步骤2：基线采集
- 变更执行前采集当前核心指标的基线值
- 基线指标必须包含（Golden Signals）：
  - **延迟**：P50/P95/P99响应时间
  - **流量**：QPS/TPS
  - **错误**：错误率（HTTP 5xx比例、业务异常比例）
  - **饱和度**：CPU/内存/连接数使用率
- 基线采集周期：至少过去30分钟的稳定数据

### 步骤3：分批执行与监控
每个批次执行后：
- **即时检查**（放量后30秒内）：服务是否存活、健康检查是否通过
- **短期观察**（放量后1-5分钟）：错误率、响应时间是否有突变
- **稳定验证**（放量后5-15分钟）：各指标是否回归稳定，无持续恶化趋势
- 5秒粒度采集监控数据，实时对比基线

### 步骤4：异常判定
触发告警的条件（任一满足即暂停放量）：
- 错误率：相对基线上升 > 2倍 或 绝对值 > 1%
- P99响应时间：相对基线上升 > 3倍 或 绝对值 > 3秒
- 成功率：下降 > 1个百分点
- CPU/内存：突增 > 30%（相对增量）
- 日志异常：新增ERROR日志模式 > 10条/分钟

### 步骤5：放量决策
- **继续放量**：所有指标在基线±20%范围内稳定5分钟以上
- **暂停观察**：指标轻微波动（±20-30%），延长观察时间至15分钟
- **回滚决策**：指标严重偏离基线（>30%）或出现关键错误
- 每次放量决策需记录当时的指标快照作为决策依据

## 输入规范

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| change_id | string | 是 | 变更工单标识 |
| canary_strategy | object | 是 | 灰度策略（批次/比例/观察时间） |
| baseline_metrics | object | 是 | 基线指标快照 |
| threshold_config | object | 否 | 自定义告警阈值（默认使用标准阈值） |

## 最佳实践

1. **首批尽可能小**：第一批灰度控制在1%以内，用最小代价验证变更基本正确
2. **观察时间不可压缩**：即使指标完美也要等够观察时间，部分问题有延迟表现
3. **关注长尾指标**：P99比P50更能发现问题，关注指标分布而非仅关注均值
4. **业务指标优先**：核心业务成功率（下单成功率、支付成功率）比技术指标更直接反映影响
5. **夜间流量注意**：低流量时段指标波动大，需适当放宽阈值或延长观察时间

## 约束与限制

- 灰度监控有效性依赖监控系统的数据采集实时性（5秒粒度要求）
- 某些变更无法灰度（如数据库Schema变更、全局配置），需用其他策略替代
- 低流量时段（如凌晨）指标样本不足，统计显著性较差
- 灰度维度的选择可能导致样本偏差（如按地域灰度可能遇到区域性问题）
