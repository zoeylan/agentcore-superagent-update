# 变更风险评分 (Risk Scoring)

## 概述

变更风险评分是将变更的多维度风险因素量化为统一的风险分值和风险等级的技能。通过加权计算变更类型、影响范围、执行时间、团队历史成功率、回滚复杂度等因子，输出客观、可比较的风险评分，直接决定变更的审批流程路由。该技能是将主观的风险判断转化为可重复、可审计的标准化流程的关键。

## 适用场景

- 每一个变更申请的风险等级判定
- 批量变更的优先级排序
- 风险评估模型的校准和优化
- 变更类型是否可纳入标准变更预授权目录的评判

## 执行步骤

### 步骤1：风险因子采集
收集以下维度的数据：
- **变更类型权重**：代码发布(基础分30)、配置变更(20)、数据库变更(50)、基础设施变更(60)、安全变更(45)
- **影响范围系数**：影响面分析结果的impact_score（0-100）
- **执行时间风险**：工作时间(系数1.0)、维护窗口(0.8)、非维护窗口夜间(1.3)、周五(1.5)
- **团队历史成功率**：该团队近90天同类变更的历史成功率
- **回滚复杂度**：自动回滚(系数0.5)、手动回滚可逆(1.0)、部分不可逆(1.5)、完全不可逆(2.0)
- **变更频次**：首次执行(系数1.3)、历史成功≥5次(0.8)、历史成功≥20次(0.5)

### 步骤2：风险分值计算
```
风险总分 = (变更类型权重 × 0.25) + (影响范围系数 × 0.30) + (时间风险系数 × 变更类型权重 × 0.15) + ((1-历史成功率) × 100 × 0.15) + (回滚复杂度 × 20 × 0.15)
```
附加规则：
- 涉及核心交易链路：总分 × 1.3
- 涉及数据不可逆操作：总分 × 1.5
- 执行时间在变更冻结期：直接判定为"禁止执行"

### 步骤3：风险等级判定
- **标准变更**（≤25分）：匹配预授权目录 + 历史成功率≥99.9% + 影响面评分≤20
- **普通变更**（26-60分）：需CAB审批，工作日24小时内完成审批
- **高风险变更**（61-85分）：需VP+审批 + 限维护窗口执行
- **极高风险变更**（>85分）：需CTO审批 + 维护窗口 + 灰度方案强制 + 双人复核

### 步骤4：审批建议生成
- 根据风险等级映射审批流程
- 标注需要额外关注的风险点
- 给出执行建议（如建议加长灰度验证时间、增加监控指标等）

## 输入规范

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| change_type | enum | 是 | 变更类型 |
| impact_score | number | 是 | 影响面评分(0-100) |
| planned_time | datetime | 是 | 计划执行时间 |
| team_id | string | 是 | 执行团队标识 |
| rollback_type | enum | 是 | 回滚方式类型 |
| execution_count | number | 是 | 同类变更历史执行次数 |
| critical_path | boolean | 是 | 是否涉及核心交易链路 |
| irreversible | boolean | 是 | 是否包含不可逆操作 |

{
  "risk_score": 72,
  "risk_level": "高风险变更",
  "score_breakdown": {

    "history_factor": 12.0,

## 最佳实践

1. **模型持续校准**：每月基于实际变更结果（成功/失败）回测评分模型的准确性，调整权重
2. **边界case人工审核**：风险分在等级临界值±5分范围内的变更，建议人工确认等级判定
3. **团队能力动态评估**：新组建团队或人员变动大的团队，历史成功率参考需打折
4. **组合风险叠加**：同一维护窗口内多个变更的组合风险大于单个变更的简单加总
5. **季节性因子**：大促前/节假日期间的风险系数应适当提高

## 约束与限制

- 评分模型基于历史统计数据，对全新类型的变更（无历史参考）准确性有限
- 团队历史成功率需≥10次样本才具备统计意义，否则使用行业平均值
- 风险评分是辅助决策工具，不替代人工的最终判断
- 评分模型的权重参数需根据组织实际情况定期调整
