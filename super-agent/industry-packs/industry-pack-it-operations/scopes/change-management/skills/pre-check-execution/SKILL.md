# 变更执行前检查 (Pre-check Execution)

## 概述

Pre-check是变更执行前的最后一道安全关卡，通过8项必检清单的逐一验证，确保变更在所有前置条件就绪的情况下才开始执行。任何一项检查未通过都必须阻断变更执行。这不是形式主义的"打勾"环节，而是用结构化的检查消除人为遗忘和疏忽带来的风险——业界数据显示，约15%的变更失败可追溯到执行前准备不充分。

## 适用场景

- 每一次生产变更执行前的强制检查
- 灰度发布各批次放量前的增量检查
- 回滚方案执行前的快速Pre-check
- 变更被中断后恢复执行时的重新检查

## 执行步骤

### 8项必检清单

#### 检查项1：审批状态确认
- 确认变更工单状态为"已审批"且在有效期内
- 确认审批流程完整（无跳步、无过期）
- 确认执行时间在审批允许的窗口内

#### 检查项2：回滚方案就绪验证
- 回滚脚本/工具已准备并可访问
- 回滚目标版本（旧版本制品）在制品库中可获取
- 回滚预估时间确认（≤变更时间50%）
- 回滚方案已通过验证审核

#### 检查项3：监控大盘已打开确认
- 核心监控大盘已打开并正常显示
- 监控指标基线已标记（用于对比变更前后）
- 告警通道畅通（验证告警接收人能正常收到通知）
- 自定义的变更专属监控视图已就绪

#### 检查项4：相关方通知完成确认
- 上下游依赖团队已通知并确认知晓
- 值班运维已通知（知道当前有变更在执行）
- 业务方已通知（如变更可能影响用户体验）
- 客服团队已预警（如可能引发用户咨询/投诉）

#### 检查项5：执行人和复核人就位确认
- 主执行人在线并已准备好操作
- 复核人在线并已打开监控大盘
- 高风险变更：DBA/安全/网络专家按需就位
- 确认所有人员的联系方式通畅

#### 检查项6：依赖服务状态正常确认
- 变更对象的上下游依赖服务当前运行正常
- 无正在进行中的其他影响同一链路的变更
- 当前无活跃故障影响变更执行环境
- 依赖的外部服务（如CDN、DNS、云服务）状态正常

#### 检查项7：资源余量充足确认
- CPU/内存/磁盘余量满足变更需求（如新版本部署需要更多资源）
- 数据库连接池、线程池等有足够余量
- 网络带宽满足变更期间的额外需求
- 变更工具/平台本身的可用性确认

#### 检查项8：变更脚本/工具就绪确认
- 变更脚本已上传到执行环境并通过MD5/SHA校验
- 自动化工具配置正确（如Ansible playbook、K8s manifest）
- 变更参数已最终确认（环境变量、配置值等）
- 灰度策略配置已就绪（流量分组、权重设置等）

## 输入规范

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| change_id | string | 是 | 变更工单标识 |
| executor | string | 是 | 主执行人 |
| reviewer | string | 是 | 复核人 |
| target_environment | string | 是 | 目标环境 |
| change_artifacts | array | 是 | 变更制品列表 |

## 最佳实践

1. **检查顺序优化**：先检查"快速失败"项（如审批是否有效），再检查耗时项（如资源余量）
2. **自动化为主**：8项中至少6项应能自动化检查，仅人员就位需人工确认
3. **超时处理**：如Pre-check超过15分钟未全部完成，需重新评估是否继续
4. **记录留痕**：Pre-check的每一项结果和时间戳必须自动记录，作为变更执行的准入凭证
5. **动态感知**：检查项6（依赖服务状态）应实时感知，避免检查时正常但执行时已异常

## 约束与限制

- Pre-check结果有时效性，超过30分钟未执行变更需重新检查
- 自动化检查依赖与监控系统、CMDB、工单系统的API集成
- 人员就位确认依赖人工响应，可能是流程瓶颈
- Pre-check通过不代表变更一定成功，仅确保前置条件满足
