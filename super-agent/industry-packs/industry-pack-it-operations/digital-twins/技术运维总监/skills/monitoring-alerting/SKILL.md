# 监控体系设计与告警治理

## 场景描述
设计和优化全链路监控体系，制定告警策略，解决告警风暴和告警疲劳问题。帮助团队建立以SLO为核心的可观测性体系，实现从被动告警到主动发现的转型。

## 触发条件
- 用户需要设计或优化监控方案
- 用户面临告警风暴或告警疲劳问题
- 用户需要制定SLO/SLI体系
- 用户需要选型监控工具或设计Dashboard
- 涉及关键词：监控、告警、Prometheus、Grafana、SLO、SLI、可观测性、Dashboard

## 执行框架

### 1. 监控分层体系

#### 基础设施层（Infrastructure）
| 指标 | 采集方式 | 告警阈值 | 说明 |
|------|----------|----------|------|
| CPU使用率 | node_exporter | >80%持续5min | 排除短暂毛刺 |
| 内存使用率 | node_exporter | >85% | 预留OOM缓冲 |
| 磁盘使用率 | node_exporter | >85%/90%(预警/严重) | 含增长趋势预测 |
| 磁盘IO | node_exporter | await>50ms | 区分HDD/SSD阈值 |
| 网络丢包率 | blackbox_exporter | >0.1% | 结合MTR定位 |
| TCP连接数 | node_exporter | >60000 | 接近端口耗尽 |

#### 中间件层（Middleware）
- **MySQL**：QPS/TPS、慢查询数、连接池使用率、主从延迟、Deadlock次数
- **Redis**：命中率(<95%告警)、内存使用率、连接数、bigkey检测
- **Kafka**：消费Lag、ISR收缩、分区Leader分布不均
- **Nginx**：5xx占比、活跃连接数、请求排队时长

#### 应用层（Application）
- RED指标：Rate(请求速率)、Errors(错误率)、Duration(延迟)
- 四个黄金信号：延迟、流量、错误、饱和度
- JVM/Go Runtime：GC暂停时间、Goroutine数、线程池活跃数

#### 业务层（Business）
- 核心转化漏斗指标
- 订单/支付成功率
- 用户登录成功率
- API调用量环比

### 2. SLO体系设计

#### SLI选择原则
- 可用性SLI：成功请求数 / 总请求数
- 延迟SLI：延迟低于阈值的请求占比（取P99而非平均值）
- 数据新鲜度SLI：数据更新延迟在可接受范围内的比例

#### SLO设定方法
- 基于历史数据：取过去30天的P95作为基线
- 基于业务需求：与产品/业务方协商用户可接受的底线
- 典型值：核心服务99.95%，非核心99.9%，内部工具99%
- Error Budget = 1 - SLO（如SLO=99.95%，月Error Budget≈21.6分钟）

#### Error Budget消耗策略
- Budget充足(>50%)：允许快速迭代，接受一定风险
- Budget紧张(20-50%)：收紧变更审批，增加灰度比例
- Budget耗尽(<20%)：冻结非紧急变更，聚焦稳定性改进

### 3. 告警治理

#### 告警分级
- **P0-Critical**：业务中断，5分钟内响应，电话通知
- **P1-High**：核心功能受损，15分钟内响应，IM+短信
- **P2-Medium**：性能下降，1小时内响应，IM通知
- **P3-Low**：潜在风险，工作时间处理，邮件通知

#### 告警抑制与聚合
- 时间抑制：相同告警5分钟内不重复发送
- 层级抑制：上游故障时抑制下游告警
- 聚合策略：同类告警合并，展示影响全貌
- 静默窗口：变更期间对相关服务静默

#### 告警质量指标
- 告警响应率 > 95%
- 告警有效率 > 70%（非误报）
- MTTA(平均确认时间) < 5分钟
- 告警→工单转化率追踪

### 4. Dashboard设计原则
- **层次化**：全局概览 → 服务详情 → 实例粒度
- **目标导向**：每个Dashboard回答一个明确问题
- **减少认知负担**：关键指标一屏可见，颜色语义一致
- **可操作性**：异常指标可直接下钻到原因

## Prometheus最佳实践
- 命名规范：`<namespace>_<subsystem>_<name>_<unit>`
- 避免高基数Label：URL路径参数化，用户ID绝不作为Label
- Recording Rules预聚合高频查询
- 联邦集群架构应对大规模场景
- Thanos/VictoriaMetrics解决长期存储需求
