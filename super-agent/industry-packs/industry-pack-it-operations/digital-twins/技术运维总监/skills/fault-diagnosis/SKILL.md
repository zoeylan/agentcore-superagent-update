# 故障诊断与应急响应

## 场景描述
当生产环境出现告警或业务异常时，快速进行故障定位、影响评估和应急处置。覆盖从单机层面的性能瓶颈到分布式系统的级联故障等各种场景。

## 触发条件
- 用户描述生产环境告警或异常现象
- 用户提供系统指标异常数据（CPU、内存、磁盘、网络）
- 用户询问故障排查思路或应急处置方案
- 涉及关键词：故障、告警、宕机、超时、OOM、丢包、慢查询、502/503/504

## 执行框架

### 1. 快速评估（30秒内）
- 确认影响面：哪些业务/用户受影响？影响程度？
- 判断故障等级：P0(全站不可用) / P1(核心功能受损) / P2(非核心功能异常) / P3(性能下降)
- 决定是否需要立即启动应急响应流程

### 2. 止血优先
- 优先恢复服务可用性，而非追查根因
- 常见止血手段：回滚变更、切换流量、重启服务、限流降级、扩容
- 评估止血方案本身的风险

### 3. 系统性排查
按照OSI模型自底向上排查：
- **基础设施层**：物理机/VM/容器的CPU、内存、磁盘IO、网络连通性
- **操作系统层**：内核日志(dmesg)、OOM记录、文件描述符耗尽、TCP连接状态
- **中间件层**：数据库连接池、缓存命中率、消息队列积压
- **应用层**：错误日志、链路追踪、GC状态、线程池
- **业务层**：流量突增、数据异常、上下游依赖

### 4. 输出规范
- 给出明确的排查命令和预期输出
- 提供决策树：如果A则做X，如果B则做Y
- 量化指标阈值：明确什么数值算正常，什么算异常
- 标注风险点和注意事项

## 专业知识库

### Linux性能诊断
- CPU：`top` → `pidstat` → `perf top` → `perf record + flamegraph`
- 内存：`free` → `/proc/meminfo` → `slabtop` → `valgrind`
- 磁盘：`iostat -x` → `iotop` → `blktrace`
- 网络：`ss -s` → `netstat -ant | awk` → `tcpdump` → `wireshark`

### 常见故障模式
- **TCP连接风暴**：TIME_WAIT堆积 → 调整tw_reuse/增大端口范围
- **OOM Kill**：区分真实内存泄漏 vs limits设置不当（对比RSS增长趋势）
- **磁盘IO饱和**：区分随机IO vs 顺序IO，确认是日志写入还是数据库刷盘
- **级联故障**：超时传播 → 检查熔断器配置和超时层级设置
- **DNS解析故障**：先确认/etc/resolv.conf和nscd缓存状态

### K8s环境特有排查
- Pod CrashLoopBackOff → `kubectl logs --previous` + 检查探针配置
- Pod Pending → 资源不足 / NodeAffinity / PVC绑定失败
- Service不通 → Endpoint是否就绪 / NetworkPolicy / kube-proxy规则

## 沟通原则
- 紧急情况下用最简洁的语言
- 每个建议都标注优先级和预估耗时
- 明确区分"已确认的结论"和"需要进一步验证的假设"
