# 变更管理与发布策略

## 场景描述
为生产环境的变更操作提供风险评估、方案设计和发布策略指导。涵盖应用发布、基础设施变更、配置调整、数据库DDL等各类变更场景，确保变更可控、可追溯、可回滚。

## 触发条件
- 用户需要制定变更方案或发布计划
- 用户询问变更风险评估方法
- 用户需要设计灰度发布或蓝绿部署策略
- 涉及关键词：发布、上线、变更、迁移、升级、回滚、灰度、蓝绿、金丝雀

## 执行框架

### 1. 变更分类与定级
| 等级 | 定义 | 审批要求 | 变更窗口 |
|------|------|----------|----------|
| S级 | 影响全站/核心链路 | CTO+总监审批 | 业务低峰期，需值班 |
| A级 | 影响单业务线 | 总监审批 | 工作日低峰期 |
| B级 | 影响单服务 | 经理审批 | 工作日 |
| C级 | 配置微调 | 自审 | 随时 |

### 2. 变更方案模板
```
【变更标题】
【变更类型】应用发布 / 基础设施 / 数据库 / 配置 / 网络
【影响范围】涉及的服务、业务线、用户群体
【变更时间】开始时间 → 预计完成时间
【前置条件】依赖的准备工作
【变更步骤】详细到可执行的step-by-step
【验证方法】如何确认变更成功
【回滚方案】回滚步骤 + 回滚耗时预估
【风险评估】可能的异常情况及应对措施
【通知计划】变更前/中/后的通知对象
```

### 3. 发布策略选择

#### 滚动更新（Rolling Update）
- 适用：无状态服务、兼容性变更
- 关键参数：maxSurge、maxUnavailable、minReadySeconds
- 注意：新旧版本共存期间的接口兼容性

#### 金丝雀发布（Canary）
- 适用：功能变更、不确定性较高的版本
- 流量比例：1% → 5% → 20% → 50% → 100%
- 观察指标：错误率、延迟P99、业务转化率
- 最小观察时间：每级至少15分钟

#### 蓝绿部署（Blue-Green）
- 适用：大版本升级、数据库schema变更
- 前提：足够的资源冗余（2倍容量）
- 优势：秒级回滚
- 风险：数据库兼容性、会话状态

#### 特殊变更注意事项
- **数据库DDL**：大表加索引用pt-osc/gh-ost，避免锁表
- **K8s集群升级**：先升级控制面，再逐节点drain+升级
- **网络变更**：双人操作，保留console接入的回退路径
- **存储扩容**：LVM在线扩容，确认文件系统支持online resize

### 4. 回滚决策矩阵
- 错误率超过基线2倍 → 立即回滚
- P99延迟超过SLO阈值 → 观察5分钟，持续则回滚
- 核心业务指标下降10%以上 → 立即回滚
- 单机异常 → 摘除异常节点，继续观察

## 最佳实践
- 变更必须有明确的Owner和Reviewer
- 高风险变更必须有"变更演练"记录
- 所有变更操作必须有审计日志
- 变更窗口期间禁止并行执行其他变更
- Feature Flag优先于代码分支管理
- 配置变更与代码变更解耦，支持热加载
