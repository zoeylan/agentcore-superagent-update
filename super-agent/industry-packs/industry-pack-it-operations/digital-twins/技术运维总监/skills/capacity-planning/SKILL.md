# 容量规划与架构评审

## 场景描述
基于业务增长趋势和技术指标，进行系统容量评估、扩容规划和架构合理性评审。涵盖日常容量管理、大促保障、成本优化和技术架构风险评估。

## 触发条件
- 用户需要评估系统容量是否充足
- 用户需要制定扩容计划或大促保障方案
- 用户需要进行架构评审或技术选型
- 用户关注云资源成本优化
- 涉及关键词：容量、扩容、大促、压测、架构评审、成本优化、FinOps

## 执行框架

### 1. 容量评估方法论

#### 水位线模型
- **绿色水位(≤60%)**：健康，可承受日常波动
- **黄色水位(60-80%)**：关注，需启动扩容计划
- **红色水位(≥80%)**：紧急，立即扩容或限流

#### 容量公式
```
所需容量 = 峰值流量 × 安全系数(1.5-2.0) / 单实例承载能力
```

#### 关键维度
- **计算容量**：CPU/内存利用率，请求并发数
- **存储容量**：磁盘增长速度，数据生命周期
- **网络容量**：带宽利用率，连接数上限
- **中间件容量**：连接池大小，队列深度

### 2. 压测与基线建立

#### 压测类型
- **基准测试**：确定单实例极限性能
- **容量验证**：确认当前集群能承载目标流量
- **极限压测**：找出系统崩溃点
- **稳定性测试**：长时间持续压力下的表现

#### 压测注意事项
- 必须在独立环境或有流量隔离的生产环境进行
- 压测数据与真实数据的差异校正（通常真实场景比压测差20-30%）
- 逐步加压，观察拐点：响应时间突增 = 瓶颈出现
- 关注短板效应：CPU/内存/IO/网络/连接池哪个先饱和
- 压测后必须检查：数据是否污染、资源是否回收

#### 大促保障方案框架
```
1. 流量预估：基于历史同比+业务增长系数
2. 瓶颈分析：全链路各环节容量盘点
3. 扩容方案：需要多少资源、何时扩容、何时缩容
4. 降级预案：各级别限流降级策略
5. 演练计划：故障注入测试 + 应急预案验证
6. 值班安排：核心时段人员在位保障
```

### 3. 架构评审要点

#### 可用性评审
- 单点故障识别：任何单一组件故障是否导致业务中断？
- 容灾架构：同城双活/异地多活的数据同步方案
- 故障域隔离：不同AZ/Region的独立性
- RTO/RPO承诺与实际能力的差距

#### 可扩展性评审
- 水平扩展能力：是否存在有状态瓶颈？
- 数据分片策略：分片键选择是否合理？热点问题？
- 缓存策略：缓存穿透/击穿/雪崩的防护
- 异步化程度：核心链路的同步调用是否过长？

#### 运维友好性评审
- 可观测性：关键路径是否有完整Metrics/Logs/Traces？
- 变更安全：配置是否支持热加载？发布是否支持灰度？
- 故障自愈：健康检查+自动重启+自动扩容
- 文档完备：架构图、依赖关系、运维手册

### 4. 成本优化（FinOps）

#### 资源利用率优化
- 识别闲置资源：CPU<10%持续7天的实例
- Right-sizing：基于实际使用量调整规格
- 预留实例/Savings Plans规划（稳定负载用预留，弹性负载用按量）
- Spot实例在无状态计算场景的应用

#### 成本可视化
- 按业务线/团队/环境分摊成本
- 建立单位成本指标：每笔交易成本、每个用户成本
- 设置成本异常告警：日环比增长>20%

#### 常见浪费场景
- 测试环境7×24运行（应工作时间启停）
- 日志存储无生命周期管理（热→温→冷→删除）
- 过度冗余的容灾设计（3AZ对于非核心服务是浪费）
- 未清理的快照、镜像和废弃资源
