# Audit Trail Management（审计追踪管理）

## 技能概述

本技能负责审核全链路审计日志的采集、存储、查询和合规管理。确保每一条内容从进入审核系统到最终处置的全过程都有完整、不可篡改的记录，满足监管合规追溯要求（180天保留期）。支持高效的审计查询、合规报表生成和定期审计抽检。

## 适用场景

- 审核全流程日志的自动采集和结构化存储
- 监管机构要求的合规追溯查询
- 定期合规审计抽检（季度1%抽检）
- 用户申诉时的审核过程复查
- 审核系统合规性评估和报告

## 执行步骤

### Step 1: 审计事件采集
- 在审核流程的每个关键节点自动生成审计事件：
  - **前置拦截节点**：请求到达/检测结果/拦截决策/净化改写
  - **审核引擎节点**：审核开始/各维度结果/综合评分/处理耗时
  - **分流决策节点**：置信度/分流通道/决策依据
  - **人工复核节点**：工单创建/分配/开始复核/提交结论/升级
  - **最终处置节点**：放行/拒绝/修改/申诉/二次处理
- 每个事件记录：时间戳、操作人（系统/人工）、操作类型、输入输出、决策依据

### Step 2: 日志结构化存储
- 标准化日志格式：
  ```json
  {
    "trace_id": "全局追踪ID（同一条内容全链路使用相同ID）",
    "event_id": "事件唯一ID",
    "timestamp": "ISO-8601时间戳",
    "node": "审核节点标识",
    "actor": "操作者（system_agent/reviewer_id）",
    "action": "操作类型",
    "input_hash": "输入内容hash",
    "output": "操作输出结果",
    "decision_basis": "决策依据说明",
    "metadata": "附加元数据"
  }
  ```
- 日志写入采用追加模式（append-only），不允许修改或删除
- 关键字段建立索引：trace_id、timestamp、node、action

### Step 3: 数据完整性保障
- 每条日志生成SHA-256摘要
- 使用Merkle Tree结构保证日志链不可篡改
- 定期（每小时）生成日志完整性校验报告
- 发现数据异常（缺失/篡改）立即触发安全告警

### Step 4: 合规查询服务
- 支持多维度查询：
  - 按trace_id查询单条内容的完整审核历程
  - 按时间范围+条件筛选批量日志
  - 按审核员查询其所有复核记录
  - 按规则ID查询该规则的所有触发记录
- 查询结果支持导出（JSON/CSV/PDF报告格式）
- 查询操作本身也记入审计日志（谁查了什么）

### Step 5: 定期审计抽检
- 每季度执行合规审计抽检：
  - 随机抽取1%已审核内容
  - 由独立审计人员（非日常审核团队）进行复核
  - 对比抽检结论与原始审核结论的一致性
  - 不一致率>5%触发全面排查
- 抽检结果形成审计报告，归档备查

### Step 6: 数据生命周期管理
- 热数据（30天内）：高性能存储，支持实时查询
- 温数据（30-180天）：标准存储，支持分钟级查询
- 冷数据（180天后）：归档存储，仅在合规要求下恢复查询
- 到期数据的安全销毁（满足数据最小化原则）

## 输入规格

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| action | enum | 是 | log/query/audit/export/verify |
| audit_event | object | 条件 | 审计事件详情（log时必填） |
| query_params | object | 条件 | 查询参数（query时必填） |
| audit_config | object | 条件 | 抽检配置（audit时必填） |
| export_format | enum | 条件 | 导出格式（export时必填） |

## 输出规格

| 字段 | 类型 | 说明 |
|------|------|------|
| event_id | string | 事件ID（log操作时） |
| query_results | array | 查询结果列表 |
| total_count | int | 查询命中总数 |
| integrity_status | object | 完整性校验状态 |
| audit_report | object | 审计抽检报告 |
| export_url | string | 导出文件下载链接 |

## 最佳实践

1. **全量记录**：宁可多记也不遗漏，审计日志的完整性是合规的基础
2. **不可篡改**：采用append-only+密码学校验，确保日志不被事后修改
3. **高效索引**：预设常用查询场景的索引，监管检查时能快速响应
4. **权限控制**：审计日志的访问权限严格管控，操作留痕
5. **成本控制**：热温冷分层存储，平衡合规需求和存储成本

## 约束条件

- 日志写入延迟<100ms（不影响审核主流程性能）
- 热数据查询响应<1秒，温数据<30秒
- 存储保留期严格180天（不提前删除、到期及时清理）
- 日志完整性校验每小时执行一次
- 审计日志系统本身的可用性>99.99%
- 单条trace的完整日志链可在10秒内重建
