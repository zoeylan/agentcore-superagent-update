# Grayscale Release Control（灰度发布控制）

## 技能概述

本技能管理审核规则的渐进式灰度发布流程，确保新规则从开发到生产环境的平滑过渡。通过NoOp（观察模式）→ Shadow（影子模式）→ Prod（生产模式）三阶段验证机制，将规则变更的风险降到最低，避免一次性全量上线导致的大规模误判或漏判。

## 适用场景

- 新审核规则的标准上线流程
- 现有规则的重大变更（阈值调整/逻辑重构）
- 审核模型版本升级
- 监管热点事件的紧急规则加速上线
- 规则下线的灰度退出

## 执行步骤

### Step 1: 发布计划制定
- 确定灰度策略：标准流程（NoOp≥24h → Shadow≥48h → Prod）或加速流程（Shadow≥4h → Prod）
- 定义各阶段的验收指标：
  - NoOp阶段：触发频率评估、无明显异常模式
  - Shadow阶段：与人工/现有规则一致率>95%、FP率<预设阈值
  - Prod阶段：FN率<2%、FP率<1.5%
- 设置流量分配比例（Shadow阶段可设置部分流量生效）
- 定义回滚条件和回滚方案

### Step 2: NoOp阶段执行（观察模式）
- 规则部署到生产环境但仅记录不拦截
- 收集规则触发数据：触发频率、触发内容样本、触发分布（时段/渠道/内容类型）
- 评估触发量是否符合预期（过高可能存在规则过泛，过低可能规则失效）
- 检测是否存在异常触发模式（如集中在某一类正常内容上）
- 最短观察周期：24小时（紧急规则可跳过此阶段）

### Step 3: Shadow阶段执行（影子模式）
- 规则实际执行判定但不对外生效，记录判定结果
- 将规则判定结果与人工审核结果（或现有规则结果）进行对比
- 计算一致率：一致率 = 与基准结果一致的样本数 / 总样本数
- 分析不一致案例：是规则更准（发现了人工遗漏）还是规则误判
- 推进条件：一致率>95% 且 已验证样本量>100条 且 观察时间≥48小时（加速通道≥4小时）

### Step 4: Prod阶段切入（生产模式）
- 逐步放量：10% → 30% → 50% → 100%（每步观察≥2小时）
- 实时监控核心指标：FN率、FP率、触发频率
- 设置自动熔断阈值：FP率突增>5% 或 触发频率异常波动>200%时自动回滚
- 全量生效后进入稳定监控期（7天内保持高频监控）

### Step 5: 发布完成确认
- 全量生效后7天稳定期内指标正常
- 输出灰度发布总结报告（各阶段数据/发现的问题/最终效果）
- 规则状态从"deploying"更新为"active"
- 存档发布流程记录

### Step 6: 回滚执行（异常时）
- 触发回滚条件时，立即将规则状态回退到上一稳定阶段
- 回滚操作<5秒完成（配置热更新）
- 记录回滚原因和现场数据
- 通知策略中枢进行规则修正

## 输入规格

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| rule_id | string | 是 | 待发布的规则ID |
| release_type | enum | 是 | standard/accelerated/rollback |
| acceptance_criteria | object | 是 | 各阶段验收指标 |
| traffic_config | object | 否 | 流量分配配置 |
| rollback_conditions | object | 否 | 自定义回滚条件 |
| skip_noop | boolean | 否 | 是否跳过NoOp阶段（仅紧急规则） |

## 输出规格

| 字段 | 类型 | 说明 |
|------|------|------|
| release_id | string | 发布流程唯一ID |
| current_stage | enum | noop/shadow/prod/completed/rolled_back |
| stage_metrics | object | 当前阶段的监控指标 |
| progress_percentage | float | 总体进度百分比 |
| can_advance | boolean | 是否满足推进到下一阶段的条件 |
| issues_found | array | 发现的问题列表 |
| rollback_triggered | boolean | 是否触发了回滚 |
| summary_report | object | 发布总结报告（完成时） |

## 最佳实践

1. **不跳过Shadow**：即使是紧急规则，Shadow验证也不可省略——4小时是最低要求
2. **自动熔断**：必须配置自动回滚阈值，不依赖人工发现问题
3. **小步快跑**：Prod阶段逐步放量，不要一次性100%切入
4. **数据量要求**：Shadow阶段至少需要100条验证样本才具有统计显著性
5. **完整记录**：灰度全过程的数据和决策记录保留，用于复盘和改进

## 约束条件

- NoOp阶段最短24小时（紧急规则例外）
- Shadow阶段最短48小时/4小时（标准/加速），一致率必须>95%
- Prod阶段逐步放量，每步观察≥2小时
- 回滚操作响应时间<5秒
- 同时处于灰度中的规则不超过10条（避免相互干扰）
