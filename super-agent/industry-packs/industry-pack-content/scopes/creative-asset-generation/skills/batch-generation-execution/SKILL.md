# 批量生成执行（Batch Generation Execution）

## 技能概述

本技能负责大规模素材生成任务的并发调度、执行监控和异常处理。核心挑战在于：在保证生成质量的前提下，最大化吞吐量和资源利用率，同时优雅处理各种异常情况（超时/限流/安全拦截/配额耗尽）。支持单次任务100+素材的批量生产场景。

## 适用场景

- 电商大促批量商品主图生成（100-1000张/批次）
- Campaign系列素材的并行生产
- 多尺寸适配的批量裁切和生成
- 虚拟试穿的多服装×多角度组合生成

## 执行步骤

### Step 1: 任务队列构建
将子任务列表转化为可执行的任务队列：

```json
{
  "batch_id": "BATCH-20240101-001",
  "total_tasks": 100,
  "concurrency_limit": 10,
  "queue": [
    {
      "task_id": "T-001",
      "prompt": "...",
      "model": "flux-pro",
      "params": {},
      "priority": 1,
      "retry_count": 0,
      "max_retries": 3,
      "timeout_ms": 60000,
      "status": "pending"
    }
  ]
}
```

### Step 2: 并发调度执行
- 根据`concurrency_limit`控制同时执行的任务数
- 优先级高的任务优先调度
- 支持动态调整并发度（模型响应变慢时自动降低并发）
- 实施令牌桶限流，防止触发模型API的rate limit

### Step 3: 实时监控与告警
监控指标及阈值：
- **成功率**：实时计算 `成功数/已完成数`
  - >90%：正常（绿色）
  - 80-90%：注意（黄色）
  - <80%：告警（橙色）
  - <70%：暂停批次（红色）
- **平均延迟**：跟踪每个任务的响应时间
  - 延迟突增50%以上：降低并发度
- **配额消耗**：跟踪API配额使用情况
  - 剩余配额<20%：告警通知规划师

### Step 4: 异常处理策略

| 异常类型 | 处理策略 |
|----------|----------|
| 模型超时(>60s) | 重试1次，仍超时标记失败 |
| HTTP 429 (限流) | 指数退避重试(2s→4s→8s) |
| HTTP 5xx (服务错误) | 重试最多3次，失败后切备用模型 |
| 内容安全拦截 | 不重试，标记原因，上报分析 |
| 配额耗尽 | 暂停当前模型任务，切备用模型或排队等待 |
| 批次失败率>30% | 暂停整个批次，通知创意规划师 |

### Step 5: 结果汇总与交付
批次完成后输出执行报告：

```json
{
  "batch_id": "BATCH-20240101-001",
  "summary": {
    "total": 100,
    "succeeded": 93,
    "failed": 5,
    "safety_blocked": 2,
    "success_rate": 0.93,
    "avg_latency_ms": 12500,
    "total_cost": "$8.40",
    "total_time": "15min"
  },
  "failures": [
    { "task_id": "T-023", "reason": "timeout after 3 retries", "model": "flux-pro" }
  ],
  "safety_blocks": [
    { "task_id": "T-067", "reason": "content_policy_violation", "detail": "..." }
  ]
}
```

## 输入规格

| 输入项 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| task_queue | array | 是 | 完整的任务队列（含提示词+模型+参数） |
| concurrency_config | object | 是 | 并发配置（限制数/模型API密钥） |
| monitoring_thresholds | object | 否 | 监控阈值配置 |
| fallback_models | array | 否 | 备用模型列表 |

## 输出规格

| 输出项 | 类型 | 说明 |
|--------|------|------|
| generated_assets | array | 生成结果文件列表（含元数据） |
| batch_report | object | 批次执行报告 |
| exception_log | array | 异常事件日志 |
| cost_breakdown | object | 成本明细 |

## 最佳实践

1. **先跑pilot批次**：100+任务先跑10个验证流程畅通，再全量投入
2. **并发度渐进提升**：从低并发开始，观察模型响应稳定后再逐步提升
3. **错误分类处理**：不同错误类型有不同的最优处理策略，不要一刀切重试
4. **保存中间状态**：支持断点续传，批次中断后可从失败点恢复
5. **成本实时追踪**：大批次可能产生意外高成本，设置成本上限告警

## 常见陷阱

- 并发度设太高触发API限流，反而降低整体吞吐
- 重试风暴：所有失败任务同时重试，造成更大压力
- 未设置超时导致单个任务阻塞队列
- 配额用完才发现，大量任务白排队
- 忽略了不同模型的rate limit差异（DALL-E vs Flux vs MJ）
