# A/B测试设计（A/B Testing Design）

## 技能概述

本技能负责在创意素材生成过程中设计和管理A/B测试方案，通过对比不同提示词版本、模型选择或创意方向的生成效果，数据驱动地选择最优方案。特别适用于新风格探索、关键创意决策、以及提示词模板优化迭代场景。

## 适用场景

- 关键创意素材（品牌主KV）需要多方案对比选择
- 新提示词模板上线前的效果验证
- 多模型对比选择最适合当前任务的模型
- 同一主题不同风格方向的探索性生成
- 提示词参数（CFG/Steps/风格权重）的调优

## 执行步骤

### Step 1: 测试变量确定
明确本次A/B测试的变量类型：
- **提示词变量**：同一需求的不同描述方式/关键词组合
- **模型变量**：同一提示词在不同模型上的效果对比
- **参数变量**：同一提示词+模型的不同参数配置对比
- **风格变量**：同一主题的不同创意方向/视觉风格

### Step 2: 测试方案设计

```json
{
  "test_id": "AB-001",
  "test_type": "prompt_variant",
  "hypothesis": "增加具体光照描述可以提升商品图的质感评分",
  "control": {
    "label": "Version A - 基础版",
    "prompt": "...",
    "model": "flux-pro",
    "params": {}
  },
  "variants": [
    {
      "label": "Version B - 增强光照描述",
      "prompt": "...(增加了studio lighting细节)",
      "model": "flux-pro",
      "params": {}
    },
    {
      "label": "Version C - 增强材质描述",
      "prompt": "...(增加了材质纹理细节)",
      "model": "flux-pro",
      "params": {}
    }
  ],
  "sample_size": 5,
  "evaluation_criteria": ["质量评分", "创意匹配度", "生成成功率"],
  "success_metric": "质量评分提升≥0.5分"
}
```

### Step 3: 样本生成控制
- 每个版本生成相同数量的样本（通常3-5张）
- 使用相同的seed以排除随机性干扰（适用于对比提示词差异）
- 或使用不同seed以评估稳定性（适用于评估某版本的一致性）
- 记录每次生成的完整参数和元数据

### Step 4: 结果评估与决策
- 收集质量检验员对每个版本的评分
- 统计各版本的平均分/最高分/最低分/方差
- 分析是否达到预设的success_metric
- 输出明确的结论和建议

### Step 5: 知识沉淀
- 有效的提示词模式记录到模板库
- 无效的方向记录为避坑清单
- 更新模型能力矩阵中的场景评分

## 输入规格

| 输入项 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| test_objective | string | 是 | 测试目标描述 |
| base_requirement | object | 是 | 基础创意需求 |
| variable_type | enum | 是 | 测试变量类型 |
| sample_size | number | 否 | 每版本样本量（默认5） |
| evaluation_criteria | array | 否 | 评估维度 |

## 输出规格

| 输出项 | 类型 | 说明 |
|--------|------|------|
| test_plan | object | A/B测试方案 |
| test_results | object | 测试结果汇总 |
| winner_decision | object | 获胜版本及理由 |
| learnings | array | 沉淀的经验教训 |

## 最佳实践

1. **单一变量原则**：每次测试只改变一个变量，否则无法归因
2. **样本量充足**：单版本至少3张样本，重要决策至少5张
3. **盲评机制**：评估时不告知评分者哪个是control/variant
4. **设定明确阈值**：测试前确定"什么算赢"的标准，避免事后合理化
5. **控制测试成本**：A/B测试有成本，非关键决策不必每次都做

## 常见陷阱

- 变量过多导致组合爆炸（3个变量×3个选项=27种组合）
- 样本量过小导致结论不可靠（1张图的好坏可能是随机的）
- 忽略了种子(seed)对结果的影响
- 过度A/B测试导致交付延迟
- 未记录测试结论导致重复测试同一问题
