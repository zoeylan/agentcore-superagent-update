# 跨部门协作 (Cross-Department Collaboration)

## 概述
当工单处理需要其他部门（技术、财务、物流、产品等）的支持时，发起、跟踪和协调跨部门协作请求的能力。确保信息完整传递、进度透明可控、客户及时获得反馈。

## 适用场景
- 退款审批需要财务部门确认
- 技术故障需要研发团队排查
- 物流异常需要仓储/快递方核实
- 产品功能问题需要产品团队确认
- 合同/政策类问题需要法务确认

## 执行步骤

### 1. 协作需求评估
- 确认问题确实超出客服权限范围需要跨部门协作
- 识别需要协作的目标部门和具体需求
- 评估协作紧急度（是否在SLA时限压力下）

### 2. 协作请求发起
构建标准化协作请求，包含：
- **问题描述**：清晰简洁的问题现象和背景
- **已排查内容**：客服侧已完成的排查和确认的信息
- **具体需求**：需要对方执行的动作或确认的信息
- **影响范围**：受影响的客户数量/金额/业务影响
- **时间要求**：基于工单SLA计算的回复截止时间
- **客户情绪**：告知协作方客户当前的情绪和期望

### 3. 进度跟踪
- 协作发起后4小时无响应 → 发送催促提醒
- 协作发起后8小时无响应 → 升级通知对方部门主管
- 跟踪进度更新，及时向客户反馈进展
- 客户每次追问时主动同步最新进展状态

### 4. 结果整合
- 收到协作回复后，验证是否完整回答了所有问题
- 将技术性/专业性回复翻译为客户可理解的语言
- 整合协作结果到完整解决方案中
- 如果协作回复不完整，追问补充

## 输入规范
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| ticket_id | string | 是 | 关联工单ID |
| target_department | enum | 是 | 目标协作部门 |
| request_type | string | 是 | 协作类型（审批/排查/确认/操作） |
| request_detail | string | 是 | 详细协作需求描述 |
| urgency | enum | 是 | 紧急度（urgent/normal） |
| sla_deadline | datetime | 是 | SLA截止时间 |

## 输出规范
| 字段 | 类型 | 说明 |
|------|------|------|
| collaboration_id | string | 协作请求编号 |
| status | enum | 状态（pending/in_progress/completed/overdue） |
| response_content | string | 协作方回复内容 |
| response_time | datetime | 回复时间 |
| integrated_solution | string | 整合后的解决方案 |

## 最佳实践
- 协作请求描述越清晰，对方响应越快越准确
- 提前排查客服能力范围内的信息，减少不必要的跨部门沟通
- 对客户保持透明，告知"已转交XX部门处理，预计X小时内有结果"
- 建立常见协作场景的SOP模板，减少信息遗漏

## 质量指标
- 协作请求响应时间 <= 4小时
- 信息传递完整性 100%（不因信息缺失被退回）
- 协作结果客户满意度 >= 4.0/5
- 因协作导致的SLA超时率 < 3%
