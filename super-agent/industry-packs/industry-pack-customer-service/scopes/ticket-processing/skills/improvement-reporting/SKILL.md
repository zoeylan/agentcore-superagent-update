# 改进报告输出 (Improvement Reporting)

## 概述
将满意度分析结果转化为结构化的改进报告和行动建议，推动涉及坐席培训、流程优化、知识库完善等方面的持续改进闭环。

## 适用场景
- 周度/月度满意度改进报告输出
- 针对特定低分问题的专项改进建议
- 坐席绩效改进辅导方案
- 服务流程优化建议书
- 管理层汇报和决策支撑

## 执行步骤

### 1. 改进方向识别
基于反馈分析结果，确定改进优先级：
- **紧急改进**：CSAT<3分占比超过15%的领域
- **重点改进**：连续下降趋势的维度
- **持续优化**：已达标但有提升空间的领域
- 使用影响度×可行性矩阵排序

### 2. 改进措施设计
针对各类问题制定具体改进措施：

**坐席层面**：
- 低分坐席一对一辅导计划
- 针对性技能培训课程建议
- 优秀案例学习和话术共享
- 服务意识强化和激励方案

**流程层面**：
- 高频低分环节的流程简化建议
- 客户等待环节的体验优化
- 转派规则优化（减少不必要的多次流转）
- 回复时效标准的合理性调整

**知识层面**：
- 低解决率知识内容的更新建议
- 知识盲区的补充需求提交
- FAQ回复模板的优化建议
- 常见误解/错误回答的纠正

### 3. 报告结构化输出
标准改进报告包含：
- **执行摘要**：一页纸概述核心发现和建议（给管理层）
- **数据详情**：完整的数据分析和图表
- **问题清单**：按优先级排序的问题列表
- **改进计划**：具体措施 + 负责人 + 时限 + 预期效果
- **跟踪看板**：前期改进措施的执行进度和效果

### 4. 效果跟踪
- 每项改进措施设定效果指标和验证周期
- 通常观察2-4周后评估效果
- 有效措施固化为长期标准
- 无效措施分析原因，调整后重试或放弃

## 输入规范
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| analysis_results | object | 是 | 反馈分析结果 |
| report_type | enum | 是 | 报告类型（weekly/monthly/special） |
| target_audience | enum | 是 | 目标受众（team_lead/manager/director） |
| previous_actions | array | 否 | 前期改进措施及其状态 |

## 输出规范
| 字段 | 类型 | 说明 |
|------|------|------|
| report_title | string | 报告标题 |
| executive_summary | string | 执行摘要 |
| key_findings | array | 关键发现列表 |
| improvement_actions | array | 改进行动列表（含负责人、时限） |
| tracking_dashboard | object | 历史改进措施跟踪状态 |
| next_review_date | date | 下次review日期 |

## 最佳实践
- 报告语言简洁有力，决策层看摘要、执行层看详情
- 每次报告不超过5条核心改进建议（避免行动过载）
- 改进措施必须SMART（具体、可衡量、可执行、相关、有时限）
- 正面反馈也要记录和表彰，不只盯着问题

## 约束条件
- 周报在每周一上午10点前输出
- 月报在次月5日前输出
- 涉及坐席个人数据的报告需注意隐私保护
- 改进建议需考虑现有资源约束，不提不切实际的建议
