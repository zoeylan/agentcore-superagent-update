# 解决方案生成 (Solution Generation)

## 概述
基于知识库检索结果和客户具体情况，生成完整、可操作的问题解决方案和客户回复内容。该技能将知识转化为个性化的服务输出，直接影响客户体验和首次解决率。

## 适用场景
- 基于知识库内容生成标准化解决方案
- 综合多源信息制定定制化处理方案
- 根据客户反馈调整和优化已有方案
- 复杂问题的多步骤解决路径设计

## 执行步骤

### 1. 方案框架设计
根据问题类型选择方案结构：
- **咨询类**：直接回答 + 补充说明 + 相关推荐
- **故障排查类**：症状确认 → 逐步排查 → 解决操作 → 验证确认
- **变更操作类**：前置条件确认 → 操作步骤 → 预期结果 → 注意事项
- **复合问题类**：分解为子问题 → 逐一解决 → 整合验证

### 2. 内容生成
- 引用知识库标准答案作为核心内容
- 结合客户具体信息进行个性化（如具体的订单号、产品版本、账户状态）
- 补充异常处理说明（"如果按步骤操作后仍未解决，请..."）
- 添加预期结果描述（让客户能自行验证是否解决）

### 3. 话术包装
根据渠道和场景适配沟通风格：
- **同理心开头**：先确认理解客户问题和感受
- **方案主体**：清晰、分步骤、可操作
- **结尾确认**：询问是否解决 + 提供进一步帮助承诺
- **语气调整**：根据客户情绪状态调整正式/亲和程度

### 4. 质量自检
- 方案完整性：是否覆盖所有可能的场景分支
- 可操作性：客户是否能独立按步骤执行
- 准确性：引用的知识是否在有效期内、是否适用于当前版本
- 安全性：是否涉及敏感操作需要额外确认

## 输入规范
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| ticket_summary | string | 是 | 工单问题摘要 |
| knowledge_results | array | 是 | 知识库检索结果 |
| customer_context | object | 是 | 客户上下文信息 |
| channel | enum | 是 | 回复渠道 |
| attempt_number | int | 否 | 第几次方案（首次/调整） |
| previous_feedback | string | 否 | 客户对上次方案的反馈 |

## 输出规范
| 字段 | 类型 | 说明 |
|------|------|------|
| solution_summary | string | 方案摘要（内部记录用） |
| customer_reply | string | 发送给客户的回复内容 |
| action_steps | array | 操作步骤列表 |
| expected_outcome | string | 预期结果描述 |
| escalation_needed | boolean | 是否需要升级处理 |
| knowledge_refs | array | 引用的知识库文章ID |

## 最佳实践
- 首次方案力求一次解决，宁可信息全面冗余，不要遗漏关键步骤
- 第二次方案需明确分析首次未解决的原因，避免重复同样的建议
- 对于无法完全解决的问题，坦诚告知客户当前进展和后续安排
- 技术类方案需区分客户技术水平，非技术用户使用通俗语言

## 约束条件
- 单次方案生成不超过30秒
- 回复内容长度控制：在线聊天<500字、邮件<1500字
- 涉及金额、期限等承诺必须有依据，不可随意承诺
- 两次方案未解决必须建议升级，不得无限循环
