# 客户沟通 (Customer Communication)

## 概述
根据客户渠道、情绪状态和问题类型，生成专业、共情且有效的客户沟通内容。包括首次回复、进度更新、方案解释、等待安抚等多种沟通场景的话术生成能力。

## 适用场景
- 工单首次响应回复
- 解决方案的客户说明
- 处理进度的主动更新
- 等待期间的安抚沟通
- 客户追问的及时回应
- 关单前的确认沟通

## 执行步骤

### 1. 沟通场景识别
判断当前沟通属于哪种场景：
- **首次响应**：确认收到问题 + 告知处理安排
- **方案提供**：同理心开头 + 解决步骤 + 确认询问
- **进度更新**：当前状态 + 预计时间 + 致歉/感谢
- **追问回应**：确认理解追问 + 补充说明
- **等待安抚**：致歉 + 原因说明 + 时间承诺
- **关单确认**：确认解决 + 后续支持承诺

### 2. 语气与风格适配
- **客户情绪正面/中性**：专业友好、简洁明快
- **客户情绪负面**：增加同理心表达、避免模板感
- **客户情绪愤怒**：先致歉安抚、不辩解、表达重视
- **VIP客户**：适当提升正式度和尊重感
- **年轻客户群体**：适当轻松化、可使用简单表情

### 3. 渠道格式适配
- **在线聊天**：短句、分段、即时感、可用简单emoji
- **邮件**：正式称谓、完整结构、签名落款
- **电话脚本**：口语化、关键信息重复确认
- **微信**：亲和力、控制长度、善用分段
- **短信**：极简、核心信息、含关键链接

### 4. 内容生成与审核
- 根据场景模板生成沟通内容
- 检查是否包含必要的确认询问（引导客户反馈）
- 确认无敏感信息泄露（不在非安全渠道透露账户细节）
- 确认承诺内容可兑现（不做超出能力的承诺）

## 输入规范
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| scenario | enum | 是 | 沟通场景类型 |
| channel | enum | 是 | 沟通渠道 |
| customer_sentiment | enum | 是 | 客户情绪状态 |
| customer_level | enum | 否 | 客户等级 |
| content_body | string | 是 | 需要传达的核心内容 |
| ticket_context | object | 是 | 工单上下文 |

## 输出规范
| 字段 | 类型 | 说明 |
|------|------|------|
| message_content | string | 生成的沟通内容 |
| tone | string | 使用的语气风格 |
| key_points | array | 核心要点列表 |
| requires_response | boolean | 是否需要客户回复 |
| follow_up_reminder | datetime | 若无回复何时跟进 |

## 最佳实践
- "三明治法则"处理负面场景：同理→方案→积极结尾
- 避免使用否定性起始语（"不行"、"不能"、"没办法"）
- 用"我来帮您"替代"你需要"，主动承担而非推给客户
- 涉及等待时给出具体时间而非"尽快"
- 每次沟通必须有明确的"下一步"指引

## 约束条件
- 不使用行业黑话或技术术语面对普通客户
- 不在非安全渠道传递敏感信息（密码、完整身份证号等）
- 不做超出服务权限的承诺（如承诺全额退款但无审批权限）
- 不与客户发生争辩，遇到无理要求礼貌引导或升级
