# 产能监控 (Capacity Monitoring)

## 概述
实时监控客服团队的处理产能（坐席在线数、工作负载、排队情况），评估当前和未来的产能是否满足工单处理需求，为调度决策和资源规划提供数据支撑。

## 适用场景
- 实时评估各技能组的处理能力余量
- 高峰期来临前的产能预判和预警
- 支撑智能路由的负载均衡决策
- 排班优化和人力资源规划

## 执行步骤

### 1. 产能数据采集
实时采集以下维度数据：
- **坐席状态分布**：在线/忙碌/小休/离线人数
- **技能组容量**：各技能组可用坐席数 vs 当前排队数
- **个人负载**：每个坐席当前处理中的工单数
- **处理速率**：近1小时平均每人处理工单数/时
- **到达速率**：近1小时各渠道新到工单数/时

### 2. 产能利用率计算
```
技能组利用率 = 当前处理中工单数 / (可用坐席数 × 最大并发数) × 100%
排队等待比 = 排队工单数 / 可用坐席数
预计消化时间 = 排队工单数 / (可用坐席数 × 平均处理速率)
```

### 3. 产能预警
| 利用率 | 状态 | 动作建议 |
|--------|------|----------|
| < 50% | 空闲 | 可接受跨组支援请求 |
| 50-75% | 正常 | 正常运转 |
| 75-90% | 繁忙 | 关注来单趋势，准备应急 |
| 90-100% | 饱和 | 启动溢出策略，通知管理层 |
| > 100% | 过载 | 紧急扩容，启用应急队列 |

### 4. 趋势预测
- 基于历史同期数据预测未来2小时的来单量
- 结合当前排班计划评估未来产能缺口
- 特殊日期（大促、月底、系统上线）提前预测峰值

### 5. 调度建议输出
- 产能不足时：建议启用哪些后备坐席、哪些技能组可支援
- 产能过剩时：建议哪些坐席可安排培训或休息
- 持续高负载时：建议调整排班或申请增员

## 输入规范
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| skill_group | string | 否 | 技能组名称（为空则监控全部） |
| time_range | int | 否 | 监控时间范围（分钟），默认60 |
| include_forecast | boolean | 否 | 是否包含未来预测 |

## 输出规范
| 字段 | 类型 | 说明 |
|------|------|------|
| overall_utilization | float | 整体利用率百分比 |
| group_details | array | 各技能组详细数据 |
| queue_depth | int | 当前总排队数 |
| estimated_wait_time | int | 预计平均等待时间（秒） |
| capacity_status | enum | 产能状态（idle/normal/busy/saturated/overloaded） |
| forecast | object | 未来2小时预测 |
| recommendations | array | 调度建议列表 |

## 最佳实践
- 产能数据刷新频率不低于每5分钟一次
- 在预测到产能紧张前提前30分钟告警（给管理层调度时间）
- 区分"技能性产能不足"（特定技能缺人）和"总量性产能不足"
- 产能数据与SLA监控联动，优先保障高优先级工单的产能

## 约束条件
- 不直接执行人员调度（只提供建议）
- 监控数据实时性要求：延迟 < 1分钟
- 历史数据保留用于基线计算：至少90天
