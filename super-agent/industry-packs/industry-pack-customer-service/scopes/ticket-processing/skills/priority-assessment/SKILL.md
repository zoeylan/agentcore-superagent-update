# 优先级智能判定 (Priority Assessment)

## 概述
综合评估客户请求的紧急程度、业务影响范围和客户价值，输出P1-P4优先级标签并确定对应的SLA时限要求。该技能确保高影响、高价值的请求得到优先处理。

## 适用场景
- 新工单创建时的初始优先级评定
- 工单处理过程中因新信息出现的优先级动态调整
- 批量事件发生时的优先级批量重评

## 执行步骤

### 1. 问题影响度评估
分析问题对客户业务/使用的影响程度：
- **关键影响**：业务完全中断、资金安全风险、数据泄露风险
- **重大影响**：核心功能不可用但有替代方案、批量用户受影响
- **一般影响**：非核心功能异常、单用户受影响、有临时workaround
- **轻微影响**：体验类问题、非紧急咨询、优化建议

### 2. 客户价值评估
调取客户画像进行价值分级：
- **VIP客户**（CLV前5%或明确VIP标记）：自动提升一个优先级
- **高价值客户**（CLV前20%、年消费TOP客户）：同等条件优先处理
- **普通客户**：按标准优先级处理
- **新客户**（注册<30天）：适当关注，影响留存

### 3. 渠道与时效因素
- 监管渠道（12315转办、银保监转办）：自动设为P1
- 社交媒体公开渠道（微博@、抖音评论）：至少P2（舆情风险）
- 重复来电/来单（同一问题24h内第3次联系）：自动升级一个优先级
- 临近营业时间结束的请求：考虑跨日SLA影响

### 4. 优先级输出
| 优先级 | 定义 | 首次响应 | 解决时限 |
|--------|------|----------|----------|
| P1-紧急 | 影响业务运行/资金安全/监管要求 | 30分钟 | 4小时 |
| P2-高 | 核心功能受阻/VIP客户/舆情风险 | 2小时 | 8小时 |
| P3-中 | 一般功能问题/普通变更需求 | 8小时 | 24小时 |
| P4-低 | 咨询建议/非紧急优化/信息获取 | 24小时 | 72小时 |

## 输入规范
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| category | enum | 是 | 工单分类结果 |
| intent | string | 是 | 意图标签 |
| customer_profile | object | 是 | 客户画像（VIP等级、CLV、历史记录） |
| channel | enum | 是 | 接入渠道 |
| impact_description | string | 否 | 影响范围描述 |
| repeat_count | int | 否 | 同一问题重复联系次数 |

## 输出规范
| 字段 | 类型 | 说明 |
|------|------|------|
| priority | enum | P1/P2/P3/P4 |
| sla_response_deadline | datetime | 首次响应截止时间 |
| sla_resolution_deadline | datetime | 解决截止时间 |
| priority_factors | array | 影响优先级判定的因素列表 |
| auto_upgrade_applied | boolean | 是否触发了自动升级规则 |
| upgrade_reason | string | 自动升级原因说明 |

## 最佳实践
- 优先级判定宁高勿低：不确定时选择更高的优先级
- 多因素叠加时取最高优先级（如VIP客户+一般问题=至少P3而非P4）
- P1工单必须附带明确的业务影响描述，不能仅凭客户主观感受
- 大促等特殊时期可预设全局优先级策略（如订单类问题统一升至P2）

## 约束条件
- 优先级判定必须在5秒内完成
- 修改已有工单优先级必须记录变更原因
- P1工单数量应控制在总量的5%以内（超出说明标准过松）
