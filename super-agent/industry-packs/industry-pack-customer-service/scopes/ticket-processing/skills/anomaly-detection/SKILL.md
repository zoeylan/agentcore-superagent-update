# 异常模式检测 (Anomaly Detection)

## 概述
持续监控工单处理系统的运行状态，识别异常模式（如批量超时、反复流转、异常积压），为及时干预和应急响应提供预警支撑。

## 适用场景
- 检测工单批量涌入异常（可能的系统故障或事件）
- 识别反复流转的"踢皮球"工单
- 发现长期挂起未处理的遗漏工单
- 坐席产能异常监控（突然大量积压）
- 系统性SLA风险的早期预警

## 执行步骤

### 1. 基线建立
- 统计各时段正常的工单到达率（分时段、分渠道、分类型）
- 建立各坐席/技能组的正常处理能力基线
- 建立各类型工单的正常处理时长基线
- 设定偏离阈值（通常为基线的±2个标准差）

### 2. 实时异常检测
监控以下异常模式：

**来单异常**：
- 某渠道/类型来单量突然超过基线3倍 → 可能的系统故障或事件
- 特定错误关键词短时间内密集出现 → 可能的批量故障

**处理异常**：
- 同一工单被转派超过2次 → "踢皮球"风险
- 工单状态停留"待分配"超过30分钟 → 路由异常
- 工单状态停留"处理中"超过正常处理时长2倍 → 可能遇到困难

**产能异常**：
- 某坐席当前工单数超过阈值上限 → 过载风险
- 某技能组排队数突增 → 产能不足
- 整体积压量（Backlog）超过日均处理量50% → 系统性风险

### 3. 异常分级与响应
| 异常级别 | 定义 | 响应动作 |
|----------|------|----------|
| 注意 | 偏离基线但尚可控 | 记录并观察 |
| 预警 | 持续偏离或影响范围扩大 | 通知组长 |
| 告警 | 批量影响或严重偏离 | 通知主管+建议应急措施 |
| 紧急 | 系统性故障或大面积影响 | 启动应急预案 |

### 4. 应急建议生成
针对告警级及以上异常，生成应急处理建议：
- 批量来单：建议临时扩容、启用溢出队列、AI自动分流
- 产能不足：建议跨组支援、临时调整优先级策略
- 系统故障：建议启动故障公告、批量延长SLA、降级处理

## 输入规范
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| monitoring_scope | enum | 是 | 监控范围（全局/技能组/坐席） |
| time_window | int | 是 | 检测时间窗口（分钟） |
| baseline_data | object | 是 | 基线数据 |
| current_metrics | object | 是 | 当前指标数据 |

## 输出规范
| 字段 | 类型 | 说明 |
|------|------|------|
| anomalies | array | 检测到的异常列表 |
| anomalies[].type | string | 异常类型 |
| anomalies[].severity | enum | 严重级别 |
| anomalies[].description | string | 异常描述 |
| anomalies[].affected_scope | string | 影响范围 |
| anomalies[].suggested_action | string | 建议响应动作 |
| system_health_score | float | 系统健康度评分 0-100 |

## 最佳实践
- 基线数据需按工作日/周末/节假日/大促期分别建立
- 避免过度告警（告警疲劳），阈值设置需要平衡灵敏度和误报率
- 重大异常需关联历史相似事件，提供经验参考
- 异常消除后记录事件报告，作为基线优化依据

## 质量指标
- 异常检测准确率 >= 85%（报出的异常确实是异常）
- 重大事件漏报率 = 0%（不允许遗漏严重异常）
- 平均检测延迟 < 5分钟
- 告警误报率 < 15%
