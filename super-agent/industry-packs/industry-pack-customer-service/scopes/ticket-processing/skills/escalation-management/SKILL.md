# 升级管理 (Escalation Management)

## 概述
当工单超出SLA时限或处理能力时，执行结构化的逐级升级流程，确保问题得到更高权限和更专业的资源介入处理。该技能保障不会有工单因超时而被遗忘。

## 适用场景
- SLA超时自动触发升级
- 工单多次转派无法解决时触发升级
- 坐席主动申请升级（问题超出处理能力）
- 客户要求升级处理

## 执行步骤

### 1. 升级条件判定
满足以下任一条件触发升级：
- SLA消耗超过100%（超时）
- 同一工单被转派超过2次
- 坐席主动标记"需升级"
- 客户明确要求"找领导/主管"
- 两次方案均被客户拒绝

### 2. 升级层级确定
| 触发条件 | 升级目标 | 响应要求 |
|----------|----------|----------|
| SLA超时50% | 组长 | 30分钟内接手 |
| SLA超时100% | 主管 | 15分钟内响应 |
| SLA超时200% | 部门总监 | 立即响应 |
| 多次转派 | 组长指定专人 | 1小时内接手 |
| 客户要求 | 值班组长 | 即时转接 |

### 3. 升级信息打包
生成升级通知，包含：
- **工单摘要**：客户问题一句话概述
- **处理历史**：已尝试的解决方案和客户反馈
- **超时信息**：超出SLA的程度和时长
- **客户状态**：当前情绪、VIP等级、历史投诉记录
- **建议动作**：基于情况建议的处理方向
- **升级原因**：明确的升级触发条件

### 4. 升级执行与跟踪
- 将工单分配至升级目标，同时发送预警通知
- 设置升级后的响应计时（升级后仍有SLA要求）
- 如果升级后仍未在要求时间内响应，继续向上一级升级
- 升级完成后记录升级事件（用于后续分析）

### 5. 升级后监控
- 确认升级目标已接手（状态变为"处理中"）
- 如果超过响应要求时间未接手，启动二次催促
- 升级工单解决后记录实际处理时长和解决方案

## 输入规范
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| ticket_id | string | 是 | 工单编号 |
| escalation_trigger | enum | 是 | 升级触发条件 |
| current_handler | string | 是 | 当前处理人 |
| sla_breach_percentage | float | 否 | SLA超时百分比 |
| transfer_count | int | 否 | 已转派次数 |
| ticket_history | array | 是 | 工单处理历史记录 |

## 输出规范
| 字段 | 类型 | 说明 |
|------|------|------|
| escalation_id | string | 升级事件ID |
| escalation_level | enum | 升级层级（team_lead/supervisor/director） |
| target_handler | string | 升级目标处理人 |
| escalation_summary | string | 升级摘要信息 |
| response_deadline | datetime | 接手截止时间 |
| notification_sent | boolean | 通知是否发送成功 |

## 最佳实践
- 升级不是"甩锅"，升级信息必须包含完整上下文减少重复沟通
- 升级后原坐席仍可能协助（如提供补充信息），不完全脱责
- 对频繁触发升级的坐席/问题类型进行根因分析
- 升级工单的解决率目标>=90%，高于普通工单

## 质量指标
- 升级后1小时内接手率 >= 95%
- 升级工单最终解决率 >= 90%
- 升级信息完整性评分 >= 4.5/5
- 虚假升级率（不必要的升级）< 5%
