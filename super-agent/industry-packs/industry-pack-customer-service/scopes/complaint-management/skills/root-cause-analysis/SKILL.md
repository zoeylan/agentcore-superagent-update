# 根因分析方法论

## 概述
根因分析（Root Cause Analysis, RCA）技能用于从单个或批量投诉中识别引发问题的根本原因，区分表象和本质，为系统性改进提供决策依据。该技能运用多种分析方法论，确保改进措施针对根因而非症状。

## 适用场景
- 同类投诉频繁发生时的根因挖掘
- 新增投诉类型的原因追溯
- 投诉量突然上升时的异常诊断
- 改进措施效果不佳时的深层原因分析
- 月度/季度投诉分析报告编制

## 执行步骤

### 1. 问题定义
- 明确分析对象：具体的投诉类型/模式
- 量化问题规模：发生频次、影响客户数、时间分布
- 设定分析范围：时间窗口、涉及产品线、客户群体

### 2. 数据收集与聚类
- 收集目标时间窗口内的相关投诉案件
- 按多维度标签聚类：投诉类型、产品线、客户行为路径、发生时段
- 识别TOP集中模式和异常分布
- 提取典型案例作为深度分析样本

### 3. 根因分析方法运用

**5-Why分析法**（适用于单一事件深度分析）：
```
问题：客户投诉充值未到账
Why1: 系统未记录充值→ Why2: 支付回调超时 → Why3: 服务器负载过高
→ Why4: 大促期间未做容量扩展 → Why5: 缺少容量规划流程
根因：缺少大促前的系统容量规划和压测流程
```

**鱼骨图分析法**（适用于多因素问题）：
- 人员因素：技能不足、操作失误、培训缺失
- 流程因素：流程缺失、流程繁琐、规则矛盾
- 系统因素：系统bug、性能不足、兼容性问题
- 产品因素：功能缺陷、设计不合理、说明不清晰
- 外部因素：第三方故障、政策变化、恶意攻击

**帕累托分析法**（适用于优先级排序）：
- 按投诉量排序，识别贡献80%投诉的TOP20%原因
- 优先解决高频根因，实现最大改进收益

### 4. 根因验证
- 验证假设根因与投诉发生的因果关系
- 检查根因解释是否覆盖所有相关投诉
- 确认根因是否可被改进措施消除
- 评估消除根因后的预期改善幅度

### 5. 改进建议制定
- 针对根因提出具体改进措施
- 明确责任部门和执行时限
- 评估改进的成本和难度
- 设定效果验证指标和目标值

## 输入规范
```json
{
  "analysis_scope": {
    "time_range": "分析时间范围",
    "complaint_types": ["目标投诉类型"],
    "sample_cases": ["典型案件编号"],
    "data_volume": "涉及投诉案件数"
  },
  "trigger": "analysis_requested|threshold_breach|periodic_review"
}
```

## 输出规范
```json
{
  "rca_report": {
    "problem_statement": "问题描述",
    "analysis_method": "使用的分析方法",
    "root_causes": [
      {
        "cause": "根因描述",
        "category": "人员|流程|系统|产品|外部",
        "evidence": "支撑证据",
        "impact_scope": "影响范围",
        "improvement_suggestion": "改进建议",
        "owner": "建议责任部门",
        "priority": "high|medium|low"
      }
    ],
    "expected_improvement": "预期改善幅度",
    "verification_plan": "效果验证计划"
  }
}
```

## 最佳实践
- **深度优先**：不要停留在表面原因，至少追问3层"为什么"
- **数据驱动**：结论必须有数据支撑，避免主观臆断
- **系统视角**：单个投诉可能是系统性问题的冰山一角
- **可行性优先**：改进建议必须可落地执行，不提空泛建议
- **闭环验证**：每个改进措施都要有明确的验证时间点和成功标准

## 约束条件
- 分析结论必须基于足够的样本量（通常≥10个案例）
- 不以单个极端案例推断系统性问题
- 改进建议不涉及具体人员处罚（归组织和流程问题）
- 分析报告中的客户信息必须脱敏处理
