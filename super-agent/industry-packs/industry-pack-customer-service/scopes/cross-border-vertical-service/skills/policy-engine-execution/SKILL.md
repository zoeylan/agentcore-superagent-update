# Policy Engine规则执行 (Policy Engine Execution)

## 概述

Policy Engine规则执行技能负责对跨境电商和行业垂直场景中涉及资金变动或高风险操作的业务请求进行自动化规则验证，输出PERMIT（允许执行）、DENY（拒绝并说明原因）或ESCALATE（超阈值需人工审批）三种标准化决策。该技能是保障业务合规性和风控安全的核心防线，将原本需要人工逐条审核的决策过程自动化，同时确保决策准确率>=95%。

## 适用场景

- 退款操作的金额阈值验证（退款>500元需ESCALATE）
- 航空退改签的时限和舱位规则校验
- 金融交易的额度检查和风控拦截
- 优惠券/折扣码发放的资格验证
- 高额转账操作的审批触发（>5000元）
- 特殊旅客政策（婴儿票、军人优惠等）的合规验证

## 详细执行步骤

### 步骤1：请求解析与规则定位
1. 接收业务操作请求，解析核心参数：
   - operation_type: 操作类型（refund/exchange/cancel/transfer/coupon等）
   - amount: 涉及金额（如有）
   - currency: 原始币种
   - customer_id: 客户标识
   - order_id: 关联订单（如有）
   - context: 业务上下文（平台、品类、时间等）
2. 根据operation_type定位适用的规则集（Rule Set）
3. 加载规则集的最新版本（规则支持热更新）

### 步骤2：前置条件验证
1. 验证客户身份是否已通过认证（双重验证：订单号+手机尾号）
2. 验证操作请求的数据完整性（必要字段是否齐全）
3. 验证操作时效性（如退换货是否在平台允许的时限内）
4. 检查客户是否有未完成的同类操作（防重复提交）
5. 任一前置条件不满足 → 输出DENY并明确说明哪个条件未满足

### 步骤3：规则匹配与评估
1. 按优先级顺序执行规则链：
   - 黑名单规则（最高优先级）：客户是否在风控黑名单中
   - 硬性限制规则：是否超出平台/行业的绝对限制
   - 金额阈值规则：是否超出自动处理权限
   - 频次限制规则：同一客户短期内操作频次是否异常
   - 业务逻辑规则：具体业务场景的专属规则
2. 规则评估采用"短路"模式：命中DENY规则立即返回，不继续评估
3. 所有规则均通过且金额在阈值内 → PERMIT
4. 所有规则均通过但金额超阈值 → ESCALATE

### 步骤4：决策输出
1. **PERMIT决策**：
   - 确认操作可执行，输出授权token
   - 记录决策日志（规则版本、命中规则、决策时间）
   - 通知下游执行Agent可以继续操作
2. **DENY决策**：
   - 输出拒绝原因（面向客户的友好说明+面向系统的技术原因）
   - 提供替代方案建议（如"退款超期但可申请店铺优惠券补偿"）
   - 记录决策日志
3. **ESCALATE决策**：
   - 生成审批工单（含完整业务上下文、风险评估、建议审批意见）
   - 推送至对应审批人（基于金额段和业务类型）
   - 向客户发送"正在为您审批，预计30分钟内处理"的通知
   - 启动审批超时监控（30分钟）

### 步骤5：决策审计与反馈
1. 所有决策写入不可篡改的审计日志
2. 标记DENY决策的后续跟踪（是否被人工推翻）
3. 人工推翻的DENY决策纳入规则优化队列
4. 按日/周维度统计决策分布和异常模式

## 输入规格

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| operation_type | string | 是 | 操作类型 |
| amount | float | 否 | 涉及金额（人民币） |
| currency | string | 否 | 原始币种 |
| customer_id | string | 是 | 客户唯一标识 |
| order_id | string | 否 | 关联订单号 |
| identity_verified | boolean | 是 | 客户身份是否已验证 |
| business_context | object | 是 | 业务上下文（平台、品类、时间等） |

## 输出规格

| 字段 | 类型 | 说明 |
|------|------|------|
| decision | string | PERMIT / DENY / ESCALATE |
| rule_matched | string | 命中的规则ID和名称 |
| reason | string | 决策原因说明 |
| customer_message | string | 面向客户的友好说明 |
| alternative_suggestion | string | 替代方案建议（DENY时） |
| approval_ticket_id | string | 审批工单ID（ESCALATE时） |
| authorization_token | string | 执行授权token（PERMIT时） |

## 最佳实践

1. **规则版本管理**：每次规则变更必须有版本号，决策日志记录所用规则版本，便于追溯
2. **灰度发布**：新规则先在10%流量上灰度验证，确认无异常后再全量生效
3. **定期Review**：每月review被人工推翻的DENY决策，推翻率>10%的规则需要调优
4. **多币种处理**：统一转换为人民币进行阈值判定，汇率使用当日中间价
5. **熔断机制**：当规则引擎异常率>5%时，触发熔断，所有操作默认ESCALATE到人工

## 约束条件

- 规则判定准确率>=95%
- 单次规则评估耗时<500ms
- 人工推翻DENY决策的比例<10%
- ESCALATE到人工后2小时内必须完成审批
- 审计日志保留期限>=2年
- 规则更新后10分钟内全局生效
